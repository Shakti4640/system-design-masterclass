🎯 HYPOTHETICAL PROBLEM

Your restaurant chain (Projects 1-3) is scaling (Project 4)
→ 200 restaurants, 50,000 orders/day
→ Your menu has 500 items across all locations
→ Every customer who opens the app → fetches the menu

HERE IS THE DISASTER:

  Menu Request Pattern (Monday):
  ┌────────────────────────────────────────────────────────┐
  │ 9:00 AM  → 2,000 users open app → 2,000 DB queries   │
  │ 9:01 AM  → 1,800 users open app → 1,800 DB queries   │
  │ 9:02 AM  → 2,200 users open app → 2,200 DB queries   │
  │ ...                                                    │
  │ 12:00 PM → 8,000 users (lunch rush) → 8,000 DB queries│
  │                                                        │
  │ Total menu queries today: 180,000                      │
  │ Menu changed today: 0 times                            │
  │ Identical results returned: 180,000 times              │
  │ Wasted database work: 99.9994%                         │
  └────────────────────────────────────────────────────────┘

  Each menu query:
  → Parse SQL
  → Read from disk (or DB buffer)
  → JOIN menu_items + categories
  → Format 500 items into JSON
  → Send response
  → Time: 15-50ms per query
  → DB CPU used: ~5% per concurrent query

  With 180,000 queries:
  → Total DB time: 45-150 minutes of pure processing
  → For data that NEVER CHANGED

THE LIBRARIAN ANALOGY:
  → You walk into a library and ask "Do you have Harry Potter?"
  → Librarian walks to shelf, finds it, comes back: "Yes, Shelf 7, Row 3"
  → You leave
  → Next person asks the SAME question
  → Librarian walks to shelf AGAIN, finds it AGAIN, comes back AGAIN
  → This happens 500 times today
  → Same answer every time

SMART LIBRARIAN:
  → First person asks "Do you have Harry Potter?"
  → Librarian walks to shelf, finds it
  → Librarian WRITES ON A STICKY NOTE: "Harry Potter → Shelf 7, Row 3"
  → Sticks note on the counter
  → Next 499 people ask → looks at sticky note → instant answer
  → Only walks to shelf ONCE
  → The sticky note IS the cache

THIS is Caching:
  → Store the RESULT of expensive operations
  → Serve from fast storage (memory) instead of slow storage (disk/DB)
  → The single most impactful performance optimization in computing

🛠️ HOW CAN WE SOLVE IT

THE CACHE LAYER IN THE ARCHITECTURE:

  WITHOUT CACHE:
  Client → Server → Database → Server → Client
                    (slow, every time)

  WITH CACHE:
  Client → Server → Cache HIT? → Yes → Server → Client (fast!)
                       ↓
                      No (miss)
                       ↓
                    Database → Server → Store in Cache → Client

  VISUAL:
  ┌────────┐      ┌────────────┐      ┌──────────┐
  │        │      │            │ HIT  │          │
  │ Client │─────→│   Server   │─────→│  CACHE   │ → Response (1ms)
  │        │      │            │      │ (Memory) │
  └────────┘      └─────┬──────┘      └──────────┘
                        │ MISS              ↑
                        ▼                   │
                  ┌──────────┐         Store result
                  │ DATABASE │──────────────┘
                  │  (Disk)  │
                  └──────────┘
                  Response (15-50ms)


FIVE CACHING STRATEGIES:

Strategy 1: CACHE-ASIDE (Lazy Loading)
──────────────────────────────────────
  → Application manages the cache explicitly
  → Read: Check cache → miss → query DB → store in cache → return
  → Write: Update DB → delete from cache (next read will refresh)
  → MOST COMMON pattern

  ┌─────────────────────────────────────────────────────┐
  │  def get_menu():                                     │
  │      result = cache.get("menu")       # Check cache │
  │      if result is not None:                          │
  │          return result                 # Cache HIT  │
  │                                                      │
  │      result = db.query("SELECT ...")   # Cache MISS │
  │      cache.set("menu", result, ttl=300) # Store     │
  │      return result                                   │
  └─────────────────────────────────────────────────────┘

Strategy 2: WRITE-THROUGH
──────────────────────────
  → Write to cache AND database simultaneously
  → Read always from cache (always fresh)
  → Higher write latency (two writes per operation)
  → Cache is always consistent with DB

  ┌─────────────────────────────────────────────────────┐
  │  def update_item(id, data):                          │
  │      db.update(id, data)               # Write DB   │
  │      cache.set(f"item:{id}", data)     # Write Cache│
  │      return data                                     │
  └─────────────────────────────────────────────────────┘

Strategy 3: WRITE-BEHIND (Write-Back)
──────────────────────────────────────
  → Write to cache FIRST, return immediately
  → Asynchronously write to database later
  → Fastest writes, but risk of data loss if cache crashes
  → Used for: analytics, counters, non-critical data

  ┌─────────────────────────────────────────────────────┐
  │  def increment_view_count(item_id):                  │
  │      cache.increment(f"views:{item_id}") # Fast!    │
  │      # Background job writes to DB every 60 seconds │
  └─────────────────────────────────────────────────────┘

Strategy 4: READ-THROUGH
─────────────────────────
  → Cache itself is responsible for loading data
  → Application asks cache, cache asks DB if miss
  → Application doesn't know about DB directly
  → Simpler application code, cache does the work

Strategy 5: REFRESH-AHEAD
──────────────────────────
  → Cache proactively refreshes entries BEFORE they expire
  → Predicts which entries will be needed soon
  → Zero cache misses for hot data
  → Complex to implement correctly

📌 WHAT COULD BE DONE

Option A: In-Process Dictionary Cache
  → Python dict in the same process
  → Fastest possible (no network hop)
  → But: each server has its own copy
  → And: lost when server restarts
  → When: Single server, prototype, small datasets

Option B: External Cache Server (Redis / Memcached)
  → Separate server dedicated to caching
  → Shared across all application servers
  → Survives app server restarts
  → When: Multiple servers, production systems
  → Redis: supports data structures, persistence
  → Memcached: simpler, pure key-value

Option C: Multi-Level Cache
  → L1: In-process (fastest, small)
  → L2: External Redis (fast, larger)
  → L3: Database (slow, complete)
  → Check L1 → miss → check L2 → miss → query DB
  → When: High-performance systems needing maximum speed

Option D: HTTP Cache Headers
  → Server tells client: "cache this for 1 hour"
  → Cache-Control: max-age=3600
  → Client doesn't even make a request
  → When: Static/semi-static content, reducing server load

Option E: CDN Caching (Project 12 deep dive)
  → Cache at edge servers worldwide
  → User in Tokyo → served from Tokyo cache, not US server
  → When: Global user base, static content

For THIS project → Option A (in-process) + Option B concepts
→ Understand fundamentals with simple implementation
→ Same patterns apply to Redis/Memcached at scale

✅ WHAT SHOULD DO

1. ALWAYS set a TTL (Time To Live)
   → Every cache entry MUST expire
   → Without TTL → stale data lives forever
   → Menu cached for 5 minutes → at worst, 5 min stale
   → Menu cached forever → price change never reflected
   → Better slightly stale than infinitely stale

2. ALWAYS handle cache misses gracefully
   → Cache is an OPTIMIZATION, not a requirement
   → If cache is down → fall back to database
   → If cache miss → query database, don't error
   → System must work WITHOUT cache (just slower)

3. ALWAYS cache the FINAL result, not intermediate data
   → Cache the formatted JSON response, not raw DB rows
   → Saves formatting time on every hit
   → Trade memory for speed

4. ALWAYS use meaningful cache keys
   ✅ cache_key = "menu_items:category=pizza:page=1:limit=10"
   ❌ cache_key = "data_123"
   → Keys should describe what's cached
   → Include all parameters that affect the result

5. ALWAYS monitor cache hit rate
   → Good: > 80% hit rate
   → Great: > 95% hit rate
   → Bad: < 50% hit rate (cache isn't helping much)
   → If hit rate is low → wrong data is being cached

6. ALWAYS warm the cache for critical data
   → Don't wait for first user to trigger cache population
   → On server startup → pre-load popular data into cache
   → Prevents "cold start" latency for first users

7. ALWAYS size the cache appropriately
   → Too small → constant evictions → low hit rate
   → Too large → wasting memory that could be used elsewhere
   → Monitor and tune based on actual usage patterns

❌ WHAT SHOULD NOT DO

1. NEVER cache data that changes every request
   ❌ Cache user's real-time location (changes every second)
   ❌ Cache current stock price (changes millisecond)
   → Caching rapidly changing data = always stale = useless

2. NEVER cache without a TTL
   ❌ cache.set("menu", data)  ← no expiration
   → Menu price changes → cache never updates
   → Users see wrong prices for hours/days
   → ALWAYS: cache.set("menu", data, ttl=300)

3. NEVER treat cache as primary storage
   ❌ "We only store it in cache, no database needed"
   → Cache server restarts → ALL data gone
   → Cache eviction → data silently disappears
   → Cache is EPHEMERAL — database is PERMANENT
   → (Exception: Project 23 covers cache persistence)

4. NEVER cache sensitive data without protection
   ❌ Caching user passwords, credit card numbers
   ❌ Caching authentication tokens in shared cache without encryption
   → If cache is compromised → all data exposed
   → If caching auth data → encrypt and use short TTLs

5. NEVER ignore cache stampede
   → Popular cache entry expires
   → 1,000 concurrent requests all see cache miss
   → All 1,000 hit the database simultaneously
   → Database overwhelmed
   → FIX: Lock mechanism, stale-while-revalidate (covered below)

6. NEVER assume cache is always faster
   → Cache miss + failed lookup + DB query > just DB query
   → Very small datasets might be faster to just query
   → Network hop to external cache > in-memory DB buffer
   → Measure to verify cache actually helps

7. NEVER use the same TTL for everything
   ❌ All entries expire in 60 seconds
   → Menu (changes daily): should cache for hours
   → Order status (changes frequently): cache for seconds
   → Match TTL to data change frequency

🚫 WHEN NOT TO USE

1. Caching is NOT useful when:
   → Data changes on every read (random number, current timestamp)
   → Every request has unique parameters (search with unique query)
   → Dataset is tiny (10 rows — DB query is already <1ms)
   → Write-heavy workload (cache invalidated faster than used)
   → Consistency is critical (bank balance — must always be accurate)

2. In-Process cache is NOT suitable when:
   → Running multiple servers (each has different cache state)
   → Cache needs to survive server restarts
   → Cache size exceeds server memory
   → Need atomic operations across cache entries

3. External cache is NOT suitable when:
   → Network latency to cache > benefit of caching
      (rare, but possible for very fast DB queries)
   → Budget doesn't allow additional infrastructure
   → Single-server deployment (in-process is simpler)

4. Long TTLs are NOT suitable when:
   → Data freshness is critical for business logic
   → Users making financial decisions based on cached data
   → Regulatory requirements demand real-time accuracy

PART 1: HOW LRU (LEAST RECENTLY USED) CACHE WORKS
═══════════════════════════════════════════════════════

  Data Structure: Hash Map + Doubly Linked List
  
  Hash Map: O(1) lookup by key
  Linked List: O(1) reordering of access time

  ┌─────────────────────────────────────────────────────┐
  │  HASH MAP                                            │
  │  ┌──────────┬──────────┐                            │
  │  │ "menu"   │ → Node 3 │                            │
  │  │ "item:1" │ → Node 1 │                            │
  │  │ "item:5" │ → Node 2 │                            │
  │  └──────────┴──────────┘                            │
  │                                                      │
  │  DOUBLY LINKED LIST (ordered by access time)         │
  │                                                      │
  │  HEAD (most recent)              TAIL (least recent) │
  │    ↓                                ↓                │
  │  [Node 3] ←→ [Node 1] ←→ [Node 2]                  │
  │  "menu"       "item:1"    "item:5"                   │
  │  (just used)              (oldest access)            │
  └─────────────────────────────────────────────────────┘

  OPERATIONS:

  GET "menu":
    1. Hash map lookup → O(1) → find Node 3
    2. Move Node 3 to HEAD of linked list → O(1)
    3. Return value → CACHE HIT

  GET "item:99" (not in cache):
    1. Hash map lookup → not found → CACHE MISS
    2. Fetch from database
    3. Create new node, insert at HEAD
    4. If cache is full → remove TAIL node (least recently used)
    5. Store in hash map

  SET "orders:recent":
    1. Create node with key + value + TTL
    2. Insert at HEAD of linked list
    3. Store in hash map
    4. If over capacity → evict TAIL

  WHY LRU WORKS:
    → Recently accessed items likely to be accessed again
    → "Temporal locality" — if you read it now, you'll read it soon
    → Menu just requested → likely requested again in seconds
    → Item not accessed in 10 minutes → probably not needed

PART 2: HOW TTL (TIME TO LIVE) WORKS
═══════════════════════════════════════════════════════

  When entry is stored:
    entry = {
        "key": "menu",
        "value": [...menu data...],
        "created_at": 1704067200,    # Unix timestamp
        "ttl_seconds": 300,          # 5 minutes
        "expires_at": 1704067500     # created_at + ttl
    }

  When entry is accessed:
    current_time = time.time()       # 1704067350
    if current_time > entry.expires_at:
        # EXPIRED — treat as cache miss
        delete(entry)
        return None
    else:
        # VALID — return cached value
        return entry.value

  TWO EXPIRATION STRATEGIES:

  1. Lazy Expiration (Check on Access):
     → Don't actively scan for expired entries
     → Only check when someone asks for the key
     → Expired entries sit in memory until accessed
     → Simple but wastes memory

  2. Active Expiration (Background Cleanup):
     → Periodically scan for expired entries
     → Remove them proactively
     → Frees memory sooner
     → Uses CPU for scanning
     → Redis does BOTH: lazy + periodic sampling

PART 3: HOW LFU (LEAST FREQUENTLY USED) WORKS
═══════════════════════════════════════════════════════

  Each entry tracks ACCESS COUNT:

  ┌──────────┬───────┬──────────────┐
  │ Key      │ Count │ Value        │
  ├──────────┼───────┼──────────────┤
  │ "menu"   │  847  │ [...data...] │ ← Very popular, keep!
  │ "item:1" │  234  │ [...data...] │
  │ "item:42"│    3  │ [...data...] │ ← Rarely used, evict first
  │ "item:7" │   89  │ [...data...] │
  └──────────┴───────┴──────────────┘

  When cache is full → evict entry with LOWEST count

  Problem: "Frequency Aging"
    → Item was popular last week (count=10,000)
    → Not accessed this week at all
    → But count is still high → never evicted
    → Solution: Decay counts over time
    → Or use "LFU with aging" / "Window LFU"

  LRU vs LFU:
  ┌──────────┬──────────────────────┬──────────────────────┐
  │          │ LRU                  │ LFU                  │
  ├──────────┼──────────────────────┼──────────────────────┤
  │ Evicts   │ Oldest access        │ Fewest accesses      │
  │ Good for │ Recency matters      │ Popularity matters   │
  │ Bad for  │ Scan pollution       │ Historical bias      │
  │          │ (one-time scan evicts│ (old popular items   │
  │          │  hot items)          │  never evicted)      │
  │ Used by  │ Most caches default  │ Redis supports both  │
  └──────────┴──────────────────────┴──────────────────────┘

PART 4: CACHE STAMPEDE — THE THUNDERING HERD PROBLEM
═══════════════════════════════════════════════════════

  THE PROBLEM:
  
  Time 0:00 → "menu" cached, TTL = 300s
  Time 4:59 → 500 concurrent users actively using the app
  Time 5:00 → "menu" cache entry EXPIRES
  Time 5:00 → All 500 users request /menu simultaneously
  Time 5:00 → All 500 see CACHE MISS
  Time 5:00 → All 500 query the database
  Time 5:00 → Database gets 500 identical queries AT ONCE
  Time 5:00 → Database overloaded, slow responses, maybe crash

  ┌─────────────────────────────────────────────────────┐
  │  Cache expires                                       │
  │       ↓                                              │
  │  User 1 → cache miss → DB query ──→ ┐              │
  │  User 2 → cache miss → DB query ──→ │              │
  │  User 3 → cache miss → DB query ──→ │ 500 queries! │
  │  ...                                 │              │
  │  User 500 → cache miss → DB query → ┘              │
  │                                                      │
  │  Database: "HELP! 500 identical queries!"           │
  └─────────────────────────────────────────────────────┘

  SOLUTION 1: Cache Lock (Mutex)
  → First request that sees cache miss → acquires lock
  → All other requests → WAIT for lock
  → First request → queries DB → updates cache → releases lock
  → All waiting requests → now see cache HIT
  → Only 1 DB query instead of 500

  SOLUTION 2: Stale-While-Revalidate
  → Cache entry has TWO expiration times:
    → "soft" expiry: 5 minutes (data is "stale" but usable)
    → "hard" expiry: 10 minutes (data is too old, must refresh)
  → After soft expiry: serve stale data, refresh in background
  → Users get instant (slightly stale) response
  → No one waits for DB query

  SOLUTION 3: Early/Probabilistic Refresh
  → As TTL approaches, some requests randomly trigger refresh
  → Before actual expiry → cache is already refreshed
  → Spreads the refresh load over time

═══════════════════════════════════════════════════════
PART 5: CACHE INVALIDATION STRATEGIES
═══════════════════════════════════════════════════════

  THE FUNDAMENTAL PROBLEM:
  → Data changes in database
  → Cache still has OLD data
  → Users see stale information
  → HOW and WHEN do you update the cache?

  Strategy A: TTL-Based (Time Expiration)
  → Set TTL = 300 seconds
  → After 5 minutes, cache entry dies automatically
  → Next request triggers fresh DB query
  → Simple, but up to 5 minutes of stale data
  → MOST COMMON for most use cases

  Strategy B: Event-Based (Active Invalidation)
  → When menu item is updated → explicitly delete cache
  → cache.delete("menu") on every write
  → Fresh data on next read
  → Zero stale period
  → Requires discipline — every write path must invalidate

  Strategy C: Version-Based
  → Cache key includes version: "menu:v42"
  → When data changes → increment version → "menu:v43"
  → Old cache entry is never read again (orphaned, expires via TTL)
  → No explicit deletion needed

  Strategy D: Tag-Based
  → Cache entries tagged with dependencies
  → "menu" tagged with ["category:pizza", "category:burger"]
  → When pizza category changes → invalidate all tagged entries
  → Powerful but complex to implement

  THE HARD TRUTH:
  → Perfect cache invalidation is IMPOSSIBLE in distributed systems
  → There is ALWAYS a window where cache can be stale
  → The question is: how SMALL can that window be?
  → And: does your business TOLERATE that staleness?

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Cache Stampede (Thundering Herd)
  → Described above in internal mechanics
  → FIX: Mutex/lock, stale-while-revalidate, early refresh

PITFALL 2: Cache Penetration
  → Requesting data that DOESN'T EXIST in DB
  → Key: "item:999999" → cache miss → DB query → no result
  → Next request: same miss → same query → still no result
  → Cache never populated → DB hit EVERY time
  → Attacker can exploit: request millions of non-existent IDs
  → FIX: Cache the NULL result too
    → cache.set("item:999999", NULL, ttl=60)
  → FIX: Bloom filter (check if key CAN exist before querying)

PITFALL 3: Cache Avalanche
  → Many cache entries expire at the SAME TIME
  → All entries were created at the same time with same TTL
  → Massive DB load when they all expire simultaneously
  → FIX: Add random jitter to TTL
    → ttl = base_ttl + random(0, 60)  # Spread expirations

PITFALL 4: Stale Data Served After Write
  → User updates their profile
  → Response says "Updated!"
  → User refreshes page → sees OLD data (from cache)
  → Confusing and trust-breaking
  → FIX: Invalidate cache IMMEDIATELY after write
  → FIX: Write-through pattern for user-facing data

PITFALL 5: Cache Key Collision
  → Two different queries produce same cache key
  → "items:page=1" for pizza restaurant
  → "items:page=1" for burger restaurant
  → Same key → wrong data served
  → FIX: Include ALL distinguishing parameters in key
    → "restaurant:5:items:category=pizza:page=1:limit=10"

PITFALL 6: Over-Caching (Caching Everything)
  → Cache every single DB query result
  → Memory usage explodes
  → Eviction rate is high → low hit rate
  → More complexity for minimal benefit
  → FIX: Cache only HOT data (frequently accessed)
  → 80/20 rule: 20% of data serves 80% of requests

PITFALL 7: Inconsistent Cache in Multi-Server Setup
  → Server 1 updates DB and invalidates ITS cache
  → Server 2 still has old data in ITS cache
  → Different users see different data
  → FIX: External shared cache (Redis) — not in-process
  → Or: broadcast invalidation to all servers

PITFALL 8: Serialization Overhead
  → Storing complex objects in cache
  → Serialize to JSON/bytes on write, deserialize on read
  → If object is large → serialization cost > DB query cost
  → FIX: Cache only what you need, not entire objects
  → FIX: Use efficient serialization (msgpack, protobuf)

# SECTION 1: LRU CACHE ENGINE (from scratch)
# ============================================================

class LRUCache:
    """
    Least Recently Used cache with TTL support.
    
    Internal structure:
    - OrderedDict for O(1) access + O(1) reordering
    - Each entry stores: value, created_at, ttl, access_count
    
    When full → evicts the least recently used entry
    When TTL expires → entry treated as miss on next access
    """
    
    def __init__(self, max_size=1000, default_ttl=300, name="LRU-Cache"):
        self.max_size = max_size
        self.default_ttl = default_ttl
        self.name = name
        self.store = OrderedDict()
        self.lock = threading.Lock()
        
        # Metrics
        self.hits = 0
        self.misses = 0
        self.evictions = 0
        self.expirations = 0
        self.writes = 0
    
    def get(self, key):
        """
        Retrieve value from cache.
        Returns (value, hit) tuple.
        hit=True means served from cache, hit=False means cache miss.
        """
        with self.lock:
            if key not in self.store:
                self.misses += 1
                return None, False
            
            entry = self.store[key]
            
            # Check TTL expiration
            if self._is_expired(entry):
                del self.store[key]
                self.expirations += 1
                self.misses += 1
                return None, False
            
            # Move to end (most recently used)
            self.store.move_to_end(key)
            entry["access_count"] += 1
            entry["last_accessed"] = time.time()
            
            self.hits += 1
            return entry["value"], True
    
    def set(self, key, value, ttl=None):
        """Store value in cache with optional custom TTL."""
        if ttl is None:
            ttl = self.default_ttl
        
        with self.lock:
            # If key exists, update it
            if key in self.store:
                self.store.move_to_end(key)
                self.store[key] = self._create_entry(value, ttl)
                self.writes += 1
                return
            
            # If at capacity, evict LRU (first item)
            if len(self.store) >= self.max_size:
                evicted_key, _ = self.store.popitem(last=False)
                self.evictions += 1
            
            # Insert new entry at end (most recent)
            self.store[key] = self._create_entry(value, ttl)
            self.writes += 1
    
    def delete(self, key):
        """Explicitly remove entry from cache (invalidation)."""
        with self.lock:
            if key in self.store:
                del self.store[key]
                return True
            return False
    
    def delete_pattern(self, pattern):
        """Delete all keys matching a prefix pattern."""
        with self.lock:
            keys_to_delete = [
                k for k in self.store.keys()
                if k.startswith(pattern)
            ]
            for key in keys_to_delete:
                del self.store[key]
            return len(keys_to_delete)
    
    def clear(self):
        """Remove all entries."""
        with self.lock:
            self.store.clear()
    
    def size(self):
        """Current number of entries."""
        return len(self.store)
    
    def get_or_set(self, key, loader_fn, ttl=None):
        """
        Cache-aside pattern in one call.
        If key exists and valid → return cached value.
        If miss → call loader_fn() → cache result → return.
        """
        value, hit = self.get(key)
        if hit:
            return value, True
        
        # Cache miss — load from source
        value = loader_fn()
        self.set(key, value, ttl)
        return value, False
    
    def _create_entry(self, value, ttl):
        now = time.time()
        return {
            "value": value,
            "created_at": now,
            "ttl": ttl,
            "expires_at": now + ttl,
            "access_count": 0,
            "last_accessed": now
        }
    
    def _is_expired(self, entry):
        return time.time() > entry["expires_at"]
    
    def get_metrics(self):
        """Return cache performance metrics."""
        total_requests = self.hits + self.misses
        hit_rate = (self.hits / total_requests * 100) if total_requests > 0 else 0
        
        return {
            "name": self.name,
            "type": "LRU",
            "max_size": self.max_size,
            "current_size": len(self.store),
            "utilization_pct": round(
                len(self.store) / self.max_size * 100, 1
            ),
            "default_ttl_sec": self.default_ttl,
            "total_requests": total_requests,
            "hits": self.hits,
            "misses": self.misses,
            "hit_rate_pct": round(hit_rate, 2),
            "evictions": self.evictions,
            "expirations": self.expirations,
            "writes": self.writes
        }
    
    def reset_metrics(self):
        self.hits = 0
        self.misses = 0
        self.evictions = 0
        self.expirations = 0
        self.writes = 0
    
    def inspect(self, limit=10):
        """Peek at cache contents for debugging."""
        entries = []
        now = time.time()
        with self.lock:
            for key, entry in list(self.store.items())[-limit:]:
                entries.append({
                    "key": key,
                    "ttl_remaining_sec": round(
                        max(0, entry["expires_at"] - now), 1
                    ),
                    "access_count": entry["access_count"],
                    "expired": self._is_expired(entry),
                    "age_sec": round(now - entry["created_at"], 1)
                })
        return entries

class LFUCache:
    """
    Least Frequently Used cache with TTL support.
    Evicts the entry with the LOWEST access count.
    Better for workloads where popularity matters more than recency.
    """
    
    def __init__(self, max_size=1000, default_ttl=300, name="LFU-Cache"):
        self.max_size = max_size
        self.default_ttl = default_ttl
        self.name = name
        self.store = {}  # key → entry
        self.freq_map = defaultdict(OrderedDict)  # freq → {key: entry}
        self.min_freq = 0
        self.lock = threading.Lock()
        
        # Metrics
        self.hits = 0
        self.misses = 0
        self.evictions = 0
        self.expirations = 0
        self.writes = 0
    
    def get(self, key):
        with self.lock:
            if key not in self.store:
                self.misses += 1
                return None, False
            
            entry = self.store[key]
            
            if time.time() > entry["expires_at"]:
                self._remove_entry(key)
                self.expirations += 1
                self.misses += 1
                return None, False
            
            # Increase frequency
            self._increase_freq(key)
            entry["last_accessed"] = time.time()
            
            self.hits += 1
            return entry["value"], True
    
    def set(self, key, value, ttl=None):
        if ttl is None:
            ttl = self.default_ttl
        
        with self.lock:
            if key in self.store:
                self.store[key]["value"] = value
                self.store[key]["expires_at"] = time.time() + ttl
                self._increase_freq(key)
                self.writes += 1
                return
            
            if len(self.store) >= self.max_size:
                self._evict()
            
            now = time.time()
            entry = {
                "value": value,
                "freq": 1,
                "created_at": now,
                "expires_at": now + ttl,
                "last_accessed": now,
                "access_count": 0
            }
            
            self.store[key] = entry
            self.freq_map[1][key] = entry
            self.min_freq = 1
            self.writes += 1
    
    def delete(self, key):
        with self.lock:
            if key in self.store:
                self._remove_entry(key)
                return True
            return False
    
    def _increase_freq(self, key):
        entry = self.store[key]
        old_freq = entry["freq"]
        new_freq = old_freq + 1
        
        del self.freq_map[old_freq][key]
        if not self.freq_map[old_freq]:
            del self.freq_map[old_freq]
            if self.min_freq == old_freq:
                self.min_freq = new_freq
        
        entry["freq"] = new_freq
        entry["access_count"] += 1
        self.freq_map[new_freq][key] = entry
    
    def _evict(self):
        if not self.freq_map[self.min_freq]:
            return
        
        evicted_key, _ = self.freq_map[self.min_freq].popitem(last=False)
        if not self.freq_map[self.min_freq]:
            del self.freq_map[self.min_freq]
        
        del self.store[evicted_key]
        self.evictions += 1
    
    def _remove_entry(self, key):
        entry = self.store[key]
        freq = entry["freq"]
        del self.freq_map[freq][key]
        if not self.freq_map[freq]:
            del self.freq_map[freq]
        del self.store[key]
    
    def get_or_set(self, key, loader_fn, ttl=None):
        value, hit = self.get(key)
        if hit:
            return value, True
        value = loader_fn()
        self.set(key, value, ttl)
        return value, False
    
    def size(self):
        return len(self.store)
    
    def clear(self):
        with self.lock:
            self.store.clear()
            self.freq_map.clear()
            self.min_freq = 0
    
    def get_metrics(self):
        total = self.hits + self.misses
        hit_rate = (self.hits / total * 100) if total > 0 else 0
        return {
            "name": self.name,
            "type": "LFU",
            "max_size": self.max_size,
            "current_size": len(self.store),
            "utilization_pct": round(
                len(self.store) / self.max_size * 100, 1
            ),
            "total_requests": total,
            "hits": self.hits,
            "misses": self.misses,
            "hit_rate_pct": round(hit_rate, 2),
            "evictions": self.evictions,
            "expirations": self.expirations,
            "writes": self.writes
        }
    
    def reset_metrics(self):
        self.hits = 0
        self.misses = 0
        self.evictions = 0
        self.expirations = 0
        self.writes = 0

class StampedeProtectedCache:
    """
    Wraps any cache with stampede (thundering herd) protection.
    Uses mutex locks per key to prevent multiple simultaneous
    DB queries for the same expired key.
    Also supports stale-while-revalidate.
    """
    
    def __init__(self, cache, stale_ttl_multiplier=2.0):
        self.cache = cache
        self.stale_multiplier = stale_ttl_multiplier
        self.key_locks = defaultdict(threading.Lock)
        self.lock_cleanup = threading.Lock()
        self.stampedes_prevented = 0
    
    def get_or_set(self, key, loader_fn, ttl=None):
        """
        Stampede-safe cache-aside pattern.
        Only ONE thread queries the DB on cache miss.
        Others wait for that thread's result.
        """
        value, hit = self.cache.get(key)
        if hit:
            return value, True
        
        # Cache miss — acquire per-key lock
        key_lock = self.key_locks[key]
        
        if key_lock.acquire(blocking=False):
            # First thread to arrive — do the actual work
            try:
                # Double-check (another thread might have filled it)
                value, hit = self.cache.get(key)
                if hit:
                    return value, True
                
                # Actually load from source
                value = loader_fn()
                self.cache.set(key, value, ttl)
                return value, False
            finally:
                key_lock.release()
        else:
            # Another thread is already loading — wait
            self.stampedes_prevented += 1
            key_lock.acquire()  # Wait for the other thread
            key_lock.release()
            
            # Now the cache should be populated
            value, hit = self.cache.get(key)
            if hit:
                return value, True
            
            # Rare edge case: other thread failed
            value = loader_fn()
            self.cache.set(key, value, ttl)
            return value, False
    
    def get_metrics(self):
        metrics = self.cache.get_metrics()
        metrics["stampedes_prevented"] = self.stampedes_prevented
        return metrics

class CacheKeyBuilder:
    """
    Generates consistent, collision-free cache keys.
    Includes all parameters that affect the result.
    """
    
    @staticmethod
    def build(resource, **params):
        """
        Build a deterministic cache key.
        
        Examples:
          build("menu_items", category="pizza", page=1, limit=10)
          → "menu_items:category=pizza:limit=10:page=1"
          
          build("order", id=42)
          → "order:id=42"
        """
        parts = [resource]
        for key in sorted(params.keys()):
            value = params[key]
            if value is not None:
                parts.append(f"{key}={value}")
        return ":".join(parts)
    
    @staticmethod
    def build_hashed(resource, **params):
        """
        For very long keys, use a hash.
        Still includes readable prefix for debugging.
        """
        key_str = CacheKeyBuilder.build(resource, **params)
        if len(key_str) > 200:
            hash_suffix = hashlib.md5(
                key_str.encode()
            ).hexdigest()[:12]
            return f"{resource}:hash:{hash_suffix}"
        return key_str

class RestaurantService:
    """
    Application service that uses cache-aside pattern.
    Demonstrates how caching integrates with business logic.
    """
    
    def __init__(self, db, cache):
        self.db = db
        self.cache = cache
        self.key = CacheKeyBuilder
    
    def get_menu(self, category=None):
        """Cache-aside: check cache first, then DB."""
        cache_key = self.key.build("menu", category=category or "all")
        
        def loader():
            return self.db.query("get_menu", category=category)
        
        if isinstance(self.cache, StampedeProtectedCache):
            return self.cache.get_or_set(cache_key, loader, ttl=300)
        else:
            return self.cache.get_or_set(cache_key, loader, ttl=300)
    
    def get_item(self, item_id):
        cache_key = self.key.build("item", id=item_id)
        
        def loader():
            return self.db.query("get_item", id=item_id)
        
        return self.cache.get_or_set(cache_key, loader, ttl=600)
    
    def update_item(self, item_id, updates):
        """Write-through: update DB then invalidate cache."""
        # Update database first
        result = self.db.query("update_item", id=item_id, updates=updates)
        
        if result:
            # Invalidate related cache entries
            self.cache.cache.delete(self.key.build("item", id=item_id)) \
                if isinstance(self.cache, StampedeProtectedCache) \
                else self.cache.delete(self.key.build("item", id=item_id))
            
            # Invalidate menu caches (item is part of menu)
            cache_obj = self.cache.cache \
                if isinstance(self.cache, StampedeProtectedCache) \
                else self.cache
            if hasattr(cache_obj, 'delete_pattern'):
                cache_obj.delete_pattern("menu:")
        
        return result
    
    def get_analytics(self):
        cache_key = self.key.build("analytics")
        
        def loader():
            return self.db.query("get_analytics")
        
        if isinstance(self.cache, StampedeProtectedCache):
            return self.cache.get_or_set(cache_key, loader, ttl=600)
        else:
            return self.cache.get_or_set(cache_key, loader, ttl=600)

class CacheBenchmark:
    """
    Runs performance tests comparing cached vs uncached operations.
    """
    
    def __init__(self):
        self.results = []
    
    def run_test(self, service, test_name, num_requests,
                 concurrent_users, operation, op_params=None):
        """Run a benchmark test."""
        print(f"\n  🔥 TEST: {test_name}")
        print(f"     Requests: {num_requests} | "
              f"Concurrent: {concurrent_users}")
        print(f"     Running...", end="", flush=True)
        
        latencies = []
        errors = 0
        cache_hits = 0
        cache_misses = 0
        start_time = time.time()
        
        def execute_request(_):
            req_start = time.time()
            try:
                if operation == "get_menu":
                    cat = op_params.get("category") if op_params else None
                    result, hit = service.get_menu(category=cat)
                elif operation == "get_item":
                    item_id = random.randint(1, 100)
                    result, hit = service.get_item(item_id)
                elif operation == "get_analytics":
                    result, hit = service.get_analytics()
                else:
                    return None
                
                elapsed = (time.time() - req_start) * 1000
                return {"latency_ms": elapsed, "hit": hit}
            except Exception as e:
                return {"error": str(e)}
        
        with ThreadPoolExecutor(max_workers=concurrent_users) as executor:
            futures = [
                executor.submit(execute_request, i)
                for i in range(num_requests)
            ]
            
            for future in as_completed(futures):
                result = future.result()
                if result is None:
                    continue
                if "error" in result:
                    errors += 1
                else:
                    latencies.append(result["latency_ms"])
                    if result["hit"]:
                        cache_hits += 1
                    else:
                        cache_misses += 1
        
        total_time = time.time() - start_time
        
        if latencies:
            latencies.sort()
            n = len(latencies)
            
            analysis = {
                "test_name": test_name,
                "total_time_sec": round(total_time, 3),
                "throughput_rps": round(len(latencies) / total_time, 1),
                "successful": len(latencies),
                "errors": errors,
                "cache_hits": cache_hits,
                "cache_misses": cache_misses,
                "hit_rate_pct": round(
                    cache_hits / max(1, cache_hits + cache_misses) * 100, 2
                ),
                "latency": {
                    "avg_ms": round(statistics.mean(latencies), 2),
                    "p50_ms": round(latencies[int(n * 0.50)], 2),
                    "p95_ms": round(latencies[int(n * 0.95)], 2),
                    "p99_ms": round(latencies[min(int(n * 0.99), n-1)], 2),

            }

        self._print_result(analysis)

        return analysis

FROM PROJECT 5 WE KNOW:

  → Or share a cache (external Redis)

THE PROBLEM:
  → We have 5 identical servers
  → Client wants to request /api/v1/menu
  → Which server should the client talk to?
  → Client can't know about all 5 servers
  → If one server dies, client shouldn't notice
  → If one server is overloaded, traffic should go elsewhere

THE TRAFFIC COP:
  → Client talks to ONE address (the load balancer)
  → Load balancer forwards to one of the 5 servers
  → Client doesn't know (or care) which server handled it
  → If Server 3 dies → load balancer stops sending traffic there
  → If Server 2 is busy → load balancer sends to Server 4 instead

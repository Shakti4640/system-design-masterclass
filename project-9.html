PROJECT 9: THE BOUNCER AT THE DOOR — API RATE LIMITER

📋 PHASE 2 (Easy) — Core Patterns

🎯 HYPOTHETICAL PROBLEM

You built a pizza shop API (Project 2)
You added a database to store orders (Project 3)
You added caching so repeated menu requests are fast (Project 5)
You added JWT authentication so you know WHO is ordering (Project 8)

Everything works beautifully... until one night:

SCENARIO A: The Greedy Script
  → A developer writes a script that hits GET /menu 50,000 times/second
  → Your server CPU spikes to 100%
  → Legitimate customers can't load the menu
  → Your database connection pool is exhausted
  → The entire pizza shop is DOWN for everyone

SCENARIO B: The Brute Force Attacker
  → Someone tries POST /login with different passwords
  → 10,000 attempts per minute
  → They're trying to guess a customer's password
  → Without limits, they WILL eventually succeed

SCENARIO C: The Accidental DDoS
  → A popular food blogger links to your API
  → 500,000 users hit your API simultaneously
  → Not malicious, but the EFFECT is the same
  → Your server melts

SCENARIO D: The Costly Endpoint
  → POST /orders triggers payment processing ($0.02 per API call to Stripe)
  → A buggy client retries 1,000 times per minute
  → You're losing $20/minute in payment API fees
  → That's $28,800/day GONE

THE QUESTION:
  → How do you let LEGITIMATE traffic through
  → While BLOCKING abusive traffic
  → Without slowing down normal users
  → Without requiring manual intervention
  → In real-time, at the EDGE of your system?

THE ANSWER: A Rate Limiter
  → The BOUNCER at your API door
  → Checks how many times you've entered recently
  → If too many → "Come back later" (HTTP 429)
  → If within limits → "Welcome in"

🔍 HOW TO APPROACH IT

Step 1: Define WHAT to limit
         → Per user? Per IP? Per API key? Per endpoint?
         → Each has different implications

Step 2: Define HOW MUCH to allow
         → 100 requests/minute? 1000 requests/hour?
         → Different endpoints may need different limits
         → GET /menu (cheap) vs POST /orders (expensive)

Step 3: Define the TIME WINDOW
         → Per second? Per minute? Per hour?
         → Fixed windows vs sliding windows

Step 4: Choose the ALGORITHM
         → Fixed Window Counter
         → Sliding Window Log
         → Sliding Window Counter
         → Token Bucket
         → Leaky Bucket
         → Each has different tradeoffs

Step 5: Decide WHERE to enforce
         → At the application level (in your code)
         → At the API Gateway level (Project 19, future)
         → At the load balancer level (Project 6)
         → At the reverse proxy level (Nginx)

Step 6: Decide WHAT to do when limit is exceeded
         → Return 429 Too Many Requests
         → Include Retry-After header
         → Queue the request for later?
         → Silently drop?

Step 7: Decide WHERE to store counters
         → In-memory (fast but lost on restart)
         → Redis/cache layer (Project 5 concepts — shared across servers)
         → Database (slow but persistent)

We will implement ALL 5 major algorithms
to understand their tradeoffs deeply:

ALGORITHM 1: FIXED WINDOW COUNTER
══════════════════════════════════
  Timeline:  |----minute 1----|----minute 2----|
  Requests:  ||||| ||||| ||   ||| ||           
  Count:     12 (limit=15)     5 (limit=15)
  
  → Divide time into fixed windows (e.g., each minute)
  → Count requests in current window
  → Reset counter when window changes

  PROBLEM: Boundary burst
  → 15 requests at 0:59 + 15 requests at 1:00 = 30 in 2 seconds
  → Both are "within limit" individually


ALGORITHM 2: SLIDING WINDOW LOG
════════════════════════════════
  → Store timestamp of EVERY request
  → When new request arrives:
    → Remove timestamps older than window size
    → Count remaining timestamps
    → If count < limit → allow
  
  Timeline:  [t1, t2, t3, ... t98, t99, t100]
             ↑ remove old ones    ↑ count these
  
  PROBLEM: Memory intensive
  → Must store every timestamp
  → 1M users × 100 requests = 100M timestamps in memory


ALGORITHM 3: SLIDING WINDOW COUNTER
════════════════════════════════════
  → Hybrid of Fixed Window + Sliding Window
  → Uses WEIGHTED count from previous + current window
  
  Previous window count: 84 (limit=100)
  Current window count: 36
  Current position: 25% into current window
  
  Weighted count = 84 × (1 - 0.25) + 36 = 84 × 0.75 + 36 = 63 + 36 = 99
  → 99 < 100 → ALLOW
  
  Low memory, good accuracy


ALGORITHM 4: TOKEN BUCKET
═════════════════════════
  → Imagine a bucket that holds tokens
  → Bucket has MAX CAPACITY (e.g., 10 tokens)
  → Tokens are ADDED at a fixed rate (e.g., 1 token/second)
  → Each request CONSUMES 1 token
  → If bucket is empty → REJECT
  → If bucket has tokens → ALLOW and remove one
  
  ┌─────────────┐
  │ 🪙🪙🪙🪙🪙  │ ← Bucket (capacity=10)
  │ 🪙🪙🪙🪙🪙  │
  └──────┬──────┘
         │ +1 token every second (refill rate)
         │
         │ -1 token per request (consumption)
         ▼
  
  ADVANTAGE: Allows SHORT BURSTS up to bucket capacity
  → Then smooths out to refill rate
  → Most commonly used in production (AWS, Stripe, etc.)


ALGORITHM 5: LEAKY BUCKET
══════════════════════════
  → Requests enter the bucket (queue)
  → Bucket processes at a FIXED RATE
  → If bucket is full → REJECT new requests
  → Smooths out bursty traffic to constant rate
  
  Incoming:  |||||||||||  (burst of 10)
  Outgoing:  | . | . | . | . | . |  (1 per second)
  
  ┌─────────────┐
  │ req req req  │ ← Queue (max size = bucket capacity)
  │ req req      │
  └──────┬──────┘
         │ drip... drip... drip (fixed output rate)
         ▼
       Server

  ADVANTAGE: Perfectly smooth output rate
  DISADVANTAGE: Adds latency (requests wait in queue)

📌 WHAT COULD BE DONE

Option A: Application-Level Rate Limiting
  → Implement in your Python/Node/Java code
  → Full control over logic
  → Per-endpoint, per-user granularity
  → But: Every server instance needs shared state (Project 5 — cache)

Option B: Reverse Proxy Rate Limiting (Nginx)
  → Configure Nginx to limit before requests hit your app
  → Very fast (C-level performance)
  → But: Less granular (usually per-IP, not per-user)

Option C: API Gateway Rate Limiting (Project 19 preview)
  → Centralized rate limiting at the gateway
  → All services behind it are protected
  → But: Single point of configuration

Option D: Distributed Rate Limiting (Redis-based)
  → Store counters in Redis (shared across all servers)
  → Works with horizontal scaling (Project 4)
  → But: Adds network hop to Redis for every request

Option E: Client-Side Rate Limiting
  → Client self-limits (polite clients)
  → Never rely on this alone (can be bypassed)
  → But: Good practice in SDKs you distribute

✅ WHAT SHOULD DO

1. ALWAYS return HTTP 429 with Retry-After header
   → Client knows WHEN to try again
   → Without it, clients just hammer you in a loop
   → Response: {"error": "Rate limit exceeded", "retry_after_seconds": 30}

2. ALWAYS rate limit by IDENTITY, not just IP
   → Behind NAT: 1000 users share one IP
   → Using JWT from Project 8: extract user_id from token
   → Rate limit per user_id, fallback to IP for unauthenticated

3. ALWAYS set DIFFERENT limits for different endpoints
   → GET /menu → 200 requests/minute (cheap, read-only)
   → POST /orders → 10 requests/minute (expensive, writes)
   → POST /login → 5 requests/minute (security-critical)

4. ALWAYS include rate limit info in response headers
   → X-RateLimit-Limit: 100        (your limit)
   → X-RateLimit-Remaining: 73     (how many left)
   → X-RateLimit-Reset: 1640000000 (when window resets, epoch)
   → Transparent to the client

5. ALWAYS have a whitelist/blacklist mechanism
   → Internal services → no rate limit (whitelisted)
   → Known attackers → permanently blocked (blacklisted)
   → Premium users → higher limits

6. ALWAYS log rate-limited requests
   → Who was limited? When? How many times?
   → Helps detect attacks and tune limits

7. ALWAYS fail OPEN or CLOSED deliberately
   → If Redis is down and you can't check limits:
     → Fail OPEN: Allow all traffic (availability > protection)
     → Fail CLOSED: Block all traffic (protection > availability)
     → Choose based on your system's priority

❌ WHAT SHOULD NOT DO

1. NEVER rate limit ONLY at the application level when scaled
   → With 10 servers (Project 4), each has its own counter
   → User hits server A 99 times, then server B 99 times
   → Each server thinks "only 99, under limit of 100"
   → But total = 198 → way over limit
   → FIX: Use shared state (Redis — Project 5 cache concept)

2. NEVER use only IP-based limiting in production
   → Corporate networks: 10,000 employees behind 1 IP
   → You'd block an entire company
   → Mobile carriers: millions share a few IPs
   → FIX: Use API key or JWT user_id as primary identifier

3. NEVER expose your rate limiting internals to attackers
   → Don't say "You have 3 attempts left on password"
   → Attackers use this to optimize their brute force
   → For login: generic "Too many attempts, try later"

4. NEVER set rate limits without measuring first
   → Measure your NORMAL traffic patterns
   → Set limits at 2-3x normal peak
   → Too tight → legitimate users get blocked
   → Too loose → doesn't protect anything

5. NEVER forget to rate limit internal services
   → A buggy microservice can DDoS your database
   → Internal traffic should also have limits
   → "Trust but verify" → even between your own services

6. NEVER implement rate limiting without considering time sync
   → Distributed servers may have slightly different clocks
   → Use NTP or atomic clocks for consistency
   → Or use token bucket (doesn't depend on wall clock)

🚫 WHEN NOT TO USE

1. DO NOT rate limit health check endpoints
   → Load balancers (Project 6) ping /health constantly
   → Rate limiting them = load balancer thinks server is dead
   → Whitelist health endpoints

2. DO NOT rate limit internal service-to-service calls the same as external
   → Internal calls are trusted (mostly)
   → Use higher limits or separate limit pools
   → But DO still have limits (buggy services exist)

3. DO NOT use rate limiting as your ONLY defense
   → Rate limiting ≠ Authentication (Project 8)
   → Rate limiting ≠ Authorization
   → Rate limiting ≠ Input Validation (Project 2)
   → Rate limiting ≠ DDoS protection (need CDN/WAF — Project 12)
   → It's ONE layer in defense-in-depth

4. DO NOT apply rate limiting to batch/async operations
   → If a user submits a batch job that takes 10 minutes
   → That's ONE request, not thousands
   → Rate limit the SUBMISSION, not the processing

5. DO NOT use rate limiting when you need back-pressure instead
   → If downstream service is slow → use circuit breaker (Project 21)
   → Rate limiter protects YOUR service
   → Circuit breaker protects THEIR service
   → Different tools, different purposes

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

ALGORITHM 1: FIXED WINDOW COUNTER — INTERNAL MECHANICS
═══════════════════════════════════════════════════════

  Data Structure: HashMap<key, {count, window_start}>
  
  On each request:
    1. current_window = floor(current_time / window_size)
    2. Lookup key in HashMap
    3. If window_start ≠ current_window:
         → Reset count to 0, update window_start
    4. If count < limit:
         → Increment count, ALLOW
    5. Else:
         → REJECT with 429

  Memory: O(1) per key (just a counter + timestamp)
  Time:   O(1) per request

  THE BOUNDARY PROBLEM (visualized):
  
  Window 1 (0:00-0:59)     Window 2 (1:00-1:59)
  ─────────────────────┬────────────────────────
                  ▓▓▓▓▓│▓▓▓▓▓
                  5 req │5 req
                  0:58  │1:01
  
  → Both windows allow 5 (limit=5)
  → But 10 requests in 3 seconds!
  → Effective rate = 2x the intended limit at boundaries


ALGORITHM 2: SLIDING WINDOW LOG — INTERNAL MECHANICS
════════════════════════════════════════════════════

  Data Structure: HashMap<key, SortedList<timestamps>>
  
  On each request:
    1. current_time = now()
    2. window_start = current_time - window_size
    3. Remove ALL timestamps < window_start (expired)
    4. If len(timestamps) < limit:
         → Add current_time to list, ALLOW
    5. Else:
         → REJECT with 429

  Memory: O(limit) per key (stores every timestamp)
  Time:   O(limit) per request (cleanup + count)

  WHY IT'S PERFECT:
  → No boundary problem
  → Always looking at exact last N seconds
  → But storing 100 timestamps × 1M users = 100M entries


ALGORITHM 3: SLIDING WINDOW COUNTER — INTERNAL MECHANICS
═══════════════════════════════════════════════════════

  Data Structure: HashMap<key, {prev_count, curr_count, curr_window}>
  
  On each request:
    1. current_window = floor(current_time / window_size)
    2. If curr_window is outdated → rotate:
         prev_count = curr_count, curr_count = 0
    3. elapsed_ratio = (current_time % window_size) / window_size
    4. weighted_count = prev_count × (1 - elapsed_ratio) + curr_count
    5. If weighted_count < limit:
         → Increment curr_count, ALLOW
    6. Else:
         → REJECT

  Memory: O(1) per key (two counters + window marker)
  Time:   O(1) per request

  WHY THE MATH WORKS:
  
  Time:  |===prev window===|===curr window===|
         |  count=80       |  count=30       |
                                 ↑
                           We are 40% into current window
  
  weighted = 80 × (1 - 0.4) + 30
           = 80 × 0.6 + 30
           = 48 + 30 = 78
  
  → Approximates "how many in the last full window"
  → 78 < 100 → ALLOW


ALGORITHM 4: TOKEN BUCKET — INTERNAL MECHANICS
══════════════════════════════════════════════

  Data Structure: HashMap<key, {tokens, last_refill_time}>
  
  On each request:
    1. current_time = now()
    2. elapsed = current_time - last_refill_time
    3. new_tokens = elapsed × refill_rate
    4. tokens = min(tokens + new_tokens, max_capacity)
    5. last_refill_time = current_time
    6. If tokens >= 1:
         → tokens -= 1, ALLOW
    7. Else:
         → REJECT

  Memory: O(1) per key (token count + timestamp)
  Time:   O(1) per request

  THE REFILL MECHANICS (lazy refill):
  
  Time:  0s     1s     2s     3s     4s     5s
  Tokens: 10     9      10     10     8      9
  Events:  -1    refill  -     -2    refill
  
  → We DON'T run a background thread adding tokens
  → We CALCULATE how many tokens should exist WHEN a request arrives
  → This is called "lazy evaluation" — no timers needed
  → This is why Token Bucket is so efficient

  TWO PARAMETERS CONTROL BEHAVIOR:
  → max_capacity (bucket size) → controls BURST size
  → refill_rate (tokens/second) → controls SUSTAINED rate
  
  Example:
    capacity=10, refill_rate=2/sec
    → Can burst up to 10 requests instantly
    → Then limited to 2 requests/second sustained
    → After 5 seconds idle: bucket refills to 10 again


ALGORITHM 5: LEAKY BUCKET — INTERNAL MECHANICS
══════════════════════════════════════════════

  Data Structure: HashMap<key, Queue<request>> + processor
  
  On each request:
    1. If queue.size < capacity:
         → Add request to queue, ACCEPT (queued)
    2. Else:
         → REJECT (bucket overflow)
  
  Background processor:
    → Every (1/rate) seconds:
      → Dequeue one request
      → Process it
      → Repeat

  Memory: O(capacity) per key (the queue)
  Time:   O(1) to enqueue

  KEY DIFFERENCE FROM TOKEN BUCKET:
  ┌─────────────────┬────────────────────┬─────────────────────┐
  │                  │ TOKEN BUCKET       │ LEAKY BUCKET        │
  ├─────────────────┼────────────────────┼─────────────────────┤
  │ Burst handling  │ Allows bursts      │ Smooths to constant │
  │ Processing      │ Immediate if token │ Queued, fixed rate   │
  │ Analogy         │ Coins in a jar     │ Water through a hole │
  │ Latency impact  │ None (pass/fail)   │ Adds queue wait time │
  │ Implementation  │ Counter-based      │ Queue-based          │
  └─────────────────┴────────────────────┴─────────────────────┘

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Race Condition in Counter Increment
  → Two requests arrive simultaneously for same user
  → Both read count=99 (limit=100)
  → Both see 99 < 100, both increment
  → Final count = 101 → exceeded limit
  → FIX: Use atomic operations
    → Redis INCR is atomic
    → Python threading.Lock (Project 1C showed this)
    → Database: UPDATE ... WHERE count < limit (atomic check-and-set)

PITFALL 2: Clock Skew in Distributed Systems
  → Server A thinks it's 12:00:00
  → Server B thinks it's 12:00:03
  → Fixed Window resets at different times on different servers
  → User gets different limits depending on which server they hit
  → FIX: Use Token Bucket (doesn't depend on wall clock alignment)
  → FIX: Use centralized time source (Redis server's clock)

PITFALL 3: Memory Leak with Sliding Window Log
  → User makes 1 request, never comes back
  → Their timestamp sits in memory forever
  → Millions of inactive users = millions of stale entries
  → FIX: TTL-based expiration (Project 5 concept)
  → FIX: Periodic cleanup of inactive keys

PITFALL 4: Rate Limit Counter Survives Server Restart (or Not)
  → In-memory counters → server restarts → all counters reset to 0
  → Attacker knows this: crash your server, get fresh limits
  → FIX: Store counters in Redis (survives app restart)
  → FIX: Redis persistence (AOF/RDB — covered in Project 23)

PITFALL 5: Not Handling Distributed Rate Limiting
  → Project 6 showed: load balancer spreads traffic across N servers
  → Each server has LOCAL counters
  → User limit=100, 5 servers → user gets 500 effective requests
  → FIX: Shared counter store (Redis)
  → FIX: Sticky sessions (Project 8) — but this limits scaling

PITFALL 6: Blocking on Rate Limit Check
  → If Redis is slow or down → every request waits
  → Entire system becomes as slow as rate limiter
  → FIX: Timeout on Redis call (50ms max)
  → FIX: Fail-open policy (if can't check → allow)
  → FIX: Local in-memory fallback with degraded accuracy

PITFALL 7: Not Differentiating Between Read and Write
  → GET /menu (costs almost nothing) has same limit as
  → POST /orders (costs database write + payment API call)
  → FIX: Per-endpoint limits, weighted by cost

📈 SCALABILITY CONSIDERATIONS

SINGLE SERVER (where we start):
  → In-memory HashMap → microsecond lookups
  → Handles millions of checks per second
  → No network hop needed
  → Problem: Lost on restart, not shared across instances

MULTIPLE SERVERS (Project 4/6 scenario):
  → Need SHARED state across all servers
  → Redis becomes the standard solution
  
  Server A ──→ ┌───────────┐
  Server B ──→ │   REDIS   │ ← Single source of truth for counters
  Server C ──→ └───────────┘
  
  → Each rate limit check = 1 network round trip to Redis
  → Redis handles ~100,000 operations/second single-threaded
  → For 10,000 requests/second → Redis is fine
  → For 1,000,000 requests/second → need Redis cluster (Project 33)

OPTIMIZING REDIS CALLS:
  → Use Redis pipeline: batch multiple commands in one round trip
  → Use Lua scripts: atomic multi-step operations on Redis server
  → Token Bucket in Redis Lua: read + calculate + update in ONE call
  → Reduces network overhead by 3-5x

AT EXTREME SCALE (>1M requests/second):
  → Hybrid approach:
    → Local in-memory counter (fast, approximate)
    → Periodic sync to Redis (accurate, slower)
    → Accept slight over-limit (99.9% accuracy is fine)
  
  → Tiered rate limiting:
    → Tier 1: Nginx connection limit (per-IP, very fast)
    → Tier 2: Application rate limiter (per-user, accurate)
    → Tier 3: API Gateway rate limiter (per-tenant, policy-driven)

NUMBERS TO KNOW:
  → Redis INCR: ~100,000 ops/sec (single instance)
  → In-memory check: ~10,000,000 ops/sec (Python)
  → In-memory check: ~100,000,000 ops/sec (C/Rust)
  → Network round trip to Redis: ~0.5-2ms
  → Rate limit overhead should be <1% of request latency

🔍 INTERPRETABILITY CONSIDERATIONS

WHAT CLIENTS SHOULD SEE:
  
  Every response (whether allowed or limited) includes:
  
  HTTP/1.1 200 OK (or 429 Too Many Requests)
  X-RateLimit-Limit: 100                 ← Your total budget
  X-RateLimit-Remaining: 67              ← How much is left
  X-RateLimit-Reset: 1700000060          ← When budget refills (epoch)
  Retry-After: 30                        ← (only on 429) Wait this long

  These headers make rate limiting TRANSPARENT
  → Client SDKs can auto-throttle
  → Developers can debug without guessing
  → Monitoring tools can track usage

  Dashboard Metrics:
  → Total requests/second (by endpoint)
  → Percentage of requests rate-limited (by user, by IP)
  → Top 10 most rate-limited users (who's hitting limits?)
  → Rate limit headroom (how close are normal users to limits?)
  → Redis latency (is rate limit check becoming bottleneck?)

  Alerts:
  → "User X has been rate-limited 500 times in 1 hour" → potential attack

# BASE INTERFACE — All rate limiters share this contract

class RateLimiter(ABC):
    """
    Every rate limiter answers ONE question:
    "Should this request from this key be ALLOWED right now?"
    """
    
    @abstractmethod
    def allow_request(self, key: str) -> dict:
        """
        Returns:
          {
            "allowed": True/False,
            "limit": max requests allowed,
            "remaining": how many left,
            "retry_after": seconds to wait (if blocked)
          }
        """
        pass

# ALGORITHM 1: FIXED WINDOW COUNTER
# ============================================================

class FixedWindowCounter(RateLimiter):
    
    def __init__(self, max_requests: int, window_size_seconds: int):
        self.max_requests = max_requests
        self.window_size = window_size_seconds
        # {key: {"count": N, "window_start": T}}
        self.windows = defaultdict(lambda: {"count": 0, "window_start": 0})
        self.lock = threading.Lock()
    
    def _get_current_window(self) -> int:
        """Which fixed window are we in right now?"""
        return int(time.time() // self.window_size)
    
    def allow_request(self, key: str) -> dict:
        with self.lock:
            current_window = self._get_current_window()
            entry = self.windows[key]
            
            # New window? Reset counter
            if entry["window_start"] != current_window:
                entry["count"] = 0
                entry["window_start"] = current_window
            
            if entry["count"] < self.max_requests:
                entry["count"] += 1
                return {
                    "allowed": True,
                    "limit": self.max_requests,
                    "remaining": self.max_requests - entry["count"],
                    "retry_after": 0
                }
            else:
                # How long until window resets?
                window_end = (current_window + 1) * self.window_size
                retry_after = window_end - time.time()
                return {
                    "allowed": False,
                    "limit": self.max_requests,
                    "remaining": 0,
                    "retry_after": round(retry_after, 1)
                }

class SlidingWindowLog(RateLimiter):
    
    def __init__(self, max_requests: int, window_size_seconds: int):
        self.max_requests = max_requests
        self.window_size = window_size_seconds
        # {key: deque of timestamps}
        self.request_logs = defaultdict(deque)
        self.lock = threading.Lock()
    
    def allow_request(self, key: str) -> dict:
        with self.lock:
            now = time.time()
            window_start = now - self.window_size
            log = self.request_logs[key]
            
            # Remove expired timestamps
            while log and log[0] <= window_start:
                log.popleft()
            
            if len(log) < self.max_requests:
                log.append(now)
                return {
                    "allowed": True,
                    "limit": self.max_requests,
                    "remaining": self.max_requests - len(log),
                    "retry_after": 0
                }
            else:
                # Earliest request in window will expire at:
                oldest = log[0]
                retry_after = oldest + self.window_size - now
                return {
                    "allowed": False,
                    "limit": self.max_requests,
                    "remaining": 0,
                    "retry_after": round(max(0, retry_after), 1)
                }

class SlidingWindowCounter(RateLimiter):
    
    def __init__(self, max_requests: int, window_size_seconds: int):
        self.max_requests = max_requests
        self.window_size = window_size_seconds
        # {key: {"prev_count": N, "curr_count": N, "curr_window": T}}
        self.counters = defaultdict(
            lambda: {"prev_count": 0, "curr_count": 0, "curr_window": 0}
        )
        self.lock = threading.Lock()
    
    def _get_current_window(self) -> int:
        return int(time.time() // self.window_size)
    
    def allow_request(self, key: str) -> dict:
        with self.lock:
            now = time.time()
            current_window = self._get_current_window()
            entry = self.counters[key]
            
            # Rotate windows if needed
            if entry["curr_window"] != current_window:
                if entry["curr_window"] == current_window - 1:
                    # Previous window becomes prev_count
                    entry["prev_count"] = entry["curr_count"]
                else:
                    # More than one window has passed, prev is 0
                    entry["prev_count"] = 0
                entry["curr_count"] = 0
                entry["curr_window"] = current_window
            
            # Calculate weighted count
            elapsed_ratio = (now % self.window_size) / self.window_size
            weighted_count = (
                entry["prev_count"] * (1 - elapsed_ratio) 
                + entry["curr_count"]
            )
            
            if weighted_count < self.max_requests:
                entry["curr_count"] += 1
                remaining = max(0, int(self.max_requests - weighted_count - 1))
                return {
                    "allowed": True,
                    "limit": self.max_requests,
                    "remaining": remaining,
                    "retry_after": 0
                }
            else:
                window_end = (current_window + 1) * self.window_size
                retry_after = window_end - now
                return {
                    "allowed": False,
                    "limit": self.max_requests,
                    "remaining": 0,
                    "retry_after": round(retry_after, 1)
                }

class TokenBucket(RateLimiter):
    
    def __init__(self, max_tokens: int, refill_rate: float):
        """
        max_tokens: Bucket capacity (burst size)
        refill_rate: Tokens added per second (sustained rate)
        """
        self.max_tokens = max_tokens
        self.refill_rate = refill_rate
        # {key: {"tokens": float, "last_refill": float}}
        self.buckets = defaultdict(
            lambda: {"tokens": max_tokens, "last_refill": time.time()}
        )
        self.lock = threading.Lock()
    
    def _refill(self, bucket: dict) -> None:
        """Lazy refill: calculate tokens that should have been added"""
        now = time.time()
        elapsed = now - bucket["last_refill"]
        new_tokens = elapsed * self.refill_rate
        bucket["tokens"] = min(
            self.max_tokens, 
            bucket["tokens"] + new_tokens
        )
        bucket["last_refill"] = now
    
    def allow_request(self, key: str, tokens_needed: int = 1) -> dict:
        with self.lock:
            bucket = self.buckets[key]
            self._refill(bucket)
            
            if bucket["tokens"] >= tokens_needed:
                bucket["tokens"] -= tokens_needed
                return {
                    "allowed": True,
                    "limit": self.max_tokens,
                    "remaining": int(bucket["tokens"]),
                    "retry_after": 0
                }
            else:
                # How long until enough tokens are available?
                tokens_deficit = tokens_needed - bucket["tokens"]
                retry_after = tokens_deficit / self.refill_rate
                return {
                    "allowed": False,
                    "limit": self.max_tokens,
                    "remaining": 0,
                    "retry_after": round(retry_after, 1)
                }

class LeakyBucket(RateLimiter):
    
    def __init__(self, capacity: int, leak_rate: float):
        """
        capacity: Maximum queue size
        leak_rate: Requests processed per second
        """
        self.capacity = capacity
        self.leak_rate = leak_rate
        # {key: {"water": float, "last_leak": float}}
        self.buckets = defaultdict(
            lambda: {"water": 0.0, "last_leak": time.time()}
        )
        self.lock = threading.Lock()
    
    def _leak(self, bucket: dict) -> None:
        """Drain water (processed requests) since last check"""
        now = time.time()
        elapsed = now - bucket["last_leak"]
        leaked = elapsed * self.leak_rate
        bucket["water"] = max(0.0, bucket["water"] - leaked)
        bucket["last_leak"] = now
    
    def allow_request(self, key: str) -> dict:
        with self.lock:
            bucket = self.buckets[key]
            self._leak(bucket)
            
            if bucket["water"] < self.capacity:
                bucket["water"] += 1
                return {
                    "allowed": True,
                    "limit": self.capacity,
                    "remaining": int(self.capacity - bucket["water"]),
                    "retry_after": 0
                }
            else:
                # How long until one slot opens?
                retry_after = 1.0 / self.leak_rate
                return {
                    "allowed": False,
                    "limit": self.capacity,
                    "remaining": 0,
                    "retry_after": round(retry_after, 1)
                }

class TokenBucketLimiter:
    """Production token bucket with per-key and per-endpoint support"""
    
    def __init__(self):
        self.lock = threading.Lock()
        self.buckets = {}
        # Cleanup stale entries periodically
        self.cleanup_thread = threading.Thread(
            target=self._periodic_cleanup, daemon=True
        )
        self.cleanup_thread.start()
    
    def _periodic_cleanup(self):
        """Remove buckets that haven't been accessed in 10 minutes"""
        while True:
            time.sleep(60)
            cutoff = time.time() - 600
            with self.lock:
                stale_keys = [
                    k for k, v in self.buckets.items()
                    if v["last_refill"] < cutoff
                ]
                for k in stale_keys:
                    del self.buckets[k]
                if stale_keys:
                    print(f"  [CLEANUP] Removed {len(stale_keys)} stale rate limit entries")
    
    def check(
        self,
        key: str,
        max_tokens: int,
        refill_rate: float,
        tokens_needed: int = 1
    ) -> dict:
        with self.lock:
            now = time.time()
            
            if key not in self.buckets:
                self.buckets[key] = {
                    "tokens": max_tokens,
                    "last_refill": now,
                    "total_allowed": 0,
                    "total_rejected": 0
                }
            
            bucket = self.buckets[key]
            
            # Lazy refill
            elapsed = now - bucket["last_refill"]
            new_tokens = elapsed * refill_rate
            bucket["tokens"] = min(max_tokens, bucket["tokens"] + new_tokens)
            bucket["last_refill"] = now
            
            if bucket["tokens"] >= tokens_needed:
                bucket["tokens"] -= tokens_needed
                bucket["total_allowed"] += 1
                return {
                    "allowed": True,
                    "limit": max_tokens,
                    "remaining": int(bucket["tokens"]),
                    "reset_at": now + (max_tokens - bucket["tokens"]) / refill_rate,
                    "retry_after": 0
                }
            else:
                bucket["total_rejected"] += 1
                deficit = tokens_needed - bucket["tokens"]
                retry_after = deficit / refill_rate
                return {
                    "allowed": False,
                    "limit": max_tokens,
                    "remaining": 0,
                    "reset_at": now + retry_after,
                    "retry_after": round(retry_after, 2)
                }
    
    def get_stats(self) -> dict:
        with self.lock:
            return {
                "active_keys": len(self.buckets),
                "keys": {
                    k: {
                        "tokens_remaining": round(v["tokens"], 1),
                        "total_allowed": v["total_allowed"],
                        "total_rejected": v["total_rejected"]
                    }
                    for k, v in self.buckets.items()
                }
            }

def extract_user_from_token(auth_header: Optional[str]) -> Optional[str]:
    """Extract user_id from Bearer token (simplified JWT decode)"""
    if not auth_header or not auth_header.startswith("Bearer "):
        return None
    
    token = auth_header[7:]  # Remove "Bearer "
    try:
        # JWT has 3 parts: header.payload.signature
        parts = token.split(".")
        if len(parts) != 3:
            return None
        
        # Decode payload (base64)
        payload = parts[1]
        # Add padding if needed
        payload += "=" * (4 - len(payload) % 4)
        decoded = base64.urlsafe_b64decode(payload)
        data = json.loads(decoded)
        return data.get("user_id", None)
    except Exception:
        return None

class RateLimitedHandler(BaseHTTPRequestHandler):
    
    # ---- Core: Rate Limit Check ----
    def check_rate_limit(self) -> Optional[dict]:
        """
        Returns None if allowed, or a rejection dict if rate limited.
        This runs BEFORE any business logic.
        """
        client_ip = self.client_address[0]
        
        # Step 1: Check blacklist (always block)
        if client_ip in BLACKLIST_IPS:
            return {
                "status": 403,
                "error": "Forbidden",
                "message": "Your IP has been blocked"
            }
        
        # Step 2: Check whitelist (always allow)
        if client_ip in WHITELIST_IPS:
            return None  # Allowed, skip rate limiting
        
        # Step 3: Determine identity
        # Priority: JWT user_id > API key > IP address
        auth_header = self.headers.get("Authorization")
        user_id = extract_user_from_token(auth_header)
        
        if user_id:
            identity = f"user:{user_id}"
        else:
            identity = f"ip:{client_ip}"
        
        # Step 4: Find matching rule
        method = self.command  # GET, POST, DELETE, etc.
        path = self.path.split("?")[0]  # Remove query params
        
        # Try exact match, then prefix match, then default
        rule_key = f"{method}:{path}"
        
        # Check prefix matches for paths like /orders/ORD-001
        matched_rule = None
        for pattern, rule in RATE_LIMIT_RULES.items():
            if pattern == "__default__":
                continue
            if rule_key.startswith(pattern):
                matched_rule = rule
                break
        
        if not matched_rule:
            matched_rule = RATE_LIMIT_RULES["__default__"]
        
        # Step 5: Build composite key (identity + endpoint)
        composite_key = f"{identity}|{rule_key}"
        
        # Step 6: Check the rate limiter
        result = rate_limiter.check(
            key=composite_key,
            max_tokens=matched_rule["max_tokens"],
            refill_rate=matched_rule["refill_rate"],
            tokens_needed=matched_rule["cost"]
        )
        
        # Store for header injection
        self._rate_limit_info = result
        self._rate_limit_identity = identity
        
        if not result["allowed"]:
            return {
                "status": 429,
                "error": "Too Many Requests",
                "message": f"Rate limit exceeded for {identity}",
                "retry_after": result["retry_after"],
                "limit": result["limit"]
            }
        
        return None  # Allowed
    
    # ---- Utility: Send Response with Rate Limit Headers ----
    def send_json(self, status_code: int, data: dict):
        self.send_response(status_code)
        self.send_header("Content-Type", "application/json")
        
        # Always inject rate limit headers
        if hasattr(self, "_rate_limit_info"):
            info = self._rate_limit_info
            self.send_header("X-RateLimit-Limit", str(info["limit"]))
            self.send_header("X-RateLimit-Remaining", str(info["remaining"]))
            self.send_header("X-RateLimit-Reset", str(int(info["reset_at"])))
            if not info["allowed"]:
                self.send_header("Retry-After", str(int(info["retry_after"]) + 1))
        
        body = json.dumps(data, indent=2).encode("utf-8")
        self.send_header("Content-Length", str(len(body)))
        self.end_headers()
        self.wfile.write(body)
    
    # ---- GET Endpoints ----
    def do_GET(self):
        # RATE LIMIT CHECK — runs first, before any logic
        rejection = self.check_rate_limit()
        if rejection:
            self.send_json(rejection["status"], rejection)
            return
        
        path = self.path.split("?")[0]
        
        if path == "/health":
            self.send_json(200, {"status": "healthy"})
        
        elif path == "/menu":
            self.send_json(200, {
                "status": 200,
                "menu": {
                    "margherita": 10,
                    "pepperoni": 13,
                    "veggie": 11
                }
            })
        
        elif path == "/rate-limit-stats":
            stats = rate_limiter.get_stats()
            self.send_json(200, {"status": 200, "rate_limiter": stats})
        
        else:
            self.send_json(404, {"status": 404, "error": "Not found"})
    
    # ---- POST Endpoints ----
    def do_POST(self):
        rejection = self.check_rate_limit()
        if rejection:
            self.send_json(rejection["status"], rejection)
            return
        
        path = self.path.split("?")[0]
        
        if path == "/orders":
            content_length = int(self.headers.get("Content-Length", 0))
            body = {}
            if content_length > 0:
                body = json.loads(self.rfile.read(content_length).decode())
            
            self.send_json(201, {
                "status": 201,
                "message": "Order created",
                "pizza": body.get("pizza", "unknown"),
                "order_id": f"ORD-{int(time.time()) % 10000:04d}"

        elif path == "/login":

def make_request(method, path, body=None, token=None):
    """Make HTTP request and return status + response"""
    url = f"{BASE_URL}{path}"
    data = json.dumps(body).encode() if body else None
    req = urllib.request.Request(url, data=data, method=method)
    req.add_header("Content-Type", "application/json")
    if token:
        req.add_header("Authorization", f"Bearer {token}")
    
    try:
        with urllib.request.urlopen(req) as resp:
            headers = dict(resp.headers)
            result = json.loads(resp.read().decode())
            return {
                "status": resp.status,
                "body": result,
                "rate_limit_remaining": headers.get("X-RateLimit-Remaining", "N/A"),
                "rate_limit_limit": headers.get("X-RateLimit-Limit", "N/A"),
                "retry_after": headers.get("Retry-After", None)
            }
    except urllib.error.HTTPError as e:
        headers = dict(e.headers)
        try:
            result = json.loads(e.read().decode())
        except Exception:
            result = {"error": "Could not parse response"}
        return {
            "status": e.code,
            "body": result,
            "rate_limit_remaining": headers.get("X-RateLimit-Remaining", "N/A"),
            "rate_limit_limit": headers.get("X-RateLimit-Limit", "N/A"),
            "retry_after": headers.get("Retry-After", None)
        }
    except Exception as e:
        return {
            "status": 0,
            "body": {"error": str(e)},
            "rate_limit_remaining": "N/A",
            "rate_limit_limit": "N/A",
            "retry_after": None
        }


def print_header(title, description):
    print(f"\n{'=' * 70}")
    print(f"  🔴 {title}")
    print(f"  {description}")
    print(f"{'=' * 70}")


def print_result_summary(results, label=""):
    status_counts = Counter(r["status"] for r in results)
    total = len(results)
    allowed = status_counts.get(200, 0) + status_counts.get(201, 0)
    blocked = status_counts.get(429, 0)
    errors = total - allowed - blocked
    
    print(f"\n  📊 {label} RESULTS:")
    print(f"  {'─' * 45}")
    print(f"  Total Requests:  {total}")
    print(f"  ✅ Allowed:       {allowed} ({allowed/total*100:.1f}%)")
    print(f"  ❌ Rate Limited:  {blocked} ({blocked/total*100:.1f}%)")
    if errors > 0:
        print(f"  ⚠️  Errors:        {errors}")
    print(f"  {'─' * 45}")
    
    # Show last few rate limit headers
    if results:
        last = results[-1]
        print(f"  Last response:")
        print(f"    Status: {last['status']}")
        print(f"    Remaining: {last['rate_limit_remaining']}")
        print(f"    Retry-After: {last['retry_after']}")

def scenario_c_accidental_ddos():

        "SCENARIO C: THE ACCIDENTAL DDoS",
        "50 different users each send 5 requests to GET /menu simultaneously"
    )
    
    all_results = {}
    lock = threading.Lock()
    
    def user_session(user_id):
        """Simulate one user making 5 requests"""
        import base64
        # Create a unique token for each user
        header = base64.urlsafe_b64encode(
            json.dumps({"alg": "none"}).encode()
        ).decode().rstrip("=")
        payload = base64.urlsafe_b64encode(
            json.dumps({"user_id": f"user_{user_id}"}).encode()
        ).decode().rstrip("=")
        token = f"{header}.{payload}.sig"
        
        user_results = []
        for _ in range(5):
            result = make_request("GET", "/menu", token=token)
            user_results.append(result)
            time.sleep(0.05)  # Slight delay between requests
        
        with lock:
            allowed = sum(1 for r in user_results if r["status"] == 200)
            blocked = sum(1 for r in user_results if r["status"] == 429)
            all_results[f"user_{user_id}"] = {
                "allowed": allowed,
                "blocked": blocked
            }
    
    # Launch 50 users simultaneously
    threads = []
    start = time.time()
    
    for i in range(50):
        t = threading.Thread(target=user_session, args=(i,))
        threads.append(t)
        t.start()
    
    for t in threads:
        t.join()
    
    total_time = time.time() - start
    
    # Analyze results
    total_allowed = sum(r["allowed"] for r in all_results.values())
    total_blocked = sum(r["blocked"] for r in all_results.values())
    users_fully_served = sum(
        1 for r in all_results.values() if r["blocked"] == 0
    )
    users_partially_blocked = sum(
        1 for r in all_results.values() if r["blocked"] > 0
    )
    
    print(f"\n  👥 Total users: {len(all_results)}")
    print(f"  📨 Total requests: {total_allowed + total_blocked}")
    print(f"  ✅ Total allowed: {total_allowed}")
    print(f"  ❌ Total blocked: {total_blocked}")
    print(f"  👤 Users fully served (0 blocks): {users_fully_served}")

    print(f"  ⏱️  Total time: {total_time:.2f}s")
    
    print(f"\n  💡 INSIGHT: Each user has their OWN rate limit bucket")
    print(f"     User A being fast does NOT affect User B's limits")

# Uses a shared store (simulating Redis — Project 5 concept)

import time
import threading
import json
from collections import defaultdict


# ============================================================
# SIMULATED REDIS — Shared state across all servers
# In production: replace with actual Redis client
# Concepts from Project 5 (caching layer)
# ============================================================

class SimulatedRedis:
    """
    Thread-safe in-memory store that acts like Redis
    In production you would use: import redis; r = redis.Redis()
    Key operations we need:
      → GET/SET (read/write a key)
      → INCR (atomic increment)
      → EXPIRE (auto-delete after TTL)
      → EVAL (run Lua script atomically)
    """
    
    def __init__(self):
        self.store = {}
        self.expiry = {}
        self.lock = threading.Lock()
        
        # Background expiry cleaner
        self.cleaner = threading.Thread(target=self._clean_expired, daemon=True)
        self.cleaner.start()
    
    def _clean_expired(self):
        while True:
            time.sleep(1)
            now = time.time()
            with self.lock:
                expired = [
                    k for k, exp in self.expiry.items()
                    if exp <= now
                ]
                for k in expired:
                    self.store.pop(k, None)
                    self.expiry.pop(k, None)
    
    def _is_expired(self, key):
        if key in self.expiry and self.expiry[key] <= time.time():
            self.store.pop(key, None)
            self.expiry.pop(key, None)
            return True
        return False
    
    def get(self, key):
        with self.lock:
            if self._is_expired(key):
                return None
            return self.store.get(key)
    
    def set(self, key, value, ex=None):
        """Set key with optional expiry in seconds"""
        with self.lock:
            self.store[key] = value
            if ex:
                self.expiry[key] = time.time() + ex
    
    def incr(self, key):
        """Atomic increment — returns new value"""
        with self.lock:
            if self._is_expired(key):
                self.store[key] = 0
            current = self.store.get(key, 0)
            self.store[key] = current + 1
            return self.store[key]
    
    def expire(self, key, seconds):
        with self.lock:
            self.expiry[key] = time.time() + seconds
    
    def ttl(self, key):
        """Time-to-live remaining"""
        with self.lock:
            if key not in self.expiry:
                return -1
            remaining = self.expiry[key] - time.time()
            return max(0, remaining)
    
    def eval_token_bucket(self, key, max_tokens, refill_rate, now):
        """
        Atomic token bucket operation (simulates Redis Lua script)
        In real Redis, this would be a EVAL command with Lua script
        ensuring all steps happen atomically on the Redis server
        """
        with self.lock:
            if self._is_expired(key):
                self.store.pop(key, None)
            
            bucket = self.store.get(key)
            
            if bucket is None:
                bucket = {
                    "tokens": max_tokens,
                    "last_refill": now
                }
            
            # Refill
            elapsed = now - bucket["last_refill"]
            new_tokens = elapsed * refill_rate
            bucket["tokens"] = min(max_tokens, bucket["tokens"] + new_tokens)
            bucket["last_refill"] = now
            
            # Try consume
            if bucket["tokens"] >= 1:
                bucket["tokens"] -= 1
                self.store[key] = bucket
                self.expiry[key] = now + (max_tokens / refill_rate) + 60
                return {
                    "allowed": True,
                    "remaining": int(bucket["tokens"])
                }
            else:
                deficit = 1 - bucket["tokens"]
                retry_after = deficit / refill_rate
                self.store[key] = bucket
                return {
                    "allowed": False,
                    "remaining": 0,
                    "retry_after": round(retry_after, 2)
                }

# DISTRIBUTED RATE LIMITER — Uses shared Redis
# ============================================================

class DistributedRateLimiter:
    """
    Rate limiter that works across multiple server instances
    All instances share the same Redis store
    """
    
    def __init__(self, redis_client: SimulatedRedis):
        self.redis = redis_client
    
    def check_token_bucket(
        self,
        key: str,
        max_tokens: int = 10,
        refill_rate: float = 1.0
    ) -> dict:
        """
        Atomic token bucket check via Redis
        One network round trip, one atomic operation
        """
        now = time.time()
        redis_key = f"ratelimit:tb:{key}"
        
        result = self.redis.eval_token_bucket(
            redis_key, max_tokens, refill_rate, now
        )
        
        return {
            "allowed": result["allowed"],
            "limit": max_tokens,
            "remaining": result["remaining"],
            "retry_after": result.get("retry_after", 0)
        }
    
    def check_fixed_window(
        self,
        key: str,
        max_requests: int = 100,
        window_seconds: int = 60
    ) -> dict:
        """
        Fixed window using Redis INCR + EXPIRE
        Two Redis commands, but INCR is atomic
        """
        window = int(time.time() // window_seconds)
        redis_key = f"ratelimit:fw:{key}:{window}"
        
        count = self.redis.incr(redis_key)
        
        # Set expiry on first request in window
        if count == 1:
            self.redis.expire(redis_key, window_seconds)
        
        if count <= max_requests:
            return {
                "allowed": True,
                "limit": max_requests,
                "remaining": max_requests - count,
                "retry_after": 0
            }
        else:
            ttl = self.redis.ttl(redis_key)
            return {
                "allowed": False,
                "limit": max_requests,
                "remaining": 0,
                "retry_after": round(ttl, 1)
            }

class APIServer:
    """Simulates one server instance behind a load balancer"""
    
    def __init__(self, server_id: str, rate_limiter: DistributedRateLimiter):
        self.server_id = server_id
        self.rate_limiter = rate_limiter
        self.requests_handled = 0
        self.requests_limited = 0
    
    def handle_request(self, user_id: str, endpoint: str) -> dict:
        """Process one request through rate limiter then business logic"""
        key = f"{user_id}:{endpoint}"
        
        result = self.rate_limiter.check_token_bucket(
            key=key,
            max_tokens=10,
            refill_rate=2.0
        )
        
        if result["allowed"]:
            self.requests_handled += 1
            return {
                "server": self.server_id,
                "status": 200,
                "message": f"Processed by {self.server_id}",
                "remaining": result["remaining"]
            }
        else:
            self.requests_limited += 1
            return {
                "server": self.server_id,
                "status": 429,
                "message": "Rate limited",
                "retry_after": result["retry_after"]

import time
import sys
import statistics

# Re-defining minimally for standalone execution
from collections import defaultdict, deque
import threading


class FixedWindowBench:
    def __init__(self, limit, window):
        self.limit = limit
        self.window = window
        self.data = defaultdict(lambda: {"c": 0, "w": 0})
        self.lock = threading.Lock()
    
    def check(self, key):
        with self.lock:
            w = int(time.time() // self.window)
            d = self.data[key]
            if d["w"] != w:
                d["c"] = 0
                d["w"] = w
            if d["c"] < self.limit:
                d["c"] += 1
                return True
            return False


class SlidingLogBench:
    def __init__(self, limit, window):
        self.limit = limit
        self.window = window
        self.logs = defaultdict(deque)
        self.lock = threading.Lock()
    
    def check(self, key):
        with self.lock:
            now = time.time()
            log = self.logs[key]
            while log and log[0] <= now - self.window:
                log.popleft()
            if len(log) < self.limit:
                log.append(now)
                return True
            return False


class TokenBucketBench:
    def __init__(self, capacity, rate):
        self.capacity = capacity
        self.rate = rate
        self.data = defaultdict(lambda: {"t": capacity, "r": time.time()})
        self.lock = threading.Lock()
    
    def check(self, key):
        with self.lock:
            now = time.time()
            d = self.data[key]
            d["t"] = min(self.capacity, d["t"] + (now - d["r"]) * self.rate)
            d["r"] = now
            if d["t"] >= 1:
                d["t"] -= 1
                return True
            return False


class SlidingCounterBench:
    def __init__(self, limit, window):
        self.limit = limit
        self.window = window
        self.data = defaultdict(lambda: {"p": 0, "c": 0, "w": 0})
        self.lock = threading.Lock()
    
    def check(self, key):
        with self.lock:
            now = time.time()
            w = int(now // self.window)
            d = self.data[key]
            if d["w"] != w:
                d["p"] = d["c"] if d["w"] == w - 1 else 0
                d["c"] = 0
                d["w"] = w
            ratio = (now % self.window) / self.window
            weighted = d["p"] * (1 - ratio) + d["c"]
            if weighted < self.limit:
                d["c"] += 1
                return True
            return False


class LeakyBucketBench:
    def __init__(self, capacity, rate):
        self.capacity = capacity
        self.rate = rate
        self.data = defaultdict(lambda: {"w": 0.0, "t": time.time()})
        self.lock = threading.Lock()
    
    def check(self, key):
        with self.lock:
            now = time.time()
            d = self.data[key]
            d["w"] = max(0, d["w"] - (now - d["t"]) * self.rate)
            d["t"] = now
            if d["w"] < self.capacity:
                d["w"] += 1
                return True
            return False


def benchmark_algorithm(name, limiter_check, num_operations, num_keys):
    """Benchmark a rate limiter algorithm"""
    keys = [f"user_{i}" for i in range(num_keys)]
    
    # Warm up
    for key in keys[:10]:
        limiter_check(key)
    
    # Benchmark
    latencies = []
    allowed = 0
    
    start = time.time()
    for i in range(num_operations):
        key = keys[i % num_keys]
        op_start = time.perf_counter()
        result = limiter_check(key)
        op_end = time.perf_counter()
        latencies.append((op_end - op_start) * 1_000_000)  # microseconds
        if result:
            allowed += 1
    total_time = time.time() - start
    
    ops_per_sec = num_operations / total_time
    avg_latency = statistics.mean(latencies)
    p50 = statistics.median(latencies)
    p99 = sorted(latencies)[int(len(latencies) * 0.99)]
    max_latency = max(latencies)
    
    return {
        "name": name,
        "ops_per_sec": ops_per_sec,
        "avg_latency_us": avg_latency,
        "p50_us": p50,
        "p99_us": p99,
        "max_us": max_latency,
        "allowed": allowed,
        "total": num_operations
    }

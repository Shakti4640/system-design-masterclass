🎯 HYPOTHETICAL PROBLEM

Your pizza shop API is now FAST:
  → Rate limiter blocks abuse (Project 9)
  → Database indexes make queries instant (Project 10)
  → Cache avoids unnecessary DB hits (Project 5)

But there's a NEW problem that no amount of indexing or caching can fix:

SCENARIO A: The Slow Order Pipeline
  → Customer clicks "Place Order"
  → Your server SYNCHRONOUSLY does ALL of this:

    ┌─────────────────────────────────────────────────────┐
    │  1. Validate order data              →    2ms       │
    │  2. Check inventory in DB            →    8ms       │
    │  3. Process payment (Stripe API)     →  800ms  ⚠️  │
    │  4. Send confirmation email (SMTP)   →  500ms  ⚠️  │
    │  5. Send SMS notification (Twilio)   →  400ms  ⚠️  │
    │  6. Update analytics dashboard       →  200ms       │
    │  7. Notify kitchen display system    →  100ms       │
    │  8. Generate invoice PDF             →  300ms       │
    ├─────────────────────────────────────────────────────┤
    │  TOTAL WAIT TIME FOR CUSTOMER:       → 2,310ms     │
    └─────────────────────────────────────────────────────┘

  → Customer stares at a spinner for 2.3 SECONDS
  → But they only NEED steps 1-2 to know "order accepted"
  → Steps 3-8 can happen AFTER the customer gets their response


SCENARIO B: The Cascading Failure
  → Twilio (SMS service) goes down for 30 minutes
  → Every order now FAILS at step 5
  → Even though the order is valid and payment succeeded
  → 30 minutes of ZERO orders because one external service is down
  → Revenue loss: $50,000

  → The email server is slow today (3 second response)
  → Every order now takes 3 EXTRA seconds
  → Customer complaints spike
  → One slow dependency makes EVERYTHING slow


SCENARIO C: The Traffic Spike
  → Super Bowl Sunday: 10x normal order volume
  → Your API can handle 500 orders/second
  → But Stripe can only process 100 payments/second
  → Without buffering: 400 orders/second FAIL
  → With a queue: all 500 orders are ACCEPTED
    → Payments are processed at Stripe's pace
    → Customer sees "Order received, payment processing..."


SCENARIO D: The Lost Work
  → Server crashes mid-processing after payment but before email
  → Payment was charged but no confirmation sent
  → Customer calls: "I was charged but got no confirmation!"
  → Work is LOST because it was all in-memory

THE ANALOGY:
  → You write a letter and go to the POST OFFICE
  → You DROP the letter in the mailbox and GO HOME
  → You DON'T wait at the post office until the letter is delivered
  → The postal system GUARANTEES delivery (eventually)
  → If one mail carrier is sick → another takes over
  → If the destination is temporarily unreachable → letter waits

  → The MAILBOX is the MESSAGE QUEUE
  → YOU are the PRODUCER (creates messages)
  → The MAIL CARRIER is the CONSUMER (processes messages)
  → The LETTER is the MESSAGE
  → "Delivery guarantee" is the DELIVERY SEMANTIC

🔍 HOW TO APPROACH IT

Step 1: Identify SYNCHRONOUS vs ASYNCHRONOUS work
         → Synchronous: Customer MUST wait for the result
           (validation, inventory check)
         → Asynchronous: Can happen AFTER response
           (email, SMS, analytics, PDF generation)

Step 2: Identify the PRODUCER and CONSUMER
         → Producer: The API endpoint that receives the order
         → Consumer: Background workers that process each task
         → They communicate through a QUEUE

Step 3: Choose the DELIVERY GUARANTEE
         → At-most-once: Message might be lost (fire and forget)
         → At-least-once: Message is never lost but might be duplicated
         → Exactly-once: Message processed exactly once (hardest)

Step 4: Design the MESSAGE FORMAT
         → What data does the consumer need?
         → Include enough context to process independently
         → Include metadata: timestamp, retry count, source

Step 5: Handle FAILURES
         → Consumer crashes mid-processing → message returns to queue
         → Consumer can't process → send to Dead Letter Queue
         → External service down → retry with exponential backoff

Step 6: Scale the CONSUMERS
         → One consumer can't keep up → add more workers
         → Queue acts as a BUFFER between production rate and consumption rate
         → This is the key scalability insight

QUEUE TYPES WE WILL IMPLEMENT:

TYPE 1: SIMPLE FIFO QUEUE
══════════════════════════
  → First In, First Out
  → Messages processed in order
  → Single consumer or multiple consumers
  
  [msg1] → [msg2] → [msg3] → [msg4]
  ↑ oldest                    newest ↑
  (next to process)    (just arrived)


TYPE 2: PRIORITY QUEUE
══════════════════════
  → Messages have priority levels
  → Higher priority processed first
  → Payment > Email > Analytics
  
  [CRITICAL: payment] → [HIGH: email] → [LOW: analytics]


TYPE 3: DELAY QUEUE
═══════════════════
  → Messages become visible after a delay
  → "Send follow-up email 24 hours after order"
  → "Retry payment in 30 seconds"
  
  [msg visible at T+30s] → [msg visible at T+60s] → [msg visible at T+24h]


TYPE 4: DEAD LETTER QUEUE (DLQ)
═══════════════════════════════
  → Messages that FAILED processing after N retries
  → Moved to a special queue for human investigation
  → Prevents poison messages from blocking the queue forever
  
  Main Queue: [msg1] [msg2] [msg3]
                              │
                              │ Failed 3 times
                              ▼
  Dead Letter Queue: [msg3] ← investigate why this keeps failing


DELIVERY SEMANTICS:
═══════════════════

  AT-MOST-ONCE:
  ─────────────
    Producer → Queue → Consumer reads → Message DELETED immediately
    If consumer crashes AFTER reading but BEFORE processing → message LOST
    
    Use when: Analytics events, logging (loss is acceptable)
    
    Producer ──→ Queue ──→ Consumer
                  │ (delete)    │
                  └─────────────┘
                  Message gone, even if consumer crashes


  AT-LEAST-ONCE:
  ──────────────
    Producer → Queue → Consumer reads → Consumer processes → Consumer ACKs
    Message only deleted AFTER acknowledgment
    If consumer crashes before ACK → message REDELIVERED to another consumer
    
    Use when: Payments, orders, emails (MUST be processed)
    Risk: Message might be processed TWICE (consumer processed but crashed before ACK)
    
    Producer ──→ Queue ──→ Consumer
                  │            │
                  │      (process...)
                  │            │
                  │      (ACK ✅) → message deleted
                  │      (CRASH ❌) → message redelivered
    
    CRITICAL: Consumer must be IDEMPOTENT
    → Processing same message twice must produce same result
    → Charging $10 twice = BAD (not idempotent)
    → "Set order status to PAID" twice = OK (idempotent)


  EXACTLY-ONCE (Hardest):
  ───────────────────────
    → Technically impossible in distributed systems
    → Approximated by: at-least-once + idempotent consumer + deduplication
    → Consumer tracks processed message IDs
    → If same message arrives again → skip it
    
    → Project 37 (Payment) covers this in depth
    → For now: use at-least-once + make consumers idempotent


MESSAGE ACKNOWLEDGMENT FLOW:
════════════════════════════

  1. Producer publishes message → Queue stores it
  2. Consumer pulls message → Queue marks it "in-flight" (invisible to others)
  3. Consumer processes message
  4. Consumer sends ACK → Queue permanently deletes message
  
  OR:
  
  4b. Consumer sends NACK (negative ack) → Queue makes message visible again
  4c. Visibility timeout expires → Queue makes message visible again
  4d. Max retries exceeded → Queue moves to Dead Letter Queue

  Timeline:
  
  t=0:  [msg1, msg2, msg3]  ← all visible
  t=1:  Consumer A takes msg1 → [msg1(invisible), msg2, msg3]
  t=2:  Consumer B takes msg2 → [msg1(invisible), msg2(invisible), msg3]
  t=3:  Consumer A ACKs msg1 → [msg2(invisible), msg3]
  t=10: Consumer B crashes, timeout expires → [msg2(visible again), msg3]
  t=11: Consumer C takes msg2 → processes → ACKs → [msg3]

📌 WHAT COULD BE DONE

Option A: In-Process Queue (Thread-Based)
  → Queue lives inside the application process
  → Background threads consume messages
  → Simple, no external dependencies
  → But: Lost if process crashes, not shared across servers

Option B: Database-Backed Queue
  → Store messages in a database table (Project 3)
  → Consumers poll the table for new messages
  → Durable (survives crashes)
  → But: Polling is wasteful, DB becomes bottleneck

Option C: Dedicated Message Queue (Redis, RabbitMQ, SQS)
  → Purpose-built for queuing
  → High throughput, low latency
  → Shared across all servers
  → But: Additional infrastructure to manage

Option D: Event Streaming (Kafka, Kinesis)
  → Messages stored in an append-only log
  → Multiple consumers can read the same messages
  → Replay-able (go back in time)
  → But: More complex, overkill for simple task queues
  → Covered more in Project 29 (Event-Driven Architecture)

✅ WHAT SHOULD DO

1. ALWAYS make consumers IDEMPOTENT
   → Same message processed twice → same outcome
   → Use unique message IDs to detect duplicates
   → "Set status to PAID" ✅ (idempotent)
   → "Increment balance by $10" ❌ (not idempotent without dedup)

2. ALWAYS include enough context in the message
   → Bad:  {"action": "process_order", "order_id": 123}
     → Consumer must query DB to get order details
     → What if DB is down? Consumer can't process
   → Good: {"action": "process_order", "order_id": 123, 
             "user_email": "a@b.com", "amount": 25.99, "items": [...]}
     → Consumer has everything it needs

3. ALWAYS set visibility timeouts appropriately
   → Too short: message redelivered while still being processed → duplicates
   → Too long: if consumer crashes, message waits too long before retry
   → Rule: timeout = 2x expected processing time

4. ALWAYS implement Dead Letter Queues
   → Poison messages (can never be processed) block the queue forever
   → After N retries → move to DLQ
   → Alert operators to investigate DLQ
   → Common cause: corrupt data, schema changes, bugs

5. ALWAYS use exponential backoff for retries
   → Retry 1: wait 1 second
   → Retry 2: wait 2 seconds
   → Retry 3: wait 4 seconds
   → Retry 4: wait 8 seconds
   → Prevents hammering a down service
   → Add jitter (random delay) to prevent thundering herd

6. ALWAYS monitor queue depth
   → Queue growing → consumers can't keep up
   → Queue always empty → might have too many consumers (wasting resources)
   → Alert when queue depth exceeds threshold
   → This tells you when to scale consumers

7. ALWAYS design for message ordering awareness
   → Most queues do NOT guarantee strict ordering with multiple consumers
   → If order matters: use single consumer per partition
   → If order doesn't matter: use multiple consumers for throughput

❌ WHAT SHOULD NOT DO

1. NEVER put large payloads in messages
   → Bad: {"action": "process_image", "image_data": "<10MB base64>"}
   → Queue is for COORDINATION, not data transfer
   → Good: {"action": "process_image", "image_url": "s3://bucket/img.jpg"}
   → Store large data elsewhere, pass a REFERENCE in the message

2. NEVER rely on message ordering across multiple consumers
   → Consumer A takes msg1, Consumer B takes msg2
   → B might finish before A → msg2 processed before msg1
   → If strict order matters → single consumer or sequence numbers
   → Most business logic doesn't actually need strict ordering

3. NEVER use a queue when synchronous is actually required
   → User asks "What's my account balance?" → MUST respond immediately
   → Don't queue this → just query the database
   → Queues are for FIRE-AND-FORGET or DEFERRED work

4. NEVER ignore the Dead Letter Queue
   → "We set up DLQ but nobody monitors it"
   → Messages pile up → customers never get emails
   → DLQ must have alerting and periodic review

5. NEVER use infinite retries
   → Some messages CAN'T be processed (corrupt data, invalid state)
   → Infinite retries → consumer stuck in loop forever
   → Set max_retries (3-5 is typical)
   → After max → DLQ

6. NEVER share queues for unrelated tasks
   → Bad: One queue for payments AND analytics AND emails
   → Slow analytics consumer blocks payment processing
   → Good: Separate queue per task type
   → Each can scale independently

7. NEVER forget to handle consumer shutdown gracefully
   → Consumer receives SIGTERM (server shutting down)
   → Must finish current message and ACK before exiting
   → Or NACK so message returns to queue
   → Abrupt kill → message stays invisible until timeout

🚫 WHEN NOT TO USE

1. DO NOT use a queue when the user needs an IMMEDIATE response
   → "Show me my order status" → query DB directly
   → "What's the menu?" → serve from cache (Project 5)
   → Queues add LATENCY — response comes later, not now

2. DO NOT use a queue for simple request-response patterns
   → API calls that must return a result to the caller
   → Use direct HTTP/gRPC calls instead
   → Queue = "I'll get back to you later"

3. DO NOT use a queue when strong consistency is needed immediately
   → "Deduct $100 from account and show new balance"
   → User expects to see updated balance IMMEDIATELY
   → Queueing the deduction → stale balance shown
   → (Project 29/30 covers eventual consistency patterns)

4. DO NOT use a queue for tiny, fast operations
   → Logging a single line → just write to file
   → Incrementing a counter → just do it in-process
   → Queue overhead (serialize, network, deserialize) > actual work

5. DO NOT use message queues as a database
   → Queues are for TRANSIENT messages, not permanent storage
   → Once consumed, message is gone
   → If you need to re-read messages → use event streaming (Project 29)

PULL vs PUSH DELIVERY:
═════════════════════

  PULL (Polling):
  → Consumer asks: "Any messages for me?"
  → Queue responds: "Yes, here's one" or "No, try again later"
  → Consumer controls pace
  → Wastes resources if queue is often empty
  → Simple to implement

  ┌──────────┐  "Any msgs?"   ┌───────┐
  │ Consumer │───────────────→│ Queue │
  │          │←───────────────│       │
  └──────────┘  "Here's msg1" └───────┘

  PUSH (Subscription):
  → Consumer registers: "Notify me when messages arrive"
  → Queue pushes messages to consumer
  → No wasted polls
  → But: Consumer must handle back-pressure
  → What if consumer is slow? Messages pile up

  ┌──────────┐                ┌───────┐
  │ Consumer │←───────────────│ Queue │
  │          │  "New msg for  │       │
  └──────────┘   you!"        └───────┘

  LONG POLLING (Best of both):
  → Consumer asks: "Any messages?"
  → If none: Queue holds the connection open for N seconds
  → If message arrives during wait → immediately sent
  → No busy-loop polling, no push complexity
  → AWS SQS uses this approach

═══════════════════════════

  ┌──────────┐

BACKPRESSURE MECHANICS:
══════════════════════

  What happens when producers are faster than consumers?

  Production rate: 1000 msgs/sec
  Consumption rate: 200 msgs/sec
  Gap: 800 msgs/sec accumulating

  t=0:    Queue depth: 0
  t=10:   Queue depth: 8,000
  t=60:   Queue depth: 48,000
  t=600:  Queue depth: 480,000 ← memory pressure!

  Solutions:
  1. Scale consumers (add more workers)
  2. Queue max size → reject new messages when full
  3. Rate limit producers (Project 9 concepts)
  4. Circuit breaker (Project 21) → stop sending if queue is too deep

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Non-Idempotent Consumers
  → Payment consumer charges credit card
  → Crashes after charging but before ACK
  → Message redelivered → charges AGAIN
  → Customer charged twice
  → FIX: Use transaction ID as idempotency key
    → Before processing: "Have I seen this message ID before?"
    → If yes → skip, send ACK
    → If no → process, record message ID, send ACK

PITFALL 2: Poison Message (Infinite Retry Loop)
  → Message has corrupt data
  → Consumer tries to process → exception
  → Message returns to queue → consumer takes it again → exception
  → Infinite loop: same message failing forever
  → Blocks other messages from being processed
  → FIX: Max retry count + Dead Letter Queue

PITFALL 3: Message Ordering Assumption
  → Producer sends: order_created → order_paid → order_shipped
  → With multiple consumers: order_shipped might be processed
    BEFORE order_created
  → Consumer tries to mark "shipped" on an order that doesn't exist
  → FIX: Design consumers to handle out-of-order messages
  → FIX: Use sequence numbers and buffering if order matters

PITFALL 4: Queue as Database Anti-Pattern
  → Storing important data ONLY in the queue
  → Once consumed → data is gone forever
  → Need to re-process? Data is lost
  → FIX: Write to database FIRST, then publish to queue
    → Database is source of truth
    → Queue is for triggering async work

PITFALL 5: Consumer Slower Than Producer (Unbounded Growth)
  → Queue grows endlessly
  → Memory exhaustion → queue crashes → ALL messages lost
  → FIX: Set max queue size
  → FIX: Monitor queue depth with alerts
  → FIX: Auto-scale consumers based on queue depth

PITFALL 6: Visibility Timeout Too Short
  → Timeout: 30 seconds
  → Processing takes 45 seconds
  → At t=30: message becomes visible again
  → Consumer-2 picks it up → now TWO consumers processing same message

# MESSAGE — The fundamental unit

class MessageState(Enum):
    PENDING = "pending"         # Waiting to be consumed
    IN_FLIGHT = "in_flight"     # Being processed by a consumer
    COMPLETED = "completed"     # Successfully processed and ACKed
    FAILED = "failed"           # Moved to DLQ after max retries


class Message:
    """A single message in the queue"""
    
    def __init__(self, body: dict, priority: int = 5,
                 delay_seconds: float = 0, message_id: str = None):
        self.id = message_id or str(uuid.uuid4())[:12]
        self.body = body
        self.priority = priority          # 1=highest, 10=lowest
        self.state = MessageState.PENDING
        self.created_at = time.time()
        self.visible_at = time.time() + delay_seconds
        self.retry_count = 0
        self.max_retries = 3
        self.last_error = None
        self.processed_by = None
        self.processing_started_at = None
        self.completed_at = None
        self.visibility_timeout = 30      # seconds
    
    def is_visible(self) -> bool:
        """Can this message be picked up by a consumer?"""
        now = time.time()
        if self.state == MessageState.PENDING and now >= self.visible_at:
            return True
        if self.state == MessageState.IN_FLIGHT:
            # Check if visibility timeout has expired
            if now > self.processing_started_at + self.visibility_timeout:
                # Return to pending state
                self.state = MessageState.PENDING
                self.processed_by = None
                return True
        return False
    
    def to_dict(self) -> dict:
        return {
            "id": self.id,
            "body": self.body,
            "priority": self.priority,
            "state": self.state.value,
            "retry_count": self.retry_count,
            "created_at": self.created_at,
            "age_seconds": round(time.time() - self.created_at, 2)
        }
    
    def __lt__(self, other):
        """For priority queue comparison"""
        if self.priority != other.priority:
            return self.priority < other.priority
        return self.created_at < other.created_at

class MessageQueue:
    """
    Full-featured message queue with:
    → FIFO and Priority ordering
    → Visibility timeout
    → Retry with exponential backoff
    → Dead Letter Queue
    → At-least-once delivery
    → Metrics and monitoring
    """
    
    def __init__(self, name: str, max_size: int = 10000,
                 use_priority: bool = False,
                 visibility_timeout: int = 30,
                 max_retries: int = 3):
        self.name = name
        self.max_size = max_size
        self.use_priority = use_priority
        self.default_visibility_timeout = visibility_timeout
        self.default_max_retries = max_retries
        
        # Storage
        self.messages = {}               # id → Message
        self.pending_ids = deque()       # FIFO order
        self.priority_heap = []          # Priority order
        
        # Dead Letter Queue
        self.dlq = deque()
        self.dlq_messages = {}
        
        # Processed message IDs (for deduplication)
        self.processed_ids = set()
        
        # Thread safety
        self.lock = threading.Lock()
        
        # Metrics
        self.metrics = {
            "total_published": 0,
            "total_consumed": 0,
            "total_acked": 0,
            "total_nacked": 0,
            "total_retries": 0,
            "total_dlq": 0,
            "total_duplicates_skipped": 0,
            "max_queue_depth": 0,
            "total_processing_time_ms": 0
        }
    
    # ---- PUBLISH (Producer side) ----
    
    def publish(self, body: dict, priority: int = 5,
                delay_seconds: float = 0,
                message_id: str = None) -> Optional[str]:
        """
        Add a message to the queue
        Returns message_id if successful, None if queue is full
        """
        with self.lock:
            # Check queue capacity
            active_count = sum(
                1 for m in self.messages.values()
                if m.state in (MessageState.PENDING, MessageState.IN_FLIGHT)
            )
            
            if active_count >= self.max_size:
                return None  # Queue full — producer must handle this
            
            # Create message
            msg = Message(
                body=body,
                priority=priority,
                delay_seconds=delay_seconds,
                message_id=message_id
            )
            msg.visibility_timeout = self.default_visibility_timeout
            msg.max_retries = self.default_max_retries
            
            # Store message
            self.messages[msg.id] = msg
            
            if self.use_priority:
                heapq.heappush(self.priority_heap, msg)
            else:
                self.pending_ids.append(msg.id)
            
            # Update metrics
            self.metrics["total_published"] += 1
            current_depth = active_count + 1
            if current_depth > self.metrics["max_queue_depth"]:
                self.metrics["max_queue_depth"] = current_depth
            
            return msg.id
    
    # ---- CONSUME (Consumer side) ----
    
    def consume(self) -> Optional[Message]:
        """
        Get the next available message from the queue
        Message becomes invisible to other consumers
        Returns None if no messages available
        """
        with self.lock:
            msg = None
            
            if self.use_priority:
                # Find highest priority visible message
                visible = []
                while self.priority_heap:
                    candidate = heapq.heappop(self.priority_heap)
                    if candidate.id not in self.messages:
                        continue  # Already deleted
                    if candidate.state == MessageState.COMPLETED:
                        continue
                    if candidate.is_visible():
                        msg = candidate
                        # Put rest back
                        for v in visible:
                            heapq.heappush(self.priority_heap, v)
                        break
                    else:
                        visible.append(candidate)
                
                if msg is None:
                    # Put everything back
                    for v in visible:
                        heapq.heappush(self.priority_heap, v)
            else:
                # FIFO: Find first visible message
                checked = 0
                total = len(self.pending_ids)
                while checked < total:
                    msg_id = self.pending_ids.popleft()
                    checked += 1
                    
                    if msg_id not in self.messages:
                        continue
                    
                    candidate = self.messages[msg_id]
                    
                    if candidate.state == MessageState.COMPLETED:
                        continue
                    
                    if candidate.is_visible():
                        msg = candidate
                        break
                    else:
                        # Not visible yet, put back
                        self.pending_ids.append(msg_id)
            
            if msg is None:
                return None
            
            # Mark as in-flight
            msg.state = MessageState.IN_FLIGHT
            msg.processing_started_at = time.time()
            
            self.metrics["total_consumed"] += 1
            
            return msg
    
    # ---- ACKNOWLEDGE (Consumer confirms processing) ----
    
    def ack(self, message_id: str) -> bool:
        """
        Acknowledge successful processing
        Message is permanently removed from queue
        """
        with self.lock:
            if message_id not in self.messages:
                return False
            
            msg = self.messages[message_id]
            msg.state = MessageState.COMPLETED
            msg.completed_at = time.time()
            
            # Track processing time
            if msg.processing_started_at:
                proc_time = (msg.completed_at - msg.processing_started_at) * 1000
                self.metrics["total_processing_time_ms"] += proc_time
            
            # Add to processed set (for deduplication)
            self.processed_ids.add(message_id)
            
            # Clean up
            del self.messages[message_id]
            
            self.metrics["total_acked"] += 1
            return True
    
    # ---- NEGATIVE ACKNOWLEDGE (Consumer reports failure) ----
    
    def nack(self, message_id: str, error: str = "Processing failed") -> str:
        """
        Report failed processing
        Returns: 'retried', 'dlq', or 'not_found'
        """
        with self.lock:
            if message_id not in self.messages:
                return "not_found"
            
            msg = self.messages[message_id]
            msg.retry_count += 1
            msg.last_error = error
            
            self.metrics["total_nacked"] += 1
            
            if msg.retry_count > msg.max_retries:
                # Move to Dead Letter Queue
                msg.state = MessageState.FAILED
                self.dlq.append(msg)
                self.dlq_messages[msg.id] = msg
                del self.messages[msg.id]
                self.metrics["total_dlq"] += 1
                return "dlq"
            else:
                # Exponential backoff: 2^retry * 1 second
                backoff = min(2 ** msg.retry_count, 60)  # Cap at 60s
                msg.state = MessageState.PENDING
                msg.visible_at = time.time() + backoff
                msg.processed_by = None
                
                # Re-add to queue
                if self.use_priority:
                    heapq.heappush(self.priority_heap, msg)
                else:
                    self.pending_ids.append(msg.id)
                
                self.metrics["total_retries"] += 1
                return "retried"
    
    # ---- MONITORING ----
    
    def get_depth(self) -> dict:
        with self.lock:
            pending = sum(
                1 for m in self.messages.values()
                if m.state == MessageState.PENDING
            )
            in_flight = sum(
                1 for m in self.messages.values()
                if m.state == MessageState.IN_FLIGHT
            )
            return {
                "pending": pending,
                "in_flight": in_flight,
                "total_active": pending + in_flight,
                "dlq_depth": len(self.dlq)
            }
    
    def get_metrics(self) -> dict:
        with self.lock:
            depth = self.get_depth()
            avg_proc_time = 0
            if self.metrics["total_acked"] > 0:
                avg_proc_time = (self.metrics["total_processing_time_ms"] /
                                self.metrics["total_acked"])
            
            return {
                **self.metrics,
                "current_depth": depth,
                "avg_processing_time_ms": round(avg_proc_time, 2),
                "retry_rate": (
                    round(self.metrics["total_retries"] /
                          max(self.metrics["total_consumed"], 1) * 100, 1)
                ),
                "dlq_rate": (
                    round(self.metrics["total_dlq"] /
                          max(self.metrics["total_published"], 1) * 100, 1)
                )
            }
    
    def get_dlq_messages(self) -> list:
        with self.lock:
            return [msg.to_dict() for msg in self.dlq]

class Worker:
    """
    Background worker that consumes messages from a queue
    and processes them using a handler function
    """
    
    def __init__(self, worker_id: str, queue: MessageQueue,
                 handler: Callable, poll_interval: float = 0.1):
        self.worker_id = worker_id
        self.queue = queue
        self.handler = handler
        self.poll_interval = poll_interval
        self.running = False
        self.thread = None
        
        # Worker stats
        self.stats = {
            "messages_processed": 0,
            "messages_failed": 0,
            "total_processing_time_ms": 0
        }
    
    def start(self):
        self.running = True
        self.thread = threading.Thread(target=self._run, daemon=True)
        self.thread.start()
    
    def stop(self):
        self.running = False
        if self.thread:
            self.thread.join(timeout=5)
    
    def _run(self):
        while self.running:
            msg = self.queue.consume()
            
            if msg is None:
                time.sleep(self.poll_interval)
                continue
            
            start = time.time()
            
            try:
                # Process message using handler function
                self.handler(msg.body, msg.id, self.worker_id)
                
                # Success — acknowledge
                self.queue.ack(msg.id)
                
                elapsed = (time.time() - start) * 1000
                self.stats["messages_processed"] += 1
                self.stats["total_processing_time_ms"] += elapsed
                
            except Exception as e:
                # Failure — negative acknowledge
                elapsed = (time.time() - start) * 1000
                self.stats["messages_failed"] += 1
                
                result = self.queue.nack(msg.id, error=str(e))
                
                if result == "dlq":
                    print(f"    ☠️  [{self.worker_id}] Message {msg.id} → "
                          f"DEAD LETTER QUEUE (retries exhausted)")

class SynchronousOrderPipeline:

class LightQueue:
    """Minimal thread-safe queue for pipeline use"""
    
    def __init__(self, name):
        self.name = name
        self.messages = deque()
        self.lock = threading.Lock()
        self.processed = 0
        self.failed = 0
        self.processing_times = []
    
    def put(self, message):
        with self.lock:
            self.messages.append(message)
    
    def get(self):
        with self.lock:
            if self.messages:
                return self.messages.popleft()
            return None
    
    def size(self):
        with self.lock:
            return len(self.messages)
    
    def record_success(self, proc_time_ms):
        with self.lock:
            self.processed += 1
            self.processing_times.append(proc_time_ms)
    
    def record_failure(self):
        with self.lock:
            self.failed += 1

class AsyncOrderPipeline:
    """
    Customer waits ONLY for validation + inventory check
    Everything else happens asynchronously via queues
    Total customer wait = ~10ms instead of ~2300ms
    """
    
    def __init__(self, num_workers_per_queue=2):
        self.services = ExternalServices()
        
        # Create separate queues for each async task type
        self.queues = {
            "payment": LightQueue("payment"),
            "email": LightQueue("email"),
            "sms": LightQueue("sms"),
            "analytics": LightQueue("analytics"),
            "kitchen": LightQueue("kitchen"),
            "invoice": LightQueue("invoice"),
        }
        
        # Dead letter queue for failed messages
        self.dlq = LightQueue("dead_letter")
        
        # Processed message IDs (idempotency — Project 11 concept)
        self.processed_ids = set()
        self.processed_lock = threading.Lock()
        
        self.stats = {
            "orders_accepted": 0,
            "orders_rejected": 0,
            "sync_latency_ms": 0,
        }
        
        self.workers = []
        self.running = False
        self.num_workers = num_workers_per_queue
    
    def start_workers(self):
        """Start background workers for each queue"""
        self.running = True
        
        # Map queue names to handler functions
        handlers = {
            "payment": self._handle_payment,
            "email": self._handle_email,
            "sms": self._handle_sms,
            "analytics": self._handle_analytics,
            "kitchen": self._handle_kitchen,
            "invoice": self._handle_invoice,
        }
        
        for queue_name, handler in handlers.items():
            for i in range(self.num_workers):
                worker_id = f"{queue_name}_worker_{i}"
                t = threading.Thread(
                    target=self._worker_loop,
                    args=(worker_id, self.queues[queue_name], handler),
                    daemon=True
                )
                self.workers.append(t)
                t.start()
    
    def stop_workers(self):
        self.running = False
        for t in self.workers:
            t.join(timeout=2)
    
    def _worker_loop(self, worker_id, queue, handler):
        """Generic worker loop — pulls from queue, processes, handles errors"""
        while self.running:
            msg = queue.get()
            if msg is None:
                time.sleep(0.05)
                continue
            
            # Idempotency check
            dedup_key = f"{queue.name}:{msg['order_id']}"
            with self.processed_lock:
                if dedup_key in self.processed_ids:
                    continue  # Skip duplicate
            
            start = time.time()
            retry_count = msg.get("_retry_count", 0)
            max_retries = 3
            
            try:
                handler(msg)
                
                proc_time = (time.time() - start) * 1000
                queue.record_success(proc_time)
                
                with self.processed_lock:
                    self.processed_ids.add(dedup_key)
                    
            except Exception as e:
                retry_count += 1
                queue.record_failure()
                
                if retry_count <= max_retries:
                    # Retry with exponential backoff
                    msg["_retry_count"] = retry_count
                    backoff = min(2 ** retry_count * 0.1, 5)
                    time.sleep(backoff)
                    queue.put(msg)  # Re-queue
                else:
                    # Move to DLQ
                    msg["_error"] = str(e)
                    msg["_final_retry"] = retry_count
                    self.dlq.put(msg)
    
    # ---- Handler functions for each task type ----
    
    def _handle_payment(self, msg):
        self.services.process_payment(msg)
    
    def _handle_email(self, msg):
        self.services.send_email(msg)
    
    def _handle_sms(self, msg):
        self.services.send_sms(msg)
    
    def _handle_analytics(self, msg):
        self.services.update_analytics(msg)
    
    def _handle_kitchen(self, msg):
        self.services.notify_kitchen(msg)
    
    def _handle_invoice(self, msg):
        self.services.generate_invoice(msg)
    
    # ---- The main order placement (what the customer waits for) ----
    
    def place_order(self, order: dict) -> dict:
        """
        Process order asynchronously
        Customer only waits for validation + inventory
        Everything else is queued for background processing
        """
        start = time.time()
        
        try:
            # ---- SYNCHRONOUS PART (customer waits) ----
            
            # Step 1: Validate
            self.services.validate_order(order)
            
            # Step 2: Check inventory
            self.services.check_inventory(order)
            
            sync_latency = (time.time() - start) * 1000
            
            # ---- ASYNCHRONOUS PART (customer does NOT wait) ----
            
            # Generate unique order ID
            order_id = order.get("order_id", f"ORD-{uuid.uuid4().hex[:8]}")
            enriched = {**order, "order_id": order_id}
            
            # Publish to all task queues
            self.queues["payment"].put(enriched.copy())
            self.queues["email"].put(enriched.copy())
            self.queues["sms"].put(enriched.copy())
            self.queues["analytics"].put(enriched.copy())
            self.queues["kitchen"].put(enriched.copy())
            self.queues["invoice"].put(enriched.copy())
            
            total_ms = (time.time() - start) * 1000
            self.stats["orders_accepted"] += 1
            self.stats["sync_latency_ms"] += sync_latency
            
            # Customer gets IMMEDIATE response
            return {
                "status": "accepted",
                "order_id": order_id,
                "message": "Order accepted! Processing in background.",
                "customer_wait_ms": round(total_ms),
                "sync_steps": ["validate", "inventory"],
                "async_steps": ["payment", "email", "sms",
                               "analytics", "kitchen", "invoice"]
            }
            
        except Exception as e:
            total_ms = (time.time() - start) * 1000
            self.stats["orders_rejected"] += 1
            
            return {
                "status": "rejected",
                "error": str(e),
                "customer_wait_ms": round(total_ms)
            }
    
    def get_full_stats(self) -> dict:
        queue_stats = {}
        for name, q in self.queues.items():
            avg_time = 0
            if q.processing_times:
                avg_time = sum(q.processing_times) / len(q.processing_times)
            queue_stats[name] = {
                "pending": q.size(),
                "processed": q.processed,
                "failed": q.failed,
                "avg_processing_ms": round(avg_time, 1)
            }
        
        return {
            "orders": self.stats,
            "queues": queue_stats,
            "dlq_depth": self.dlq.size()
        }

# PROJECT 11C: QUEUE AS SHOCK ABSORBER
# Demonstrates how queues buffer traffic spikes
# Without queue: server overloads and drops requests
# With queue: requests buffered and processed at safe rate
# ============================================================

import time
import threading
import random
from collections import deque


class TrafficSimulator:
    """
    Simulates bursty traffic hitting a service
    Shows queue buffering vs direct processing
    """
    
    def __init__(self):
        self.lock = threading.Lock()
    
    def simulate_direct_processing(self, traffic_pattern: list,
                                   max_capacity: int = 50) -> dict:
        """
        WITHOUT queue: requests hit server directly
        Server can handle max_capacity requests/second
        Excess requests are DROPPED (503 Service Unavailable)
        """
        results = {
            "accepted": 0,
            "dropped": 0,
            "timeline": [],
            "total_seconds": len(traffic_pattern)
        }
        
        for second, requests_this_second in enumerate(traffic_pattern):
            if requests_this_second <= max_capacity:
                accepted = requests_this_second
                dropped = 0
            else:
                accepted = max_capacity
                dropped = requests_this_second - max_capacity
            
            results["accepted"] += accepted
            results["dropped"] += dropped
            results["timeline"].append({
                "second": second,
                "incoming": requests_this_second,
                "accepted": accepted,
                "dropped": dropped
            })
        
        return results
    
    def simulate_queue_buffered(self, traffic_pattern: list,
                                processing_rate: int = 50,
                                max_queue_size: int = 5000) -> dict:
        """
        WITH queue: requests go to queue, workers process at steady rate
        Queue absorbs bursts, workers drain at processing_rate/second
        """
        results = {
            "accepted": 0,
            "dropped": 0,
            "processed": 0,
            "timeline": [],
            "total_seconds": len(traffic_pattern),
            "max_queue_depth_reached": 0
        }
        
        queue_depth = 0
        
        for second, requests_this_second in enumerate(traffic_pattern):
            # Incoming requests enter queue
            space_available = max_queue_size - queue_depth
            accepted = min(requests_this_second, space_available)
            dropped = requests_this_second - accepted
            queue_depth += accepted
            
            # Workers process at steady rate
            processed_this_second = min(queue_depth, processing_rate)
            queue_depth -= processed_this_second
            
            results["accepted"] += accepted
            results["dropped"] += dropped
            results["processed"] += processed_this_second
            
            if queue_depth > results["max_queue_depth_reached"]:
                results["max_queue_depth_reached"] = queue_depth
            
            results["timeline"].append({
                "second": second,
                "incoming": requests_this_second,
                "accepted": accepted,
                "dropped": dropped,
                "processed": processed_this_second,
                "queue_depth": queue_depth
            })
        
        # Drain remaining queue
        drain_seconds = 0
        while queue_depth > 0:
            processed = min(queue_depth, processing_rate)
            queue_depth -= processed
            results["processed"] += processed
            drain_seconds += 1
        
        results["drain_seconds"] = drain_seconds
        
        return results

# One event triggers multiple independent processing pipelines
# Foundation for Project 24 (Notifications) and Project 29 (Events)

import time
import threading
import uuid
from collections import defaultdict, deque


class EventBus:
    """
    Simple publish-subscribe system
    One event published → multiple subscribers each get a copy
    Each subscriber has its own queue → independent processing
    """
    
    def __init__(self):
        self.subscriptions = defaultdict(list)  # event_type → [queues]
        self.lock = threading.Lock()
        self.event_log = []
        self.stats = {
            "events_published": 0,
            "total_deliveries": 0
        }
    
    def subscribe(self, event_type: str, subscriber_name: str) -> deque:
        """Subscribe to an event type, get a dedicated queue"""
        queue = deque()
        with self.lock:
            self.subscriptions[event_type].append({
                "name": subscriber_name,
                "queue": queue
            })
        return queue
    
    def publish(self, event_type: str, data: dict):
        """Publish event to ALL subscribers of this type"""
        with self.lock:
            event = {
                "event_id": str(uuid.uuid4())[:8],
                "event_type": event_type,
                "data": data,
                "published_at": time.time()
            }
            
            subscribers = self.subscriptions.get(event_type, [])
            for sub in subscribers:
                sub["queue"].append(event.copy())
            
            self.stats["events_published"] += 1
            self.stats["total_deliveries"] += len(subscribers)
            self.event_log.append({
                "event": event_type,
                "subscribers_notified": len(subscribers)
            })
            
            return event["event_id"]


class Subscriber:
    """A subscriber that processes events from its queue"""
    
    def __init__(self, name: str, queue: deque, handler,
                 processing_time: float = 0.1):
        self.name = name
        self.queue = queue
        self.handler = handler
        self.processing_time = processing_time
        self.processed = []
        self.running = False
        self.thread = None
    
    def start(self):
        self.running = True
        self.thread = threading.Thread(target=self._run, daemon=True)
        self.thread.start()
    
    def stop(self):
        self.running = False
        if self.thread:
            self.thread.join(timeout=2)
    
    def _run(self):
        while self.running:
            if self.queue:
                event = self.queue.popleft()
                start = time.time()
                
                try:
                    result = self.handler(event, self.name)
                    elapsed = (time.time() - start) * 1000
                    self.processed.append({
                        "event_id": event["event_id"],
                        "event_type": event["event_type"],
                        "processing_time_ms": round(elapsed, 1),
                        "status": "success"
                    })
                except Exception as e:
                    self.processed.append({
                        "event_id": event["event_id"],
                        "event_type": event["event_type"],
                        "status": "failed",
                        "error": str(e)
                    })
            else:
                time.sleep(0.05)

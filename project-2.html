PROJECT 2: THE RESTAURANT MENU — DESIGNING YOUR FIRST REST API

📋 PHASE 1 (Very Easy) — Foundation Thinking

🎯 HYPOTHETICAL PROBLEM

Remember our Pizza Shop from Project 1?
→ Customers sent custom JSON like {"action": "get_menu"}
→ Every developer had to READ our specific docs
→ No standard way to distinguish "reading" from "creating" from "deleting"
→ If 10 different companies built pizza APIs, all 10 would look different

NOW imagine you're building a RESTAURANT CHAIN:
→ 200 restaurants across the country
→ Each has a MENU with items
→ Each accepts ORDERS from customers
→ Each tracks ORDER STATUS
→ Mobile apps, websites, kiosks, delivery partners ALL need to talk to your system

THE CHAOS WITHOUT STANDARDS:
→ Restaurant A: {"action": "fetch_menu"}
→ Restaurant B: {"command": "show_items"}
→ Restaurant C: {"do": "list_food"}
→ Delivery partner must write DIFFERENT code for EACH restaurant

THE QUESTION:
→ Can we agree on ONE universal language for ALL of them?
→ Where the URL tells you WHAT resource you want
→ Where the HTTP method tells you WHAT action you want
→ Where the status code tells you WHAT happened
→ Where the body format is ALWAYS predictable

THIS is REST — Representational State Transfer
→ Not a library, not a framework
→ A SET OF RULES for designing APIs that everyone understands

🔍 HOW TO APPROACH IT

FROM PROJECT 1 WE KNOW:
→ Client sends Request, Server sends Response (don't repeat — see Project 1)
→ HTTP has methods, paths, headers, body, status codes
→ Server listens on a port

NOW WE ADD STRUCTURE:

Step 1: Identify your RESOURCES (nouns, not verbs)
         → Menu Items, Orders, Customers, Reviews
         → These become your URL paths

Step 2: Map ACTIONS to HTTP Methods (verbs)
         → GET    = Read (give me data)
         → POST   = Create (make something new)
         → PUT    = Replace entirely (overwrite)
         → PATCH  = Update partially (change one field)
         → DELETE = Remove

Step 3: Design URL PATTERNS (consistent, predictable)
         → /items        = collection of all items
         → /items/42     = one specific item (ID=42)
         → /orders       = collection of all orders
         → /orders/7     = one specific order (ID=7)

Step 4: Define RESPONSE FORMAT (always same shape)
         → Success: {"success": true, "data": {...}}
         → Error:   {"success": false, "error": {...}}

Step 5: Use correct STATUS CODES (standard meanings)
         → 200, 201, 204, 400, 401, 404, 409, 500

Step 6: Handle EDGE CASES consistently
         → Missing fields, invalid types, not found, duplicates

🛠️ HOW CAN WE SOLVE IT

THE REST MENTAL MODEL:

  Everything is a RESOURCE (a noun)
  Every action is an HTTP METHOD (a verb)
  Every outcome is a STATUS CODE (a result)

  /restaurants/5                    → Restaurant #5
  /restaurants/5/menu               → Menu of restaurant #5
  /restaurants/5/menu/items         → All menu items of restaurant #5

  /items?min_price=5&max_price=20   → Price range filter

📌 WHAT COULD BE DONE

Option A: Flat API (no nesting)
  → /items, /orders, /restaurants
  → Simple but loses relationship context
  → When: Small apps with few resources

Option B: Nested API (resources within resources)
  → /restaurants/5/items, /restaurants/5/orders
  → Shows ownership and hierarchy
  → When: Clear parent-child relationships exist

Option C: Hybrid (flat + nested where needed)
  → /items/42 for direct access by ID
  → /restaurants/5/items for scoped listing
  → When: Need both convenience and context
  → THIS IS WHAT MOST REAL SYSTEMS USE

Option D: Action-based endpoints (RPC-style)
  → /cancelOrder, /getMenu, /placeOrder
  → NOT RESTful — verbs in URLs
  → When: Very specific operations that don't map to CRUD
  → Example: /orders/99/cancel (acceptable hybrid)

For THIS project → Option C (Hybrid)
→ Most practical and industry-standard

✅ WHAT SHOULD DO

1. USE NOUNS for URLs, never verbs
   ✅ GET /orders          → "Get me the orders"
   ❌ GET /getOrders       → Verb in URL = bad REST
   ❌ POST /createOrder    → HTTP method already says "create"

2. USE PLURAL NOUNS for collections
   ✅ /items     (collection)
   ✅ /items/42  (single item from collection)
   ❌ /item/42   (inconsistent — is it one or many?)

3. USE PROPER STATUS CODES (not just 200 for everything)
   → 200 OK: Successful read/update
   → 201 Created: Successful creation (POST)
   → 204 No Content: Successful delete (nothing to return)
   → 400 Bad Request: Client sent invalid data
   → 401 Unauthorized: Not authenticated
   → 403 Forbidden: Authenticated but not allowed
   → 404 Not Found: Resource doesn't exist
   → 409 Conflict: Duplicate or conflicting state
   → 422 Unprocessable Entity: Valid JSON but invalid data
   → 500 Internal Server Error: Server broke

4. RETURN the created/updated resource in response
   → POST /orders → returns the new order WITH its generated ID
   → Client doesn't need a second request to see what was created

5. USE CONSISTENT response envelope
   → Every response follows the SAME shape
   → Client parsing logic works for ALL endpoints

6. SUPPORT FILTERING and PAGINATION from day one
   → Without pagination, listing 1 million items = crash
   → Query params: ?page=1&limit=20&sort=name

7. USE LOWERCASE and HYPHENS in URLs
   ✅ /menu-items
   ❌ /menuItems (camelCase)
   ❌ /Menu_Items (mixed)

❌ WHAT SHOULD NOT DO

1. NEVER put verbs in your URLs
   ❌ POST /createItem
   ❌ GET  /fetchAllOrders
   ❌ PUT  /updateItem/42
   → The HTTP METHOD is the verb

2. NEVER return 200 for errors
   ❌ 200 {"error": "Item not found"}
   → Client code checks status first, body second
   → 200 = success in every HTTP library in every language
   → Returning 200 with error body = confusing every consumer

3. NEVER nest more than 2-3 levels deep
   ❌ /restaurants/5/floors/2/tables/7/orders/99/items/3
   → Too deep = hard to use
   → Instead: /order-items/3 with restaurant_id as a filter

4. NEVER expose database IDs directly if sensitive
   → Auto-increment IDs reveal how many records exist
   → Sequential IDs are guessable (/users/1, /users/2...)
   → Use UUIDs for public-facing resources (covered more in Project 16)

5. NEVER ignore content negotiation
   → Client says Accept: application/json
   → Server should respect it
   → Don't return HTML when JSON is requested

6. NEVER return different response shapes for same endpoint
   → GET /items returns {"items": [...]} one day
   → GET /items returns {"data": [...]} next day
   → Breaking clients = losing trust

7. NEVER skip input validation
   → Price = -50? Reject it
   → Name = ""? Reject it
   → Quantity = "abc"? Reject it
   → Validate TYPES, RANGES, REQUIRED fields

HTTP METHOD DECISION TREE:

  "What am I doing?"
       │
       ├── Reading data? ──────────────→ GET
       │
       ├── Creating NEW resource? ─────→ POST
       │
       ├── Replacing ENTIRE resource? ─→ PUT
       │
       ├── Changing PART of resource? ─→ PATCH
       │
       └── Removing a resource? ───────→ DELETE


IDEMPOTENCY RULE (critical for reliability):
  ┌─────────┬────────────┬──────────────────────────────────────┐
  │ METHOD  │ IDEMPOTENT │ MEANING                              │
  ├─────────┼────────────┼──────────────────────────────────────┤
  │ GET     │ ✅ Yes     │ Calling 10 times = same result       │
  │ PUT     │ ✅ Yes     │ Replace with same data = same state  │
  │ DELETE  │ ✅ Yes     │ Delete same thing twice = still gone │
  │ PATCH   │ ❌ Maybe   │ Depends on implementation            │
  │ POST    │ ❌ No      │ Calling 10 times = 10 new records    │
  └─────────┴────────────┴──────────────────────────────────────┘

  WHY THIS MATTERS:
  → Network fails after POST, client retries
  → Without idempotency protection → duplicate order created
  → Solution: Idempotency keys (covered in Project 29, 37)

🚫 WHEN NOT TO USE

1. REST is NOT ideal when:
   → You need real-time push from server to client
      Example: Live chat → WebSocket is better (Project 31)
   
   → You need to fetch deeply nested related data in one call
      Example: "Give me user, their orders, each order's items,
               each item's reviews" → GraphQL handles this better
   
   → You need extreme performance between internal services
      Example: 50 microservices calling each other millions of
               times → gRPC with Protocol Buffers is faster
   
   → You need event streaming
      Example: Processing millions of events → Kafka/Message Queue
               (Project 11)

2. GET with body is NOT standard
   → Some HTTP libraries strip the body from GET requests
   → If you need complex query parameters → use query string
   → If query string gets too long → consider POST with search body
      (pragmatic exception to REST rules)

3. REST doesn't define:
   → Authentication method (that's separate — Project 8)
   → Rate limiting (separate — Project 9)
   → Versioning strategy (but you should have one)
   → Caching headers (but you should set them — Project 5)

PHASE 4: CONTENT NEGOTIATION (How format is decided)
═══════════════════════════════════════════════════

  Client says: Accept: application/json
  Server checks: "Can I produce JSON?" → Yes → returns JSON
  
  Client says: Accept: application/xml
  Server checks: "Can I produce XML?" → Maybe → returns XML
  
  Client says: Accept: text/html
  Server checks: "Can I produce HTML?" → No → returns 406 Not Acceptable

  Content-Type in REQUEST = "I'm SENDING you this format"
  Accept in REQUEST       = "I WANT you to respond in this format"
  Content-Type in RESPONSE = "I'm RESPONDING in this format"

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Inconsistent Naming
  → /menuItems (camelCase) + /order_status (snake_case) + /Users (PascalCase)
  → Pick ONE convention and STICK with it everywhere
  → Recommendation: lowercase with hyphens in URLs, snake_case in JSON bodies

PITFALL 2: Not Handling Missing Fields
  → Client sends: {"pizza": "margherita"} (missing quantity)
  → Server crashes with KeyError
  → FIX: Always check for required fields, return 400 with clear message

PITFALL 3: Returning Entire Database
  → GET /orders returns ALL 5 million orders
  → Client crashes, network overloaded, server memory explodes
  → FIX: Pagination from day one (default limit = 20)

PITFALL 4: No API Versioning
  → You change response format
  → All existing clients break
  → FIX: Version your API
     → URL versioning:  /api/v1/items (most common)
     → Header versioning: Accept: application/vnd.food.v1+json
     → Query param: /api/items?version=1

PITFALL 5: Ignoring PUT vs PATCH Difference
  → PUT means "replace the ENTIRE resource"
     → If you PUT with missing fields, those fields become null
  → PATCH means "update ONLY the fields I'm sending"
  → Mixing them up = data loss or confusion

PITFALL 6: Not Returning Created Resource
  → POST /items → returns just {"message": "created"}
  → Client doesn't know the ID or server-generated fields
  → FIX: Return the full resource + Location header

PITFALL 7: Swallowing Errors
  → Server catches exception, returns 200 {"status": "ok"}
  → Client thinks everything worked
  → Data is actually corrupted
  → FIX: Let errors surface as proper 4xx/5xx codes

PITFALL 8: N+1 Query Problem (Preview)
  → GET /orders returns 100 orders
  → For each order, client calls GET /orders/{id}/items
  → 100 + 1 = 101 HTTP requests for one page
  → FIX: Include related data or support ?include=items
  → (Database solution in Project 3, Project 10)

📈 SCALABILITY CONSIDERATIONS

FROM PROJECT 1 WE KNOW:
→ Single server has connection limits
→ Threading helps but has ceiling
(See Project 1 for details — not repeated here)

NEW CONSIDERATIONS FOR REST APIs:

1. PAYLOAD SIZE
   → Every response travels over the network
   → Returning 100 fields when client needs 3 = waste
   → Solutions:
     → Field selection: GET /items?fields=name,price
     → Pagination: GET /items?page=1&limit=20
     → Compression: Accept-Encoding: gzip (reduces payload 60-80%)

2. REQUEST VOLUME
   → Popular API: millions of GET /menu requests per day
   → Menu changes once a day
   → Why process it every time?
   → Solution: Caching (Project 5)
     → Cache-Control: max-age=3600 (cache for 1 hour)
     → ETag: "abc123" (revalidate only if changed)

3. STATELESSNESS = SCALABILITY
   → REST is STATELESS by design
   → Server doesn't remember previous requests
   → Each request carries ALL needed information
   → WHY this matters:
     → Any server can handle any request
     → Add more servers = handle more requests
     → Load balancer (Project 6) can distribute freely
   → If server stores session state → only THAT server can respond
     → This BREAKS horizontal scaling (Project 4)

4. RESPONSE TIME BUDGETS
   → Good API: < 100ms for simple reads
   → Acceptable: < 500ms for writes with validation
   → Bad: > 1 second for any single API call
   → Database queries are usually the bottleneck (Project 3, 10)

5. API RATE LIMITING (Preview for Project 9)
   → Without limits: one client can flood your server
   → Need: X requests per minute per client
   → Response: 429 Too Many Requests

🔍 INTERPRETABILITY CONSIDERATIONS

FROM PROJECT 1 WE KNOW:
→ Logging, status codes, and network tools help visibility
(See Project 1 — not repeated)

NEW FOR REST APIs:

1. SELF-DOCUMENTING URLs
   → GET /api/v1/restaurants/5/orders/99
   → Anyone reading this KNOWS: "get order 99 from restaurant 5"
   → No documentation needed to understand INTENT

2. STANDARDIZED ERROR RESPONSES
   → Always include:
     → Error code (machine-readable)
     → Error message (human-readable)
     → Field-level errors (for validation)
   
   {
     "success": false,
     "error": {
       "code": "VALIDATION_ERROR",
       "message": "Request validation failed",
       "details": [
         {"field": "price", "issue": "must be positive"},
         {"field": "name", "issue": "is required"}
       ]
     }
   }

3. REQUEST TRACING
   → Client sends: X-Request-ID: uuid-abc-123
   → Server echoes it back in response
   → If something fails, both sides can reference the same ID
   → Foundation for distributed tracing (Project 22)

4. API DOCUMENTATION
   → OpenAPI/Swagger specification
   → Auto-generated from code or written separately
   → Interactive docs: try endpoints from browser
   → Contract between frontend and backend teams

5. HATEOAS (Hypermedia as the Engine of Application State)
   → Response includes LINKS to related actions
   → Client doesn't hardcode URLs
   {
     "data": {"id": 99, "status": "preparing"},
     "links": {
       "self": "/api/orders/99",
       "cancel": "/api/orders/99",
       "restaurant": "/api/restaurants/5"
     }
   }
   → Rarely fully implemented but good to know

class DataStore:
    """Simple in-memory store simulating a database"""
    
    def __init__(self):
        self.menu_items = {}
        self.orders = {}
        self.item_counter = 0
        self.order_counter = 0
        self.lock = threading.Lock()
        self._seed_data()
    
    def _seed_data(self):
        """Pre-populate with sample menu items"""
        seed_items = [
            {"name": "Margherita Pizza", "category": "pizza",
             "price": 12.99, "available": True, "prep_time_mins": 15},
            {"name": "Pepperoni Pizza", "category": "pizza",
             "price": 14.99, "available": True, "prep_time_mins": 18},
            {"name": "Caesar Salad", "category": "salad",
             "price": 8.99, "available": True, "prep_time_mins": 5},
            {"name": "Garlic Bread", "category": "appetizer",
             "price": 5.99, "available": True, "prep_time_mins": 8},
            {"name": "Tiramisu", "category": "dessert",
             "price": 7.99, "available": False, "prep_time_mins": 0},
            {"name": "Veggie Burger", "category": "burger",
             "price": 11.99, "available": True, "prep_time_mins": 12},
            {"name": "Chicken Wings", "category": "appetizer",
             "price": 9.99, "available": True, "prep_time_mins": 20},
            {"name": "Pasta Carbonara", "category": "pasta",
             "price": 13.99, "available": True, "prep_time_mins": 16},
        ]
        for item_data in seed_items:
            self.create_item(item_data)
    
    def next_item_id(self):
        with self.lock:
            self.item_counter += 1
            return self.item_counter
    
    def next_order_id(self):
        with self.lock:
            self.order_counter += 1
            return self.order_counter
    
    def create_item(self, data):
        item_id = self.next_item_id()
        now = datetime.datetime.now().isoformat()
        item = {
            "id": item_id,
            "name": data["name"],
            "category": data.get("category", "uncategorized"),
            "price": data["price"],
            "available": data.get("available", True),
            "prep_time_mins": data.get("prep_time_mins", 10),
            "created_at": now,
            "updated_at": now
        }
        self.menu_items[item_id] = item
        return item
    
    def get_item(self, item_id):
        return self.menu_items.get(item_id)
    
    def list_items(self, filters=None, page=1, limit=10, sort_by="id",
                   sort_order="asc"):
        items = list(self.menu_items.values())
        
        # Apply filters
        if filters:
            if "category" in filters:
                items = [i for i in items
                         if i["category"] == filters["category"]]
            if "available" in filters:
                avail = filters["available"].lower() == "true"
                items = [i for i in items if i["available"] == avail]
            if "min_price" in filters:
                items = [i for i in items
                         if i["price"] >= float(filters["min_price"])]
            if "max_price" in filters:
                items = [i for i in items
                         if i["price"] <= float(filters["max_price"])]
            if "search" in filters:
                term = filters["search"].lower()
                items = [i for i in items
                         if term in i["name"].lower()]
        
        # Sort
        reverse = sort_order.lower() == "desc"
        if sort_by in ("id", "name", "price", "category"):
            items.sort(key=lambda x: x.get(sort_by, ""), reverse=reverse)
        
        # Pagination
        total = len(items)
        start = (page - 1) * limit
        end = start + limit
        paginated = items[start:end]
        
        return {
            "items": paginated,
            "pagination": {
                "total": total,
                "page": page,
                "limit": limit,
                "total_pages": max(1, -(-total // limit)),  # Ceiling div
                "has_next": end < total,
                "has_prev": page > 1
            }
        }
    
    def update_item(self, item_id, data, partial=True):
        item = self.menu_items.get(item_id)
        if not item:
            return None
        
        if partial:
            # PATCH: update only provided fields
            for key, value in data.items():
                if key in ("name", "category", "price", "available",
                           "prep_time_mins"):
                    item[key] = value
        else:
            # PUT: replace entire resource (keep id and timestamps)
            item["name"] = data.get("name", "")
            item["category"] = data.get("category", "uncategorized")
            item["price"] = data.get("price", 0)
            item["available"] = data.get("available", True)
            item["prep_time_mins"] = data.get("prep_time_mins", 10)
        
        item["updated_at"] = datetime.datetime.now().isoformat()
        return item
    
    def delete_item(self, item_id):
        if item_id in self.menu_items:
            del self.menu_items[item_id]
            return True
        return False
    
    def create_order(self, data):
        order_id = self.next_order_id()
        now = datetime.datetime.now().isoformat()
        
        # Calculate total
        total = 0
        order_items = []
        for entry in data.get("items", []):
            item = self.menu_items.get(entry["item_id"])
            if item:
                line_total = item["price"] * entry.get("quantity", 1)
                total += line_total
                order_items.append({
                    "item_id": item["id"],
                    "name": item["name"],
                    "quantity": entry.get("quantity", 1),
                    "unit_price": item["price"],
                    "line_total": round(line_total, 2)
                })
        
        order = {
            "id": order_id,
            "customer_name": data.get("customer_name", "Anonymous"),
            "items": order_items,
            "total_price": round(total, 2),
            "status": "received",
            "notes": data.get("notes", ""),
            "created_at": now,
            "updated_at": now
        }
        self.orders[order_id] = order
        return order
    
    def get_order(self, order_id):
        return self.orders.get(order_id)
    
    def list_orders(self, filters=None, page=1, limit=10):
        orders = list(self.orders.values())
        
        if filters:
            if "status" in filters:
                orders = [o for o in orders
                          if o["status"] == filters["status"]]
        
        total = len(orders)
        start = (page - 1) * limit
        end = start + limit
        
        return {
            "orders": orders[start:end],
            "pagination": {
                "total": total,
                "page": page,
                "limit": limit,
                "total_pages": max(1, -(-total // limit)),
                "has_next": end < total,
                "has_prev": page > 1
            }
        }
    
    def update_order_status(self, order_id, new_status):
        order = self.orders.get(order_id)
        if not order:
            return None
        
        valid_transitions = {
            "received": ["preparing", "cancelled"],
            "preparing": ["ready", "cancelled"],
            "ready": ["delivered"],
            "delivered": [],
            "cancelled": []
        }
        
        current = order["status"]
        if new_status not in valid_transitions.get(current, []):
            return {
                "error": True,
                "message": f"Cannot transition from '{current}' to "
                           f"'{new_status}'",
                "valid_transitions": valid_transitions.get(current, [])
            }
        
        order["status"] = new_status
        order["updated_at"] = datetime.datetime.now().isoformat()
        return order

class Validator:
    """Centralized input validation"""
    
    @staticmethod
    def validate_menu_item(data, partial=False):
        errors = []
        
        if not partial or "name" in data:
            name = data.get("name", "")
            if not name or not isinstance(name, str):
                errors.append({
                    "field": "name",
                    "issue": "is required and must be a non-empty string"
                })
            elif len(name) > 100:
                errors.append({
                    "field": "name",
                    "issue": "must be 100 characters or fewer"
                })
        
        if not partial or "price" in data:
            price = data.get("price")
            if price is None:
                if not partial:
                    errors.append({
                        "field": "price",
                        "issue": "is required"
                    })
            elif not isinstance(price, (int, float)) or price < 0:
                errors.append({
                    "field": "price",
                    "issue": "must be a non-negative number"
                })
            elif price > 9999.99:
                errors.append({
                    "field": "price",
                    "issue": "must be less than 10000"
                })
        
        if "category" in data:
            valid_categories = [
                "pizza", "burger", "salad", "appetizer",
                "dessert", "pasta", "beverage", "uncategorized"
            ]
            if data["category"] not in valid_categories:
                errors.append({
                    "field": "category",
                    "issue": f"must be one of: {valid_categories}"
                })
        
        if "available" in data:
            if not isinstance(data["available"], bool):
                errors.append({
                    "field": "available",
                    "issue": "must be true or false"
                })
        
        return errors
    
    @staticmethod
    def validate_order(data):
        errors = []
        
        items = data.get("items")
        if not items or not isinstance(items, list):
            errors.append({
                "field": "items",
                "issue": "is required and must be a non-empty list"
            })
            return errors
        
        if len(items) > 50:
            errors.append({
                "field": "items",
                "issue": "cannot have more than 50 items per order"
            })
            return errors
        
        for idx, entry in enumerate(items):
            item_id = entry.get("item_id")
            if not item_id or not isinstance(item_id, int):
                errors.append({
                    "field": f"items[{idx}].item_id",
                    "issue": "is required and must be an integer"
                })
                continue
            
            menu_item = db.get_item(item_id)
            if not menu_item:
                errors.append({
                    "field": f"items[{idx}].item_id",
                    "issue": f"item {item_id} does not exist"
                })
            elif not menu_item["available"]:
                errors.append({
                    "field": f"items[{idx}].item_id",
                    "issue": f"'{menu_item['name']}' is currently unavailable"
                })
            
            qty = entry.get("quantity", 1)
            if not isinstance(qty, int) or qty < 1 or qty > 100:
                errors.append({
                    "field": f"items[{idx}].quantity",
                    "issue": "must be an integer between 1 and 100"
                })
        
        return errors

class Router:
    """
    Pattern-based URL router
    Maps METHOD + URL pattern → handler function
    Extracts path parameters like {id}
    """
    
    def __init__(self):
        self.routes = []
    
    def add_route(self, method, pattern, handler):
        # Convert /items/{id} to regex /items/(?P<id>[^/]+)
        regex_pattern = re.sub(
            r'\{(\w+)\}',
            r'(?P<\1>[^/]+)',
            pattern
        )
        regex_pattern = f'^{regex_pattern}$'
        self.routes.append({
            "method": method.upper(),
            "pattern": pattern,
            "regex": re.compile(regex_pattern),
            "handler": handler
        })
    
    def resolve(self, method, path):
        for route in self.routes:
            if route["method"] != method.upper():
                continue
            match = route["regex"].match(path)
            if match:
                params = match.groupdict()
                # Convert numeric params to int
                for key, value in params.items():
                    if value.isdigit():
                        params[key] = int(value)
                return route["handler"], params
        
        return None, {}

class RestaurantAPIHandler(BaseHTTPRequestHandler):
    
    # Class-level router (shared across all requests)
    router = Router()
    
    # ---- Response Helpers ----
    
    def send_json(self, status_code, data):
        self.send_response(status_code)
        self.send_header('Content-Type', 'application/json; charset=utf-8')
        self.send_header('Access-Control-Allow-Origin', '*')
        self.send_header('Access-Control-Allow-Methods',
                         'GET, POST, PUT, PATCH, DELETE, OPTIONS')
        self.send_header('Access-Control-Allow-Headers',
                         'Content-Type, Authorization, X-Request-ID')
        
        # Echo request ID for tracing
        request_id = self.headers.get('X-Request-ID', str(uuid.uuid4()))
        self.send_header('X-Request-ID', request_id)
        
        body = json.dumps(data, indent=2)
        self.send_header('Content-Length', str(len(body.encode('utf-8'))))
        self.end_headers()
        self.wfile.write(body.encode('utf-8'))
    
    def send_success(self, data, status_code=200, meta=None):
        response = {"success": True, "data": data}
        if meta:
            response["meta"] = meta
        self.send_json(status_code, response)
    
    def send_error(self, status_code, code, message, details=None):
        response = {
            "success": False,
            "error": {
                "code": code,
                "message": message
            }
        }
        if details:
            response["error"]["details"] = details
        self.send_json(status_code, response)
    
    def read_body(self):
        content_length = int(self.headers.get('Content-Length', 0))
        if content_length > 0:
            body = self.rfile.read(content_length)
            try:
                return json.loads(body.decode('utf-8')), None
            except json.JSONDecodeError as e:
                return None, str(e)
        return {}, None
    
    def get_query_params(self):
        parsed = urllib.parse.urlparse(self.path)
        return dict(urllib.parse.parse_qsl(parsed.query))
    
    def get_clean_path(self):
        parsed = urllib.parse.urlparse(self.path)
        return parsed.path.rstrip('/')
    
    # ---- Request Routing ----
    
    def route_request(self, method):
        path = self.get_clean_path()
        timestamp = datetime.datetime.now().strftime('%Y-%m-%d %H:%M:%S')
        request_id = self.headers.get('X-Request-ID', 'none')
        print(f"[{timestamp}] {method:6s} {self.path} "
              f"[from: {self.client_address[0]}] "
              f"[req-id: {request_id}]")
        
        handler, params = self.router.resolve(method, path)
        
        if handler:
            handler(self, **params)
        else:
            self.send_error(404, "NOT_FOUND",
                            f"Endpoint {method} {path} not found",
                            {
                                "hint": "Check /api/v1 for available endpoints",
                                "docs": "/api/v1"
                            })
    
    def do_GET(self):
        self.route_request("GET")
    
    def do_POST(self):
        self.route_request("POST")
    
    def do_PUT(self):
        self.route_request("PUT")
    
    def do_PATCH(self):
        self.route_request("PATCH")
    
    def do_DELETE(self):
        self.route_request("DELETE")
    
    def do_OPTIONS(self):
        # CORS preflight
        self.send_json(200, {"methods": ["GET","POST","PUT","PATCH","DELETE"]})
    
    def log_message(self, format, *args):
        pass  # Suppress default logging

    for order in db.orders.values():
        if order["status"] in ("received", "preparing"):

                        f"#{order['id']}",

    db.delete_item(id)

    handler.send_response(204)
    handler.send_header('Content-Length', '0')
    handler.end_headers()

        req = urllib.request.Request(url, data=data, method=method)
        req.add_header('Content-Type', 'application/json')
        req.add_header('Accept', 'application/json')
        req.add_header('X-Request-ID', str(uuid.uuid4())[:8])
        
        try:
            with urllib.request.urlopen(req) as resp:
                status = resp.status
                if resp.headers.get('Content-Length', '0') != '0':
                    result = json.loads(resp.read().decode('utf-8'))
                else:
                    result = {"_no_content": True}
        except urllib.error.HTTPError as e:
            status = e.code
            result = json.loads(e.read().decode('utf-8'))
        
        status_icon = "✅" if status < 400 else "⚠️" if status < 500 else "❌"
        print(f"  {status_icon} [{status}] {method:6s} {path}")
        if body:
            print(f"         Body: {json.dumps(body)}")
        
        if result.get("success") is False:
            err = result.get("error", {})
            print(f"         Error: {err.get('code')}: {err.get('message')}")
            if err.get("details"):
                for d in err["details"][:3]:
                    print(f"           → {d['field']}: {d['issue']}")
        elif result.get("_no_content"):
            print(f"         (No Content — Successful Delete)")
        else:
            data_val = result.get("data")
            if isinstance(data_val, list):
                print(f"         Returned: {len(data_val)} items")
                meta = result.get("meta")
                if meta:
                    print(f"         Page {meta['page']}/{meta['total_pages']}"
                          f" (total: {meta['total']})")
            elif isinstance(data_val, dict) and "id" in data_val:
                name_field = data_val.get('name',
                              data_val.get('customer_name', ''))
                print(f"         ID: {data_val['id']} | {name_field}")
                if "status" in data_val:
                    print(f"         Status: {data_val['status']}")
                if "total_price" in data_val:
                    print(f"         Total: ${data_val['total_price']}")
            elif isinstance(data_val, dict):
                for key in list(data_val.keys())[:4]:
                    print(f"         {key}: {data_val[key]}")
        
        print()
        return result

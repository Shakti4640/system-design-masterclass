🎯 HYPOTHETICAL PROBLEM

You are building an e-commerce platform like Amazon or Shopify

The situation:
→ 50,000 users are browsing a flash sale page
→ There are exactly 100 units of a hot product (limited sneakers)
→ At 12:00:00 PM sharp, the sale goes live
→ Within the first 3 seconds, 8,000 users click "BUY NOW"
→ Each user expects INSTANT confirmation or rejection
→ Payment takes 5-30 seconds to process externally

WHAT GOES WRONG without proper design:

  Scenario A — OVERSELLING:
    → User A reads stock = 1
    → User B reads stock = 1
    → Both proceed to checkout
    → Both payments succeed
    → Stock is now -1 ← IMPOSSIBLE in real world
    → You owe someone a product you don't have
    → Legal liability + refund + reputation damage

  Scenario B — UNDERSELLING:
    → You lock the ENTIRE inventory table for safety
    → Only 1 user can buy at a time
    → 7,999 users wait in a queue
    → Most users timeout or leave
    → You sold 30 units instead of 100
    → Lost revenue = catastrophic

  Scenario C — PHANTOM INVENTORY:
    → User adds item to cart
    → Holds it for 30 minutes but never pays
    → 50 users "hold" the last 50 items
    → Real buyers see "OUT OF STOCK"
    → None of the holders ever pay
    → You sold 50 instead of 100

  Scenario D — PAYMENT LIMBO:
    → User clicks buy, stock decremented
    → Payment gateway is down
    → Stock is gone but no money received
    → User refreshes, tries again → stock now 0
    → Nobody got the product, nobody paid

THE REAL CHALLENGE:
→ How do you sell EXACTLY 100 units
→ To the first 100 LEGITIMATE buyers
→ Without overselling, underselling, or locking everyone out
→ While payments are slow and unreliable
→ While 8,000 concurrent requests hit in 3 seconds

🔍 HOW TO APPROACH IT

Decompose into 5 DISTINCT sub-problems:

SUB-PROBLEM 1: INVENTORY ACCURACY
  → How to decrement stock without going negative
  → How to handle concurrent reads + writes
  → This is a CONSISTENCY problem

SUB-PROBLEM 2: ORDER LIFECYCLE
  → Order has states: Created → Reserved → Paid → Shipped → Delivered
  → Each state transition has rules
  → This is a STATE MACHINE problem

SUB-PROBLEM 3: TEMPORARY RESERVATION
  → "Hold" stock for a user while they pay
  → Release it if they don't pay within X minutes
  → This is the RESERVATION PATTERN

SUB-PROBLEM 4: PAYMENT COORDINATION
  → Payment is external and unreliable
  → Must handle success, failure, timeout
  → Already solved via Saga in Project 37 — we REUSE that here

SUB-PROBLEM 5: FLASH SALE THUNDERING HERD
  → 8,000 requests in 3 seconds
  → Database can't handle 8,000 write transactions/sec
  → Need request buffering, pre-validation, early rejection
  → This combines Project 5 (Cache), Project 9 (Rate Limit),

🛠️ HOW CAN WE SOLVE IT

SOLUTION: THREE-PHASE COMMIT FOR EACH ORDER

PHASE 1 — RESERVE (Optimistic → Atomic Decrement)
  ┌────────┐         ┌──────────┐         ┌──────────┐
  │ Client │ ──BUY─→ │  Order   │ ──RSV─→ │Inventory │
  │        │         │  Service │         │ Service  │
  │        │ ←─ACK── │          │ ←─OK─── │          │
  └────────┘         └──────────┘         └──────────┘
  
  Inventory does:
    UPDATE products 
    SET stock = stock - 1 
    WHERE id = X AND stock > 0
  
  If affected_rows = 1 → reservation SUCCESS
  If affected_rows = 0 → OUT OF STOCK (instant rejection)
  
  NO LOCKS NEEDED — the WHERE clause IS the lock
  This is OPTIMISTIC CONCURRENCY at the database level

PHASE 2 — PAY (Saga Pattern from Project 37)
  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │  Order   │ ──PAY─→ │ Payment  │ ──API─→ │ External │
  │  Service │         │ Service  │         │ Gateway  │
  │          │ ←─RES── │          │ ←─RES── │          │
  └──────────┘         └──────────┘         └──────────┘
  
  If payment SUCCESS → Order moves to CONFIRMED
  If payment FAILS   → COMPENSATE: restore stock (+1)
  If payment TIMEOUT → Wait, retry, then compensate

PHASE 3 — CONFIRM or ROLLBACK
  SUCCESS PATH:
    Order → CONFIRMED → triggers fulfillment pipeline
  
  FAILURE PATH:
    Order → CANCELLED → stock restored → user notified
  
  TIMEOUT PATH:
    Reservation expires after T minutes
    Background job sweeps expired reservations
    Stock restored automatically

THE RESERVATION TIMEOUT IS CRITICAL:
  ┌─────────────────────────────────────────────────┐
  │  Time=0:00  User clicks BUY                     │
  │  Time=0:01  Stock decremented (reserved)         │
  │  Time=0:02  Payment initiated                    │
  │  ...                                             │
  │  Time=10:00 Payment still pending                │
  │  Time=15:00 RESERVATION EXPIRES                  │
  │  Time=15:01 Stock automatically restored         │
  │  Time=15:02 Order marked EXPIRED                 │
  │  Time=15:03 If payment comes through AFTER this  │
  │             → REFUND (compensation)              │
  └─────────────────────────────────────────────────┘

📌 WHAT COULD BE DONE

APPROACH A: PESSIMISTIC LOCKING (SELECT FOR UPDATE)
  → Lock the row when reading stock
  → Other transactions wait
  → Guarantees consistency
  → TERRIBLE for flash sales (serializes all requests)
  → Throughput: ~100-500 orders/sec

APPROACH B: OPTIMISTIC LOCKING (Version Column)
  → Read stock + version number
  → On update: WHERE version = read_version
  → If version changed → retry
  → Better than pessimistic, but high retry rate under contention
  → Throughput: ~500-2000 orders/sec

APPROACH C: ATOMIC DECREMENT (Our chosen approach)
  → UPDATE SET stock = stock - 1 WHERE stock > 0
  → Single atomic SQL statement
  → Database handles concurrency internally
  → No explicit locks, no version checks
  → Throughput: ~2000-10,000 orders/sec

APPROACH D: REDIS PRE-DECREMENT + DB SYNC
  → Stock count cached in Redis
  → DECR command is atomic and ~100,000 ops/sec
  → If Redis returns >= 0 → proceed to DB
  → If Redis returns < 0  → instant rejection
  → Filters 7,900 of 8,000 requests BEFORE hitting DB
  → Throughput: ~50,000+ orders/sec at rejection layer

APPROACH E: QUEUE-BASED SERIALIZATION
  → All buy requests go to a queue
  → Single consumer processes sequentially
  → Perfect consistency, but added latency
  → Good for ultra-high-value items (concert tickets)

OUR DESIGN: APPROACH D + APPROACH C combined
  → Redis as fast pre-filter (Project 5, 23)
  → Atomic DB decrement as source of truth
  → Saga for payment coordination (Project 37)
  → Reservation timeout for stuck orders

✅ WHAT SHOULD DO

1. USE ATOMIC OPERATIONS FOR STOCK DECREMENT
   → Never do: read stock → check → update (race condition)
   → Always do: UPDATE ... SET stock = stock - qty WHERE stock >= qty
   → The database is your concurrency control

2. IMPLEMENT RESERVATION WITH EXPIRY
   → Every stock hold has a TTL
   → Background sweeper restores expired holds
   → Never hold stock indefinitely

3. USE IDEMPOTENCY KEYS ON EVERY ORDER
   → Project 29 introduced idempotency
   → Client sends unique key with each purchase attempt
   → If same key comes twice → return existing result
   → Prevents double-ordering on retry/refresh

4. SEPARATE READ AND WRITE PATHS
   → Project 30 CQRS pattern
   → Reads (browsing, stock display): hit read replicas/cache
   → Writes (buy, cancel): hit primary DB
   → Stock display can be EVENTUALLY CONSISTENT (show "Low Stock" not exact count)

5. IMPLEMENT ORDER STATE MACHINE WITH STRICT TRANSITIONS
   → CREATED → RESERVED → PAYMENT_PENDING → CONFIRMED → SHIPPED → DELIVERED
   → Only valid transitions allowed
   → Every transition logged with timestamp
   → No skipping states

6. PRE-VALIDATE BEFORE EXPENSIVE OPERATIONS
   → Check cache for stock > 0 before hitting DB
   → Validate user, address, payment method before reservation
   → Reject fast, confirm slow

7. USE COMPENSATING TRANSACTIONS FOR FAILURES
   → Payment fails → restore stock (Project 37 Saga)
   → Never leave the system in an inconsistent state

❌ WHAT SHOULD NOT DO

1. NEVER READ-THEN-WRITE WITHOUT ATOMIC GUARANTEE
   → stock = SELECT stock FROM products WHERE id = X
   → IF stock > 0: UPDATE products SET stock = stock - 1
   → Between SELECT and UPDATE, 100 other requests sneak in
   → Result: stock goes to -99

2. NEVER LOCK ENTIRE TABLES
   → "LOCK TABLE products" during flash sale = system freeze
   → Lock only the specific row, or better: use atomic ops

3. NEVER DECREMENT STOCK ON CART-ADD
   → Users add 5 items to cart, buy 1
   → Cart is NOT commitment
   → Only decrement on explicit "PLACE ORDER" action

4. NEVER TRUST CLIENT-SENT PRICES
   → Client says "price: $1.00" for a $500 item
   → ALWAYS read price from YOUR database at order time
   → Client only sends: product_id + quantity

5. NEVER SKIP THE RESERVATION TIMEOUT
   → "We'll clean up later" = never
   → Stuck reservations = phantom out-of-stock
   → Scenario C from our problem statement

6. NEVER PROCESS PAYMENT BEFORE RESERVATION
   → What if payment succeeds but stock is gone?
   → Now you have money but no product → must refund
   → ALWAYS: Reserve → Pay → Confirm (in this order)

7. NEVER SHOW REAL-TIME EXACT STOCK TO USERS
   → 8,000 users see "1 left" → all click buy → 7,999 failures
   → Show "In Stock" / "Low Stock" / "Out of Stock"
   → Exact count is internal only

8. NEVER USE DISTRIBUTED LOCK FOR EVERY PURCHASE
   → Project 27 locks are expensive (Redis roundtrip)
   → Atomic DB operations are sufficient for stock
   → Use distributed locks only for cross-service coordination

🚫 WHEN NOT TO USE

1. RESERVATION PATTERN is NOT for:
   → Digital goods with unlimited stock (no scarcity)
   → Subscription services (no inventory concept)
   → When stock replenishes faster than it sells

2. REDIS PRE-FILTER is NOT for:
   → Products with very low traffic (overhead not justified)
   → When stock changes from multiple sources (warehouse sync)
     Redis count could drift from DB — need reconciliation

3. ATOMIC DECREMENT is NOT for:
   → Complex multi-product bundles with dependencies
     ("Buy laptop + charger as a set")
   → Need distributed transaction or saga instead

4. OPTIMISTIC LOCKING is NOT for:
   → Extremely high contention (flash sale on 1 item)
   → Retry storms will DDoS your own database
   → Use atomic decrement or queue instead

5. QUEUE SERIALIZATION is NOT for:
   → Normal shopping (latency too high for everyday purchases)
   → When user expects instant response
   → Adds 100ms-1s latency per order in queue

OPTIMISTIC LOCKING — HOW VERSION COLUMN WORKS
═══════════════════════════════════════════════════════════

  Table: products
  ┌─────┬──────────┬───────┬─────────┐
  │ id  │ name     │ stock │ version │
  ├─────┼──────────┼───────┼─────────┤
  │ 42  │ Sneakers │ 5     │ 17      │
  └─────┴──────────┴───────┴─────────┘

  Thread A reads: stock=5, version=17
  Thread B reads: stock=5, version=17

  Thread A updates:
    UPDATE products SET stock=4, version=18
    WHERE id=42 AND version=17
    → affected_rows = 1 ✅ (version matched)
    → version is now 18

  Thread B updates:
    UPDATE products SET stock=4, version=18
    WHERE id=42 AND version=17
    → affected_rows = 0 ❌ (version is 18 now, not 17)
    → Thread B must RETRY (re-read, re-check)

  WHY WE PREFER ATOMIC DECREMENT OVER THIS:
    → No retry needed
    → Single statement, no read-then-write
    → Database handles it internally
    → But optimistic locking is useful for
      complex updates (price change + stock change together)

═══════════════════════════════════════════════════════════
REDIS AS PRE-FILTER — THE MATH
═══════════════════════════════════════════════════════════

  Flash sale: 8,000 requests, 100 items

  WITHOUT Redis:
    8,000 DB write transactions
    Each takes ~5ms
    Serial bottleneck: ~200 tx/sec on single row
    Time to process all: 40 seconds
    Most users timeout → terrible UX

  WITH Redis pre-filter:
    8,000 Redis DECR operations → ~80ms total (100K ops/sec)
    First 100 get value >= 0 → proceed to DB
    Remaining 7,900 get value < 0 → instant rejection
    Only 100 DB transactions needed
    Time: ~0.5 seconds total
    7,900 users get "Sold Out" in <10ms → honest UX

  Redis DECR is atomic:
    → Single-threaded event loop
    → No race conditions possible
    → DECR returns the NEW value after decrement
    → If returned value < 0 → we over-decremented → INCR back

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: THE ABA PROBLEM IN STOCK
  → Stock is 5 → User A reserves (stock=4)
  → User A cancels (stock=5 again)
  → User B reads stock=5, thinks nothing happened
  → But the VERSION changed underneath
  → FIX: Version column catches this, or atomic ops don't care

PITFALL 2: REDIS-DB STOCK DRIFT
  → Redis says stock=10, DB says stock=8
  → Caused by: failed DB write after Redis DECR, no compensation
  → FIX: Always INCR Redis back on DB failure
  → FIX: Periodic reconciliation job (every 1 minute)
    SELECT available_stock FROM products WHERE id = X
    SET product:{id}:stock = {db_value}  ← Redis overwrite

PITFALL 3: RESERVATION SWEEPER RACE CONDITION
  → Sweeper marks reservation as EXPIRED at T+15:00
  → Payment gateway confirms SUCCESS at T+15:01
  → Stock was already restored
  → Now stock is +1 more than it should be
  → FIX: Check reservation status BEFORE confirming payment
    IF reservation.status == 'EXPIRED':
      → Refund the payment instead of confirming
      → Or re-reserve if stock is still available

PITFALL 4: THUNDERING HERD ON STOCK RESTORE
  → 50 reservations expire at exactly the same time
  → 50 stock restores hit the DB simultaneously
  → + 50 Redis INCR operations
  → Sudden spike
  → FIX: Jitter the sweep (random offset per reservation)
  → FIX: Batch the restores into single UPDATE

PITFALL 5: DOUBLE-SPEND VIA REFRESH
  → User clicks "Pay" → page loading → user refreshes → clicks "Pay" again
  → Two payment requests for same order
  → FIX: Idempotency key per order (Project 29)
  → FIX: Order status check before payment initiation
    IF order.status != 'RESERVED': reject

PITFALL 6: CART HOARDING ATTACK
  → Malicious user scripts: reserve 100 items across 100 accounts
  → All items held → real users see "Out of Stock"
  → Reservations expire → repeat
  → FIX: Rate limit reservations per IP / per user
  → FIX: Short reservation window (5 min not 30 min)
  → FIX: Require verified account for flash sales

PITFALL 7: MULTI-ITEM ORDER PARTIAL FAILURE
  → User orders: Sneakers (1 left) + T-Shirt (50 left)
  → Sneakers reserved successfully
  → T-Shirt reservation fails (DB error)
  → Now sneakers are held but order can't complete
  → FIX: Reserve ALL items atomically
    → Either all succeed or all fail
    → Single transaction for all items in same shard
    → Saga for items across shards (Project 37)

PITFALL 8: PRICE CHANGED BETWEEN BROWSE AND BUY
  → User sees price $50 on product page (cached)
  → Admin changes price to $75
  → User clicks buy, expects $50
  → FIX: Read price at reservation time from DB
  → FIX: Show final price on checkout confirmation
  → FIX: If price changed > X% → warn user before proceeding

═══════════════════════════════════════════════════
BOTTLENECK ANALYSIS FOR FLASH SALE:
═══════════════════════════════════════════════════

  HOTSPOT: Single product row in DB
    → 8,000 writes to same row = row-level lock contention
    → PostgreSQL: ~2,000-5,000 simple updates/sec per row

class OrderStatus(Enum):
    CREATED = "CREATED"
    RESERVED = "RESERVED"
    PAYMENT_PENDING = "PAYMENT_PENDING"
    CONFIRMED = "CONFIRMED"
    SHIPPED = "SHIPPED"
    DELIVERED = "DELIVERED"

    PAYMENT_FAILED = "PAYMENT_FAILED"
    EXPIRED = "EXPIRED"
    CANCELLED = "CANCELLED"
    REFUNDED = "REFUNDED"


# Valid transitions: current_state → set of allowed next states
VALID_TRANSITIONS = {
    OrderStatus.CREATED: {OrderStatus.RESERVED, OrderStatus.CANCELLED},
    OrderStatus.RESERVED: {
        OrderStatus.PAYMENT_PENDING,
        OrderStatus.EXPIRED,
        OrderStatus.CANCELLED
    },
    OrderStatus.PAYMENT_PENDING: {
        OrderStatus.CONFIRMED,
        OrderStatus.PAYMENT_FAILED,
        OrderStatus.EXPIRED
    },
    OrderStatus.CONFIRMED: {OrderStatus.SHIPPED, OrderStatus.REFUNDED},
    OrderStatus.SHIPPED: {OrderStatus.DELIVERED},
    OrderStatus.DELIVERED: {OrderStatus.COMPLETED, OrderStatus.REFUNDED},
    OrderStatus.COMPLETED: set(),
    OrderStatus.PAYMENT_FAILED: {OrderStatus.CANCELLED},
    OrderStatus.EXPIRED: {OrderStatus.CANCELLED, OrderStatus.REFUNDED},
    OrderStatus.CANCELLED: set(),
    OrderStatus.REFUNDED: set(),
}


class OrderStateMachine:
    """Enforces valid state transitions with audit trail"""

    def __init__(self, order_id: str, initial_status: OrderStatus = OrderStatus.CREATED):
        self.order_id = order_id
        self.current_status = initial_status
        self.history = []
        self._record_transition(None, initial_status, "system", "Order created")

    def transition(self, new_status: OrderStatus, actor: str, reason: str = "") -> bool:
        if new_status not in VALID_TRANSITIONS.get(self.current_status, set()):
            print(
                f"    [STATE MACHINE] ❌ ILLEGAL: {self.order_id} "
                f"cannot go from {self.current_status.value} → {new_status.value}"
            )
            return False

        old_status = self.current_status
        self.current_status = new_status
        self._record_transition(old_status, new_status, actor, reason)
        return True

    def _record_transition(self, from_status, to_status, actor, reason):
        entry = {
            "timestamp": datetime.now().isoformat(),
            "from": from_status.value if from_status else None,
            "to": to_status.value,
            "actor": actor,
            "reason": reason,
        }
        self.history.append(entry)

    def get_audit_trail(self):
        return self.history

class SimulatedRedis:
    """Thread-safe atomic counter operations simulating Redis"""

    def __init__(self):
        self._data = {}
        self._lock = threading.Lock()
        self._op_count = 0

    def set_stock(self, product_id: str, count: int):
        with self._lock:
            self._data[f"product:{product_id}:stock"] = count

    def atomic_decrement(self, product_id: str, amount: int = 1) -> int:
        """
        Simulates Redis DECRBY — returns value AFTER decrement.
        In real Redis this is a single atomic operation.
        No race condition possible because Redis is single-threaded.
        """
        key = f"product:{product_id}:stock"
        with self._lock:
            self._op_count += 1
            current = self._data.get(key, 0)
            new_value = current - amount
            self._data[key] = new_value
            return new_value

    def atomic_increment(self, product_id: str, amount: int = 1) -> int:
        """Simulates Redis INCRBY — restore stock on failure/expiry"""
        key = f"product:{product_id}:stock"
        with self._lock:
            self._op_count += 1
            current = self._data.get(key, 0)
            new_value = current + amount
            self._data[key] = new_value
            return new_value

    def get_stock(self, product_id: str) -> int:
        with self._lock:
            return self._data.get(f"product:{product_id}:stock", 0)

    def get_op_count(self):
        return self._op_count

class InventoryDB:
    """
    Simulates a database with atomic stock operations.
    The key insight: UPDATE ... WHERE stock >= qty
    is a SINGLE atomic statement in real databases.
    """

    def __init__(self):
        self._products = {}
        self._lock = threading.Lock()
        self._transaction_count = 0

    def add_product(self, product_id: str, name: str, price: float, total_stock: int):
        with self._lock:
            self._products[product_id] = {
                "product_id": product_id,
                "name": name,
                "price": price,
                "total_stock": total_stock,
                "available_stock": total_stock,
                "reserved_stock": 0,
                "sold_stock": 0,
                "version": 1,
            }

    def atomic_reserve(self, product_id: str, quantity: int) -> bool:
        """
        Simulates: UPDATE inventory
                   SET available_stock = available_stock - qty,
                       reserved_stock = reserved_stock + qty,
                       version = version + 1
                   WHERE product_id = ? AND available_stock >= qty

        Returns True if reservation succeeded (affected_rows = 1)
        Returns False if insufficient stock (affected_rows = 0)
        """
        with self._lock:
            self._transaction_count += 1
            product = self._products.get(product_id)
            if not product:
                return False

            if product["available_stock"] >= quantity:
                product["available_stock"] -= quantity
                product["reserved_stock"] += quantity
                product["version"] += 1
                return True
            return False

    def confirm_sale(self, product_id: str, quantity: int) -> bool:
        """Reservation → Sold: reserved_stock → sold_stock"""
        with self._lock:
            product = self._products.get(product_id)
            if not product:
                return False

            if product["reserved_stock"] >= quantity:
                product["reserved_stock"] -= quantity
                product["sold_stock"] += quantity
                product["version"] += 1
                return True
            return False

    def release_reservation(self, product_id: str, quantity: int) -> bool:
        """Compensation: reserved_stock → available_stock"""
        with self._lock:
            product = self._products.get(product_id)
            if not product:
                return False

            if product["reserved_stock"] >= quantity:
                product["reserved_stock"] -= quantity
                product["available_stock"] += quantity
                product["version"] += 1
                return True
            return False

    def get_product(self, product_id: str) -> Optional[dict]:
        with self._lock:
            p = self._products.get(product_id)
            return dict(p) if p else None

    def check_invariant(self, product_id: str) -> bool:
        """total = available + reserved + sold (must ALWAYS hold)"""
        with self._lock:
            p = self._products.get(product_id)
            if not p:
                return False
            computed_total = (
                p["available_stock"]
                + p["reserved_stock"]
                + p["sold_stock"]
            )
            return computed_total == p["total_stock"]

    def get_transaction_count(self):
        return self._transaction_count

class ReservationStore:
    """Tracks all active reservations with expiry times"""

    def __init__(self, default_ttl_seconds: int = 900):
        self._reservations = {}
        self._lock = threading.Lock()
        self.default_ttl = default_ttl_seconds

    def create_reservation(
        self, reservation_id: str, order_id: str,
        product_id: str, quantity: int,
        ttl_seconds: int = None
    ):
        ttl = ttl_seconds or self.default_ttl
        with self._lock:
            self._reservations[reservation_id] = {
                "reservation_id": reservation_id,
                "order_id": order_id,
                "product_id": product_id,
                "quantity": quantity,
                "status": "ACTIVE",
                "created_at": datetime.now(),
                "expires_at": datetime.now() + timedelta(seconds=ttl),
            }

    def get_reservation(self, reservation_id: str) -> Optional[dict]:
        with self._lock:
            r = self._reservations.get(reservation_id)
            return dict(r) if r else None

    def mark_confirmed(self, reservation_id: str) -> bool:
        with self._lock:
            r = self._reservations.get(reservation_id)
            if r and r["status"] == "ACTIVE":
                r["status"] = "CONFIRMED"
                r["confirmed_at"] = datetime.now()
                return True
            return False

    def mark_expired(self, reservation_id: str) -> bool:
        with self._lock:
            r = self._reservations.get(reservation_id)
            if r and r["status"] == "ACTIVE":
                r["status"] = "EXPIRED"
                r["expired_at"] = datetime.now()
                return True
            return False

    def get_expired_reservations(self) -> list:
        """Find all reservations past their TTL"""
        now = datetime.now()
        with self._lock:
            expired = []
            for rid, r in self._reservations.items():
                if r["status"] == "ACTIVE" and r["expires_at"] <= now:
                    expired.append(dict(r))
            return expired

    def get_stats(self) -> dict:
        with self._lock:
            stats = defaultdict(int)
            for r in self._reservations.values():
                stats[r["status"]] += 1
            return dict(stats)

class IdempotencyStore:
    """Prevents duplicate order processing on retries"""

    def __init__(self):
        self._keys = {}
        self._lock = threading.Lock()

    def check_and_set(self, idempotency_key: str, result: dict) -> Optional[dict]:
        """
        If key exists → return cached result (duplicate detected)
        If key is new → store result and return None
        """
        with self._lock:
            if idempotency_key in self._keys:
                return self._keys[idempotency_key]
            self._keys[idempotency_key] = result
            return None

    def exists(self, idempotency_key: str) -> bool:
        with self._lock:
            return idempotency_key in self._keys

    def get(self, idempotency_key: str) -> Optional[dict]:
        with self._lock:
            return self._keys.get(idempotency_key)

class PaymentGateway:
    """
    Simulates external payment processing.
    In reality: Stripe, PayPal, etc.
    Intentionally unreliable to test compensation logic.
    """

    def __init__(self, success_rate: float = 0.80, timeout_rate: float = 0.05):
        self.success_rate = success_rate
        self.timeout_rate = timeout_rate
        self._payments = {}
        self._lock = threading.Lock()
        import random
        self._random = random

    def process_payment(
        self, payment_id: str, order_id: str, amount: float
    ) -> dict:
        # Simulate processing delay
        time.sleep(self._random.uniform(0.01, 0.05))

        roll = self._random.random()

        if roll < self.timeout_rate:
            result = {
                "payment_id": payment_id,
                "status": "TIMEOUT",
                "message": "Gateway did not respond in time",
            }
        elif roll < self.timeout_rate + (1 - self.success_rate - self.timeout_rate):
            result = {
                "payment_id": payment_id,
                "status": "FAILED",
                "message": "Insufficient funds",
            }
        else:
            result = {
                "payment_id": payment_id,
                "status": "SUCCESS",
                "message": "Payment captured",
                "transaction_ref": f"TXN-{uuid.uuid4().hex[:8].upper()}",
            }

        with self._lock:
            self._payments[payment_id] = result
        return result

    def refund(self, payment_id: str) -> dict:
        time.sleep(self._random.uniform(0.01, 0.03))
        return {
            "payment_id": payment_id,
            "status": "REFUNDED",
            "message": "Refund processed",
        }

class OrderService:

    def __init__(
        self,
        redis: SimulatedRedis,
        inventory_db: InventoryDB,
        reservation_store: ReservationStore,
        idempotency_store: IdempotencyStore,
        payment_gateway: PaymentGateway,
        reservation_ttl_seconds: int = 5,
    ):
        self.redis = redis
        self.inventory_db = inventory_db
        self.reservations = reservation_store
        self.idempotency = idempotency_store
        self.payment = payment_gateway
        self.reservation_ttl = reservation_ttl_seconds

        # Order storage
        self._orders = {}
        self._state_machines = {}
        self._lock = threading.Lock()

        # Metrics
        self.metrics = {
            "total_requests": 0,
            "rejected_by_redis": 0,
            "rejected_by_db": 0,
            "reserved": 0,
            "payment_success": 0,
            "payment_failed": 0,
            "payment_timeout": 0,
            "expired": 0,
            "compensated": 0,
            "duplicate_requests": 0,
        }
        self._metrics_lock = threading.Lock()

    def _inc_metric(self, key: str, amount: int = 1):
        with self._metrics_lock:
            self.metrics[key] += amount

    # ============================================================
    # MAIN ENTRY POINT: Place an Order
    # ============================================================

    def place_order(
        self,
        user_id: str,
        product_id: str,
        quantity: int,
        idempotency_key: str,
    ) -> dict:

        self._inc_metric("total_requests")

        # ── STEP 0: IDEMPOTENCY CHECK ──
        cached = self.idempotency.get(idempotency_key)
        if cached is not None:
            self._inc_metric("duplicate_requests")
            return {**cached, "_note": "DUPLICATE — returned cached result"}

        # ── STEP 1: REDIS PRE-FILTER (fast rejection) ──
        redis_stock = self.redis.atomic_decrement(product_id, quantity)

        if redis_stock < 0:
            # Over-decremented — fix Redis
            self.redis.atomic_increment(product_id, quantity)
            self._inc_metric("rejected_by_redis")
            result = {
                "success": False,
                "reason": "OUT_OF_STOCK",
                "layer": "redis_pre_filter",
            }
            self.idempotency.check_and_set(idempotency_key, result)
            return result

        # ── STEP 2: DATABASE ATOMIC RESERVATION ──
        db_success = self.inventory_db.atomic_reserve(product_id, quantity)

        if not db_success:
            # Redis said yes, DB said no — fix Redis
            self.redis.atomic_increment(product_id, quantity)
            self._inc_metric("rejected_by_db")
            result = {
                "success": False,
                "reason": "OUT_OF_STOCK",
                "layer": "database",
            }
            self.idempotency.check_and_set(idempotency_key, result)
            return result

        # ── STEP 3: CREATE ORDER + RESERVATION ──
        order_id = f"ORD-{uuid.uuid4().hex[:8].upper()}"
        reservation_id = f"RSV-{uuid.uuid4().hex[:8].upper()}"

        # Create order record
        product = self.inventory_db.get_product(product_id)
        order = {
            "order_id": order_id,
            "user_id": user_id,
            "product_id": product_id,
            "product_name": product["name"],
            "quantity": quantity,
            "unit_price": product["price"],
            "total_price": product["price"] * quantity,
            "reservation_id": reservation_id,
            "created_at": datetime.now().isoformat(),
        }

        # Create state machine
        state_machine = OrderStateMachine(order_id)

        # Transition: CREATED → RESERVED
        state_machine.transition(
            OrderStatus.RESERVED, "inventory-service", "Stock reserved"
        )

        # Create reservation with TTL
        self.reservations.create_reservation(
            reservation_id=reservation_id,
            order_id=order_id,
            product_id=product_id,
            quantity=quantity,
            ttl_seconds=self.reservation_ttl,
        )

        with self._lock:
            self._orders[order_id] = order
            self._state_machines[order_id] = state_machine

        self._inc_metric("reserved")

        # ── STEP 4: PROCESS PAYMENT (Saga — Project 37) ──
        state_machine.transition(
            OrderStatus.PAYMENT_PENDING, "order-service", "Payment initiated"
        )

        payment_id = f"PAY-{uuid.uuid4().hex[:8].upper()}"
        payment_result = self.payment.process_payment(
            payment_id=payment_id,
            order_id=order_id,
            amount=order["total_price"],
        )

        # ── STEP 5: HANDLE PAYMENT RESULT ──

        if payment_result["status"] == "SUCCESS":
            return self._handle_payment_success(
                order_id, product_id, quantity,
                reservation_id, state_machine,
                payment_result, idempotency_key,
            )

        elif payment_result["status"] == "FAILED":
            return self._handle_payment_failure(
                order_id, product_id, quantity,
                reservation_id, state_machine,
                payment_result, idempotency_key,
            )

        else:  # TIMEOUT
            return self._handle_payment_timeout(
                order_id, product_id, quantity,
                reservation_id, state_machine,
                payment_result, idempotency_key,
            )

    # ============================================================
    # PAYMENT SUCCESS → Confirm Order
    # ============================================================

    def _handle_payment_success(
        self, order_id, product_id, quantity,
        reservation_id, state_machine,
        payment_result, idempotency_key,
    ):
        # Check reservation still active (not expired by sweeper)
        reservation = self.reservations.get_reservation(reservation_id)

        if reservation and reservation["status"] == "ACTIVE":
            # Confirm: reserved_stock → sold_stock
            self.inventory_db.confirm_sale(product_id, quantity)
            self.reservations.mark_confirmed(reservation_id)
            state_machine.transition(
                OrderStatus.CONFIRMED, "payment-gateway", "Payment successful"
            )
            self._inc_metric("payment_success")

            result = {
                "success": True,
                "order_id": order_id,
                "status": "CONFIRMED",
                "payment_ref": payment_result.get("transaction_ref"),
                "total_charged": self._orders[order_id]["total_price"],
            }
        else:
            # Reservation expired while payment was processing
            # Must REFUND the payment
            self.payment.refund(payment_result.get("payment_id", ""))
            state_machine.transition(
                OrderStatus.EXPIRED, "reservation-sweeper", "Reservation expired during payment"
            )
            state_machine.transition(
                OrderStatus.REFUNDED, "payment-service", "Late payment refunded"
            )
            self._inc_metric("compensated")

            result = {
                "success": False,
                "order_id": order_id,
                "status": "EXPIRED_AND_REFUNDED",
                "reason": "Reservation expired before payment confirmed",
            }

        self.idempotency.check_and_set(idempotency_key, result)
        return result

    # ============================================================
    # PAYMENT FAILURE → Compensate (Restore Stock)
    # ============================================================

    def _handle_payment_failure(
        self, order_id, product_id, quantity,
        reservation_id, state_machine,
        payment_result, idempotency_key,
    ):
        # COMPENSATE: Restore stock
        self.inventory_db.release_reservation(product_id, quantity)
        self.redis.atomic_increment(product_id, quantity)
        self.reservations.mark_expired(reservation_id)

        state_machine.transition(
            OrderStatus.PAYMENT_FAILED, "payment-gateway",
            payment_result.get("message", "Payment failed"),
        )
        state_machine.transition(
            OrderStatus.CANCELLED, "order-service", "Auto-cancelled after payment failure"
        )

        self._inc_metric("payment_failed")
        self._inc_metric("compensated")

        result = {
            "success": False,
            "order_id": order_id,
            "status": "PAYMENT_FAILED",
            "reason": payment_result.get("message"),
        }
        self.idempotency.check_and_set(idempotency_key, result)
        return result

    # ============================================================
    # PAYMENT TIMEOUT → Retry then Compensate
    # ============================================================

    def _handle_payment_timeout(
        self, order_id, product_id, quantity,
        reservation_id, state_machine,
        payment_result, idempotency_key,
    ):
        # In production: retry 2-3 times with backoff
        # For demo: treat as failure and compensate

        self.inventory_db.release_reservation(product_id, quantity)
        self.redis.atomic_increment(product_id, quantity)
        self.reservations.mark_expired(reservation_id)

        state_machine.transition(
            OrderStatus.PAYMENT_FAILED, "payment-gateway", "Payment timeout"
        )
        state_machine.transition(
            OrderStatus.CANCELLED, "order-service", "Auto-cancelled after payment timeout"
        )

        self._inc_metric("payment_timeout")
        self._inc_metric("compensated")

        result = {
            "success": False,
            "order_id": order_id,
            "status": "PAYMENT_TIMEOUT",
            "reason": "Payment gateway did not respond",
        }
        self.idempotency.check_and_set(idempotency_key, result)
        return result

    # ============================================================
    # RESERVATION SWEEPER (Background Job)
    # ============================================================

    def sweep_expired_reservations(self):
        """Called periodically to clean up expired reservations"""
        expired = self.reservations.get_expired_reservations()

        for r in expired:
            reservation_id = r["reservation_id"]
            order_id = r["order_id"]
            product_id = r["product_id"]
            quantity = r["quantity"]

            # Restore stock
            self.inventory_db.release_reservation(product_id, quantity)
            self.redis.atomic_increment(product_id, quantity)
            self.reservations.mark_expired(reservation_id)

            # Update order state
            with self._lock:
                sm = self._state_machines.get(order_id)
                if sm and sm.current_status in (
                    OrderStatus.RESERVED,
                    OrderStatus.PAYMENT_PENDING,
                ):
                    sm.transition(
                        OrderStatus.EXPIRED, "reservation-sweeper",
                        "Reservation TTL exceeded",
                    )
                    sm.transition(
                        OrderStatus.CANCELLED, "reservation-sweeper",
                        "Auto-cancelled after expiry",
                    )

            self._inc_metric("expired")
            self._inc_metric("compensated")

        return len(expired)

    # ============================================================
    # QUERY METHODS (Read Path — CQRS read model, Project 30)
    # ============================================================

    def get_order(self, order_id: str) -> Optional[dict]:
        with self._lock:
            order = self._orders.get(order_id)
            sm = self._state_machines.get(order_id)
            if order and sm:
                return {
                    **order,
                    "current_status": sm.current_status.value,
                    "audit_trail": sm.get_audit_trail(),
                }
            return None

    def get_metrics(self) -> dict:
        with self._metrics_lock:
            return dict(self.metrics)

# SECTION 8: BACKGROUND SWEEPER THREAD
# Runs periodically to clean up expired reservations
# Similar concept to Project 36 (Task Scheduler)
# ================================================================

class ReservationSweeper:
    """
    Background daemon that periodically checks for
    expired reservations and restores stock.
    
    Without this → Scenario C from our problem:
    Users hold stock forever, real buyers see "Out of Stock"
    """

    def __init__(self, order_service: OrderService, interval_seconds: float = 1.0):
        self.order_service = order_service
        self.interval = interval_seconds
        self._running = False
        self._thread = None
        self.sweep_count = 0
        self.total_expired_found = 0

    def start(self):
        self._running = True
        self._thread = threading.Thread(target=self._sweep_loop, daemon=True)
        self._thread.start()

    def stop(self):
        self._running = False
        if self._thread:
            self._thread.join(timeout=5)

    def _sweep_loop(self):
        while self._running:
            time.sleep(self.interval)
            expired_count = self.order_service.sweep_expired_reservations()
            self.sweep_count += 1
            self.total_expired_found += expired_count

class OptimisticLockingDemo:
    """
    Demonstrates the version-column approach and its retry behavior.
    Contrast with the atomic decrement used in main system.
    """

    def __init__(self):
        self.stock = 10
        self.version = 1
        self._lock = threading.Lock()  # Simulates DB row-level behavior
        self.retry_count = 0
        self.success_count = 0
        self.total_attempts = 0

    def read(self):
        """Simulates SELECT stock, version FROM products WHERE id = X"""
        with self._lock:
            return self.stock, self.version

    def update_with_version_check(self, expected_version: int) -> bool:
        """
        Simulates:
          UPDATE products
          SET stock = stock - 1, version = version + 1
          WHERE id = X AND version = expected_version AND stock > 0
        """
        with self._lock:
            self.total_attempts += 1
            if self.version == expected_version and self.stock > 0:
                self.stock -= 1
                self.version += 1
                self.success_count += 1
                return True
            return False

    def attempt_purchase_with_retries(self, buyer_id: int, max_retries: int = 5):
        """Buyer reads, then tries to update. Retries on version mismatch."""
        for attempt in range(max_retries):
            stock, version = self.read()

            if stock <= 0:
                return False, attempt

            # Simulate processing delay (real world: validation, etc.)
            time.sleep(0.001)

            success = self.update_with_version_check(version)
            if success:
                return True, attempt

            # Version mismatch — someone else bought in between
            self.retry_count += 1
            time.sleep(0.001)  # Backoff before retry

        return False, max_retries

    @staticmethod
    def run_demo():
        print("\n" + "=" * 70)
        print("  📎 OPTIMISTIC LOCKING DEMO (Contrast with Atomic Decrement)")

        results_lock = threading.Lock()

        def buyer_thread(bid):
            success, retries = demo.attempt_purchase_with_retries(bid)
            with results_lock:
                if success:
                    results["success"] += 1
                else:
                    results["failed"] += 1
                results["total_retries"] += retries

        # 50 buyers competing for 10 items
        threads = []
        for i in range(50):
            t = threading.Thread(target=buyer_thread, args=(i,))
            threads.append(t)
            t.start()

        for t in threads:

class MultiItemOrderDemo:
    """
    Demonstrates the all-or-nothing reservation for multi-item orders.
    Either ALL items are reserved, or NONE are.
    """

    def __init__(self):
        self.inventory_db = InventoryDB()
        self.redis = SimulatedRedis()

    def setup(self):
        products = [
            ("PROD-A", "Sneakers", 199.99, 5),
            ("PROD-B", "T-Shirt", 29.99, 50),
            ("PROD-C", "Hat", 19.99, 2),
        ]
        for pid, name, price, stock in products:
            self.inventory_db.add_product(pid, name, price, stock)
            self.redis.set_stock(pid, stock)

    def reserve_multi_item(self, items: list) -> dict:
        """
        items = [
            {"product_id": "PROD-A", "quantity": 1},
            {"product_id": "PROD-C", "quantity": 1},
        ]

        ALL must succeed, or ALL must be rolled back.
        """
        reserved_items = []

        for item in items:
            pid = item["product_id"]
            qty = item["quantity"]

            # Try Redis pre-check
            redis_result = self.redis.atomic_decrement(pid, qty)
            if redis_result < 0:
                self.redis.atomic_increment(pid, qty)
                # ROLLBACK all previously reserved items
                self._rollback_reserved(reserved_items)
                return {
                    "success": False,
                    "reason": f"Insufficient stock for {pid}",
                    "failed_at": pid,
                    "rolled_back": [r["product_id"] for r in reserved_items],
                }

            # Try DB reservation
            db_success = self.inventory_db.atomic_reserve(pid, qty)
            if not db_success:
                self.redis.atomic_increment(pid, qty)
                self._rollback_reserved(reserved_items)
                return {
                    "success": False,
                    "reason": f"DB reservation failed for {pid}",
                    "failed_at": pid,
                    "rolled_back": [r["product_id"] for r in reserved_items],
                }

            reserved_items.append({"product_id": pid, "quantity": qty})

        return {
            "success": True,
            "reserved": [r["product_id"] for r in reserved_items],
        }

    def _rollback_reserved(self, reserved_items: list):
        """Compensation: undo all successful reservations"""
        for item in reserved_items:
            self.inventory_db.release_reservation(
                item["product_id"], item["quantity"]
            )
            self.redis.atomic_increment(item["product_id"], item["quantity"])

    @staticmethod
    def run_demo():
        print("\n" + "=" * 70)
        print("  📦 MULTI-ITEM ORDER DEMO (All-or-Nothing Reservation)")

        transitions = [
            # Valid path
            (OrderStatus.RESERVED, "inventory-svc", "Stock held"),
            (OrderStatus.PAYMENT_PENDING, "order-svc", "Payment started"),
            (OrderStatus.CONFIRMED, "payment-gw", "Payment captured"),
            # ILLEGAL: Try to go back to RESERVED from CONFIRMED
            (OrderStatus.RESERVED, "hacker", "Trying to re-reserve"),
            # ILLEGAL: Try to skip to DELIVERED from CONFIRMED
            (OrderStatus.DELIVERED, "hacker", "Trying to skip shipping"),
            # Valid: CONFIRMED → SHIPPED
            (OrderStatus.SHIPPED, "fulfillment-svc", "Package dispatched"),
            # Valid: SHIPPED → DELIVERED
            (OrderStatus.DELIVERED, "delivery-partner", "Package received"),
            # ILLEGAL: Try to cancel after delivery
            (OrderStatus.CANCELLED, "user", "Changed mind after delivery"),
            # Valid: DELIVERED → COMPLETED
            (OrderStatus.COMPLETED, "system", "Order lifecycle complete"),
            # ILLEGAL: Try anything after COMPLETED
            (OrderStatus.REFUNDED, "user", "Want refund after completion"),
        ]

        for target, actor, reason in transitions:
            current = sm.current_status.value
            success = sm.transition(target, actor, reason)
            symbol = "✅" if success else "❌"
            result = "ALLOWED" if success else "BLOCKED"
            print(f"    {symbol} {current:>20} → {target.value:<20} [{result}] ({reason})")

        print(f"\n  Final state: {sm.current_status.value}")
        print(f"\n  Full Audit Trail:")
        for entry in sm.get_audit_trail():
            arrow = f"{entry['from']} → " if entry['from'] else "INIT → "
            print(f"    [{entry['timestamp'][-12:]}] {arrow}{entry['to']} ({entry['actor']})")

        print(f"{'=' * 70}")

📋 PHASE 3 (Medium) — Professional Techniques

🎯 HYPOTHETICAL PROBLEM

You run a global e-commerce platform (like Amazon)
→ A customer places an order for a laptop
→ The MOMENT that order is confirmed, ALL of these must happen:

  1. Inventory Service: Decrement stock count for that laptop
  2. Payment Service: Charge the customer's credit card
  3. Shipping Service: Create a shipment and assign a courier
  4. Notification Service: Send order confirmation email + SMS
  5. Analytics Service: Update sales dashboards in real-time
  6. Recommendation Service: Update "frequently bought together"
  7. Search Service: Update product popularity ranking
  8. Fraud Detection: Analyze order for suspicious patterns
  9. Loyalty Service: Award points to the customer
  10. Autocomplete Service (Project 28): Boost "laptop" in trending

NAIVE APPROACH — Direct Service Calls:

  OrderService.placeOrder() {
      inventoryService.decrementStock()    // HTTP call
      paymentService.charge()              // HTTP call
      shippingService.createShipment()     // HTTP call
      notificationService.sendEmail()      // HTTP call
      analyticsService.recordSale()        // HTTP call
      recommendationService.update()       // HTTP call
      searchService.updateRanking()        // HTTP call
      fraudService.analyze()               // HTTP call
      loyaltyService.awardPoints()         // HTTP call
      autocompleteService.boostTerm()      // HTTP call
  }

WHY THIS FAILS:

  Problem 1: TIGHT COUPLING
  → OrderService KNOWS about all 10 downstream services
  → Adding an 11th service = modify OrderService code
  → Removing a service = modify OrderService code
  → Each change risks breaking the order flow

  Problem 2: CASCADING FAILURES
  → If NotificationService is down → entire order fails?
  → Customer can't buy a laptop because email server is down?
  → Absurd — email is NOT critical to placing an order

  Problem 3: LATENCY ACCUMULATION
  → Each HTTP call: 50-200ms
  → 10 calls sequentially: 500-2000ms
  → Customer waits 2 seconds for "Order Placed" — terrible UX

  Problem 4: PARTIAL FAILURES
  → Payment succeeds but Shipping fails
  → Now what? Rollback payment? Retry shipping?
  → Complex distributed transaction nightmare

  Problem 5: SCALABILITY MISMATCH
  → OrderService handles 1000 orders/sec
  → AnalyticsService can only handle 100 events/sec
  → Fast producer overwhelms slow consumer → data loss

EVENT-DRIVEN SOLUTION:
  → OrderService publishes ONE event: "OrderPlaced"
  → EACH downstream service subscribes independently
  → They consume at their OWN pace
  → They fail independently without affecting orders
  → Adding new consumers requires ZERO changes to OrderService

🔍 HOW TO APPROACH IT

Step 1: Identify the EVENT PRODUCERS
         → OrderService → "OrderPlaced", "OrderCancelled"
         → PaymentService → "PaymentProcessed", "PaymentFailed"
         → ShippingService → "ShipmentCreated", "ShipmentDelivered"
         → UserService → "UserRegistered", "UserUpdated"

Step 2: Identify the EVENT CONSUMERS
         → Each downstream service subscribes to relevant events
         → InventoryService listens to "OrderPlaced"
         → NotificationService listens to "OrderPlaced", "ShipmentDelivered"
         → AnalyticsService listens to EVERYTHING

Step 3: Define the EVENT SCHEMA
         → What data does each event carry?
         → Event ID, type, timestamp, source, payload
         → Schema must be versioned (events evolve over time)

Step 4: Choose the EVENT DELIVERY mechanism
         → Event Bus (pub/sub): broadcast to all subscribers
         → Event Stream (log): ordered, persistent, replayable
         → Message Queue (Project 11): point-to-point, one consumer

Step 5: Handle FAILURES gracefully
         → At-least-once delivery (may receive duplicates)
         → Idempotent consumers (processing same event twice = same result)
         → Dead letter queue (events that fail repeatedly)

Step 6: Handle ORDERING when needed
         → Events for same order must be processed IN ORDER
         → Events for different orders can be parallel
         → Partition by order_id (same idea as sharding — Project 18)

KEY CONCEPTS:

  TOPIC: Named channel for a category of events
    → "orders" topic carries all order-related events
    → "payments" topic carries all payment-related events

  PARTITION: Subdivision of a topic for parallelism
    → Events with same key go to same partition (ordering)
    → Different partitions processed in parallel (throughput)
    → Key = order_id → all events for one order stay ordered

  CONSUMER GROUP: Set of consumers sharing the workload
    → "inventory-group" has 3 instances of InventoryService
    → Each instance processes a SUBSET of partitions
    → Partition 0 → Instance A, Partition 1 → Instance B, etc.
    → Horizontal scaling of consumers

  OFFSET: Position in the partition log
    → Each consumer tracks where it left off
    → On restart: resume from last committed offset
    → Can replay from beginning (event sourcing)


THREE PATTERNS:

  Pattern 1: EVENT NOTIFICATION
  → "Something happened" — minimal data
  → Consumer must query source for full details
  → Loose coupling but extra network calls
  → Example: {"type": "OrderPlaced", "order_id": "ORD-123"}

  Pattern 2: EVENT-CARRIED STATE TRANSFER
  → Event carries ALL data consumer needs
  → No callback to source needed
  → More data on the wire but fully decoupled
  → Example: {"type": "OrderPlaced", "order_id": "ORD-123",
              "customer": {...}, "items": [...], "total": 999}

  Pattern 3: EVENT SOURCING
  → Events ARE the source of truth (not a side effect)
  → Current state = replay all events from beginning
  → Every state change is an immutable event in the log
  → Example: Bank account balance = sum of all deposit/withdrawal events
  → Gives complete audit trail + time travel debugging

📌 WHAT COULD BE DONE

Option A: Simple Pub/Sub (Redis Pub/Sub)
  → Fire and forget — no persistence
  → If consumer is offline → misses the event forever
  → Good for: real-time notifications where loss is acceptable
  → Bad for: anything requiring reliability

Option B: Message Queue with Topics (RabbitMQ style — Project 11)
  → Messages consumed and acknowledged
  → Messages deleted after successful consumption
  → Cannot replay past messages
  → Good for: task distribution, work queues

Option C: Event Stream / Log (Kafka style — CHOSEN)
  → Events appended to immutable, ordered log
  → Consumers track their own position (offset)
  → Events retained for configurable duration (days/weeks)
  → Can replay from any point in time
  → Multiple consumer groups read independently
  → Good for: event sourcing, analytics, audit trails

Option D: Database as Event Store
  → Events stored in database table
  → Consumers poll for new events
  → Simple but doesn't scale well
  → Good for: small systems, CQRS event store

FOR THIS PROJECT:
  → Option C as primary (Kafka-style event stream)
  → Option D as secondary (event store for sourcing)
  → Both patterns demonstrated end-to-end

✅ WHAT SHOULD DO

1. DESIGN events as IMMUTABLE FACTS
   → "OrderPlaced" is a fact — it happened
   → You NEVER delete or modify an event
   → If order is cancelled → new event "OrderCancelled"
   → Append-only log — complete audit trail

2. INCLUDE all necessary data in the event
   → Consumer should NOT need to call back to producer
   → Event-carried state transfer = full decoupling
   → If producer is down, consumer still works from event data

3. MAKE every consumer IDEMPOTENT
   → Same event processed twice → same result
   → WHY: at-least-once delivery means duplicates happen
   → HOW: track processed event IDs → skip if already seen
   → HOW: use idempotent operations (SET not INCREMENT)

4. USE event schemas with versioning
   → Events evolve over time (new fields, removed fields)
   → Version 1: {"order_id": "123"}
   → Version 2: {"order_id": "123", "priority": "express"}
   → Consumers must handle both versions gracefully
   → Never break backward compatibility

5. PARTITION by entity ID for ordering
   → All events for order "ORD-123" → same partition
   → Guarantees in-order processing per order
   → Different orders processed in parallel → throughput

6. IMPLEMENT dead letter queues
   → Event fails processing 3 times → move to DLQ
   → Don't let one bad event block all others
   → Monitor DLQ → alert → manual intervention

7. TRACK consumer lag
   → Producer is at offset 10000
   → Consumer is at offset 9500
   → Lag = 500 events behind
   → Alert if lag exceeds threshold
   → Indicates consumer can't keep up → scale it

8. LOG correlation IDs across events (Project 22)
   → Order "ORD-123" generates events across 10 services
   → Correlation ID ties them all together
   → Can trace the full lifecycle of one order

❌ WHAT SHOULD NOT DO

1. NEVER use events for synchronous request-response
   → Events are asynchronous by nature
   → If you need immediate response → use direct API call
   → Event = "fire and eventually process"
   → API = "call and wait for response"

2. NEVER put huge payloads in events
   → Event with 50MB image attachment = bad
   → Store blob in object storage (Project 25)
   → Event carries reference: {"image_url": "s3://..."}
   → Keep events small (< 1MB, ideally < 100KB)

3. NEVER assume event ordering across partitions
   → Partition 0: [A, B, C] — ordered within partition ✅
   → Partition 1: [D, E, F] — ordered within partition ✅
   → Across partitions: no guarantee A before D
   → If ordering matters → same partition key

4. NEVER modify published events
   → Event is an immutable fact
   → Found a bug in event data? Publish a CORRECTION event
   → "OrderAmountCorrected" not "modify OrderPlaced"

5. NEVER skip idempotency because "it probably won't happen"
   → Network retries WILL cause duplicates
   → Consumer restarts WILL reprocess some events
   → Murphy's Law applies to distributed systems

6. NEVER create circular event chains
   → Service A publishes event → Service B reacts → publishes event
   → Service A reacts to B's event → publishes again → infinite loop
   → FIX: Clear event ownership boundaries
   → FIX: Saga pattern for multi-step workflows (Project 37)

7. NEVER couple event schema to internal data model
   → Internal model changes frequently
   → Event schema is a PUBLIC CONTRACT
   → Decouple: internal model → event mapper → event schema
   → Consumers depend on event schema, not your internals

8. NEVER ignore consumer group management
   → Adding/removing consumer instances must rebalance partitions
   → Without rebalancing → some partitions unprocessed
   → Without coordination → duplicate processing

CHOOSING EVENT DELIVERY GUARANTEES:
  → At-most-once:  Fire and forget (logs, metrics — loss OK)
  → At-least-once: Retry until acknowledged (orders — MUST deliver)
  → Exactly-once:  Idempotent consumer + at-least-once delivery
                   (payments — Project 37 covers this)

🚫 WHEN NOT TO USE

1. SIMPLE CRUD APPLICATIONS
   → Blog with 100 users → direct DB writes are fine
   → Events add complexity with zero benefit
   → Rule: if < 3 services react to a change → direct calls are simpler

2. WHEN STRONG CONSISTENCY IS REQUIRED IMMEDIATELY
   → "Is this seat available RIGHT NOW?"
   → Event-driven = eventually consistent
   → The event might not have propagated yet
   → Use direct query for immediate consistency

3. WHEN DEBUGGING MUST BE SIMPLE
   → Event-driven = async flows across many services
   → One request → 10 events → 10 consumers → hard to trace
   → Small team + simple product → direct calls easier to debug
   → Use distributed tracing (Project 22) if you must go event-driven

4. WHEN ORDERING IS CRITICAL ACROSS ENTITIES
   → "Process all orders strictly in global order"
   → Events guarantee ordering PER PARTITION (per entity)
   → Global ordering requires single partition → kills throughput
   → Consider: do you REALLY need global ordering?

5. WHEN TEAM IS SMALL AND MOVING FAST
   → Event-driven adds operational complexity
   → Need to manage event bus infrastructure
   → Need to handle schema evolution, dead letters, lag monitoring
   → 2-person startup → probably too much overhead

MECHANISM 1: EVENT BUS (PUB/SUB WITH TOPICS)
═══════════════════════════════════════════════════════

PUBLISH FLOW:
  
  Producer → EventBus.publish("orders", event)
  
  Internally:
  1. Determine partition: hash(event.partition_key) % num_partitions
     → event.partition_key = "ORD-123"
     → hash("ORD-123") % 3 = 1
     → Goes to Partition 1
  
  2. Append event to partition log:
     Partition 1: [..., event_998, event_999, NEW_EVENT_1000]
     → Offset 1000 assigned
     → Event is now DURABLE (written to disk/replicated)
  
  3. Notify waiting consumers (if any are polling)

  ┌──────────────────────────────────────────────────────┐
  │  Topic: "orders"                                     │
  │                                                      │
  │  Partition 0: [E1, E4, E7, E10, E13, ...]           │
  │  Partition 1: [E2, E5, E8, E11, E14, ...]           │
  │  Partition 2: [E3, E6, E9, E12, E15, ...]           │
  │                                                      │
  │  Events distributed by: hash(order_id) % 3           │
  │  Within each partition: STRICTLY ORDERED              │
  │  Across partitions: NO ordering guarantee             │

SUBSCRIBE FLOW:

  Consumer Group "inventory-svc" with 3 instances:
  
  Instance A → assigned Partition 0
  Instance B → assigned Partition 1

  1. Poll: "Give me events from my partition starting at my offset"
  2. Process batch of events
  3. Commit offset: "I've processed up to offset 1000"
  4. Repeat

MECHANISM 2: CONSUMER GROUP REBALANCING
═══════════════════════════════════════════════════════

WHEN IT HAPPENS:
  → New consumer instance joins the group
  → Existing consumer instance crashes/leaves
  → New partitions added to topic

BEFORE: 2 instances, 3 partitions
  Instance A → Partition 0, 1
  Instance B → Partition 2

NEW INSTANCE C JOINS:

  1. Group coordinator detects new member
  2. Triggers rebalance
  3. ALL consumers pause processing
  4. Partitions reassigned:
     Instance A → Partition 0
     Instance B → Partition 1
     Instance C → Partition 2
  5. Each instance resumes from its committed offset
  6. Processing continues

INSTANCE B CRASHES:

  1. Heartbeat timeout → coordinator detects failure
  2. Triggers rebalance
  3. Remaining consumers get B's partitions:
     Instance A → Partition 0, 1
     Instance C → Partition 2
  4. Partition 1 resumes from B's last committed offset
  5. Events B processed but didn't commit → reprocessed
     → THIS IS WHY idempotency is critical

MECHANISM 3: EVENT SOURCING — EVENTS AS SOURCE OF TRUTH
═══════════════════════════════════════════════════════

TRADITIONAL (State-Based):
  → Database stores CURRENT state: balance = $500
  → UPDATE account SET balance = 500 WHERE id = 123
  → Previous states are LOST
  → "Why is the balance $500?" → No idea

EVENT SOURCED:
  → Event Store stores ALL state changes:
    [
      {type: "AccountCreated", amount: 0,     time: T1}
      {type: "Deposited",      amount: 1000,  time: T2}
      {type: "Withdrawn",      amount: 200,   time: T3}
      {type: "Deposited",      amount: 500,   time: T4}
      {type: "Withdrawn",      amount: 800,   time: T5}
    ]
  → Current balance = 0 + 1000 - 200 + 500 - 800 = $500
  → Balance at time T3 = 0 + 1000 - 200 = $800
  → Complete audit trail → compliance, debugging, time travel

REBUILDING STATE (Projection):
  
  def project_balance(events):
      balance = 0
      for event in events:
          if event.type == "Deposited":
              balance += event.amount
          elif event.type == "Withdrawn":
              balance -= event.amount
      return balance
  
  → Same events → same state (deterministic)
  → Can rebuild ANY view from events
  → Can create NEW views without changing event store

SNAPSHOT OPTIMIZATION:
  → Replaying 1 million events is slow
  → Periodically save a SNAPSHOT: {balance: 500, at_event: 99999}
  → On load: start from snapshot, replay only events after 99999
  → Tradeoff: snapshot frequency vs replay speed

MECHANISM 4: IDEMPOTENT CONSUMERS
═══════════════════════════════════════════════════════

WHY DUPLICATES HAPPEN:

  Timeline:
  1. Consumer receives event E1 (offset 100)
  2. Consumer processes E1 successfully
  3. Consumer is about to commit offset 100...
  4. Consumer CRASHES before commit
  5. Consumer restarts
  6. Reads from last committed offset (99)
  7. Receives event E1 AGAIN (offset 100)
  8. Processes E1 a SECOND time ← DUPLICATE

IDEMPOTENCY STRATEGIES:

  Strategy 1: Idempotency Key (Deduplication Table)
    
    processed_events = set()  # or database table
    
    def handle_event(event):
        if event.id in processed_events:
            return  # Already processed — skip
        
        do_actual_work(event)
        processed_events.add(event.id)
        commit_offset()
    
    → Track every processed event ID
    → Check before processing
    → Skip if already seen

  Strategy 2: Idempotent Operations
    
    # NON-IDEMPOTENT (dangerous with duplicates):
    UPDATE account SET balance = balance + 100
    # First time: 500 + 100 = 600
    # Duplicate:  600 + 100 = 700 ← WRONG
    
    # IDEMPOTENT (safe with duplicates):
    UPDATE account SET balance = 600 WHERE event_id = 'E1'
    # First time: sets to 600
    # Duplicate:  sets to 600 again ← SAME RESULT ✅

  Strategy 3: Conditional Writes (Optimistic Concurrency)
    
    UPDATE orders SET status = 'shipped'
    WHERE order_id = 'ORD-123' AND status = 'paid'
    # Only succeeds if status is still 'paid'
    # Duplicate: status is already 'shipped' → 0 rows affected → safe

MECHANISM 5: DEAD LETTER QUEUE (DLQ)
═══════════════════════════════════════════════════════

WHAT HAPPENS WHEN AN EVENT FAILS:

  Attempt 1: Process event → EXCEPTION (bad data)
  Attempt 2: Retry after 1s → EXCEPTION (still bad)
  Attempt 3: Retry after 5s → EXCEPTION (permanently bad)
  
  NOW WHAT?
  → Can't skip it (might lose important data)
  → Can't retry forever (blocks other events)
  → SOLUTION: Move to Dead Letter Queue (DLQ)

  Main Queue: [E1, E2, E3, E4, E5]
                         ↓ fails 3x
  Dead Letter Queue: [E3]
  
  Main Queue continues: [E1, E2, E4, E5] → processing resumes

  DLQ Handling:
  → Alert operations team
  → Manual inspection: why did E3 fail?
  → Fix the issue (bad data, bug, etc.)
  → Replay E3 from DLQ back to main queue
  → Or discard if truly invalid

MECHANISM 6: EVENTUAL CONSISTENCY
═══════════════════════════════════════════════════════

THE REALITY OF EVENT-DRIVEN SYSTEMS:

  T=0ms:  OrderService writes order to its DB
  T=1ms:  OrderService publishes "OrderPlaced" event
  T=5ms:  Event arrives at InventoryService
  T=10ms: InventoryService processes event, updates its DB
  T=8ms:  Event arrives at NotificationService
  T=50ms: NotificationService sends email

  Between T=0ms and T=10ms:
  → OrderService says: "Order exists, stock reserved"
  → InventoryService says: "Stock NOT yet reserved"
  → INCONSISTENT for 10ms

  After T=10ms:
  → Both services agree → CONSISTENT

  This is EVENTUAL CONSISTENCY:
  → System is temporarily inconsistent
  → But EVENTUALLY converges to consistent state
  → As long as events are delivered and processed

  ACCEPTABLE because:
  → 10ms inconsistency window is invisible to humans
  → Each service is internally consistent at all times
  → The overall system converges quickly
  → Alternative (distributed transactions) is MUCH slower

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: EVENT ORDERING VIOLATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  → "OrderPlaced" and "OrderCancelled" for same order
  → If processed out of order → cancel then place → order exists!
  → FIX: Same partition key (order_id) → same partition → ordered
  → FIX: Include sequence number in events → consumer detects gaps

PITFALL 2: CONSUMER LAG GROWING UNBOUNDED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  → Producer: 10,000 events/sec
  → Consumer: 5,000 events/sec
  → Lag grows by 5,000 every second
  → After 1 hour: 18 million events behind
  → FIX: Scale consumer instances (add more to consumer group)
  → FIX: Optimize consumer processing (batch writes, etc.)
  → FIX: Alert on lag threshold

PITFALL 3: POISON PILL EVENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━
  → One malformed event causes consumer to crash
  → Consumer restarts → reads same event → crashes again
  → Infinite crash loop
  → FIX: Dead letter queue after N retries
  → FIX: Try-catch around EVERY event processing

PITFALL 4: SCHEMA EVOLUTION BREAKS CONSUMERS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  → Producer adds required field "priority" to OrderPlaced
  → Old events in the log don't have "priority"
  → New consumer crashes reading old events
  → FIX: New fields must be OPTIONAL with defaults

# PARTITION — Ordered log of events (like Kafka partition)

class Partition:
    """
    An ordered, append-only log of events.
    
    Like a single Kafka partition:
    → Events appended at the end
    → Each event gets an OFFSET (sequential number)
    → Consumers read from a specific offset
    → Events retained for configurable duration
    
    Within one partition: STRICT ORDERING guaranteed
    """
    
    def __init__(self, partition_id: int, retention_seconds: float = 86400):
        self.partition_id = partition_id
        self.retention_seconds = retention_seconds
        self.events: List[Tuple[int, Event]] = []  # [(offset, event)]
        self.next_offset = 0
        self._lock = threading.Lock()
    
    def append(self, event: Event) -> int:
        """Append event, return assigned offset"""
        with self._lock:
            offset = self.next_offset
            self.events.append((offset, event))
            self.next_offset += 1
            return offset
    
    def read(self, from_offset: int, max_count: int = 100
             ) -> List[Tuple[int, Event]]:
        """Read events starting from offset, up to max_count"""
        with self._lock:
            result = []
            for offset, event in self.events:
                if offset >= from_offset and len(result) < max_count:
                    result.append((offset, event))
            return result
    
    def latest_offset(self) -> int:
        with self._lock:
            return self.next_offset - 1 if self.events else -1
    
    def cleanup_expired(self):
        """Remove events older than retention period"""
        cutoff = time.time() - self.retention_seconds
        with self._lock:
            self.events = [
                (off, evt) for off, evt in self.events
                if evt.timestamp > cutoff
            ]
    
    @property
    def size(self) -> int:
        return len(self.events)

# TOPIC — Named channel with multiple partitions
# ================================================================

class Topic:
    """
    Named channel for a category of events.
    
    Contains N partitions for parallelism:
    → Events distributed across partitions by partition_key
    → Same key → same partition (ordering guarantee per entity)
    → Different keys → may go to different partitions (parallelism)
    """
    
    def __init__(self, name: str, num_partitions: int = 3,
                 retention_seconds: float = 86400):
        self.name = name
        self.num_partitions = num_partitions
        self.partitions = [
            Partition(i, retention_seconds)
            for i in range(num_partitions)
        ]
        self.total_events = 0
        self._lock = threading.Lock()
    
    def get_partition_for_key(self, partition_key: str) -> int:
        """Deterministic partition assignment using hash"""
        hash_val = int(hashlib.md5(
            partition_key.encode()
        ).hexdigest(), 16)
        return hash_val % self.num_partitions
    
    def publish(self, event: Event) -> Tuple[int, int]:
        """
        Publish event to appropriate partition.
        Returns (partition_id, offset).
        """
        partition_id = self.get_partition_for_key(event.aggregate_id)
        partition = self.partitions[partition_id]
        offset = partition.append(event)
        
        with self._lock:
            self.total_events += 1
        
        return partition_id, offset
    
    def get_stats(self) -> Dict:
        return {
            "name": self.name,
            "partitions": self.num_partitions,
            "total_events": self.total_events,
            "partition_sizes": [p.size for p in self.partitions],
            "latest_offsets": [p.latest_offset() for p in self.partitions]
        }

# CONSUMER OFFSET TRACKER — Where each consumer left off
# ================================================================

class OffsetTracker:
    """
    Tracks committed offsets per consumer group per partition.
    
    Consumer groups read independently:
    → "inventory-svc" might be at offset 500
    → "notification-svc" might be at offset 480
    → Same events, different progress
    """
    
    def __init__(self):
        # {group_id: {partition_id: committed_offset}}
        self.offsets: Dict[str, Dict[int, int]] = {}
        self._lock = threading.Lock()
    
    def commit(self, group_id: str, partition_id: int, offset: int):
        with self._lock:
            if group_id not in self.offsets:
                self.offsets[group_id] = {}
            self.offsets[group_id][partition_id] = offset
    
    def get_offset(self, group_id: str, partition_id: int) -> int:
        """Returns last committed offset, or -1 if never committed"""
        with self._lock:
            return self.offsets.get(group_id, {}).get(partition_id, -1)
    
    def get_all_offsets(self, group_id: str) -> Dict[int, int]:
        with self._lock:
            return dict(self.offsets.get(group_id, {}))
    
    def calculate_lag(self, group_id: str,
                      topic: Topic) -> Dict[int, int]:
        """Calculate consumer lag per partition"""
        lag = {}
        with self._lock:
            group_offsets = self.offsets.get(group_id, {})
            for partition in topic.partitions:
                pid = partition.partition_id
                committed = group_offsets.get(pid, -1)
                latest = partition.latest_offset()
                lag[pid] = max(0, latest - committed)
        return lag

class DeadLetterQueue:
    """
    Stores events that failed processing after max retries.
    
    Purpose:
    → Don't lose failed events
    → Don't block healthy events
    → Allow manual inspection and replay
    """
    
    def __init__(self):
        self.entries: List[Dict] = []
        self._lock = threading.Lock()
    
    def add(self, event: Event, consumer_group: str,
            error: str, attempt_count: int):
        with self._lock:
            self.entries.append({
                "event": event.to_dict(),
                "consumer_group": consumer_group,
                "error": error,
                "attempts": attempt_count,
                "failed_at": time.time(),
                "failed_at_str": time.strftime("%H:%M:%S"),
                "status": "pending_review"
            })
    
    def get_pending(self) -> List[Dict]:
        with self._lock:
            return [e for e in self.entries
                    if e["status"] == "pending_review"]
    
    def mark_resolved(self, index: int):
        with self._lock:
            if 0 <= index < len(self.entries):
                self.entries[index]["status"] = "resolved"
    
    @property
    def size(self) -> int:
        return len(self.entries)
    
    def print_contents(self):
        with self._lock:
            print(f"\n  {'='*70}")
            print(f"  DEAD LETTER QUEUE ({len(self.entries)} entries)")
            print(f"  {'='*70}")
            if not self.entries:
                print(f"  (empty)")
                return
            for i, entry in enumerate(self.entries):
                evt = entry["event"]
                print(f"  [{i}] {entry['failed_at_str']} | "
                      f"{evt['event_type']} | "
                      f"agg={evt['aggregate_id']} | "
                      f"group={entry['consumer_group']} | "
                      f"attempts={entry['attempts']} | "
                      f"error={entry['error'][:40]}")
            print(f"  {'='*70}")

class EventBus:
    """
    Kafka-style event bus with topics, partitions, and consumer groups.
    
    Core responsibilities:
    1. Manage topics and their partitions
    2. Route published events to correct partition
    3. Track consumer group offsets
    4. Manage dead letter queue
    5. Provide monitoring metrics
    """
    
    def __init__(self):
        self.topics: Dict[str, Topic] = {}
        self.offset_tracker = OffsetTracker()
        self.dlq = DeadLetterQueue()
        self._subscribers: Dict[str, Dict[str, List[Callable]]] = {}
        # {topic_name: {group_id: [handler_func, ...]}}
        self._lock = threading.Lock()
        self._running = False
        self._consumer_threads: List[threading.Thread] = []
        self.total_published = 0
        self.total_delivered = 0
        self.publish_log: List[Dict] = []
    
    def create_topic(self, name: str, num_partitions: int = 3,
                     retention_seconds: float = 86400):
        """Create a new topic"""
        with self._lock:
            if name not in self.topics:
                self.topics[name] = Topic(name, num_partitions,
                                          retention_seconds)
                print(f"  [EVENT BUS] Topic '{name}' created "
                      f"({num_partitions} partitions)")
    
    def publish(self, topic_name: str, event: Event) -> Tuple[int, int]:
        """
        Publish event to a topic.
        Returns (partition_id, offset).
        """
        if topic_name not in self.topics:
            raise ValueError(f"Topic '{topic_name}' does not exist")
        
        topic = self.topics[topic_name]
        partition_id, offset = topic.publish(event)
        
        with self._lock:
            self.total_published += 1
            self.publish_log.append({
                "time": time.strftime("%H:%M:%S"),
                "topic": topic_name,
                "partition": partition_id,
                "offset": offset,
                "event_type": event.event_type,
                "aggregate_id": event.aggregate_id,
                "event_id": event.event_id[:8]
            })
        
        return partition_id, offset
    
    def subscribe(self, topic_name: str, group_id: str,
                  handler: Callable[[Event], bool],
                  max_retries: int = 3):
        """
        Subscribe a consumer group to a topic.
        
        handler: function that processes an event
                 returns True on success, False on failure
        
        Multiple handlers in same group → work sharing
        Multiple groups on same topic → each gets ALL events
        """
        with self._lock:
            if topic_name not in self._subscribers:
                self._subscribers[topic_name] = {}
            if group_id not in self._subscribers[topic_name]:
                self._subscribers[topic_name][group_id] = []
            
            self._subscribers[topic_name][group_id].append({
                "handler": handler,
                "max_retries": max_retries,
                "group_id": group_id,
                "topic_name": topic_name
            })
        
        print(f"  [EVENT BUS] Group '{group_id}' subscribed "
              f"to '{topic_name}'")
    
    def start_consuming(self, poll_interval: float = 0.1):
        """Start background consumer threads"""
        self._running = True
        
        with self._lock:
            for topic_name, groups in self._subscribers.items():
                for group_id, handlers in groups.items():
                    topic = self.topics[topic_name]
                    for partition in topic.partitions:
                        thread = threading.Thread(
                            target=self._consumer_loop,
                            args=(topic_name, group_id, 
                                  partition.partition_id,
                                  handlers[0],  # First handler
                                  poll_interval),
                            daemon=True,
                            name=f"consumer-{group_id}-p{partition.partition_id}"
                        )
                        self._consumer_threads.append(thread)
                        thread.start()
    
    def _consumer_loop(self, topic_name: str, group_id: str,
                       partition_id: int, handler_info: Dict,
                       poll_interval: float):
        """Background loop that polls partition and processes events"""
        handler = handler_info["handler"]
        max_retries = handler_info["max_retries"]
        topic = self.topics[topic_name]
        partition = topic.partitions[partition_id]
        
        while self._running:
            # Get current offset for this consumer group + partition
            last_offset = self.offset_tracker.get_offset(
                group_id, partition_id
            )
            
            # Read new events
            events = partition.read(
                from_offset=last_offset + 1, max_count=10
            )
            
            for offset, event in events:
                success = False
                attempts = 0
                last_error = ""
                
                while attempts < max_retries and not success:
                    attempts += 1
                    try:
                        result = handler(event)
                        if result:
                            success = True
                        else:
                            last_error = "Handler returned False"
                    except Exception as e:
                        last_error = str(e)
                        time.sleep(0.01 * attempts)  # Brief backoff
                
                if success:
                    # Commit offset
                    self.offset_tracker.commit(
                        group_id, partition_id, offset
                    )
                    with self._lock:
                        self.total_delivered += 1
                else:
                    # Move to DLQ
                    self.dlq.add(event, group_id, last_error, attempts)
                    # Skip this event — commit offset to move on
                    self.offset_tracker.commit(
                        group_id, partition_id, offset
                    )
            
            time.sleep(poll_interval)
    
    def stop_consuming(self):
        """Stop all consumer threads"""
        self._running = False
        for thread in self._consumer_threads:
            thread.join(timeout=2)
        self._consumer_threads.clear()
    
    def get_consumer_lag(self, group_id: str,
                         topic_name: str) -> Dict[int, int]:
        """Get lag per partition for a consumer group"""
        if topic_name not in self.topics:
            return {}
        return self.offset_tracker.calculate_lag(
            group_id, self.topics[topic_name]
        )
    
    def get_stats(self) -> Dict:
        topic_stats = {
            name: topic.get_stats()
            for name, topic in self.topics.items()
        }
        return {
            "total_published": self.total_published,
            "total_delivered": self.total_delivered,
            "dlq_size": self.dlq.size,
            "topics": topic_stats,
            "subscriber_groups": {
                topic: list(groups.keys())
                for topic, groups in self._subscribers.items()
            }
        }
    
    def print_publish_log(self, last_n: int = 20):
        print(f"\n  {'='*75}")
        print(f"  EVENT PUBLISH LOG (last {last_n})")
        print(f"  {'='*75}")
        print(f"  {'Time':<10} {'Topic':<12} {'P#':<4} "
              f"{'Offset':<8} {'Type':<22} {'AggID':<15} {'EventID'}")
        print(f"  {'-'*10} {'-'*12} {'-'*4} "
              f"{'-'*8} {'-'*22} {'-'*15} {'-'*8}")
        with self._lock:
            for entry in self.publish_log[-last_n:]:
                print(f"  {entry['time']:<10} "
                      f"{entry['topic']:<12} "
                      f"P{entry['partition']:<3} "
                      f"{entry['offset']:<8} "
                      f"{entry['event_type']:<22} "
                      f"{entry['aggregate_id']:<15} "
                      f"{entry['event_id']}")
        print(f"  {'='*75}")
    
    def print_dashboard(self):
        stats = self.get_stats()
        print(f"\n  {'='*65}")
        print(f"  EVENT BUS DASHBOARD")
        print(f"  {'='*65}")
        print(f"  Published: {stats['total_published']} | "
              f"Delivered: {stats['total_delivered']} | "
              f"DLQ: {stats['dlq_size']}")
        
        for name, ts in stats['topics'].items():
            print(f"\n  Topic: '{name}'")
            print(f"    Partitions: {ts['partitions']} | "
                  f"Total events: {ts['total_events']}")
            print(f"    Partition sizes: {ts['partition_sizes']}")
            print(f"    Latest offsets: {ts['latest_offsets']}")
            
            groups = stats['subscriber_groups'].get(name, [])
            for group in groups:
                lag = self.get_consumer_lag(group, name)
                total_lag = sum(lag.values())
                print(f"    Group '{group}': lag={lag} total={total_lag}")
        
        print(f"  {'='*65}")

class EventStore:
    """
    Append-only store for event sourcing.
    
    Key difference from Event Bus:
    → Event Bus = delivery mechanism (transit)
    → Event Store = permanent storage (persistence)
    
    Events are stored per AGGREGATE (entity):
    → All events for order "ORD-123" stored together
    → Replay them to reconstruct current state
    → Snapshot optimization for long event streams
    """
    
    def __init__(self):
        # {aggregate_id: [(sequence_number, event)]}

class IdempotencyTracker:
    """
    Tracks which events have been processed by this consumer.
    
    WHY THIS IS CRITICAL:
    → At-least-once delivery means consumer MAY receive duplicates
    → Consumer crashes after processing but before committing offset
    → On restart, same event delivered again
    → Without idempotency: duplicate charges, duplicate shipments
    → With idempotency: "already processed, skip" → safe
    
    Three strategies implemented:
    1. Event ID tracking (general purpose)
    2. Idempotent operations (SET not INCREMENT)
    3. Conditional writes (version check)
    """

    def __init__(self, max_tracked: int = 100000):
        self.processed_ids: OrderedDict = OrderedDict()
        self.max_tracked = max_tracked
        self._lock = threading.Lock()
        self.duplicates_caught = 0
        self.total_checked = 0

    def is_duplicate(self, event_id: str) -> bool:
        """Check if this event was already processed"""
        with self._lock:
            self.total_checked += 1
            if event_id in self.processed_ids:
                self.duplicates_caught += 1
                return True
            return False

    def mark_processed(self, event_id: str):
        """Record that this event has been processed"""
        with self._lock:
            self.processed_ids[event_id] = time.time()
            # Evict oldest entries if over capacity
            while len(self.processed_ids) > self.max_tracked:
                self.processed_ids.popitem(last=False)

    def get_stats(self) -> Dict:
        return {
            "tracked": len(self.processed_ids),
            "total_checked": self.total_checked,
            "duplicates_caught": self.duplicates_caught,
            "duplicate_rate": (
                f"{self.duplicates_caught / max(1, self.total_checked):.2%}"
            )
        }

class EventBus:
    """Central event routing with consumer group management"""

    def __init__(self):
        self.topics: Dict[str, Topic] = {}
        self.offsets: Dict[str, Dict[int, int]] = {}
        # {group_id: {partition_id: offset}}
        self.dlq = DeadLetterQueue()
        self._subscribers: Dict[str, Dict[str, Dict]] = {}
        # {topic: {group: {handler, max_retries}}}
        self._lock = threading.Lock()
        self._running = False
        self._threads: List[threading.Thread] = []
        self.total_published = 0
        self.total_delivered = 0
        self.publish_log: List[Dict] = []

    def create_topic(self, name: str, num_partitions: int = 3):
        if name not in self.topics:
            self.topics[name] = Topic(name, num_partitions)

    def publish(self, topic_name: str, event: Event) -> Tuple[int, int]:
        topic = self.topics[topic_name]
        pid, offset = topic.publish(event)
        with self._lock:
            self.total_published += 1
            self.publish_log.append({
                "time": event.time_str,
                "topic": topic_name,
                "partition": pid,
                "offset": offset,
                "type": event.event_type,
                "agg": event.aggregate_id,
                "eid": event.event_id[:8]
            })
        return pid, offset

    def subscribe(self, topic_name: str, group_id: str,
                  handler: Callable, max_retries: int = 3):
        with self._lock:
            if topic_name not in self._subscribers:
                self._subscribers[topic_name] = {}
            self._subscribers[topic_name][group_id] = {
                "handler": handler,
                "max_retries": max_retries
            }

    def _get_offset(self, group_id: str, pid: int) -> int:
        return self.offsets.get(group_id, {}).get(pid, -1)

    def _commit_offset(self, group_id: str, pid: int, offset: int):
        if group_id not in self.offsets:
            self.offsets[group_id] = {}
        self.offsets[group_id][pid] = offset

    def start_consuming(self, poll_interval: float = 0.05):
        self._running = True
        for topic_name, groups in self._subscribers.items():
            for group_id, info in groups.items():
                topic = self.topics[topic_name]
                for partition in topic.partitions:
                    t = threading.Thread(
                        target=self._consume_loop,
                        args=(topic_name, group_id,
                              partition.partition_id,
                              info["handler"],
                              info["max_retries"],
                              poll_interval),
                        daemon=True
                    )
                    self._threads.append(t)
                    t.start()

    def _consume_loop(self, topic_name, group_id, pid,
                      handler, max_retries, poll_interval):
        topic = self.topics[topic_name]
        partition = topic.partitions[pid]

        while self._running:
            last_off = self._get_offset(group_id, pid)
            events = partition.read(last_off + 1, max_count=10)

            for offset, event in events:
                success = False
                attempts = 0
                error_msg = ""

                while attempts < max_retries and not success:
                    attempts += 1
                    try:
                        result = handler(event)
                        success = bool(result)
                        if not success:
                            error_msg = "Handler returned False"
                    except Exception as e:
                        error_msg = str(e)
                        time.sleep(0.01 * attempts)

                if success:
                    self._commit_offset(group_id, pid, offset)
                    with self._lock:
                        self.total_delivered += 1
                else:
                    self.dlq.add(event, group_id, error_msg, attempts)
                    self._commit_offset(group_id, pid, offset)

            time.sleep(poll_interval)

    def stop_consuming(self):
        self._running = False
        for t in self._threads:
            t.join(timeout=2)
        self._threads.clear()

    def get_lag(self, group_id: str, topic_name: str) -> Dict:
        topic = self.topics.get(topic_name)
        if not topic:
            return {}
        lag = {}
        for p in topic.partitions:
            committed = self._get_offset(group_id, p.partition_id)
            latest = p.latest_offset()
            lag[p.partition_id] = max(0, latest - committed)
        return lag

    def print_publish_log(self, last_n: int = 25):
        print(f"\n  {'='*80}")
        print(f"  EVENT PUBLISH LOG")
        print(f"  {'='*80}")
        print(f"  {'Time':<10} {'Topic':<12} {'P#':<4} {'Off':<6} "

                      f"P{e['partition']:<3} {e['offset']:<6} "
                      f"{e['type']:<22} {e['agg']:<15} {e['eid']}")
        print(f"  {'='*80}")

    def print_dashboard(self):

class TransactionalOutbox:
    """
    Solves the dual-write problem:
    
    PROBLEM:
    → Service writes to DB AND publishes event
    → DB write succeeds, event publish fails → inconsistency
    → Event published, DB write fails → inconsistency
    
    SOLUTION (Outbox Pattern):
    1. Write business data + event to SAME database (one transaction)
    2. Separate "relay" process reads outbox table
    3. Relay publishes events to event bus
    4. Relay marks events as published
    
    Guarantee: If business data exists → event WILL be published
    """

    def __init__(self, event_bus: EventBus):
        self.event_bus = event_bus
        self._outbox: List[Dict] = []
        self._lock = threading.Lock()
        self._relay_running = False
        self._relay_thread: Optional[threading.Thread] = None
        self.total_relayed = 0

    def write_with_event(self, db_operation: Callable,
                         topic: str, event: Event) -> bool:
        """
        Atomically execute DB operation AND store event in outbox.
        In production: both happen in same DB transaction.
        """
        with self._lock:
            try:
                # Step 1: Execute business logic (DB write)
                db_result = db_operation()
                if not db_result:
                    return False

                # Step 2: Store event in outbox (same "transaction")
                self._outbox.append({
                    "topic": topic,
                    "event": event,
                    "status": "pending",
                    "created_at": time.time()
                })
                return True
            except Exception as e:
                # Both rolled back (simulated)
                return False

    def start_relay(self, interval: float = 0.1):
        """Start background relay that publishes outbox events"""
        self._relay_running = True
        self._relay_thread = threading.Thread(
            target=self._relay_loop,
            args=(interval,),
            daemon=True
        )
        self._relay_thread.start()

    def _relay_loop(self, interval: float):
        while self._relay_running:
            with self._lock:
                pending = [
                    entry for entry in self._outbox
                    if entry["status"] == "pending"
                ]

            for entry in pending:
                try:
                    self.event_bus.publish(
                        entry["topic"], entry["event"]
                    )
                    entry["status"] = "published"
                    entry["published_at"] = time.time()
                    self.total_relayed += 1
                except Exception:
                    pass  # Will retry next cycle

            time.sleep(interval)

    def stop_relay(self):
        self._relay_running = False
        if self._relay_thread:
            self._relay_thread.join(timeout=2)

    def get_stats(self) -> Dict:
        with self._lock:
            pending = sum(1 for e in self._outbox
                          if e["status"] == "pending")
            published = sum(1 for e in self._outbox
                            if e["status"] == "published")
            return {
                "total": len(self._outbox),
                "pending": pending,
                "published": published,
                "relayed": self.total_relayed
            }

class InventoryService:
    """
    Consumes: ORDER_PLACED → reserve stock
    Consumes: ORDER_CANCELLED → release stock
    Produces: STOCK_RESERVED, STOCK_RELEASED, STOCK_DEPLETED
    """

    def __init__(self, event_bus: EventBus):
        self.event_bus = event_bus
        self.stock: Dict[str, int] = {}
        self.reservations: Dict[str, Dict] = {}
        self.idempotency = IdempotencyTracker()
        self._lock = threading.Lock()
        self.processed_events: List[Dict] = []

    def initialize_stock(self, products: Dict[str, int]):
        self.stock = dict(products)

    def handle_event(self, event: Event) -> bool:
        # Idempotency check
        if self.idempotency.is_duplicate(event.event_id):
            return True  # Already processed — safe to skip

        try:
            if event.event_type == EventType.ORDER_PLACED.value:
                return self._handle_order_placed(event)
            elif event.event_type == EventType.ORDER_CANCELLED.value:
                return self._handle_order_cancelled(event)
            return True  # Ignore unrelated events
        except Exception as e:
            return False

    def _handle_order_placed(self, event: Event) -> bool:
        order_id = event.aggregate_id
        items = event.payload.get("items", [])

        with self._lock:
            # Check all items available
            for item in items:
                product = item["product_id"]
                qty = item["quantity"]
                available = self.stock.get(product, 0)
                if available < qty:
                    # Publish stock depleted event
                    depleted_event = Event(
                        event_type=EventType.STOCK_DEPLETED.value,
                        aggregate_id=product,
                        source="inventory-service",
                        correlation_id=event.correlation_id,
                        causation_id=event.event_id,
                        payload={"product_id": product,
                                 "available": available,
                                 "requested": qty,
                                 "order_id": order_id}
                    )
                    self.event_bus.publish("inventory", depleted_event)
                    self.idempotency.mark_processed(event.event_id)
                    self._log("STOCK_DEPLETED", order_id, product)
                    return True

            # Reserve stock for all items
            for item in items:
                product = item["product_id"]
                qty = item["quantity"]
                self.stock[product] -= qty

            self.reservations[order_id] = {
                "items": items,
                "reserved_at": time.time()
            }

            # Publish stock reserved event
            reserved_event = Event(
                event_type=EventType.STOCK_RESERVED.value,
                aggregate_id=order_id,
                source="inventory-service",
                correlation_id=event.correlation_id,
                causation_id=event.event_id,
                payload={"order_id": order_id, "items": items}
            )
            self.event_bus.publish("inventory", reserved_event)

        self.idempotency.mark_processed(event.event_id)
        self._log("RESERVED", order_id,
                  str([i["product_id"] for i in items]))
        return True

    def _handle_order_cancelled(self, event: Event) -> bool:
        order_id = event.aggregate_id

        with self._lock:
            if order_id in self.reservations:
                items = self.reservations[order_id]["items"]
                for item in items:
                    self.stock[item["product_id"]] += item["quantity"]
                del self.reservations[order_id]

                released_event = Event(
                    event_type=EventType.STOCK_RELEASED.value,
                    aggregate_id=order_id,
                    source="inventory-service",
                    correlation_id=event.correlation_id,
                    causation_id=event.event_id,
                    payload={"order_id": order_id, "items": items}
                )
                self.event_bus.publish("inventory", released_event)

        self.idempotency.mark_processed(event.event_id)
        self._log("RELEASED", order_id, "stock returned")
        return True

    def _log(self, action: str, order_id: str, details: str):
        self.processed_events.append({
            "time": time.strftime("%H:%M:%S"),
            "action": action,
            "order_id": order_id,
            "details": details
        })

    def print_state(self):
        print(f"\n  INVENTORY SERVICE STATE:")
        print(f"    Stock: {self.stock}")
        print(f"    Reservations: {len(self.reservations)}")
        print(f"    Idempotency: {self.idempotency.get_stats()}")

class AnalyticsService:
    """
    Consumes: ALL events from ALL topics
    Builds real-time counters and aggregations
    No side effects — pure data collection
    """

    def __init__(self):
        self.event_counts: Dict[str, int] = defaultdict(int)
        self.revenue: float = 0
        self.orders_by_status: Dict[str, int] = defaultdict(int)
        self.events_per_second: List[Tuple[float, int]] = []
        self.idempotency = IdempotencyTracker()
        self._lock = threading.Lock()
        self._window_start = time.time()
        self._window_count = 0

    def handle_event(self, event: Event) -> bool:
        if self.idempotency.is_duplicate(event.event_id):
            return True

        with self._lock:
            self.event_counts[event.event_type] += 1

            if event.event_type == EventType.ORDER_PLACED.value:
                self.orders_by_status["placed"] += 1
                total = event.payload.get("total", 0)
                self.revenue += total

            elif event.event_type == EventType.ORDER_CANCELLED.value:
                self.orders_by_status["cancelled"] += 1
                total = event.payload.get("total", 0)
                self.revenue -= total

            elif event.event_type == EventType.ORDER_SHIPPED.value:
                self.orders_by_status["shipped"] += 1

            elif event.event_type == EventType.ORDER_DELIVERED.value:
                self.orders_by_status["delivered"] += 1

            # Track throughput
            self._window_count += 1
            now = time.time()
            if now - self._window_start >= 1.0:
                self.events_per_second.append(
                    (self._window_start, self._window_count)
                )
                self._window_start = now
                self._window_count = 0

        self.idempotency.mark_processed(event.event_id)
        return True

    def print_dashboard(self):
        with self._lock:
            print(f"\n  {'='*55}")
            print(f"  ANALYTICS DASHBOARD")
            print(f"  {'='*55}")
            print(f"  Revenue: ${self.revenue:,.2f}")
            print(f"  Orders: {dict(self.orders_by_status)}")
            print(f"  Event counts:")
            for etype, count in sorted(self.event_counts.items()):
                print(f"    {etype:<30} {count}")
            if self.events_per_second:
                avg_eps = sum(c for _, c in self.events_per_second) / len(
                    self.events_per_second)
                print(f"  Avg events/sec: {avg_eps:.1f}")
            print(f"  Idempotency: {self.idempotency.get_stats()}")
            print(f"  {'='*55}")

class OrderAggregate:
    """
    Event-sourced order — state rebuilt from events.
    
    Instead of storing current state directly:
    → Store sequence of events (facts that happened)
    → Rebuild state by replaying events
    → Complete audit trail for free
    → Can time-travel to any point
    
    This is the Event Sourcing pattern from the theory section.
    """

    def __init__(self, order_id: str):
        self.order_id = order_id
        self.status = "new"
        self.items: List[Dict] = []
        self.total: float = 0
        self.customer_email: str = ""
        self.customer_id: str = ""
        self.tracking_id: str = ""
        self.version: int = -1
        self.events_applied: int = 0

    def apply_event(self, event: Event):
        """Apply a single event to update state"""
        if event.event_type == EventType.ORDER_PLACED.value:
            self.status = "placed"
            self.items = event.payload.get("items", [])
            self.total = event.payload.get("total", 0)
            self.customer_email = event.payload.get("customer_email", "")
            self.customer_id = event.payload.get("customer_id", "")

        elif event.event_type == EventType.ORDER_CONFIRMED.value:
            self.status = "confirmed"

        elif event.event_type == EventType.ORDER_SHIPPED.value:
            self.status = "shipped"
            self.tracking_id = event.payload.get("tracking_id", "")

        elif event.event_type == EventType.ORDER_DELIVERED.value:
            self.status = "delivered"

        elif event.event_type == EventType.ORDER_CANCELLED.value:
            self.status = "cancelled"

        self.events_applied += 1
        self.version += 1

    def get_state(self) -> Dict:
        return {
            "order_id": self.order_id,
            "status": self.status,
            "items": self.items,
            "total": self.total,
            "customer_email": self.customer_email,
            "tracking_id": self.tracking_id,
            "version": self.version,
            "events_applied": self.events_applied
        }

    @staticmethod
    def rebuild_from_events(order_id: str,
                            events: List[Event]) -> 'OrderAggregate':
        """Reconstruct order state by replaying all events"""
        order = OrderAggregate(order_id)
        for event in events:
            order.apply_event(event)
        return order

    @staticmethod
    def rebuild_from_snapshot(order_id: str, snapshot: Dict,
                              events_after: List[Event]
                              ) -> 'OrderAggregate':
        """Reconstruct from snapshot + events after snapshot"""
        order = OrderAggregate(order_id)
        # Apply snapshot state
        order.status = snapshot.get("status", "new")
        order.items = snapshot.get("items", [])
        order.total = snapshot.get("total", 0)
        order.customer_email = snapshot.get("customer_email", "")
        order.tracking_id = snapshot.get("tracking_id", "")
        order.version = snapshot.get("version", -1)
        order.events_applied = snapshot.get("events_applied", 0)
        # Apply subsequent events
        for event in events_after:
            order.apply_event(event)
        return order

class OrderService:
    """
    The order service that:
    1. Creates orders (writes to its DB)
    2. Publishes events via transactional outbox
    3. Uses event sourcing for order state
    """

    def __init__(self, event_bus: EventBus, event_store: EventStore):
        self.event_bus = event_bus
        self.event_store = event_store
        self.outbox = TransactionalOutbox(event_bus)
        self.orders_db: Dict[str, Dict] = {}  # Simulated database
        self._lock = threading.Lock()

    def place_order(self, customer_id: str, customer_email: str,
                    items: List[Dict]) -> Dict:
        """Place a new order — produces OrderPlaced event"""
        order_id = f"ORD-{uuid.uuid4().hex[:8].upper()}"
        total = sum(item["price"] * item["quantity"] for item in items)

        correlation_id = str(uuid.uuid4())

        event = Event(
            event_type=EventType.ORDER_PLACED.value,
            aggregate_id=order_id,
            source="order-service",
            correlation_id=correlation_id,
            payload={
                "customer_id": customer_id,
                "customer_email": customer_email,
                "items": items,
                "total": total
            }
        )

        # Transactional outbox: DB write + event in one "transaction"
        def db_write():
            with self._lock:
                self.orders_db[order_id] = {
                    "order_id": order_id,
                    "status": "placed",
                    "customer_id": customer_id,
                    "total": total,
                    "created_at": time.strftime("%H:%M:%S")
                }
            return True

        success = self.outbox.write_with_event(
            db_write, "orders", event
        )

        # Also store in event store (for event sourcing)
        self.event_store.append(order_id, event)

        if success:
            return {"success": True, "order_id": order_id,
                    "total": total, "correlation_id": correlation_id}
        return {"success": False, "error": "Failed to place order"}

    def ship_order(self, order_id: str, tracking_id: str) -> Dict:
        """Ship an order — produces OrderShipped event"""
        if order_id not in self.orders_db:
            return {"success": False, "error": "Order not found"}

        event = Event(
            event_type=EventType.ORDER_SHIPPED.value,
            aggregate_id=order_id,
            source="order-service",
            payload={
                "tracking_id": tracking_id,
                "customer_email": self.orders_db[order_id].get(
                    "customer_id", "") + "@email.com"
            }
        )

        def db_update():
            with self._lock:
                self.orders_db[order_id]["status"] = "shipped"
            return True

        self.outbox.write_with_event(db_update, "orders", event)
        self.event_store.append(order_id, event)

        return {"success": True, "order_id": order_id,
                "tracking_id": tracking_id}

    def cancel_order(self, order_id: str, reason: str = "") -> Dict:
        """Cancel an order — produces OrderCancelled event"""
        if order_id not in self.orders_db:
            return {"success": False, "error": "Order not found"}

        event = Event(
            event_type=EventType.ORDER_CANCELLED.value,
            aggregate_id=order_id,
            source="order-service",
            payload={
                "reason": reason,
                "total": self.orders_db[order_id].get("total", 0)
            }
        )

        def db_update():
            with self._lock:
                self.orders_db[order_id]["status"] = "cancelled"
            return True

        self.outbox.write_with_event(db_update, "orders", event)
        self.event_store.append(order_id, event)

        return {"success": True, "order_id": order_id}

            time.sleep(delays.get(group_name, 0.01))
            return True
        return handler

    # Subscribe three groups to same topic
    for group in ["analytics", "notifications", "audit-log"]:
        bus.subscribe("user-actions", group, make_handler(group))
        print(f"  Subscribed group '{group}' to 'user-actions'")

    # Publish events
    print(f"\n  --- Publishing 10 user action events ---")
    for i in range(10):
        event = Event(
            event_type=random.choice([
                "user.clicked", "user.searched",
                "user.purchased", "user.viewed"
            ]),
            aggregate_id=f"user-{random.randint(1,5):03d}",
            source="web-app",
            payload={"action_number": i}
        )
        bus.publish("user-actions", event)

    # Start consuming
    bus.start_consuming(poll_interval=0.05)
    time.sleep(2.0)
    bus.stop_consuming()

    # Results
    print(f"\n  --- Events received per group ---")
    for group, events in group_results.items():
        print(f"  '{group}': received {len(events)} events")
        for e in events[:3]:
            print(f"    → {e['event_type']} | {e['aggregate']}")
        if len(events) > 3:
            print(f"    → ... and {len(events)-3} more")

    total_unique_events = 10
    print(f"\n  ✅ Published {total_unique_events} events")
    print(f"  ✅ EACH consumer group received them independently")
    print(f"  ✅ Total deliveries: "
          f"{sum(len(v) for v in group_results.values())} "
          f"(events × groups)")

🌉 BRIDGE TO PROJECT 30

RIGHT NOW:
  → We have events flowing between services
  → Each service maintains its OWN data store
  → OrderService stores orders in its DB
  → InventoryService stores stock in its DB
  → AnalyticsService aggregates counters
  → NotificationService tracks sent messages

THE PROBLEM THIS CREATES:
  → OrderService's DB is OPTIMIZED FOR WRITES
     - Normalized tables, foreign keys, ACID transactions
     - Fast inserts: "Place order" → INSERT into orders table
  
  → But READS have completely different needs:
     - Customer dashboard: "Show me my orders with status, items, 
       tracking, and estimated delivery"
     - Admin dashboard: "Show me revenue by category, 
       top customers, inventory turnover"
     - Search: "Find orders by customer name, date range, 
       product, status"
  
  → Serving these reads from the WRITE-optimized DB:
     - Requires complex JOINs across 5-10 tables
     - Queries take 200-500ms (too slow for dashboards)
     - Heavy read queries compete with writes for DB resources
     - Adding indexes for reads slows down writes
     - ONE schema cannot optimize for BOTH patterns

  → This is the READ-WRITE IMPEDANCE MISMATCH

CONCRETE EXAMPLE:
  Write path (fast, simple):
    INSERT INTO orders (id, customer_id, total, status) VALUES (...)
    → 1ms, single table, no joins

  Read path (slow, complex):
    SELECT o.*, c.name, c.email, 
           GROUP_CONCAT(oi.product_name), 
           s.tracking_id, s.estimated_delivery,
           p.status as payment_status
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    JOIN order_items oi ON o.id = oi.order_id
    LEFT JOIN shipments s ON o.id = s.order_id
    LEFT JOIN payments p ON o.id = p.order_id
    WHERE o.customer_id = 123
    ORDER BY o.created_at DESC
    → 200ms, 5 table join, index contention with writes

WHAT PROJECT 30 SOLVES:
  → CQRS: Command Query Responsibility Segregation
  → SEPARATE the write model from the read model
  → Write side: normalized, optimized for inserts/updates
  → Read side: denormalized, pre-joined, optimized for queries
  → Events (Project 29) SYNC the read model from write events
  
  The flow:
  1. Command: "Place Order" → Write to normalized DB
  2. Event: "OrderPlaced" published (Project 29)
  3. Read Model Updater consumes event
  4. Updates DENORMALIZED read-optimized view
  5. Query: "Show my orders" → reads from pre-joined view → 2ms

  Write DB:                    Read DB:
  ┌─────────────┐              ┌──────────────────────────┐
  │ orders      │              │ order_dashboard_view     │
  │ order_items │   Events →   │ (pre-joined, denormalized│
  │ customers   │   ────────→  │  customer_name, items,   │
  │ shipments   │              │  tracking, status, total) │
  │ payments    │              └──────────────────────────┘
  └─────────────┘              
  Normalized                    Denormalized
  5 tables                      1 table
  INSERT: 1ms                   SELECT: 2ms
  JOIN query: 200ms             No joins needed

THE KEY INSIGHT:
  → Project 29 gave us the EVENT PIPELINE between services
  → Project 30 uses that SAME pipeline to build READ-OPTIMIZED views
  → Events are the BRIDGE between write-side and read-side
  → This completes the modern data architecture pattern:
     Commands → Write Store → Events → Read Store → Queries
  → The read store is called a "Materialized View" or "Projection"
  → Same concept as Event Sourcing projections (Project 29)
     but applied at the ARCHITECTURE level, not just one service

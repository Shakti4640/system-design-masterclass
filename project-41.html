🎯 HYPOTHETICAL PROBLEM

You are building a search engine like Google, Elasticsearch, or Algolia

The situation:
→ 10 BILLION web pages / documents to search through
→ Each document: average 5,000 words
→ Total raw text: ~50 TERABYTES
→ 100,000 search queries per second (QPS)
→ Each query must return top-10 results in <200ms
→ Documents are added/updated at 50,000 per minute
→ Users expect:
    "machine learning engineer" → exact phrase match
    "machne lerning" → typo-tolerant results
    "ML engineer" → synonym-aware
    "python developer San Francisco remote" → multi-term relevance ranking

WHAT GOES WRONG without proper design:

  Scenario A — FULL TABLE SCAN:
    → Store 10B documents in a database
    → Query: SELECT * FROM docs WHERE text LIKE '%machine learning%'
    → Scans ALL 10B rows, checking each one
    → At 1ms per row = 10 MILLION seconds = 115 DAYS per query
    → Obviously impossible

  Scenario B — SINGLE INVERTED INDEX:
    → Build an index: word → list of document IDs
    → "machine" appears in 500 million documents
    → "learning" appears in 300 million documents
    → Intersection: still millions of documents to score and rank
    → Single machine: 16GB RAM for index, but index is 2TB
    → Doesn't fit. Even if it did, scoring millions takes seconds.

  Scenario C — INDEX ON ONE MACHINE:
    → Somehow fit the index on one powerful machine
    → 100,000 QPS × each query reads ~50MB of posting lists
    → That's 5 TB/sec of I/O from memory
    → Single machine memory bandwidth: ~50 GB/sec
    → 100x overloaded → queries timeout

  Scenario D — STALE INDEX:
    → Build index once, serve queries from it
    → 50,000 new documents per minute
    → After 1 hour: 3 million documents not searchable
    → News article published 5 minutes ago? Not found.
    → "Your search engine is always behind"

  Scenario E — RELEVANCE DISASTER:
    → Return documents that CONTAIN the words
    → But rank them randomly or by recency only
    → "machine learning" query returns a grocery list 
      that mentions "machine wash" and "learning to cook"
    → Users leave after 3 bad searches

THE REAL CHALLENGE:
→ How do you search 10 BILLION documents
→ In under 200ms per query
→ At 100,000 queries per second
→ With intelligent relevance ranking
→ While continuously indexing new content
→ With typo tolerance and synonym support
→ Distributed across hundreds of machines

🔍 HOW TO APPROACH IT

Decompose into 7 DISTINCT sub-problems:

SUB-PROBLEM 1: INVERTED INDEX
  → The CORE data structure that makes search possible
  → Instead of: document → words (forward index)
  → Build: word → documents (inverted index)
  → Like the index at the back of a textbook
  → "machine" → [doc_42, doc_789, doc_1234, ...]

SUB-PROBLEM 2: RELEVANCE RANKING
  → Not all matches are equal
  → TF-IDF: Term Frequency × Inverse Document Frequency
  → BM25: improved TF-IDF with length normalization
  → "machine learning" in a 100-word doc about ML
    ranks higher than in a 10,000-word doc that mentions it once

SUB-PROBLEM 3: INDEX DISTRIBUTION (Sharding)
  → 2TB index doesn't fit on one machine
  → Split across N machines
  → Two strategies: document-partitioned vs term-partitioned
  → Query must hit all shards, merge results
  → Already learned sharding in Project 18 — apply it here

SUB-PROBLEM 4: QUERY PROCESSING
  → Parse user query into terms
  → Look up each term in inverted index
  → Intersect/union posting lists
  → Score each candidate document
  → Return top-K by score

SUB-PROBLEM 5: CRAWL & INDEX PIPELINE
  → Discover new documents (web crawler)
  → Parse and extract text
  → Tokenize, normalize, stem words
  → Update inverted index
  → Near-real-time: document searchable within minutes
  → Uses task scheduler concepts from Project 36

SUB-PROBLEM 6: QUERY ENHANCEMENT
  → Tokenization: "machine-learning" → ["machine", "learning"]
  → Stemming: "running" → "run", "engineers" → "engineer"
  → Typo tolerance: "machne" → "machine" (edit distance)
  → Synonyms: "ML" → "machine learning"
  → Phrase matching: "machine learning" as exact phrase

SUB-PROBLEM 7: CACHING & OPTIMIZATION
  → Popular queries cached (Project 5, 33)
  → Query result cache: "machine learning" → cached top-10
  → Posting list cache: hot terms in memory
  → Early termination: stop scoring after enough good results

═══════════════════════════════════════════════════════
SOLUTION CORE: THE INVERTED INDEX
═══════════════════════════════════════════════════════

FORWARD INDEX (what we normally think of):
  doc_1 → ["the", "quick", "brown", "fox"]
  doc_2 → ["the", "lazy", "brown", "dog"]
  doc_3 → ["quick", "fox", "jumps", "high"]

  To find "fox": must scan ALL documents → O(N)

INVERTED INDEX (the search engine's secret weapon):
  "the"    → [doc_1, doc_2]
  "quick"  → [doc_1, doc_3]
  "brown"  → [doc_1, doc_2]
  "fox"    → [doc_1, doc_3]
  "lazy"   → [doc_2]
  "dog"    → [doc_2]
  "jumps"  → [doc_3]
  "high"   → [doc_3]

  To find "fox": look up ONE entry → O(1) to find posting list
  Then read posting list → O(K) where K = matching documents

  This is the FUNDAMENTAL insight of all search engines.

POSTING LIST STRUCTURE (what's stored per term):
  "machine" → [
    {doc_id: 42,   term_freq: 5, positions: [12, 45, 78, 102, 230]},
    {doc_id: 789,  term_freq: 3, positions: [1, 56, 89]},
    {doc_id: 1234, term_freq: 1, positions: [445]},
    ...
  ]

  Stored sorted by doc_id for efficient intersection.
  
  Contains:
  → doc_id: which document
  → term_freq: how many times the term appears (for scoring)
  → positions: WHERE in the document (for phrase matching)


═══════════════════════════════════════════════════════
RELEVANCE SCORING: TF-IDF AND BM25
═══════════════════════════════════════════════════════

TF-IDF (Term Frequency × Inverse Document Frequency):

  TF(term, doc) = count of term in doc / total words in doc
    → "machine" appears 5 times in 100-word doc → TF = 0.05
    → Measures: how important is this term IN THIS document?

  IDF(term) = log(total_documents / documents_containing_term)
    → "the" appears in 9 billion of 10 billion docs → IDF = log(10B/9B) ≈ 0.1
    → "quantum" appears in 1 million docs → IDF = log(10B/1M) ≈ 9.2
    → Measures: how RARE is this term across ALL documents?
    → Rare terms are MORE informative

  TF-IDF(term, doc) = TF × IDF
    → "the" in any doc: 0.05 × 0.1 = 0.005 (low — boring word)
    → "quantum" in physics paper: 0.05 × 9.2 = 0.46 (high — meaningful)

  MULTI-TERM QUERY:
    score(doc, query) = Σ TF-IDF(term, doc) for each term in query

BM25 (Better than TF-IDF):

  BM25 adds two improvements:
  1. Term frequency saturation: 5 occurrences isn't 5x better than 1
     → Uses: tf / (tf + k1) where k1 ≈ 1.2
     → tf=1: 0.45, tf=5: 0.81, tf=100: 0.99 (saturates)
  
  2. Document length normalization: long docs shouldn't be penalized
     → Uses: (1 - b + b × doc_length / avg_doc_length) where b ≈ 0.75
     → Short doc with term once = more relevant than long doc with term once

  FORMULA:
    BM25(term, doc) = IDF × (tf × (k1 + 1)) / (tf + k1 × (1 - b + b × dl/avgdl))

  WHERE:
    tf    = term frequency in document
    dl    = document length
    avgdl = average document length across all documents
    k1    = 1.2 (term frequency saturation parameter)
    b     = 0.75 (document length normalization parameter)


═══════════════════════════════════════════════════════
INDEX SHARDING: DOCUMENT-PARTITIONED
═══════════════════════════════════════════════════════

  Two strategies for distributing the inverted index:

  STRATEGY A — DOCUMENT-PARTITIONED (our choice):
    Each shard has a COMPLETE inverted index for a SUBSET of documents
    
    Shard 1: docs 0-3.3B      (full index for those docs)
    Shard 2: docs 3.3B-6.6B   (full index for those docs)
    Shard 3: docs 6.6B-10B    (full index for those docs)
    
    Query "machine learning":
      → Send to ALL 3 shards in parallel
      → Each shard returns its local top-K
      → Merger combines and re-ranks
    
    PROS: Each shard is independent, easy to scale
    CONS: Every query hits every shard (fan-out)

  STRATEGY B — TERM-PARTITIONED:
    Each shard owns a SUBSET of terms for ALL documents
    
    Shard 1: terms A-H   (posting lists for all docs)
    Shard 2: terms I-P
    Shard 3: terms Q-Z
    
    Query "machine learning":
      → "machine" → Shard 2, "learning" → Shard 2
      → Only 1 shard needed!
    
    PROS: Fewer shards per query (sometimes)
    CONS: Hot terms ("the") create hot shards, intersection across shards is complex

  WHY DOCUMENT-PARTITIONED WINS:
    → Simpler to build and maintain
    → Each shard is self-contained (can score locally)
    → Adding new documents: just add to one shard
    → Google, Elasticsearch, Solr all use this approach


═══════════════════════════════════════════════════════
QUERY PROCESSING PIPELINE
═══════════════════════════════════════════════════════

  User types: "Machine Learning Engineer in SF"

  STEP 1: TOKENIZE
    → Split into tokens: ["Machine", "Learning", "Engineer", "in", "SF"]

  STEP 2: NORMALIZE
    → Lowercase: ["machine", "learning", "engineer", "in", "sf"]

  STEP 3: REMOVE STOP WORDS
    → Remove common words: ["machine", "learning", "engineer", "sf"]
    → "in" is too common to be useful

  STEP 4: STEM / LEMMATIZE
    → "engineer" → "engin" (stem)
    → "learning" → "learn"
    → Matches "engineers", "engineering", "learned", etc.

  STEP 5: EXPAND (synonyms)
    → "sf" → "sf" OR "san francisco"
    → "ml" → "ml" OR "machine learning"

  STEP 6: SCATTER TO SHARDS
    → Send processed query to all index shards in parallel
    → Each shard: look up posting lists, intersect, score, return top-K

  STEP 7: GATHER AND MERGE
    → Collect top-K from each shard
    → Merge into global top-K
    → Re-rank if needed (global statistics)

  STEP 8: RETURN RESULTS
    → Top-10 with snippets (highlighted matching text)

📌 WHAT COULD BE DONE

OPTION A: BRUTE FORCE SCAN
  → Check every document for every query
  → O(N × Q) where N = docs, Q = query terms
  → Works for <1000 documents
  → Completely impossible at scale

OPTION B: DATABASE FULL-TEXT INDEX
  → PostgreSQL tsvector / MySQL FULLTEXT
  → Uses inverted index internally
  → Works up to ~10M documents on single machine
  → No distributed query, limited ranking

OPTION C: SINGLE-NODE SEARCH ENGINE
  → Lucene (Java), Whoosh (Python), Tantivy (Rust)
  → Full inverted index with BM25 scoring
  → Works up to ~100M documents per node
  → No distribution, no horizontal scaling

OPTION D: DISTRIBUTED SEARCH (Elasticsearch / Solr)
  → Document-partitioned inverted index
  → Scatter-gather query processing
  → Built-in replication and failover
  → Handles billions of documents
  → This is the production standard

OPTION E: CUSTOM DISTRIBUTED SEARCH (our approach)
  → Build from first principles to understand internals
  → Inverted index + BM25 + document sharding
  → Query parser + scatter-gather + result merger
  → Typo tolerance + synonym expansion
  → Near-real-time indexing pipeline
  → This is what we build in this project

✅ WHAT SHOULD DO

1. BUILD INVERTED INDEX WITH POSITION DATA
   → Store term positions for phrase matching
   → "machine learning" as phrase: "machine" at pos N,
     "learning" at pos N+1 in same document
   → Without positions: can only do bag-of-words matching

2. USE BM25 OVER TF-IDF
   → BM25 handles document length variation better
   → TF-IDF over-rewards long documents with repeated terms
   → BM25 is the standard in Elasticsearch, Lucene, Solr

3. SHARD BY DOCUMENT, NOT BY TERM
   → Each shard is independent, self-contained
   → Can score locally with complete information
   → Adding documents: route to one shard
   → Project 18 sharding applied to search index

4. USE SORTED POSTING LISTS FOR EFFICIENT INTERSECTION
   → Posting lists sorted by doc_id
   → Two-pointer merge for AND queries: O(min(|A|, |B|))
   → Skip lists for even faster intersection
   → Without sorting: O(|A| × |B|) — much worse

5. CACHE POPULAR QUERY RESULTS
   → "weather today" queried 1M times per day
   → Cache result for 5 minutes
   → Project 5, 33 caching applied to search
   → Cache invalidation: on index update affecting cached results

6. IMPLEMENT EARLY TERMINATION
   → Don't score ALL matching documents
   → Score first N candidates, keep top-K
   → If a new candidate can't possibly beat current K-th score → skip
   → Reduces scoring work by 90%+ for common queries

7. TOKENIZE AND STEM CONSISTENTLY
   → Index and query MUST use the SAME tokenizer
   → If index stems "running" → "run"
   → Query must also stem "running" → "run"
   → Mismatch = search doesn't work

8. BUILD SNIPPETS WITH HIGHLIGHT
   → Don't just return document titles
   → Show the RELEVANT portion with matching terms highlighted
   → "...the field of **machine learning** has grown rapidly
     as more **engineers** enter..."

❌ WHAT SHOULD NOT DO

1. NEVER SCAN ALL DOCUMENTS PER QUERY
   → That's what the inverted index prevents
   → Even scanning 1% of 10B docs = 100M scans = too slow

2. NEVER STORE POSTING LISTS UNSORTED
   → Sorted: intersection is O(N) with two pointers
   → Unsorted: intersection is O(N²) — catastrophic at scale
   → Always maintain sorted order by doc_id

3. NEVER SKIP STOP WORD REMOVAL
   → "the" appears in 9B documents
   → Looking up "the" returns 9B posting entries
   → Completely useless for ranking AND wastes I/O
   → UNLESS doing phrase search ("The Beatles" — "the" matters)

4. NEVER USE DIFFERENT TOKENIZERS FOR INDEX AND QUERY
   → Index uses stemmer A: "running" → "run"
   → Query uses stemmer B: "running" → "runn"
   → "run" ≠ "runn" → zero results for a valid query
   → ALWAYS use identical text processing pipeline

5. NEVER RETURN RESULTS WITHOUT RELEVANCE SCORING
   → Returning by doc_id order or insertion order = useless
   → The WHOLE POINT of search is relevance ranking
   → Even a simple TF-IDF is better than no scoring

6. NEVER HOLD THE ENTIRE INDEX IN MEMORY ON ONE MACHINE
   → 10B documents × ~200 bytes per posting = 2TB+ index
   → No single machine has 2TB RAM
   → Must shard across machines (Project 18)

7. NEVER BLOCK QUERIES DURING INDEX UPDATES
   → New documents being indexed shouldn't lock the query path
   → Use segment-based architecture: new data goes to new segment
   → Queries search all segments
   → Background merge combines segments

8. NEVER IGNORE QUERY LATENCY TAIL
   → P50 latency: 50ms (great!)
   → P99 latency: 5,000ms (terrible for 1% of users)
   → Monitor and optimize for tail latency
   → Common cause: one slow shard in scatter-gather

🚫 WHEN NOT TO USE

1. INVERTED INDEX IS NOT FOR:
   → Numeric range queries (use B-Tree — Project 10)
   → Graph traversal (use adjacency list — Project 40)
   → Time-series data (use specialized time-series DB)
   → Exact key-value lookup (use hash map — Project 23)

2. BM25 IS NOT FOR:
   → Semantic search ("car" should match "automobile")
     → Use vector embeddings instead
   → Question answering ("What is the capital of France?")
     → Use NLP models, not keyword matching
   → Image/video search
     → Use content-based features, not text index

3. DOCUMENT-PARTITIONED SHARDING IS NOT FOR:
   → When queries always target specific terms (use term-partitioned)
   → When document count is small (<1M, single node is fine)
   → When queries need global aggregations across ALL documents
     (term-partitioned may be more efficient)

4. FULL-TEXT SEARCH IS NOT FOR:
   → Structured queries (SQL WHERE clauses)
   → Exact match lookups (hash index is O(1))
   → Sorting by non-text fields (use database index)

═══════════════════════════════════════════════════════════
MECHANICS 2: QUERY EXECUTION — TWO-POINTER INTERSECTION
═══════════════════════════════════════════════════════════

  Query: "quick brown fox" (AND semantics)

  Posting lists (sorted by doc_id):
    "quick" → [12, 42, 88, 150, 300]
    "brown" → [7, 42, 55, 88, 200, 300, 450]
    "fox"   → [42, 88, 99, 300]

  TWO-POINTER INTERSECTION ALGORITHM:
    Start with smallest list: "fox" [42, 88, 99, 300]
    
    For each doc_id in smallest list:
      Binary search in other lists:
        42 in "quick"? YES ✅  42 in "brown"? YES ✅  → MATCH
        88 in "quick"? YES ✅  88 in "brown"? YES ✅  → MATCH
        99 in "quick"? NO ❌                          → SKIP
        300 in "quick"? YES ✅ 300 in "brown"? YES ✅ → MATCH
    
    Result: docs [42, 88, 300] contain ALL three terms

  COMPLEXITY:
    → Start with shortest posting list: length S
    → Binary search in K other lists of length L
    → Total: O(S × K × log(L))
    → Much better than scanning all documents!

  SKIP LIST OPTIMIZATION:
    Instead of scanning posting list linearly:
    ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
    │ 7│12│42│55│88│99│150│200│300│450│  ← posting list
    └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
         ↑              ↑              ↑
       skip level 1: [12, 88, 300]
                ↑                   ↑
       skip level 2: [12, 300]
    
    Looking for doc 200:
    Level 2: 12 < 200, 300 > 200 → drop to level 1 at 12
    Level 1: 12 < 200, 88 < 200, 300 > 200 → drop to level 0 at 88
    Level 0: 88 < 200, 99 < 200, 150 < 200, 200 = 200 → FOUND!
    
    Skipped 4 elements instead of scanning all 10
    O(√N) average with skip lists

MECHANICS 3: BM25 SCORING — STEP BY STEP
═══════════════════════════════════════════════════════════

  Given:
    N = 10,000,000 (total documents)
    avgdl = 500 (average document length in words)
    k1 = 1.2, b = 0.75

  Query: "machine learning"
  Document: doc_42 (length = 200 words, tf_machine = 3, tf_learning = 5)
  
  Documents containing "machine": 500,000
  Documents containing "learning": 800,000

  IDF("machine") = log((N - df + 0.5) / (df + 0.5))
                 = log((10M - 500K + 0.5) / (500K + 0.5))
                 = log(19.0) = 2.944

  IDF("learning") = log((10M - 800K + 0.5) / (800K + 0.5))
                  = log(11.5) = 2.442

  Length normalization for doc_42:
    norm = 1 - b + b × (dl / avgdl)
         = 1 - 0.75 + 0.75 × (200 / 500)
         = 0.25 + 0.30 = 0.55

  BM25("machine", doc_42):
    = IDF × (tf × (k1 + 1)) / (tf + k1 × norm)
    = 2.944 × (3 × 2.2) / (3 + 1.2 × 0.55)
    = 2.944 × 6.6 / 3.66
    = 5.308

  BM25("learning", doc_42):
    = 2.442 × (5 × 2.2) / (5 + 1.2 × 0.55)
    = 2.442 × 11.0 / 5.66
    = 4.745

  TOTAL SCORE for doc_42:
    = BM25("machine") + BM25("learning")
    = 5.308 + 4.745
    = 10.053

  This score is compared against all other matching documents.
  Top-K by score are returned.

MECHANICS 4: SCATTER-GATHER DISTRIBUTED QUERY
═══════════════════════════════════════════════════════════

  Index split across 3 shards:
    Shard 1: docs 0 - 3.3M
    Shard 2: docs 3.3M - 6.6M  
    Shard 3: docs 6.6M - 10M

  Query: "machine learning", top-10 requested

  SCATTER PHASE:
    Query Router sends to ALL shards in parallel:
    ┌───────────┐     ┌─────────┐
    │  Router   │────→│ Shard 1 │ → local top-10 (scored)
    │           │────→│ Shard 2 │ → local top-10 (scored)
    │           │────→│ Shard 3 │ → local top-10 (scored)
    └───────────┘     └─────────┘

  Each shard:
    1. Look up "machine" posting list (local subset)
    2. Look up "learning" posting list (local subset)
    3. Intersect posting lists
    4. Score each matching doc with BM25
    5. Return top-10 by score

  GATHER PHASE:
    Router receives 3 × 10 = 30 results
    Merge-sort by score
    Take global top-10

  LATENCY:
    Scatter: 1-2ms (network fan-out)
    Shard processing: 20-50ms each (in parallel)
    Gather + merge: 1-2ms
    Total: ~25-55ms (dominated by slowest shard)

  THE STRAGGLER PROBLEM:
    → 2 shards respond in 30ms, 1 shard takes 200ms
    → Total latency = 200ms (waiting for slowest)
    → FIX: Hedged requests — send to replica if shard is slow
    → FIX: Tail-at-scale mitigation (Google's approach)

MECHANICS 5: TYPO TOLERANCE — EDIT DISTANCE + N-GRAMS
═══════════════════════════════════════════════════════════

  User types: "machne lerning" (two typos)

  METHOD 1: EDIT DISTANCE (Levenshtein)
    → "machne" vs "machine": distance = 1 (insert 'i')
    → "lerning" vs "learning": distance = 1 (insert 'a')
    → For each query term, find dictionary terms within distance 1-2
    → Expensive: comparing against all 1M unique terms

  METHOD 2: N-GRAM INDEX (faster)
    → Break each term into character n-grams (n=2 or 3):
      "machine" → {"ma", "ac", "ch", "hi", "in", "ne"}
      "machne"  → {"ma", "ac", "ch", "hn", "ne"}
    
    → Build index: ngram → [terms containing this ngram]
      "ma" → ["machine", "manual", "match", ...]
      "ch" → ["machine", "chocolate", "church", ...]
    
    → For query "machne":
      Look up each ngram: "ma", "ac", "ch", "hn", "ne"
      Intersect/count: which terms share the most ngrams?
      "machine" shares 4 of 6 ngrams → likely match!
    
    → Then verify with edit distance (only for top candidates)
    
  METHOD 3: PHONETIC MATCHING (Soundex / Metaphone)
    → "Smith" and "Smyth" sound the same
    → Map words to phonetic codes
    → Match codes instead of spellings

  WE USE: N-gram index for candidate generation + edit distance for verification

MECHANICS 7: NEAR-REAL-TIME INDEXING — SEGMENT ARCHITECTURE
═══════════════════════════════════════════════════════════

  Problem: How to add new documents without blocking queries?

  SOLUTION: Segment-based architecture (Lucene model)

  ┌─────────────────────────────────────────────────────┐
  │  INDEX = collection of IMMUTABLE segments            │
  │                                                     │
  │  Segment 1 (large, old):  500K docs, on disk        │
  │  Segment 2 (medium):      100K docs, on disk        │
  │  Segment 3 (small, new):  10K docs, in memory       │
  │  Segment 4 (tiny, newest): 500 docs, in memory      │
  └─────────────────────────────────────────────────────┘

  NEW DOCUMENT ARRIVES:
    → Added to in-memory buffer (Segment 4)
    → Immediately searchable!
    → When buffer reaches threshold → flush to disk as new segment

  QUERY SEARCHES ALL SEGMENTS:
    → Query hits Segment 1 + 2 + 3 + 4
    → Merge results across segments
    → More segments = more work per query

  BACKGROUND MERGE:
    → Periodically merge small segments into larger ones
    → Segment 3 + 4 → merged Segment 5
    → Reduces segment count → faster queries
    → Like LSM-tree compaction (used in many databases)

  DELETIONS:
    → Mark document as deleted in a "deletion bitset"
    → Filter out deleted docs during query
    → Physically removed during segment merge
    → Similar to CRDT tombstones (Project 39)

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: INDEX-QUERY TOKENIZER MISMATCH
  → Index stems "running" → "run"
  → Query doesn't stem → searches for "running"
  → ZERO results for a perfectly valid query
  → FIX: Share EXACT same tokenizer config between indexer and query parser

  → Remove "to" and "be" from index

  → Shard 1 has 100K docs with "machine", Shard 2 has 500K

  → FIX: Precompute global term statistics, share across shards
  → FIX: Or accept small inaccuracy (Google does this)

  → "the" has a posting list of 9 BILLION entries

  → FIX: Stop words never queried alone

  → 9 of 10 shards respond in 30ms
  → 1 shard is doing garbage collection: responds in 500ms
  → User sees 500ms latency
  → FIX: Hedged requests — after 50ms, send duplicate to replica

  → Query: "a" (matches basically every document)

  → Loading into memory: crash
  → FIX: Streaming intersection (never load full list)

# SECTION 1: TEXT PROCESSING PIPELINE
# Shared between indexer and query parser (MUST be identical)

class TextProcessor:
    """
    Text processing pipeline used by BOTH indexer and query parser.
    Using the same processor for both is CRITICAL —
    mismatch = search doesn't work (see pitfalls).
    """

    STOP_WORDS = frozenset({
        "a", "an", "the", "is", "are", "was", "were", "be", "been",
        "being", "have", "has", "had", "do", "does", "did", "will",
        "would", "could", "should", "may", "might", "can", "shall",
        "in", "on", "at", "to", "for", "of", "with", "by", "from",
        "as", "into", "through", "during", "before", "after",
        "and", "but", "or", "nor", "not", "so", "yet",
        "it", "its", "this", "that", "these", "those",
        "i", "me", "my", "we", "our", "you", "your", "he", "she",
    })

    SYNONYMS = {
        "ml": ["machine learning"],
        "ai": ["artificial intelligence"],
        "js": ["javascript"],
        "py": ["python"],
        "db": ["database"],
        "sf": ["san francisco"],
        "nyc": ["new york city"],
        "dev": ["developer"],
        "eng": ["engineer"],
        "sr": ["senior"],
        "jr": ["junior"],
    }

    @staticmethod
    def tokenize(text: str) -> List[str]:
        """Split text into individual tokens"""
        text = text.lower()
        tokens = re.findall(r'[a-z0-9]+', text)
        return tokens

    @staticmethod
    def simple_stem(word: str) -> str:
        """
        Simplified Porter-like stemming.
        Production: use NLTK PorterStemmer or Snowball.
        """
        if len(word) <= 3:
            return word

        suffixes = [
            ("ational", "ate"), ("tional", "tion"), ("enci", "ence"),
            ("anci", "ance"), ("izer", "ize"), ("ling", "l"),
            ("ation", "ate"), ("ator", "ate"), ("ness", ""),
            ("ment", ""), ("ful", ""), ("ous", ""),
            ("ive", ""), ("ing", ""), ("ers", "er"),
            ("ies", "y"), ("tion", "t"), ("sion", "s"),
            ("able", ""), ("ible", ""), ("ally", "al"),
            ("ly", ""), ("ed", ""), ("es", ""),
            ("er", ""), ("s", ""),
        ]

        for suffix, replacement in suffixes:
            if word.endswith(suffix) and len(word) - len(suffix) >= 2:
                return word[:-len(suffix)] + replacement

        return word

    @classmethod
    def process_for_index(cls, text: str) -> List[Tuple[str, int]]:
        """
        Process text for indexing.
        Returns: [(term, position), ...]
        Keeps position info for phrase matching.
        """
        tokens = cls.tokenize(text)
        result = []
        for pos, token in enumerate(tokens):
            if token not in cls.STOP_WORDS:
                stemmed = cls.simple_stem(token)
                if len(stemmed) >= 2:
                    result.append((stemmed, pos))
        return result

    @classmethod
    def process_for_query(
        cls, query: str, expand_synonyms: bool = True,
        remove_stop_words: bool = True,
    ) -> List[str]:
        """
        Process query text. Must use SAME pipeline as indexing.
        Optionally expands synonyms.
        """
        tokens = cls.tokenize(query)
        processed = []

        for token in tokens:
            if remove_stop_words and token in cls.STOP_WORDS:
                continue

            stemmed = cls.simple_stem(token)
            if len(stemmed) >= 2:
                processed.append(stemmed)

            # Expand synonyms
            if expand_synonyms and token in cls.SYNONYMS:
                for synonym in cls.SYNONYMS[token]:
                    syn_tokens = cls.tokenize(synonym)
                    for st in syn_tokens:
                        if st not in cls.STOP_WORDS:
                            s = cls.simple_stem(st)
                            if len(s) >= 2 and s not in processed:
                                processed.append(s)

        return processed

@dataclass
class Posting:
    """Single entry in a posting list"""
    doc_id: str
    term_frequency: int
    positions: List[int] = field(default_factory=list)

    def __lt__(self, other):
        return self.doc_id < other.doc_id


@dataclass
class DocumentMetadata:
    """Stored per document for scoring and snippets"""
    doc_id: str
    title: str
    length: int          # Number of terms after processing
    raw_text: str        # For snippet generation
    url: str = ""
    indexed_at: float = 0.0


class InvertedIndex:
    """
    Core inverted index with posting lists.
    
    Structure:
      term → sorted list of Posting objects
      Each Posting: (doc_id, term_frequency, positions)
    
    Supports:
      → Term lookup: O(1)
      → Posting list traversal: O(K) where K = matching docs
      → Intersection: O(min(|A|, |B|)) with two-pointer
    """

    def __init__(self, shard_id: str = "main"):
        self.shard_id = shard_id
        self.index: Dict[str, List[Posting]] = defaultdict(list)
        self.documents: Dict[str, DocumentMetadata] = {}
        self.total_docs = 0
        self.total_term_count = 0
        self.avg_doc_length = 0.0
        self.term_doc_freq: Dict[str, int] = defaultdict(int)
        self._lock = threading.Lock()

    def add_document(self, doc_id: str, title: str, text: str, url: str = ""):
        """Index a single document"""
        # Process text
        term_positions = TextProcessor.process_for_index(text)

        if not term_positions:
            return

        doc_length = len(term_positions)

        # Store document metadata
        metadata = DocumentMetadata(
            doc_id=doc_id,
            title=title,
            length=doc_length,
            raw_text=text,
            url=url,
            indexed_at=time.time(),
        )

        # Build term → positions mapping for this document
        term_data = defaultdict(list)
        for term, position in term_positions:
            term_data[term].append(position)

        with self._lock:
            self.documents[doc_id] = metadata
            self.total_docs += 1
            self.total_term_count += doc_length

            # Update average document length
            self.avg_doc_length = self.total_term_count / self.total_docs

            # Add to inverted index
            for term, positions in term_data.items():
                posting = Posting(
                    doc_id=doc_id,
                    term_frequency=len(positions),
                    positions=sorted(positions),
                )

                # Insert in sorted order by doc_id
                posting_list = self.index[term]
                # Binary search for insert position
                lo, hi = 0, len(posting_list)
                while lo < hi:
                    mid = (lo + hi) // 2
                    if posting_list[mid].doc_id < doc_id:
                        lo = mid + 1
                    else:
                        hi = mid
                posting_list.insert(lo, posting)

                # Update document frequency for this term
                self.term_doc_freq[term] += 1

    def get_posting_list(self, term: str) -> List[Posting]:
        """Get the posting list for a term"""
        return self.index.get(term, [])

    def get_doc_frequency(self, term: str) -> int:
        """How many documents contain this term"""
        return self.term_doc_freq.get(term, 0)

    def get_document(self, doc_id: str) -> Optional[DocumentMetadata]:
        return self.documents.get(doc_id)

    def get_stats(self) -> dict:
        with self._lock:
            total_postings = sum(len(pl) for pl in self.index.values())
            return {
                "shard_id": self.shard_id,
                "total_documents": self.total_docs,
                "unique_terms": len(self.index),
                "total_postings": total_postings,
                "avg_doc_length": round(self.avg_doc_length, 1),
                "avg_postings_per_term": round(
                    total_postings / max(len(self.index), 1), 1
                ),
            }

class BM25Scorer:
    """
    BM25 (Best Match 25) scoring algorithm.
    Industry standard for text relevance ranking.
    """

    def __init__(self, k1: float = 1.2, b: float = 0.75):
        self.k1 = k1
        self.b = b

    def compute_idf(self, doc_freq: int, total_docs: int) -> float:
        if doc_freq == 0:
            return 0.0
        return math.log(
            (total_docs - doc_freq + 0.5) / (doc_freq + 0.5) + 1.0
        )

    def score_document(
        self,
        query_terms: List[str],
        doc_id: str,
        index: InvertedIndex,
    ) -> float:
        doc = index.get_document(doc_id)
        if not doc:
            return 0.0

        total_score = 0.0
        doc_length = doc.length
        avg_dl = index.avg_doc_length

        for term in query_terms:
            df = index.get_doc_frequency(term)
            idf = self.compute_idf(df, index.total_docs)

            posting_list = index.get_posting_list(term)
            tf = 0
            for posting in posting_list:
                if posting.doc_id == doc_id:
                    tf = posting.term_frequency
                    break

            if tf == 0:
                continue

            numerator = tf * (self.k1 + 1)
            denominator = tf + self.k1 * (
                1 - self.b + self.b * (doc_length / max(avg_dl, 1))
            )
            term_score = idf * (numerator / denominator)
            total_score += term_score

        return total_score

    def score_batch(
        self,
        query_terms: List[str],
        candidate_doc_ids: Set[str],
        index: InvertedIndex,
        top_k: int = 10,
    ) -> List[Tuple[str, float]]:
        """Score a batch of candidate documents, return top-K"""
        heap = []

        for doc_id in candidate_doc_ids:
            score = self.score_document(query_terms, doc_id, index)
            if score > 0:
                if len(heap) < top_k:
                    heapq.heappush(heap, (score, doc_id))
                elif score > heap[0][0]:
                    heapq.heapreplace(heap, (score, doc_id))

        # Extract sorted results (highest score first)
        results = []
        while heap:
            score, doc_id = heapq.heappop(heap)
            results.append((doc_id, score))
        results.reverse()

        return results

# SECTION 4: POSTING LIST INTERSECTION ENGINE
# Two-pointer merge for AND queries
# ================================================================

class PostingListIntersector:
    """
    Efficient intersection of sorted posting lists.
    
    AND query: document must appear in ALL posting lists
    OR query: document must appear in ANY posting list
    
    Two-pointer intersection: O(min(|A|, |B|))
    """

    @staticmethod
    def intersect_two(
        list_a: List[Posting], list_b: List[Posting]
    ) -> Set[str]:
        """Intersect two sorted posting lists using two-pointer technique"""
        result = set()
        i, j = 0, 0

        while i < len(list_a) and j < len(list_b):
            if list_a[i].doc_id == list_b[j].doc_id:
                result.add(list_a[i].doc_id)
                i += 1
                j += 1
            elif list_a[i].doc_id < list_b[j].doc_id:
                i += 1
            else:
                j += 1

        return result

    @staticmethod
    def intersect_multiple(posting_lists: List[List[Posting]]) -> Set[str]:
        """
        Intersect multiple posting lists.
        Strategy: start with shortest list, intersect progressively.
        This minimizes work because intersection can only shrink.
        """
        if not posting_lists:
            return set()

        if len(posting_lists) == 1:
            return {p.doc_id for p in posting_lists[0]}

        # Sort by list length (shortest first)
        sorted_lists = sorted(posting_lists, key=len)

        # Start with shortest list
        result = {p.doc_id for p in sorted_lists[0]}

        # Progressively intersect with remaining lists
        for posting_list in sorted_lists[1:]:
            list_ids = {p.doc_id for p in posting_list}
            result = result & list_ids

            # Early termination: if result is empty, stop
            if not result:
                return set()

        return result

    @staticmethod
    def union_multiple(posting_lists: List[List[Posting]]) -> Set[str]:
        """Union of multiple posting lists (OR query)"""
        result = set()
        for posting_list in posting_lists:
            for posting in posting_list:
                result.add(posting.doc_id)
        return result

    @staticmethod
    def phrase_match(
        index: InvertedIndex, terms: List[str]
    ) -> Set[str]:
        """
        Find documents where terms appear as an exact phrase.
        Uses position data to verify adjacency.
        
        "machine learning" → "machine" at pos N, "learning" at pos N+1
        """
        if len(terms) < 2:
            postings = index.get_posting_list(terms[0]) if terms else []
            return {p.doc_id for p in postings}

        # Step 1: Get posting lists for all terms
        all_postings = []
        for term in terms:
            pl = index.get_posting_list(term)
            if not pl:
                return set()  # Term not in index → no phrase match
            all_postings.append(pl)

        # Step 2: Find documents containing ALL terms
        candidate_docs = PostingListIntersector.intersect_multiple(all_postings)

        # Step 3: For each candidate, verify position adjacency
        phrase_matches = set()

        for doc_id in candidate_docs:
            # Collect positions for each term in this document
            term_positions = []
            for i, term in enumerate(terms):
                positions = []
                for posting in all_postings[i]:
                    if posting.doc_id == doc_id:
                        positions = posting.positions
                        break
                term_positions.append(positions)

            # Check if there's a sequence of consecutive positions
            if PostingListIntersector._check_phrase_positions(term_positions):
                phrase_matches.add(doc_id)

        return phrase_matches

    @staticmethod
    def _check_phrase_positions(term_positions: List[List[int]]) -> bool:
        """
        Check if there exists a sequence of positions where
        term[0] is at pos P, term[1] at P+1, term[2] at P+2, etc.
        """
        if not term_positions or not term_positions[0]:
            return False

        # For each starting position of the first term
        for start_pos in term_positions[0]:
            found = True
            for offset, positions in enumerate(term_positions[1:], 1):
                expected_pos = start_pos + offset
                if expected_pos not in positions:
                    found = False
                    break
            if found:
                return True

        return False

# SECTION 5: TYPO TOLERANCE ENGINE
# Edit distance + N-gram candidate generation
# ================================================================

class TypoTolerance:
    """
    Handles misspelled queries by:
    1. Building an n-gram index of all known terms
    2. For a misspelled term, find terms sharing n-grams
    3. Verify with edit distance
    4. Return best correction
    """

    def __init__(self, ngram_size: int = 2):
        self.ngram_size = ngram_size
        self.ngram_index: Dict[str, Set[str]] = defaultdict(set)
        self.vocabulary: Set[str] = set()
        self.term_frequencies: Dict[str, int] = defaultdict(int)

    def build_from_index(self, inverted_index: InvertedIndex):
        """Build n-gram index from all terms in the inverted index"""
        for term in inverted_index.index.keys():
            self.add_term(term, inverted_index.get_doc_frequency(term))

    def add_term(self, term: str, frequency: int = 1):
        """Add a term to the vocabulary and n-gram index"""
        self.vocabulary.add(term)
        self.term_frequencies[term] = frequency

        # Generate n-grams
        padded = f"${term}$"  # Boundary markers
        for i in range(len(padded) - self.ngram_size + 1):
            ngram = padded[i:i + self.ngram_size]
            self.ngram_index[ngram].add(term)

    def get_ngrams(self, term: str) -> Set[str]:
        """Generate n-grams for a term"""
        padded = f"${term}$"
        return {
            padded[i:i + self.ngram_size]
            for i in range(len(padded) - self.ngram_size + 1)
        }

    def suggest_correction(
        self, misspelled: str, max_distance: int = 2, top_k: int = 3
    ) -> List[dict]:
        """
        Find the best corrections for a misspelled term.
        
        Step 1: Find candidate terms sharing n-grams (fast filter)
        Step 2: Compute edit distance for candidates (precise check)
        Step 3: Rank by edit distance, then by frequency
        """
        if misspelled in self.vocabulary:
            return [{"term": misspelled, "distance": 0, "frequency": self.term_frequencies[misspelled]}]

        # Step 1: N-gram candidate generation
        query_ngrams = self.get_ngrams(misspelled)
        candidate_counts = defaultdict(int)

        for ngram in query_ngrams:
            for term in self.ngram_index.get(ngram, set()):
                candidate_counts[term] += 1

        # Filter: candidates must share at least 1 n-gram
        # Sort by shared n-gram count (more shared = more likely match)
        candidates = sorted(
            candidate_counts.items(), key=lambda x: x[1], reverse=True
        )[:100]  # Limit candidates for performance

        # Step 2: Compute edit distance
        corrections = []
        for term, shared_count in candidates:
            distance = self._edit_distance(misspelled, term)
            if distance <= max_distance:
                corrections.append({
                    "term": term,
                    "distance": distance,
                    "frequency": self.term_frequencies.get(term, 0),
                    "shared_ngrams": shared_count,
                })

        # Step 3: Rank by distance (lower better), then frequency (higher better)
        corrections.sort(key=lambda x: (x["distance"], -x["frequency"]))

        return corrections[:top_k]

    @staticmethod
    def _edit_distance(s1: str, s2: str) -> int:
        """
        Levenshtein edit distance.
        Minimum number of single-character edits (insert, delete, replace)
        to transform s1 into s2.
        """
        if len(s1) < len(s2):
            return TypoTolerance._edit_distance(s2, s1)

        if len(s2) == 0:
            return len(s1)

        previous_row = list(range(len(s2) + 1))
        for i, c1 in enumerate(s1):
            current_row = [i + 1]
            for j, c2 in enumerate(s2):
                insertions = previous_row[j + 1] + 1
                deletions = current_row[j] + 1
                substitutions = previous_row[j] + (c1 != c2)
                current_row.append(min(insertions, deletions, substitutions))
            previous_row = current_row

        return previous_row[-1]

class SnippetGenerator:
    """
    Generate search result snippets that show the most relevant
    portion of a document with query terms highlighted.
    """

    @staticmethod
    def generate(
        raw_text: str,
        query_terms: List[str],
        snippet_length: int = 150,
        highlight_start: str = "**",
        highlight_end: str = "**",
    ) -> str:
        """Find the best window of text containing query terms"""
        words = raw_text.split()
        if not words:
            return ""

        # Score each position by how many query terms are nearby
        stemmed_words = [TextProcessor.simple_stem(w.lower()) for w in words]
        query_set = set(query_terms)

        # Find positions of query term matches
        match_positions = []
        for i, sw in enumerate(stemmed_words):
            cleaned = re.sub(r'[^a-z0-9]', '', sw)
            if cleaned in query_set:
                match_positions.append(i)

        if not match_positions:
            # No matches — return beginning of document
            snippet_words = words[:20]
            return " ".join(snippet_words) + "..."

        # Find the densest window of matches
        best_start = max(0, match_positions[0] - 5)
        best_score = 0

        window_size = 25  # words in snippet
        for start in range(max(0, match_positions[0] - 10),
                           min(len(words), match_positions[-1] + 10)):
            end = min(start + window_size, len(words))
            score = sum(
                1 for pos in match_positions
                if start <= pos < end
            )
            if score > best_score:
                best_score = score
                best_start = start

        # Extract snippet and highlight terms
        snippet_words = words[best_start:best_start + window_size]
        highlighted = []

        for word in snippet_words:
            cleaned = re.sub(r'[^a-z0-9]', '', word.lower())
            stemmed = TextProcessor.simple_stem(cleaned)
            if stemmed in query_set:
                highlighted.append(f"{highlight_start}{word}{highlight_end}")
            else:
                highlighted.append(word)

        prefix = "..." if best_start > 0 else ""
        suffix = "..." if best_start + window_size < len(words) else ""

        return f"{prefix}{' '.join(highlighted)}{suffix}"

class QueryCache:
    """
    LRU cache for search query results.
    Popular queries are served from cache without hitting the index.
    """

    def __init__(self, max_size: int = 1000, ttl_seconds: float = 300):
        self.max_size = max_size
        self.ttl = ttl_seconds
        self._cache: Dict[str, dict] = {}
        self._access_order: List[str] = []
        self._lock = threading.Lock()
        self.hits = 0
        self.misses = 0

    def _cache_key(self, query: str) -> str:
        return hashlib.md5(query.lower().strip().encode()).hexdigest()

    def get(self, query: str) -> Optional[dict]:
        key = self._cache_key(query)
        with self._lock:
            if key in self._cache:
                entry = self._cache[key]
                if time.time() - entry["cached_at"] < self.ttl:
                    self.hits += 1
                    # Move to end (most recently used)
                    if key in self._access_order:
                        self._access_order.remove(key)
                    self._access_order.append(key)
                    return entry["result"]
                else:
                    # Expired
                    del self._cache[key]
                    if key in self._access_order:
                        self._access_order.remove(key)

            self.misses += 1
            return None

    def put(self, query: str, result: dict):
        key = self._cache_key(query)
        with self._lock:
            # Evict if at capacity
            while len(self._cache) >= self.max_size and self._access_order:
                evict_key = self._access_order.pop(0)
                self._cache.pop(evict_key, None)

            self._cache[key] = {
                "result": result,
                "cached_at": time.time(),
            }
            self._access_order.append(key)

    def get_stats(self) -> dict:
        total = self.hits + self.misses
        return {
            "cache_size": len(self._cache),
            "max_size": self.max_size,
            "hits": self.hits,
            "misses": self.misses,
            "hit_rate": f"{(self.hits / max(total, 1)) * 100:.1f}%",
        }

class SearchShard:
    """
    A single shard of the distributed search engine.
    Contains a complete inverted index for a subset of documents.
    Processes queries independently and returns local top-K.
    """

    def __init__(self, shard_id: str):
        self.shard_id = shard_id
        self.index = InvertedIndex(shard_id=shard_id)
        self.scorer = BM25Scorer()
        self.query_count = 0
        self.total_query_time_ms = 0

    def add_document(self, doc_id: str, title: str, text: str, url: str = ""):
        self.index.add_document(doc_id, title, text, url)

    def search(
        self, query_terms: List[str], top_k: int = 10, mode: str = "and"
    ) -> dict:
        """
        Execute a search query on this shard.
        Returns local top-K results with scores.
        """
        start_time = time.time()
        self.query_count += 1

        # Step 1: Get posting lists for all query terms
        posting_lists = []
        for term in query_terms:
            pl = self.index.get_posting_list(term)
            if pl:
                posting_lists.append(pl)

        if not posting_lists:
            elapsed = (time.time() - start_time) * 1000
            self.total_query_time_ms += elapsed
            return {
                "shard_id": self.shard_id,
                "results": [],
                "candidates": 0,
                "time_ms": round(elapsed, 3),
            }

        # Step 2: Find candidate documents (intersection or union)
        if mode == "and":
            candidates = PostingListIntersector.intersect_multiple(posting_lists)
        else:
            candidates = PostingListIntersector.union_multiple(posting_lists)

        # Step 3: Score candidates with BM25
        scored = self.scorer.score_batch(
            query_terms, candidates, self.index, top_k
        )

        # Step 4: Build result objects
        results = []
        for doc_id, score in scored:
            doc = self.index.get_document(doc_id)
            if doc:
                snippet = SnippetGenerator.generate(
                    doc.raw_text, query_terms
                )
                results.append({
                    "doc_id": doc_id,
                    "title": doc.title,
                    "score": round(score, 4),
                    "snippet": snippet,
                    "url": doc.url,
                    "shard_id": self.shard_id,
                })

        elapsed = (time.time() - start_time) * 1000
        self.total_query_time_ms += elapsed

        return {
            "shard_id": self.shard_id,
            "results": results,
            "candidates": len(candidates),
            "time_ms": round(elapsed, 3),
        }

# SECTION 9: DISTRIBUTED SEARCH ENGINE (Scatter-Gather)
# ================================================================

class DistributedSearchEngine:
    """
    Complete distributed search engine.
    
    Architecture:
    → Multiple shards, each holding a subset of documents
    → Query scatter: send to all shards in parallel
    → Result gather: merge shard results into global top-K
    → Query cache for popular queries
    → Typo tolerance for misspelled queries
    """

    def __init__(self, num_shards: int = 3):
        self.num_shards = num_shards
        self.shards = {
            f"shard_{i}": SearchShard(f"shard_{i}")
            for i in range(num_shards)
        }
        self.cache = QueryCache(max_size=500, ttl_seconds=300)
        self.typo_engine = TypoTolerance()
        self.total_queries = 0
        self.text_processor = TextProcessor()

    def _route_document(self, doc_id: str) -> str:
        """Determine which shard a document belongs to (hash-based)"""
        shard_num = hash(doc_id) % self.num_shards
        return f"shard_{shard_num}"

    def index_document(self, doc_id: str, title: str, text: str, url: str = ""):
        """Index a document into the appropriate shard"""
        shard_id = self._route_document(doc_id)
        self.shards[shard_id].add_document(doc_id, title, text, url)

    def build_typo_index(self):
        """Build typo tolerance index from all shards' vocabularies"""
        for shard in self.shards.values():
            self.typo_engine.build_from_index(shard.index)

    def search(
        self,
        query: str,
        top_k: int = 10,
        mode: str = "and",
        use_cache: bool = True,
        fix_typos: bool = True,
        expand_synonyms: bool = True,
    ) -> dict:
        """
        Full search pipeline:
        1. Check cache
        2. Process query (tokenize, stem, synonyms)
        3. Fix typos
        4. Scatter to all shards
        5. Gather and merge results
        6. Cache result
        """
        self.total_queries += 1
        start_time = time.time()

        # Step 1: Cache check
        if use_cache:
            cached = self.cache.get(query)
            if cached is not None:
                elapsed = (time.time() - start_time) * 1000
                return {
                    **cached,
                    "from_cache": True,
                    "total_time_ms": round(elapsed, 3),
                }

        # Step 2: Process query
        query_terms = TextProcessor.process_for_query(
            query, expand_synonyms=expand_synonyms
        )

        if not query_terms:
            return {
                "query": query,
                "results": [],
                "total_results": 0,
                "total_time_ms": 0,
                "message": "No valid search terms after processing",
            }

        # Step 3: Typo correction
        corrections = {}
        corrected_terms = []
        if fix_typos:
            for term in query_terms:
                suggestions = self.typo_engine.suggest_correction(term, max_distance=2, top_k=1)
                if suggestions and suggestions[0]["distance"] > 0:
                    corrections[term] = suggestions[0]["term"]
                    corrected_terms.append(suggestions[0]["term"])
                else:
                    corrected_terms.append(term)

            if corrections:
                query_terms = corrected_terms

        # Step 4: Scatter to all shards (simulate parallel execution)
        shard_results = []
        shard_threads = []
        shard_responses = {}

        def query_shard(shard_id, shard):
            result = shard.search(query_terms, top_k=top_k, mode=mode)
            shard_responses[shard_id] = result

        for shard_id, shard in self.shards.items():
            t = threading.Thread(
                target=query_shard, args=(shard_id, shard)
            )
            shard_threads.append(t)
            t.start()

        for t in shard_threads:
            t.join()

        # Step 5: Gather and merge results
        all_results = []
        total_candidates = 0
        shard_timings = {}

        for shard_id, response in shard_responses.items():
            all_results.extend(response["results"])
            total_candidates += response["candidates"]
            shard_timings[shard_id] = response["time_ms"]

        # Sort by score globally and take top-K
        all_results.sort(key=lambda x: x["score"], reverse=True)
        final_results = all_results[:top_k]

        elapsed = (time.time() - start_time) * 1000

        # Step 6: Build response
        response = {
            "query": query,
            "processed_terms": query_terms,
            "corrections": corrections if corrections else None,
            "results": final_results,
            "total_results": len(all_results),
            "total_candidates": total_candidates,
            "shard_timings": shard_timings,
            "total_time_ms": round(elapsed, 3),
            "from_cache": False,
        }

        # Step 7: Cache the result
        if use_cache:
            self.cache.put(query, response)

        return response

    def phrase_search(self, phrase: str, top_k: int = 10) -> dict:
        """Search for an exact phrase across all shards"""
        start_time = time.time()

        terms = TextProcessor.process_for_query(
            phrase, expand_synonyms=False, remove_stop_words=False
        )

        if len(terms) < 2:
            return self.search(phrase, top_k=top_k)

        all_matches = set()
        for shard in self.shards.values():
            matches = PostingListIntersector.phrase_match(shard.index, terms)
            all_matches.update(matches)

        # Score matched documents
        scorer = BM25Scorer()
        scored_results = []

        for doc_id in all_matches:
            shard_id = self._route_document(doc_id)
            shard = self.shards[shard_id]
            score = scorer.score_document(terms, doc_id, shard.index)
            doc = shard.index.get_document(doc_id)
            if doc and score > 0:
                snippet = SnippetGenerator.generate(doc.raw_text, terms)
                scored_results.append({
                    "doc_id": doc_id,
                    "title": doc.title,
                    "score": round(score, 4),
                    "snippet": snippet,
                    "url": doc.url,
                })

        scored_results.sort(key=lambda x: x["score"], reverse=True)
        elapsed = (time.time() - start_time) * 1000

        return {
            "query": f'"{phrase}"',
            "type": "phrase_search",
            "processed_terms": terms,
            "results": scored_results[:top_k],
            "total_phrase_matches": len(all_matches),
            "total_time_ms": round(elapsed, 3),
        }

    def get_stats(self) -> dict:
        shard_stats = {}
        total_docs = 0
        total_terms = 0

        for shard_id, shard in self.shards.items():
            stats = shard.index.get_stats()
            shard_stats[shard_id] = stats
            total_docs += stats["total_documents"]
            total_terms += stats["unique_terms"]

        return {
            "total_documents": total_docs,
            "total_unique_terms": total_terms,
            "num_shards": self.num_shards,
            "shard_stats": shard_stats,
            "cache_stats": self.cache.get_stats(),
            "typo_vocabulary_size": len(self.typo_engine.vocabulary),
            "total_queries_served": self.total_queries,
        }

            num_sentences = random.randint(5, 15)
            sentences = []

            for s in range(num_sentences):
                if s == 0 or random.random() < 0.7:
                    context = random.choice(topic_data["context"])
                    terms = random.sample(
                        topic_data["terms"],
                        min(random.randint(2, 4), len(topic_data["terms"]))
                    )
                else:
                    context = random.choice(secondary_data["context"])
                    terms = random.sample(
                        secondary_data["terms"],
                        min(random.randint(1, 3), len(secondary_data["terms"]))
                    )

                sentence = f"{context} {' and '.join(terms)}."
                sentences.append(sentence)

            text = " ".join(sentences)
            title_terms = random.sample(topic_data["terms"], min(2, len(topic_data["terms"])))
            title = f"Guide to {' and '.join(title_terms).title()}"

            documents.append({
                "doc_id": f"doc_{i:05d}",
                "title": title,
                "text": text,
                "url": f"https://example.com/articles/{primary_topic}/{i}",
                "topic": primary_topic,
            })

        return documents

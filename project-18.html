🎯 HYPOTHETICAL PROBLEM

You run a successful URL shortener (Project 16) and Pastebin (Project 17)
→ Business exploded: 50 million new URLs per month
→ Database has 2 billion rows
→ Single PostgreSQL server: 1.5TB disk, 90% full
→ Write latency: was 5ms, now 50ms (10x slower)
→ INSERT throughput: maxed at 5,000/sec (need 15,000/sec)
→ Indexes are enormous: 200GB just for indexes
→ VACUUM takes 6 hours (locks performance)
→ Full backup takes 4 hours (recovery window too long)

You already did:
  → Project 10: Indexes are optimized (can't squeeze more)
  → Project 13: Read replicas handle read traffic
  → But replicas DON'T help with writes
  → ALL writes go to ONE primary server
  → That ONE server is the bottleneck

The database is like a LIBRARY:
  → You have ONE giant building with 2 billion books
  → Every new book must be cataloged in this ONE building
  → The catalog system is overwhelmed
  → The building is running out of shelf space
  → Finding books takes longer as shelves fill up

THE SOLUTION: Build MULTIPLE library branches
  → Split the books across 8 buildings
  → Each building has ~250 million books
  → Each building has its own catalog
  → New books go to the appropriate building
  → Finding a book: figure out WHICH building, then look there

THIS IS SHARDING:
  → Split one database into multiple independent databases
  → Each shard holds a SUBSET of the data
  → Each shard is a fully independent database server
  → Application decides WHICH shard to query

🔍 HOW TO APPROACH IT

SHARDING REQUIRES ANSWERING 5 FUNDAMENTAL QUESTIONS:

Question 1: WHAT to shard on? (Shard Key)
  → Which column determines which shard a row goes to?
  → URL shortener: shard on short_code
  → Pastebin: shard on paste_id
  → E-commerce: shard on user_id? order_id? product_id?
  → THIS IS THE MOST IMPORTANT DECISION

Question 2: HOW to map key → shard? (Sharding Strategy)
  → Range-based: keys a-m → Shard 0, n-z → Shard 1
  → Hash-based: hash(key) % num_shards = shard_number
  → Directory-based: lookup table says key X → Shard Y
  → Consistent hashing: Project 14 ring-based mapping

Question 3: HOW MANY shards? (Capacity Planning)
  → Too few: each shard still too big
  → Too many: management overhead, cross-shard queries expensive
  → Rule: plan for 2-5 years of growth
  → Start with enough shards that each is < 30% capacity

Question 4: WHAT ABOUT queries that span shards? (Cross-Shard)
  → SELECT COUNT(*) FROM urls → must query ALL shards
  → JOIN across shards is extremely expensive
  → Design schema to AVOID cross-shard queries
  → Co-locate related data on the same shard

Question 5: HOW TO add/remove shards? (Rebalancing)
  → Traffic grows → need more shards
  → Must move data between shards without downtime
  → This is the HARDEST part of sharding
  → Consistent hashing (Project 14) helps minimize data movement

🛠️ HOW CAN WE SOLVE IT

THREE SHARDING STRATEGIES:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STRATEGY 1: RANGE-BASED SHARDING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Split data by RANGES of the shard key.

  Example (URL shortener by short_code first character):
    Shard 0: codes starting with 0-9     (10 chars)
    Shard 1: codes starting with a-m     (13 chars)
    Shard 2: codes starting with n-z     (13 chars)
    Shard 3: codes starting with A-M     (13 chars)
    Shard 4: codes starting with N-Z     (13 chars)

  Example (E-commerce by date):
    Shard 0: orders from 2023
    Shard 1: orders from 2024
    Shard 2: orders from 2025

  Routing:
    short_code = "Xk9mP2"
    first_char = "X" → falls in N-Z range → Shard 4

  ┌─────────────────────────────────────────────┐
  │   0-9   │   a-m   │   n-z   │  A-M  │  N-Z │
  │ Shard 0 │ Shard 1 │ Shard 2 │ Srd 3 │ Srd 4│
  └─────────────────────────────────────────────┘

  Pros:
    → Simple to understand and implement
    → Range queries are efficient (all data in one shard)
    → Easy to add new ranges (append new shard for future dates)
  
  Cons:
    → HOTSPOT PROBLEM: some ranges much busier than others
      (new orders all go to "2025" shard → overloaded)
    → UNEVEN DISTRIBUTION: names starting with "S" more common
    → Must manually choose range boundaries
    → Rebalancing requires moving data between shards


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STRATEGY 2: HASH-BASED SHARDING                ← MOST COMMON
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Apply a hash function to shard key, modulo number of shards.

  shard_number = hash(shard_key) % num_shards

  Example:
    num_shards = 4
    hash("Xk9mP2") = 2847593 → 2847593 % 4 = 1 → Shard 1
    hash("Ab3kQ7") = 1938472 → 1938472 % 4 = 0 → Shard 0
    hash("zR8nW1") = 5729384 → 5729384 % 4 = 0 → Shard 0
    hash("mT5pY9") = 3847261 → 3847261 % 4 = 1 → Shard 1

  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │   hash % 4   │   hash % 4   │   hash % 4   │   hash % 4   │
  │     = 0      │     = 1      │     = 2      │     = 3      │
  │   Shard 0    │   Shard 1    │   Shard 2    │   Shard 3    │
  └──────────────┴──────────────┴──────────────┴──────────────┘

  Pros:
    → EVEN DISTRIBUTION: hash function spreads data uniformly
    → No hotspots (assuming good hash function)
    → Simple formula: deterministic, no lookup needed
    → Works for any data type (strings, integers, UUIDs)
  
  Cons:
    → RANGE QUERIES IMPOSSIBLE: "all codes starting with A"
      requires querying ALL shards
    → REBALANCING NIGHTMARE: changing num_shards (4→5)
      means EVERY key's shard changes
      hash("Xk9mP2") % 4 = 1, but % 5 = 3 → must MOVE data
    → Massive data migration when adding shards


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STRATEGY 3: CONSISTENT HASH SHARDING (Project 14 applied)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Use consistent hashing ring from Project 14.

  Recap from Project 14:
    → Place shard nodes on a hash ring
    → Hash the key → find position on ring
    → Walk clockwise → first node found = target shard
    → Adding a node only moves ~1/N of keys (not ALL)

  ┌───────────────────────────────────────────┐
  │                    Ring                    │
  │              Shard A (pos 0)               │
  │           ╱                  ╲             │
  │     Shard D                  Shard B       │
  │     (pos 270)                (pos 90)      │
  │           ╲                  ╱             │
  │              Shard C (pos 180)             │
  │                                            │
  │   key "Xk9mP2" hashes to pos 120          │
  │   → walk clockwise → hits Shard C (180)    │
  └───────────────────────────────────────────┘

  Adding Shard E at position 150:
    → Only keys between 120-150 move (from C to E)
    → All other keys stay where they are
    → Minimal data movement!

  Pros:
    → Minimal rebalancing when adding/removing shards
    → Even distribution with virtual nodes
    → No single point of failure for routing
  
  Cons:
    → More complex implementation
    → Range queries still impossible
    → Virtual nodes add complexity
    → Need to maintain ring state across all app servers


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STRATEGY 4: DIRECTORY-BASED SHARDING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Maintain a lookup table that maps keys to shards.

  ┌──────────────────────────────────────┐
  │ DIRECTORY SERVICE                     │
  │                                      │
  │ key_range     │ shard                │
  │ ──────────────┼──────────            │
  │ Xk9*          │ shard-2              │
  │ Ab3*          │ shard-0              │
  │ user_42       │ shard-1              │
  └──────────────────────────────────────┘

  App → asks directory "where is Xk9mP2?" → "shard-2" → query shard-2

  Pros:
    → Maximum flexibility (any key → any shard)
    → Can move individual keys without rehashing
    → Can handle uneven data (put heavy users on bigger shards)
  
  Cons:
    → Directory becomes a SINGLE POINT OF FAILURE
    → Every query needs a directory lookup first (extra hop)
    → Directory itself needs to be replicated and cached
    → More operational complexity


FOR THIS PROJECT:
  → We implement ALL FOUR strategies
  → Show when each is appropriate
  → Build a ShardRouter that supports pluggable strategies

THE KEY DECISION: Choosing the Right Shard Key

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SCENARIO: URL Shortener (Project 16)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Option A: Shard by short_code
    → GOOD: redirect lookup is single-shard (know the code)
    → GOOD: even distribution with hash sharding
    → BAD: "all URLs by user_42" requires scatter-gather
    → BEST FOR: read-heavy redirect workload ✓

  Option B: Shard by user_id
    → GOOD: "my URLs" query is single-shard
    → BAD: redirect lookup needs directory (code → user → shard)
    → BAD: some users create millions of URLs (hotspot)
    → BEST FOR: user-centric dashboards

  Option C: Shard by creation_date
    → GOOD: recent data (hot) on fast shards, old on slow
    → BAD: new shard gets ALL current writes (hotspot)
    → BEST FOR: time-series / append-only workloads

  DECISION: Shard by short_code (hash-based)
    → Primary operation (redirect) is single-shard ✓
    → Even distribution ✓
    → Trade-off: user queries require scatter-gather

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Option A: Shard by order_id
    → GOOD: order lookup is single-shard
    → BAD: "all orders by customer_123" → scatter-gather
    → BAD: "all orders for product_456" → scatter-gather

  Option B: Shard by customer_id
    → GOOD: "my orders" is single-shard
    → GOOD: can co-locate customer + orders + cart on same shard
    → BAD: some customers have millions of orders (hotspot)
    → BEST FOR: customer-centric applications ✓

  Option C: Shard by product_id
    → GOOD: product analytics is single-shard
    → BAD: customer queries → scatter-gather

  DECISION: Shard by customer_id
    → Most queries are customer-centric
    → Can co-locate related data
    → Handle hotspots with virtual shards for heavy customers

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SHARD KEY SELECTION RULES:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Rule 1: Choose the key used in MOST queries
    → If 90% of queries look up by short_code → shard by short_code
    → The most common query should be single-shard

  Rule 2: Choose a key with HIGH CARDINALITY
    → user_id: millions of distinct values → good
    → country: ~200 values → bad (too few shards)
    → boolean: 2 values → terrible

  Rule 3: Choose a key with EVEN DISTRIBUTION
    → user_id with hash: even ✓
    → last_name first letter: uneven (too many "S" names)
    → timestamp: all current traffic on latest shard

  Rule 4: Choose a key that DOESN'T CHANGE
    → user_id: never changes ✓
    → email: might change → row must move shards → expensive
    → status: changes frequently → terrible shard key

  Rule 5: AVOID keys that create CROSS-SHARD JOINS
    → If orders shard by customer_id and products shard by product_id
    → "Orders with product details" = cross-shard join → very expensive
    → Co-locate related data on the same shard key

✅ WHAT SHOULD DO

1. DESIGN schema to avoid cross-shard queries
   → Include denormalized data on each shard
   → If orders shard by customer_id, include product_name
     in the order row (don't need to JOIN products table)
   → Denormalization is the COST of sharding

2. ALWAYS include shard key in every query
   → Query without shard key → must scatter to ALL shards
   → "SELECT * FROM urls WHERE short_code = X" → single shard ✓
   → "SELECT * FROM urls WHERE user_id = Y" → ALL shards ✗
   → Design your API to always know the shard key

3. USE consistent hashing for production sharding
   → Hash-based (% N) breaks on rebalancing
   → Consistent hashing (Project 14) minimizes data movement
   → Worth the extra complexity for any growing system

4. PLAN shard count for future growth
   → Start with more shards than you need today
   → 16 or 32 logical shards (can map to fewer physical servers)
   → Logical shards can be remapped without data migration
   → "2 physical servers, each hosting 8 logical shards"
     → Later: "4 physical servers, each hosting 4 logical shards"

5. IMPLEMENT a shard routing layer
   → Application should NOT know about individual shard connections
   → ShardRouter: given a key, return the correct DB connection
   → Centralize routing logic in ONE place
   → Changing sharding strategy = change router, not all app code

6. TEST with realistic data distribution
   → Synthetic even data hides hotspot problems
   → Use production-like key distribution in testing
   → Monitor shard sizes and query distribution after deployment

7. REPLICATE each shard independently
   → Each shard is its own primary + replicas (Project 13)
   → Shard 0: primary-0 + replica-0a + replica-0b
   → Shard 1: primary-1 + replica-1a + replica-1b
   → One shard's failure doesn't affect others

❌ WHAT SHOULD NOT DO

1. NEVER shard prematurely
   → Single PostgreSQL handles millions of rows easily
   → Sharding adds MASSIVE complexity
   → First: optimize queries (Project 10)
   → Then: add read replicas (Project 13)
   → Then: add caching (Project 5)
   → ONLY THEN: consider sharding
   → "Premature sharding is the root of all evil" — adapted

2. NEVER choose a low-cardinality shard key
   → Shard by country: 200 shards max, US gets 40% of traffic
   → Shard by boolean: 2 shards, possibly very uneven
   → Need MILLIONS of distinct values for good distribution

3. NEVER do cross-shard JOINs in hot paths
   → Cross-shard JOIN = query N shards + merge in application
   → Latency = max(all shard latencies) + merge time
   → OK for analytics/reports (background, slow is acceptable)
   → NEVER for user-facing requests (must be fast)

4. NEVER assume even distribution without measuring
   → Hash functions aren't perfect
   → Some keys might cluster
   → Monitor: rows per shard, queries per shard, latency per shard
   → Rebalance when one shard is 2x bigger than average

5. NEVER use auto-increment IDs across shards naively
   → Shard 0 generates ID 1, Shard 1 also generates ID 1
   → IDs are no longer globally unique!
   → FIX: include shard_id in the ID
     → Shard 0: IDs 0-0001, 0-0002, ...
     → Shard 1: IDs 1-0001, 1-0002, ...
   → Or use UUIDs / Snowflake IDs

6. NEVER change shard key after deployment
   → Changing from shard-by-user to shard-by-order means
     moving EVERY row to a different shard
   → Essentially rebuilding the entire database
   → Choose the right key from the start

7. NEVER shard and forget to update your backup strategy
   → Backing up 8 shards independently
   → Point-in-time recovery across shards is HARD
   → Shards might be at different points in time
   → Need coordinated backup strategy

🚫 WHEN NOT TO USE

1. DON'T shard when vertical scaling still works
   → Upgrade to bigger machine (more RAM, faster SSD)
   → Much simpler than sharding
   → Modern servers: 1TB RAM, 100TB SSD, 100K IOPS
   → This handles most workloads

2. DON'T shard when read replicas solve the problem
   → If bottleneck is READ throughput → add replicas (Project 13)
   → Replicas are 10x simpler than sharding
   → Only shard when WRITE throughput is the bottleneck

3. DON'T shard when caching solves the problem
   → 90% cache hit rate means only 10% reaches DB
   → From Project 5: cache can reduce DB load by 10x
   → Cheaper and simpler than sharding

4. DON'T shard relational data with complex JOINs
   → Cross-shard JOINs are extremely expensive
   → If your schema has 10 tables with complex JOINs
   → Consider: denormalization first, or NoSQL (DynamoDB)
   → Sharding works best for simple key-value patterns

5. DON'T shard when you need strong global transactions
   → ACID across shards requires distributed transactions (2PC)
   → 2PC is slow, complex, and fragile
   → If you need "debit account A, credit account B" atomically
   → Keep A and B on the SAME shard (co-locate)

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
MECHANIC 1: HASH-BASED SHARD ROUTING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  WRITE PATH (create short URL):

  Application receives: short_code = "Xk9mP2"
  │
  ├─→ Compute: hash("Xk9mP2") = 2847593
  │
  ├─→ Compute: shard_id = 2847593 % 4 = 1
  │
  ├─→ Get connection: pool = shard_connections[1]
  │
  ├─→ Execute on Shard 1:
  │     INSERT INTO urls (short_code, long_url, ...)
  │     VALUES ('Xk9mP2', 'https://...', ...)
  │
  └─→ Return success

  READ PATH (redirect):

  Application receives: GET /Xk9mP2
  │
  ├─→ Check cache (Project 5) → cache miss
  │
  ├─→ Compute: hash("Xk9mP2") % 4 = 1 → Shard 1
  │
  ├─→ Execute on Shard 1:
  │     SELECT long_url FROM urls WHERE short_code = 'Xk9mP2'
  │
  ├─→ Populate cache
  │
  └─→ Return redirect

  SCATTER-GATHER (query without shard key):

  Application receives: "How many total URLs?"
  │
  ├─→ Send to ALL shards in parallel:
  │     Shard 0: SELECT COUNT(*) FROM urls → 500,234,891
  │     Shard 1: SELECT COUNT(*) FROM urls → 499,872,103
  │     Shard 2: SELECT COUNT(*) FROM urls → 500,102,445
  │     Shard 3: SELECT COUNT(*) FROM urls → 499,990,561
  │
  ├─→ Aggregate results: SUM = 2,000,200,000
  │
  └─→ Return total count

  Scatter-gather latency = MAX(shard latencies) + aggregation
  vs Single-shard latency = one shard only
  → Design to minimize scatter-gather queries

  THE TRICK THAT MAKES REBALANCING EASY:

  Instead of: 4 shards on 4 servers
  Use: 64 LOGICAL shards mapped to 4 PHYSICAL servers

  ┌─────────────────────────────────────────────┐
  │ Logical Shard │ Physical Server              │
  │───────────────┼──────────────────────────────│
  │ L-00 to L-15  │ physical-server-0            │
  │ L-16 to L-31  │ physical-server-1            │
  │ L-32 to L-47  │ physical-server-2            │
  │ L-48 to L-63  │ physical-server-3            │
  └─────────────────────────────────────────────┘

  Routing: hash(key) % 64 = logical_shard_number
  Mapping: logical_shard_number → physical_server

    → Move L-48 to L-51 from server-3 to server-4
    → Move L-00 to L-03 from server-0 to server-4
    → etc.
    → Application routing (hash % 64) DOESN'T CHANGE
    → Only the logical → physical mapping changes
    → Can migrate one logical shard at a time (gradual)

  This is how MongoDB, Vitess, CockroachDB work internally.

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: The Celebrity Problem (Hotspot)
  → Justin Bieber's account has 100M followers
  → Shard with his user_id gets 100x more reads
  → That shard is overwhelmed, others are idle
  → FIX: Detect hot keys → cache aggressively (Project 5)
         Or split hot key across multiple sub-shards
         Or move to dedicated shard

PITFALL 2: The Rebalancing Avalanche
  → Change from 4 to 5 shards (hash-based)
  → 80% of data needs to move (4 of 5 keys change shard)
  → During migration: data might be on old OR new shard
  → Queries must check BOTH → doubled latency
  → FIX: Use consistent hashing (only ~20% moves)
         Or use logical shards (no rehashing needed)

PITFALL 3: Cross-Shard Transaction Failure
  → Transfer $100 from User A (shard 0) to User B (shard 1)
  → Shard 0: debit $100 → SUCCESS
  → Shard 1: credit $100 → NETWORK FAILURE
  → Money disappeared! A was debited but B wasn't credited
  → FIX: Saga pattern (Project 37) — compensating transactions
         Or co-locate A and B on same shard

PITFALL 4: Schema Migration Across Shards
  → Need to add a column to ALL shards
  → Must run ALTER TABLE on each shard
  → Some shards finish fast, others slow
  → Application sees inconsistent schemas
  → FIX: Blue-green deployment per shard
         Or online schema migration tools (gh-ost, pt-osc)

PITFALL 5: Scatter-Gather Timeout
  → Query goes to all 8 shards
  → 7 respond in 10ms, 1 responds in 5000ms
  → Total query time = 5000ms (slowest shard wins)
  → FIX: Set per-shard timeouts
         Return partial results with warning
         Investigate slow shard (maybe it needs more resources)

PITFALL 6: Unbalanced Shard Growth
  → Started with 4 even shards
  → 2 years later: Shard 0 has 800M rows, Shard 3 has 300M rows
  → Hash function wasn't as uniform as expected
  → FIX: Monitor shard sizes continuously
         Use more virtual nodes in consistent hashing
         Split oversized shards proactively

PITFALL 7: Connection Pool Explosion
  → 10 app servers × 8 shards × 10 connections each = 800 connections
  → Each shard has 100 connections (way too many)
  → FIX: Connection pooler (PgBouncer) per shard
         Reduce pool size per shard
         Use connection multiplexing

class HashShardRouter:
    """
    Simplest sharding: hash(key) % num_shards
    
    Even distribution, but terrible for rebalancing.
    Adding a shard changes almost every key's assignment.
    """
    
    def __init__(self, num_shards):
        self.num_shards = num_shards
        self.name = "hash"
    
    def _hash(self, key):
        """Consistent hash function (not Python's built-in hash)"""
        return int(hashlib.md5(str(key).encode('utf-8')).hexdigest(), 16)
    
    def get_shard(self, key):
        """Determine which shard a key belongs to"""
        return self._hash(key) % self.num_shards
    
    def get_all_shards(self):
        return list(range(self.num_shards))
    
    def get_distribution(self, keys):
        """Analyze how keys distribute across shards"""
        dist = defaultdict(int)
        for key in keys:
            dist[self.get_shard(key)] += 1
        return dict(dist)
    
    def simulate_rebalance(self, keys, new_num_shards):
        """
        Show what happens when we change the number of shards.
        THIS is the problem with hash-based sharding.
        """
        old_assignments = {key: self.get_shard(key) for key in keys}
        
        new_router = HashShardRouter(new_num_shards)
        new_assignments = {key: new_router.get_shard(key) for key in keys}
        
        moved = sum(
            1 for key in keys 
            if old_assignments[key] != new_assignments[key]
        )
        
        return {
            "total_keys": len(keys),
            "keys_moved": moved,
            "move_percentage": round(moved / len(keys) * 100, 1),
            "old_shards": self.num_shards,
            "new_shards": new_num_shards
        }

class RangeShardRouter:
    """
    Shard by key ranges. Good for range queries.
    
    Each range boundary maps to a shard.
    Keys falling within a range go to that shard.
    """
    
    def __init__(self, ranges):
        """
        ranges: list of (start, end, shard_id) tuples
        Example: [(0, 999999, 0), (1000000, 1999999, 1), ...]
        """
        self.ranges = sorted(ranges, key=lambda r: r[0])
        self.starts = [r[0] for r in self.ranges]
        self.name = "range"
    
    def get_shard(self, key):
        """Find which range the key falls into"""
        if isinstance(key, str):
            # Convert string to integer for range comparison
            key = int(hashlib.md5(key.encode('utf-8')).hexdigest()[:8], 16)
        
        # Binary search for the correct range
        idx = bisect.bisect_right(self.starts, key) - 1
        if idx < 0:
            idx = 0
        
        if idx < len(self.ranges):
            start, end, shard_id = self.ranges[idx]
            if start <= key <= end:
                return shard_id
        
        # Default to last shard for overflow
        return self.ranges[-1][2]
    
    def get_shards_for_range(self, range_start, range_end):
        """
        Find which shards a range query needs to touch.
        THIS is the advantage of range sharding.
        """
        shards_needed = set()
        for start, end, shard_id in self.ranges:
            if start <= range_end and end >= range_start:
                shards_needed.add(shard_id)
        return sorted(shards_needed)
    
    def get_all_shards(self):
        return sorted(set(r[2] for r in self.ranges))
    
    def split_shard(self, shard_id, split_point, new_shard_id):
        """Split an existing shard range at a given point"""
        new_ranges = []
        for start, end, sid in self.ranges:
            if sid == shard_id and start <= split_point <= end:
                # Split this range into two
                new_ranges.append((start, split_point - 1, shard_id))
                new_ranges.append((split_point, end, new_shard_id))
            else:
                new_ranges.append((start, end, sid))
        
        self.ranges = sorted(new_ranges, key=lambda r: r[0])
        self.starts = [r[0] for r in self.ranges]
        
        return {
            "action": "split",
            "original_shard": shard_id,
            "new_shard": new_shard_id,
            "split_point": split_point,
            "total_ranges": len(self.ranges)
        }

class ConsistentHashShardRouter:
    """
    Uses consistent hashing ring from Project 14.
    Minimal data movement when adding/removing shards.
    
    This is the PRODUCTION-GRADE approach.
    """
    
    def __init__(self, virtual_nodes=150):
        self.virtual_nodes = virtual_nodes
        self.ring = {}           # position → shard_id
        self.sorted_positions = []
        self.shards = set()
        self.name = "consistent_hash"
    
    def _hash_position(self, key):
        """Map a key to a position on the ring (0 to 2^32)"""
        return int(
            hashlib.md5(str(key).encode('utf-8')).hexdigest(), 16
        ) % (2**32)
    
    def add_shard(self, shard_id):
        """Add a shard with virtual nodes to the ring"""
        self.shards.add(shard_id)
        
        for i in range(self.virtual_nodes):
            virtual_key = f"{shard_id}:vn{i}"
            position = self._hash_position(virtual_key)
            self.ring[position] = shard_id
        
        self.sorted_positions = sorted(self.ring.keys())
    
    def remove_shard(self, shard_id):
        """Remove a shard and all its virtual nodes"""
        self.shards.discard(shard_id)
        
        positions_to_remove = [
            pos for pos, sid in self.ring.items() if sid == shard_id
        ]
        for pos in positions_to_remove:
            del self.ring[pos]
        
        self.sorted_positions = sorted(self.ring.keys())
    
    def get_shard(self, key):
        """Find the shard responsible for this key"""
        if not self.sorted_positions:
            raise ValueError("No shards registered")
        
        position = self._hash_position(str(key))
        
        # Find first position on ring >= key's position
        idx = bisect.bisect_right(self.sorted_positions, position)
        
        # Wrap around if past the last position
        if idx >= len(self.sorted_positions):
            idx = 0
        
        ring_position = self.sorted_positions[idx]
        return self.ring[ring_position]
    
    def get_all_shards(self):
        return sorted(self.shards)
    
    def simulate_add_shard(self, new_shard_id, keys):
        """
        Show how adding a shard moves minimal data.
        Compare with hash-based rebalancing.
        """
        # Record current assignments
        old_assignments = {}
        for key in keys:
            old_assignments[key] = self.get_shard(key)
        
        # Add the new shard
        self.add_shard(new_shard_id)
        
        # Record new assignments
        new_assignments = {}
        moved = 0
        for key in keys:
            new_shard = self.get_shard(key)
            new_assignments[key] = new_shard
            if old_assignments[key] != new_shard:
                moved += 1
        
        return {
            "total_keys": len(keys),
            "keys_moved": moved,
            "move_percentage": round(moved / len(keys) * 100, 1),
            "expected_percentage": round(
                100.0 / len(self.shards), 1
            ),
            "new_total_shards": len(self.shards)
        }

class DirectoryShardRouter:
    """
    Lookup table maps keys (or key prefixes) to shards.
    Maximum flexibility but requires a directory service.
    """
    
    def __init__(self, default_shard=0):
        self.directory = {}       # exact key → shard
        self.prefix_directory = {}  # key prefix → shard
        self.default_shard = default_shard
        self.shards = set()
        self.name = "directory"
        self.lock = threading.Lock()
    
    def register_shard(self, shard_id):
        self.shards.add(shard_id)
    
    def assign_key(self, key, shard_id):
        """Assign a specific key to a specific shard"""
        with self.lock:
            self.directory[key] = shard_id
            self.shards.add(shard_id)
    
    def assign_prefix(self, prefix, shard_id):
        """Assign all keys with a prefix to a shard"""
        with self.lock:
            self.prefix_directory[prefix] = shard_id
            self.shards.add(shard_id)
    
    def get_shard(self, key):
        """Look up shard for a key"""
        # Check exact match first
        if key in self.directory:
            return self.directory[key]
        
        # Check prefix matches (longest prefix wins)
        best_match = None
        best_length = 0
        key_str = str(key)
        
        for prefix, shard_id in self.prefix_directory.items():
            if key_str.startswith(prefix) and len(prefix) > best_length:
                best_match = shard_id
                best_length = len(prefix)
        
        if best_match is not None:
            return best_match
        
        return self.default_shard
    
    def move_key(self, key, new_shard_id):
        """Move a key to a different shard"""
        with self.lock:
            old_shard = self.directory.get(key, self.default_shard)
            self.directory[key] = new_shard_id
            return {"key": key, "old_shard": old_shard, "new_shard": new_shard_id}
    
    def get_all_shards(self):
        return sorted(self.shards)
    
    def get_directory_size(self):
        return {
            "exact_entries": len(self.directory),
            "prefix_entries": len(self.prefix_directory),
            "total_shards": len(self.shards)
        }

class ShardInstance:
    """
    Represents one physical shard (one SQLite database).
    In production: each would be a PostgreSQL/MySQL server.
    Each shard is a FULLY INDEPENDENT database.
    """
    
    def __init__(self, shard_id, db_path):
        self.shard_id = shard_id
        self.db_path = db_path
        self.lock = threading.Lock()
        
        # Metrics
        self.reads = 0
        self.writes = 0
        self.errors = 0
        
        self._init_schema()
    
    def _init_schema(self):
        """Create tables for URL shortener data"""
        conn = sqlite3.connect(self.db_path)
        cursor = conn.cursor()
        
        cursor.execute("""
            CREATE TABLE IF NOT EXISTS urls (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                short_code TEXT NOT NULL UNIQUE,
                long_url TEXT NOT NULL,
                created_at TEXT NOT NULL,
                click_count INTEGER DEFAULT 0,
                user_id TEXT
            )
        """)
        
        cursor.execute(
            "CREATE UNIQUE INDEX IF NOT EXISTS idx_code ON urls(short_code)"
        )
        cursor.execute(
            "CREATE INDEX IF NOT EXISTS idx_user ON urls(user_id)"
        )
        
        conn.commit()
        conn.close()
    
    def insert(self, short_code, long_url, user_id=None):
        """Insert a URL mapping into this shard"""
        with self.lock:
            try:
                conn = sqlite3.connect(self.db_path)
                cursor = conn.cursor()
                cursor.execute(
                    """INSERT INTO urls (short_code, long_url, created_at, user_id)
                       VALUES (?, ?, ?, ?)""",
                    (short_code, long_url, datetime.now().isoformat(), user_id)
                )
                conn.commit()
                conn.close()
                self.writes += 1
                return {"success": True, "shard": self.shard_id}
            except sqlite3.IntegrityError:
                conn.close()
                self.errors += 1
                return {"success": False, "error": "Duplicate short_code"}
            except Exception as e:
                self.errors += 1
                return {"success": False, "error": str(e)}
    
    def lookup(self, short_code):
        """Look up a URL by short code on this shard"""
        try:
            conn = sqlite3.connect(self.db_path)
            cursor = conn.cursor()
            cursor.execute(
                "SELECT long_url, click_count, created_at, user_id "
                "FROM urls WHERE short_code = ?",
                (short_code,)
            )
            row = cursor.fetchone()
            conn.close()
            self.reads += 1
            
            if row:
                return {
                    "found": True,
                    "long_url": row[0],
                    "click_count": row[1],
                    "created_at": row[2],
                    "user_id": row[3],
                    "shard": self.shard_id
                }
            return {"found": False, "shard": self.shard_id}
            
        except Exception as e:
            self.errors += 1
            return {"found": False, "error": str(e)}
    
    def increment_clicks(self, short_code):
        """Increment click count"""
        with self.lock:
            try:
                conn = sqlite3.connect(self.db_path)
                cursor = conn.cursor()
                cursor.execute(
                    "UPDATE urls SET click_count = click_count + 1 "
                    "WHERE short_code = ?",
                    (short_code,)
                )
                conn.commit()
                conn.close()
                return True
            except Exception:
                return False
    
    def delete(self, short_code):
        """Delete a URL from this shard"""
        with self.lock:
            try:
                conn = sqlite3.connect(self.db_path)
                cursor = conn.cursor()
                cursor.execute(
                    "DELETE FROM urls WHERE short_code = ?",
                    (short_code,)
                )
                affected = cursor.rowcount
                conn.commit()
                conn.close()
                return {"success": affected > 0, "shard": self.shard_id}
            except Exception as e:
                return {"success": False, "error": str(e)}
    
    def count(self):
        """Count rows in this shard"""
        try:
            conn = sqlite3.connect(self.db_path)
            cursor = conn.cursor()
            cursor.execute("SELECT COUNT(*) FROM urls")
            count = cursor.fetchone()[0]
            conn.close()
            return count
        except Exception:
            return 0
    
    def get_all_rows(self, limit=None):
        """Get all rows from this shard (for migration)"""
        try:
            conn = sqlite3.connect(self.db_path)
            cursor = conn.cursor()
            query = "SELECT short_code, long_url, created_at, click_count, user_id FROM urls"
            if limit:
                query += f" LIMIT {limit}"
            cursor.execute(query)
            rows = cursor.fetchall()
            conn.close()
            return [
                {"short_code": r[0], "long_url": r[1], "created_at": r[2],
                 "click_count": r[3], "user_id": r[4]}
                for r in rows
            ]
        except Exception:
            return []
    
    def query_by_user(self, user_id, limit=50):
        """Query URLs by user_id (may exist on any shard)"""
        try:
            conn = sqlite3.connect(self.db_path)
            cursor = conn.cursor()
            cursor.execute(
                """SELECT short_code, long_url, click_count, created_at
                   FROM urls WHERE user_id = ?
                   ORDER BY created_at DESC LIMIT ?""",
                (user_id, limit)
            )
            rows = cursor.fetchall()
            conn.close()
            self.reads += 1
            return [
                {"short_code": r[0], "long_url": r[1],
                 "click_count": r[2], "created_at": r[3],
                 "shard": self.shard_id}
                for r in rows
            ]
        except Exception:
            return []
    
    def get_stats(self):
        """Get shard-level statistics"""
        row_count = self.count()
        db_size = os.path.getsize(self.db_path) if os.path.exists(self.db_path) else 0
        
        return {
            "shard_id": self.shard_id,
            "row_count": row_count,
            "db_size_bytes": db_size,
            "db_size_kb": round(db_size / 1024, 1),
            "reads": self.reads,
            "writes": self.writes,
            "errors": self.errors
        }

class ShardedDatabase:
    """
    The complete sharded database system.
    
    Application code talks to THIS class.
    This class routes to the correct shard.
    Application NEVER talks to individual shards directly.
    
    Supports:
      → Single-shard operations (fast, targeted)
      → Scatter-gather operations (query all shards)
      → Data migration between shards
      → Health monitoring per shard
    """
    
    def __init__(self, shard_dir="./shards", router=None, num_shards=4):
        self.shard_dir = shard_dir
        os.makedirs(shard_dir, exist_ok=True)
        
        # Initialize router
        if router:
            self.router = router
        else:
            # Default: consistent hash router
            self.router = ConsistentHashShardRouter(virtual_nodes=150)
            for i in range(num_shards):
                self.router.add_shard(i)
        
        # Initialize shard instances
        self.shards = {}
        for shard_id in self.router.get_all_shards():
            db_path = os.path.join(shard_dir, f"shard_{shard_id}.db")
            self.shards[shard_id] = ShardInstance(shard_id, db_path)
        
        # Metrics
        self.total_routed = 0
        self.total_scatter_gather = 0
        self.routing_log = []
    
    def _get_shard(self, key):
        """Route a key to the correct shard instance"""
        shard_id = self.router.get_shard(key)
        if shard_id not in self.shards:
            raise ValueError(f"Shard {shard_id} not found")
        self.total_routed += 1
        return self.shards[shard_id]
    
    # ================================================================
    # SINGLE-SHARD OPERATIONS (fast — know which shard)
    # ================================================================
    
    def insert(self, short_code, long_url, user_id=None):
        """
        Insert a URL mapping.
        Routes to correct shard based on short_code.
        """
        shard = self._get_shard(short_code)
        result = shard.insert(short_code, long_url, user_id)
        result["routed_to_shard"] = shard.shard_id
        
        self.routing_log.append({
            "operation": "insert",
            "key": short_code,
            "shard": shard.shard_id,
            "success": result["success"],
            "timestamp": datetime.now().isoformat()
        })
        
        return result
    
    def lookup(self, short_code):
        """
        Look up a URL by short code.
        Single-shard operation — O(1) routing.
        """
        shard = self._get_shard(short_code)
        result = shard.lookup(short_code)
        result["routed_to_shard"] = shard.shard_id
        return result
    
    def delete(self, short_code):
        """Delete a URL mapping from the correct shard."""
        shard = self._get_shard(short_code)
        return shard.delete(short_code)
    
    def increment_clicks(self, short_code):
        """Increment click count on the correct shard."""
        shard = self._get_shard(short_code)
        return shard.increment_clicks(short_code)
    
    # ================================================================
    # SCATTER-GATHER OPERATIONS (slow — must query all shards)
    # ================================================================
    
    def total_count(self):
        """
        Count total rows across ALL shards.
        This is a SCATTER-GATHER operation.
        Must query every shard and sum results.
        """
        self.total_scatter_gather += 1
        total = 0
        per_shard = {}
        
        for shard_id, shard in self.shards.items():
            count = shard.count()
            per_shard[shard_id] = count
            total += count
        
        return {
            "total": total,
            "per_shard": per_shard,
            "operation": "scatter-gather",
            "shards_queried": len(self.shards)
        }
    
    def query_by_user(self, user_id, limit=50):
        """
        Find all URLs by a user.
        
        SCATTER-GATHER: User's URLs can be on ANY shard
        (because we shard by short_code, not user_id).
        Must query ALL shards and merge results.
        
        This is the COST of sharding by short_code.
        If we sharded by user_id, this would be single-shard.
        """
        self.total_scatter_gather += 1
        all_results = []
        
        # Query all shards in parallel
        threads = []
        results_lock = threading.Lock()
        
        def query_shard(shard):
            rows = shard.query_by_user(user_id, limit)
            with results_lock:
                all_results.extend(rows)
        
        for shard in self.shards.values():
            t = threading.Thread(target=query_shard, args=(shard,))
            threads.append(t)
            t.start()
        
        for t in threads:
            t.join()
        
        # Sort merged results by created_at (most recent first)
        all_results.sort(key=lambda r: r.get("created_at", ""), reverse=True)
        
        # Apply limit to merged results
        return {
            "user_id": user_id,
            "total_found": len(all_results),
            "urls": all_results[:limit],
            "operation": "scatter-gather",
            "shards_queried": len(self.shards)
        }
    
    def top_urls(self, limit=10):
        """
        Get top URLs by click count across all shards.
        Another scatter-gather operation.
        """
        self.total_scatter_gather += 1
        all_urls = []
        
        for shard_id, shard in self.shards.items():
            try:
                conn = sqlite3.connect(shard.db_path)
                cursor = conn.cursor()
                cursor.execute(
                    """SELECT short_code, long_url, click_count 
                       FROM urls ORDER BY click_count DESC LIMIT ?""",
                    (limit,)
                )
                for row in cursor.fetchall():
                    all_urls.append({
                        "short_code": row[0],
                        "long_url": row[1],
                        "click_count": row[2],
                        "shard": shard_id
                    })
                conn.close()
            except Exception:
                pass
        
        # Merge and sort globally
        all_urls.sort(key=lambda x: x["click_count"], reverse=True)
        return all_urls[:limit]
    
    # ================================================================
    # SHARD MANAGEMENT
    # ================================================================
    
    def get_shard_stats(self):
        """Get statistics for every shard"""
        stats = {}
        total_rows = 0
        total_reads = 0
        total_writes = 0
        
        for shard_id, shard in sorted(self.shards.items(),
                                       key=lambda x: str(x[0])):
            s = shard.get_stats()
            stats[shard_id] = s
            total_rows += s["row_count"]
            total_reads += s["reads"]
            total_writes += s["writes"]
        
        # Calculate imbalance
        counts = [s["row_count"] for s in stats.values()]
        avg_count = total_rows / len(counts) if counts else 0
        max_count = max(counts) if counts else 0
        min_count = min(counts) if counts else 0
        
        imbalance = 0
        if avg_count > 0:
            imbalance = round((max_count - min_count) / avg_count * 100, 1)
        
        return {
            "num_shards": len(self.shards),
            "total_rows": total_rows,
            "total_reads": total_reads,
            "total_writes": total_writes,
            "total_routed": self.total_routed,
            "total_scatter_gather": self.total_scatter_gather,
            "imbalance_pct": imbalance,
            "avg_rows_per_shard": round(avg_count, 0),
            "shards": stats,
            "router_type": self.router.name
        }
    
    def add_shard(self, new_shard_id):
        """
        Add a new shard to the cluster.
        
        Steps:
          1. Create new shard instance
          2. Update router to include new shard
          3. Migrate affected keys from existing shards
        """
        if new_shard_id in self.shards:
            return {"success": False, "error": "Shard already exists"}
        
        # Create new shard
        db_path = os.path.join(self.shard_dir, f"shard_{new_shard_id}.db")
        new_shard = ShardInstance(new_shard_id, db_path)
        self.shards[new_shard_id] = new_shard
        
        # Collect all existing keys and their current assignments
        print(f"  [REBALANCE] Collecting keys from all shards...")
        all_keys = {}
        for shard_id, shard in self.shards.items():
            if shard_id == new_shard_id:
                continue
            for row in shard.get_all_rows():
                all_keys[row["short_code"]] = {
                    "data": row,
                    "current_shard": shard_id
                }
        
        # Update router
        if isinstance(self.router, ConsistentHashShardRouter):
            self.router.add_shard(new_shard_id)
        elif isinstance(self.router, HashShardRouter):
            self.router.num_shards += 1
        
        # Find keys that need to move
        migrations = []
        for key, info in all_keys.items():
            new_target = self.router.get_shard(key)
            if new_target != info["current_shard"]:
                migrations.append({
                    "key": key,
                    "data": info["data"],
                    "from_shard": info["current_shard"],
                    "to_shard": new_target
                })
        
        # Execute migrations
        print(f"  [REBALANCE] Migrating {len(migrations)} keys...")
        migrated = 0
        failed = 0
        
        for mig in migrations:
            target_shard = self.shards.get(mig["to_shard"])
            source_shard = self.shards.get(mig["from_shard"])
            
            if not target_shard or not source_shard:
                failed += 1
                continue
            
            # Insert into new shard
            result = target_shard.insert(
                mig["data"]["short_code"],
                mig["data"]["long_url"],
                mig["data"]["user_id"]
            )
            
            if result["success"]:
                # Delete from old shard
                source_shard.delete(mig["data"]["short_code"])
                migrated += 1
            else:
                failed += 1
        
        return {
            "success": True,
            "new_shard": new_shard_id,
            "total_keys": len(all_keys),
            "keys_migrated": migrated,
            "keys_failed": failed,
            "migration_pct": round(migrated / len(all_keys) * 100, 1) if all_keys else 0
        }
    
    def print_dashboard(self):
        """Print a visual dashboard of shard health"""
        stats = self.get_shard_stats()
        
        print(f"\n{'═' * 70}")
        print(f"  SHARDED DATABASE DASHBOARD")
        print(f"  Router: {stats['router_type']} | "
              f"Shards: {stats['num_shards']} | "
              f"Total Rows: {stats['total_rows']:,}")
        print(f"  Imbalance: {stats['imbalance_pct']}% | "
              f"Routed: {stats['total_routed']:,} | "
              f"Scatter-Gather: {stats['total_scatter_gather']}")
        print(f"{'═' * 70}")
        
        max_rows = max(
            (s["row_count"] for s in stats["shards"].values()), default=1
        ) or 1
        
        for shard_id in sorted(stats["shards"].keys(), key=str):
            s = stats["shards"][shard_id]
            bar_len = int(s["row_count"] / max_rows * 30)
            bar = "█" * bar_len + "░" * (30 - bar_len)
            
            print(f"  Shard {str(shard_id):<8} │{bar}│ "
                  f"{s['row_count']:>7,} rows | "
                  f"R:{s['reads']:>5} W:{s['writes']:>5} "
                  f"E:{s['errors']:>3} | "
                  f"{s['db_size_kb']:.0f}KB")
        
        print(f"{'═' * 70}\n")
    
    def cleanup(self):
        """Remove all shard databases"""
        if os.path.exists(self.shard_dir):
            shutil.rmtree(self.shard_dir)

WHAT WE LEARNED IN PROJECT 18:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Sharding splits ONE database into MULTIPLE independent databases
✅ Each shard holds a SUBSET of data, reducing per-shard load
✅ Shard key is the MOST IMPORTANT decision (determines query patterns)
✅ Hash-based: even distribution but terrible rebalancing
✅ Range-based: great for range queries but prone to hotspots
✅ Consistent hash: best rebalancing (minimal data movement)
✅ Directory-based: maximum flexibility but needs a directory service
✅ Logical shards → physical servers: decouples routing from hardware
✅ Scatter-gather: queries without shard key must hit ALL shards
✅ Cross-shard JOINs are extremely expensive → denormalize instead
✅ Global ID generation: Snowflake IDs or shard-prefixed ranges
✅ Rebalancing is the HARDEST problem in sharding

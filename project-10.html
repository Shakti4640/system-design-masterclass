PROJECT 10: THE BOOK INDEX — DATABASE INDEXING & QUERY OPTIMIZATION

🎯 HYPOTHETICAL PROBLEM

Your pizza shop (Project 3 — Database) has been running for 2 years
The orders table now has 5,000,000 rows
The customers table has 200,000 rows
The menu_items table has 150 rows

Everything was FAST when you had 1,000 orders
Now things are BREAKING:

SCENARIO A: The Slow Dashboard
  → Manager opens the dashboard every morning
  → Dashboard runs: "Show me all orders from yesterday"
  → Query: SELECT * FROM orders WHERE order_date = '2025-01-15'
  → With 1,000 rows → instant (2ms)
  → With 5,000,000 rows → 8.5 SECONDS
  → Manager stares at a loading spinner every morning

SCENARIO B: The Customer Lookup Nightmare
  → Customer calls: "Where is my order?"
  → Support agent searches by phone number
  → Query: SELECT * FROM customers WHERE phone = '555-1234'
  → Database scans ALL 200,000 customers one by one
  → Takes 1.2 seconds per lookup
  → 50 support agents doing this simultaneously = database overload

SCENARIO C: The Complex Report
  → Finance team needs: "Total revenue per pizza type, last 30 days"
  → Query: SELECT pizza_type, SUM(total_price) 
            FROM orders 
            WHERE order_date >= '2024-12-15' 
            GROUP BY pizza_type
  → Database scans 5 million rows, groups them, sums them
  → Takes 25 SECONDS
  → Finance runs this 20 times/day with different date ranges

SCENARIO D: The Sorting Catastrophe
  → API endpoint: GET /orders?sort=created_at&page=500
  → Database must SORT 5 million rows by date
  → Then skip to row 5000 and return 10
  → Sorting 5M rows takes 12 seconds
  → Just to show 10 rows on page 500

THE ANALOGY:
  → Imagine a 500-page book with NO INDEX at the back
  → Someone asks: "Find every mention of pepperoni"
  → You must read EVERY SINGLE PAGE, word by word
  → That's what the database is doing: FULL TABLE SCAN

  → Now imagine the book HAS an index:
    "Pepperoni: pages 12, 45, 67, 234, 456"
  → You flip directly to those pages
  → 1000x faster, same book, same content

THE QUESTION:
  → How do databases find rows WITHOUT reading every single one?
  → How do you decide WHICH columns to index?
  → What's the COST of having indexes?
  → When do indexes HURT instead of help?

🔍 HOW TO APPROACH IT

Step 1: Understand WHY queries are slow
         → Without an index, database does a FULL TABLE SCAN
         → Reads every row, checks if it matches your WHERE clause
         → O(N) — linear with table size
         → 5M rows × ~0.001ms per row = ~5 seconds

Step 2: Understand WHAT an index is
         → A separate data structure that maps column values → row locations
         → Like a phone book: sorted by name → points to address
         → Stored alongside the table, maintained automatically

Step 3: Understand HOW indexes work internally
         → B-Tree: balanced tree structure (most common)
         → Hash Index: exact-match only, O(1) lookup
         → Each has different strengths

Step 4: Identify WHICH queries are slow
         → Look at WHERE clauses → those columns need indexes
         → Look at ORDER BY clauses → those columns benefit from indexes
         → Look at JOIN conditions → foreign keys need indexes
         → Look at GROUP BY clauses → aggregation columns

Step 5: Understand the TRADEOFF
         → Indexes speed up READS (SELECT)
         → Indexes slow down WRITES (INSERT, UPDATE, DELETE)
         → Every write must update BOTH the table AND all its indexes
         → Too many indexes = slow writes

Step 6: Measure with EXPLAIN / QUERY PLAN
         → Don't guess — ask the database what it's doing
         → EXPLAIN SELECT ... shows the execution plan
         → Tells you: full scan vs index scan vs index seek

INDEX TYPES WE WILL IMPLEMENT AND UNDERSTAND:

TYPE 1: SINGLE-COLUMN INDEX (B-Tree)
═════════════════════════════════════
  CREATE INDEX idx_order_date ON orders(order_date);
  
  Before: Scan 5,000,000 rows → 8.5 seconds
  After:  B-Tree lookup → 5 milliseconds
  
  How: B-Tree organizes values in sorted order
       → Binary search to find the right leaf
       → Leaf points to actual row locations
       → O(log N) instead of O(N)


TYPE 2: COMPOSITE INDEX (Multi-Column)
══════════════════════════════════════
  CREATE INDEX idx_user_date ON orders(user_id, order_date);
  
  For query: WHERE user_id = 'alice' AND order_date >= '2025-01-01'
  → First narrows by user_id (leftmost column)
  → Then narrows by date within that user's orders
  → Much faster than two separate single-column indexes
  
  CRITICAL RULE: Leftmost Prefix
  → Index on (A, B, C) can be used for:
    ✅ WHERE A = ?
    ✅ WHERE A = ? AND B = ?
    ✅ WHERE A = ? AND B = ? AND C = ?
    ❌ WHERE B = ?        (skips A — can't use index)
    ❌ WHERE C = ?        (skips A and B — can't use index)
    ❌ WHERE B = ? AND C = ? (skips A — can't use index)


TYPE 3: COVERING INDEX
══════════════════════
  → An index that contains ALL columns the query needs
  → Database never needs to read the actual table row
  → "Index-only scan" — fastest possible
  
  CREATE INDEX idx_covering ON orders(user_id, order_date, total_price);
  
  SELECT total_price FROM orders 
  WHERE user_id = 'alice' AND order_date = '2025-01-15';
  
  → All data is IN the index itself
  → Zero table row lookups needed


TYPE 4: UNIQUE INDEX
════════════════════
  CREATE UNIQUE INDEX idx_email ON customers(email);
  
  → Enforces uniqueness constraint
  → Also serves as a fast lookup index
  → Database can stop after finding ONE match (knows there's only one)


TYPE 5: HASH INDEX
══════════════════
  → O(1) exact-match lookups
  → Cannot do range queries (>, <, BETWEEN)
  → Cannot do sorting (ORDER BY)
  → Available in some databases (PostgreSQL, MySQL Memory engine)


B-TREE VISUAL STRUCTURE:
═══════════════════════

  Index on order_date for 5 million rows:
  
              ┌─────────────────────┐
              │   [2024-06-15]      │          ← Root (1 node)
              │   [2024-12-01]      │
              └──┬──────┬───────┬───┘
                 │      │       │
        ┌────────┘      │       └────────┐
        ▼               ▼                ▼
  ┌──────────┐   ┌──────────┐     ┌──────────┐
  │ Jan-Jun  │   │ Jul-Nov  │     │ Dec-Jun  │    ← Internal (few nodes)
  │ pointers │   │ pointers │     │ pointers │
  └──┬───┬───┘   └──┬───┬───┘     └──┬───┬───┘
     │   │          │   │            │   │
     ▼   ▼          ▼   ▼            ▼   ▼
  ┌──────┐ ┌──────┐                            ← Leaf nodes (many)
  │Jan 1 │→│Jan 2 │→│Jan 3 │→ ... →│Jun 30│   ← Linked list!
  │row:5 │ │row:12│ │row:8 │       │row:99│
  │row:23│ │row:45│ │row:67│       │      │
  └──────┘ └──────┘ └──────┘       └──────┘
  
  Finding "order_date = 2025-01-15":
    → Start at root: 2025-01-15 < 2024-06-15? No → go right
    → Internal node: find correct child
    → Leaf node: exact match → row pointers [5, 23, 178, ...]
    → Fetch those specific rows from table
    
  Depth of tree: log₂(5,000,000) ≈ 23 levels
  But B-Tree has high branching factor (~100-500 per node)
  So actual depth: log₅₀₀(5,000,000) ≈ 2-3 levels
  → Only 2-3 disk reads to find ANY value!


FULL TABLE SCAN vs INDEX SCAN vs INDEX SEEK:
═════════════════════════════════════════════

  ┌─────────────────┬──────────────┬────────────────────────────┐
  │ Access Method    │ Performance  │ When Used                  │
  ├─────────────────┼──────────────┼────────────────────────────┤
  │ Full Table Scan │ O(N)         │ No index, or query returns │
  │                 │ Slowest      │ >20-30% of all rows        │
  ├─────────────────┼──────────────┼────────────────────────────┤
  │ Index Scan      │ O(log N + K) │ Range query with index     │
  │                 │ Medium       │ Returns subset of rows     │
  ├─────────────────┼──────────────┼────────────────────────────┤
  │ Index Seek      │ O(log N)     │ Exact match with index     │
  │                 │ Fastest      │ Unique or near-unique value│
  ├─────────────────┼──────────────┼────────────────────────────┤
  │ Index-Only Scan │ O(log N + K) │ Covering index             │
  │                 │ Fastest*     │ No table access needed     │
  └─────────────────┴──────────────┴────────────────────────────┘
  
  * Index-Only is fastest because it avoids random I/O to table

✅ WHAT SHOULD DO

1. ALWAYS index foreign keys
   → orders.customer_id should ALWAYS have an index
   → JOINs on unindexed foreign keys = full table scan of both tables
   → Most databases auto-index PRIMARY KEY but NOT foreign keys

2. ALWAYS index columns used in WHERE clauses (high-frequency queries)
   → Identify your top 10 most-run queries
   → Index the columns in their WHERE conditions
   → Prioritize by frequency × cost

3. ALWAYS check EXPLAIN before and after adding an index
   → Don't assume the index is being used
   → Database optimizer might choose full scan anyway
   → Reasons: low selectivity, outdated statistics, wrong index

4. ALWAYS consider column SELECTIVITY
   → Selectivity = number of distinct values / total rows
   → High selectivity (email, phone, user_id) → great index candidates
   → Low selectivity (gender, boolean, status with 3 values) → poor indexes
   → Index on gender (M/F) → still scans 50% of table

5. ALWAYS use composite indexes for multi-column queries
   → WHERE user_id = ? AND status = 'active'
   → Single index on user_id: narrows to user's rows, then scans for status
   → Composite index on (user_id, status): jumps directly to active orders
   → Order matters: put the MOST SELECTIVE column first

6. ALWAYS consider the write impact
   → Measure INSERT/UPDATE speed before and after adding indexes
   → Rule of thumb: 1 index adds ~10-20% overhead to writes
   → 5 indexes on one table → writes are 50-100% slower
   → Find the balance for your read/write ratio

7. ALWAYS analyze slow query logs
   → Enable slow query logging (queries > 100ms)
   → Review weekly
   → Create indexes for the most expensive recurring queries

❌ WHAT SHOULD NOT DO

1. NEVER create an index on every column "just in case"
   → 20 columns × index each = 20 indexes
   → Every INSERT now updates 20 B-Trees
   → Storage grows 3-5x
   → Write throughput drops dramatically
   → Indexes have COST — be intentional

2. NEVER index low-selectivity columns alone
   → Column "is_active" has values: true/false
   → Index on is_active → still scans 50% of table
   → Database optimizer will often IGNORE this index
   → EXCEPTION: When combined in a composite index

3. NEVER forget the leftmost prefix rule for composite indexes
   → Index on (A, B, C)
   → Query: WHERE B = ? → INDEX NOT USED
   → This is the #1 composite index mistake
   → If you need B alone, create a separate index on B

4. NEVER ignore index maintenance
   → As data is inserted/deleted, indexes develop "holes" (fragmentation)
   → Periodically REBUILD or REORGANIZE indexes
   → Stale statistics → optimizer makes wrong choices
   → Run ANALYZE TABLE periodically

5. NEVER use functions on indexed columns in WHERE
   → ❌ WHERE YEAR(order_date) = 2025
     → Database can't use index (function wraps the column)
   → ✅ WHERE order_date >= '2025-01-01' AND order_date < '2026-01-01'
     → Database CAN use index on order_date

6. NEVER assume the database will use your index
   → Optimizer has its own cost model
   → If query returns >20-30% of rows → full scan is FASTER
   → Reading random index pointers → lots of random I/O
   → Sequential full scan → one smooth read
   → Check EXPLAIN to verify

7. NEVER create redundant indexes
   → Index on (user_id) AND index on (user_id, order_date)
   → The composite index already covers WHERE user_id = ?
   → Single-column index is redundant — delete it
   → Saves storage and write overhead

🚫 WHEN NOT TO USE

1. DO NOT index tiny tables (< 1000 rows)
   → Full scan of 1000 rows takes < 1ms
   → Index lookup overhead might be MORE than just scanning
   → Database optimizer will likely ignore the index anyway
   → Indexes shine on large tables (>10,000 rows)

2. DO NOT index columns that change constantly
   → Column "last_seen_at" updated every request per user
   → Every update = B-Tree rebalance
   → If this column isn't queried in WHERE → don't index it

3. DO NOT index when query returns most of the table
   → "Find all active users" when 95% are active
   → Index says "go to row 1, row 2, row 3, ... row 950,000"
   → Random I/O for each row → SLOWER than sequential scan
   → Optimizer knows this and ignores the index

4. DO NOT use indexes as a substitute for query optimization
   → Bad query + good index is still slow
   → SELECT * when you only need 2 columns → wastes I/O
   → N+1 query problem → 1000 queries instead of 1
   → Fix the query FIRST, then add indexes

5. DO NOT create indexes during peak traffic
   → Index creation locks the table (in many databases)
   → Or uses significant CPU/I/O
   → Create indexes during maintenance windows
   → Use CONCURRENTLY option if available (PostgreSQL)

⚙️ HOW IT ONLY WORKS (INTERNAL MECHANICS)

B-TREE DEEP DIVE — HOW IT ACTUALLY WORKS:
══════════════════════════════════════════

STRUCTURE:
  → B-Tree is a balanced, sorted, self-organizing tree
  → Each node contains MULTIPLE keys (not just 2 like binary tree)
  → "B" stands for "Balanced" (or Bayer, the inventor)
  → All leaf nodes are at the SAME depth (perfectly balanced)

NODE STRUCTURE:
  ┌──────────────────────────────────────────┐
  │  [key1] [key2] [key3] ... [keyN]        │  ← Keys (sorted)
  │  [ptr0] [ptr1] [ptr2] [ptr3] ... [ptrN] │  ← Child pointers
  │  [val1] [val2] [val3] ... [valN]        │  ← Row pointers (leaf only)
  └──────────────────────────────────────────┘
  
  → Internal nodes: keys + child pointers (routing)
  → Leaf nodes: keys + row pointers (actual data locations)
  → Leaf nodes are LINKED together (doubly linked list)
    → This enables efficient RANGE SCANS

PARAMETERS:
  → Order (t): Minimum number of keys per node
  → Each node has between t-1 and 2t-1 keys
  → Typical order in databases: 100-500
  → Why? Each node = one disk page (4KB or 8KB)
  → More keys per node = fewer levels = fewer disk reads

EXAMPLE: Order t=3 (simplified for illustration)
  → Each node: 2 to 5 keys
  → For 1 million rows:
    → Binary tree depth: log₂(1,000,000) = 20 levels
    → B-Tree depth: log₅₀₀(1,000,000) ≈ 3 levels
    → 3 disk reads vs 20 disk reads!


SEARCH OPERATION (finding order_date = '2025-01-15'):
═══════════════════════════════════════════════════

  Step 1: Load root node from disk (1 disk read)
    Root: [2024-04-01, 2024-08-01, 2024-12-01]
    2025-01-15 > 2024-12-01 → follow rightmost child pointer

  Step 2: Load internal node (1 disk read)
    Node: [2025-01-01, 2025-02-01, 2025-03-01]
    2025-01-15 is between 2025-01-01 and 2025-02-01 → follow that pointer

  Step 3: Load leaf node (1 disk read)
    Leaf: [2025-01-14, 2025-01-15, 2025-01-15, 2025-01-16]
    Found! Row pointers: [row#45023, row#45089]

  Step 4: Fetch actual rows using pointers (1-2 disk reads)
    Go to row#45023 in the table → read full row data
    Go to row#45089 in the table → read full row data

  TOTAL: 3-5 disk reads (vs 5,000,000 for full scan)


INSERT OPERATION (adding a new order):
══════════════════════════════════════

  Step 1: Search for correct leaf node (same as search)
  Step 2: Insert key in sorted position within leaf
  Step 3: If leaf is FULL (2t-1 keys):
    → SPLIT the node into two halves
    → Push middle key UP to parent
    → If parent is full → split parent too (cascading)
    → This is how the tree stays BALANCED

  Before split:
    [10, 20, 30, 40, 50]  ← Full (5 keys, t=3)
    
  After inserting 35:
    Parent gets 30 pushed up
         [30]
        /    \
  [10, 20]  [35, 40, 50]  ← Two nodes, balanced


DELETE OPERATION:
═════════════════
  → Find and remove the key
  → If node becomes too empty (< t-1 keys):
    → Borrow from sibling, or
    → Merge with sibling
  → Rebalancing propagates up if needed


RANGE SCAN (WHERE date BETWEEN '2025-01-01' AND '2025-01-31'):
═════════════════════════════════════════════════════════════

  Step 1: B-Tree search for '2025-01-01' → find starting leaf
  Step 2: Follow leaf LINKED LIST forward
    Leaf1: [01-01, 01-02, 01-03] → next →
    Leaf2: [01-04, 01-05, 01-06] → next →
    ...
    LeafN: [01-30, 01-31, 02-01] → STOP (past our range)
  
  → Linked list enables sequential scan within range
  → No need to go back to root for each date
  → This is WHY B-Tree is great for range queries


HOW THE DATABASE DECIDES WHETHER TO USE AN INDEX:
═════════════════════════════════════════════════

  Query: SELECT * FROM orders WHERE status = 'active'
  
  Database Optimizer thinks:
    1. Table has 5,000,000 rows
    2. Index on 'status' exists
    3. Statistics say: 4,500,000 rows have status='active' (90%)
    4. Using index: 4,500,000 random I/O lookups
    5. Full scan: 1 sequential scan of entire table
    6. Sequential I/O is ~100x faster than random I/O
    7. DECISION: Full scan is cheaper → IGNORE the index
  
  But for: SELECT * FROM orders WHERE status = 'cancelled'
    1. Statistics say: 50,000 rows have status='cancelled' (1%)
    2. Using index: 50,000 random I/O lookups
    3. Full scan: 5,000,000 sequential reads
    4. DECISION: Index is cheaper → USE the index

  RULE OF THUMB:
    → < 5-15% of rows → Index is used
    → > 20-30% of rows → Full scan is preferred
    → In between → depends on the database and data layout


COMPOSITE INDEX INTERNAL STRUCTURE:
══════════════════════════════════

  Index on (user_id, order_date):
  
  B-Tree sorts by FIRST column, then SECOND within same first value:
  
  Leaf nodes (sorted):
  ┌────────────────────────────────────┐
  │ (alice, 2025-01-01) → row#5       │
  │ (alice, 2025-01-05) → row#23      │
  │ (alice, 2025-01-15) → row#67      │
  │ (bob,   2024-12-01) → row#12      │
  │ (bob,   2025-01-10) → row#45      │
  │ (charlie, 2025-01-01) → row#89    │
  └────────────────────────────────────┘
  
  Query: WHERE user_id = 'alice' AND order_date >= '2025-01-05'
  → Jump to 'alice' section (B-Tree search)
  → Within alice, jump to '2025-01-05' (already sorted)
  → Scan forward until user_id changes
  → ULTRA FAST

  Query: WHERE order_date = '2025-01-15' (no user_id filter)
  → Index is sorted by user_id FIRST
  → Can't jump to a date without knowing the user
  → Must scan ENTIRE index → almost as slow as full scan
  → THIS is the leftmost prefix rule


COVERING INDEX MECHANICS:
═════════════════════════

  Normal index lookup (2 phases):
    Phase 1: Search B-Tree → get row pointer
    Phase 2: Go to table → read the actual row (random I/O)
  
  Covering index (1 phase):
    Phase 1: Search B-Tree → all needed data is IN the index leaf
    Phase 2: SKIP — no need to go to the table
  
  Index on (user_id, order_date, total_price):
  
  Leaf node:
  ┌─────────────────────────────────────────────┐
  │ (alice, 2025-01-15, $25.99)                 │ ← All data here!
  │ (alice, 2025-01-16, $18.50)                 │
  └─────────────────────────────────────────────┘
  
  Query: SELECT total_price FROM orders 
         WHERE user_id = 'alice' AND order_date = '2025-01-15'
  → Found $25.99 directly in the index leaf
  → Zero table access → 50% fewer disk reads

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Index Not Used Due to Type Mismatch
  → Column user_id is INTEGER
  → Query: WHERE user_id = '123' (string, not integer)
  → Database must CAST every row to compare → can't use index
  → FIX: Match types exactly: WHERE user_id = 123

PITFALL 2: Index Not Used Due to Function on Column
  → Index on created_at
  → Query: WHERE DATE(created_at) = '2025-01-15'
  → DATE() function wraps the column → index bypassed
  → FIX: WHERE created_at >= '2025-01-15' AND created_at < '2025-01-16'

PITFALL 3: Leading Wildcard Kills Index
  → Index on name
  → Query: WHERE name LIKE '%pizza%'
  → Leading % → can't use B-Tree (doesn't know where to start)
  → FIX: Use full-text search index (Project 28 concept)
  → OK: WHERE name LIKE 'pizza%' (no leading %) → uses index

PITFALL 4: Composite Index Column Order Wrong
  → Index on (status, user_id)
  → Most queries filter by user_id first
  → WHERE user_id = 'alice' → can't use this index (leftmost rule)
  → FIX: Index on (user_id, status)

PITFALL 5: Too Many Indexes on Write-Heavy Table
  → Orders table: 10 indexes
  → Every new order: 1 table write + 10 index updates
  → Insert time: 0.5ms → 5ms (10x slower)
  → At 1000 inserts/second: 0.5s/s → 5s/s (falling behind)
  → FIX: Audit and remove unused indexes

PITFALL 6: Index Fragmentation
  → After millions of inserts/deletes, B-Tree has gaps
  → Pages are half-empty → more pages to read → slower
  → FIX: REINDEX or OPTIMIZE TABLE periodically

PITFALL 7: Stale Statistics
  → Database optimizer relies on statistics about data distribution
  → After bulk inserts, statistics are outdated
  → Optimizer makes wrong decisions (full scan vs index)
  → FIX: Run ANALYZE TABLE after major data changes

PITFALL 8: N+1 Query Problem (No Index Helps)
  → Loop: for each user, query their orders
  → 1000 users = 1000 separate queries
  → Each query uses index → fast individually
  → But 1000 queries × 2ms = 2 seconds total
  → FIX: Use JOIN or IN clause → 1 query instead of 1000
  → This is a QUERY problem, not an index problem

  → Index on user_id
  → SELECT * FROM orders WHERE user_id = 'alice'

📈 SCALABILITY CONSIDERATIONS

SINGLE TABLE, SINGLE SERVER:
  → Indexes solve most read performance problems
  → Up to ~100M rows on modern hardware
  → B-Tree depth stays at 3-4 levels
  → Index fits in memory → sub-millisecond lookups

WHEN INDEXES AREN'T ENOUGH:
  → Table exceeds RAM → index can't fit in memory
  → Disk-based B-Tree lookups: 5-10ms each
  → Still fast, but not as fast as in-memory
  → Solution: more RAM, or read replicas (Project 13)

WRITE AMPLIFICATION AT SCALE:
  → 100,000 inserts/second with 5 indexes
  → = 500,000 B-Tree modifications/second
  → Disk I/O becomes bottleneck
  → Solutions:
    → Reduce indexes on write-heavy tables
    → Batch inserts (bulk insert)
    → Write-ahead log (WAL) for crash safety
    → Async index updates (eventual consistency)

WHEN TO MOVE BEYOND INDEXES:
  → > 500M rows → consider sharding (Project 18)
  → Read replicas (Project 13): split reads across copies
  → Caching layer (Project 5): avoid hitting DB entirely
  → Search engine (Project 28/41): specialized for text search
  → Materialized views (Project 30): pre-computed aggregations

INDEX SIZE ESTIMATES:
  → Single column index (INTEGER): ~12 bytes per row
  → 5M rows × 12 bytes = ~60MB
  → Composite index (2 INTEGERs): ~20 bytes per row
  → 5M rows × 20 bytes = ~100MB
  → VARCHAR(255) index: ~varies, up to 767 bytes per entry
  → Total index size is often 50-200% of table size

NUMBERS TO KNOW:
  → B-Tree lookup: 2-4 disk reads = 0.1-1ms (SSD) / 5-20ms (HDD)
  → In-memory B-Tree lookup: 0.001-0.01ms
  → Full scan 1M rows: 100-500ms (SSD) / 5-30s (HDD)
  → Index creation on 5M rows: 10-60 seconds
  → Sweet spot: index on columns with selectivity > 10%

class BTreeNode:
    """
    A single node in the B-Tree
    Contains keys (sorted), children (if internal), values (if leaf)
    """
    
    def __init__(self, leaf: bool = False):
        self.leaf = leaf
        self.keys = []        # Sorted list of keys
        self.values = []      # Corresponding values (row pointers) — leaf only
        self.children = []    # Child node pointers — internal only
        self.next_leaf = None # Pointer to next leaf (for range scans)
        self.prev_leaf = None # Pointer to previous leaf
    
    def __repr__(self):
        return f"BTreeNode(keys={self.keys[:5]}{'...' if len(self.keys) > 5 else ''}, leaf={self.leaf})"

class BTreeIndex:
    """
    Complete B-Tree implementation
    Supports: insert, search, range scan, delete
    Tracks: comparisons, disk reads (simulated), node accesses
    """
    
    def __init__(self, order: int = 50):
        """
        order: Maximum number of children per node
        Each node has between ceil(order/2) and order children
        Each node has between ceil(order/2)-1 and order-1 keys
        """
        self.order = order
        self.max_keys = order - 1
        self.min_keys = (order + 1) // 2 - 1
        self.root = BTreeNode(leaf=True)
        self.height = 0
        self.total_keys = 0
        
        # Performance counters (simulating disk reads)
        self.stats = {
            "comparisons": 0,
            "node_accesses": 0,  # Each = 1 simulated disk read
            "splits": 0,
            "range_nodes_visited": 0
        }
    
    def reset_stats(self):
        for key in self.stats:
            self.stats[key] = 0
    
    # ---- SEARCH: Find exact key ----
    def search(self, key) -> Optional[Any]:
        """Find value associated with key, or None if not found"""
        self.reset_stats()
        node = self.root
        
        while True:
            self.stats["node_accesses"] += 1
            
            # Binary search within node's keys
            idx = self._binary_search(node.keys, key)
            
            if idx < len(node.keys) and node.keys[idx] == key:
                # Found the key
                if node.leaf:
                    return node.values[idx]
                else:
                    # In internal node — go to right child to find in leaf
                    node = node.children[idx + 1]
                    # Actually, for exact match in leaf, continue down
                    # Recheck — B+ tree stores all values in leaves
                    # Our implementation stores in both, so return here
                    return self._search_in_subtree(node, key)
            
            if node.leaf:
                return None  # Key not found
            
            # Go to appropriate child
            node = node.children[idx]
    
    def _search_in_subtree(self, node, key):
        while not node.leaf:
            self.stats["node_accesses"] += 1
            idx = self._binary_search(node.keys, key)
            node = node.children[idx]
        
        self.stats["node_accesses"] += 1
        idx = self._binary_search(node.keys, key)
        if idx < len(node.keys) and node.keys[idx] == key:
            return node.values[idx]
        return None
    
    def _binary_search(self, keys, target):
        """Find insertion point using binary search"""
        lo, hi = 0, len(keys)
        while lo < hi:
            mid = (lo + hi) // 2
            self.stats["comparisons"] += 1
            if keys[mid] < target:
                lo = mid + 1
            else:
                hi = mid
        return lo
    
    # ---- RANGE SCAN: Find all keys in [start, end] ----
    def range_scan(self, start_key, end_key) -> list:
        """Return all (key, value) pairs where start_key <= key <= end_key"""
        self.reset_stats()
        results = []
        
        # Find the leaf containing start_key
        node = self.root
        while not node.leaf:
            self.stats["node_accesses"] += 1
            idx = self._binary_search(node.keys, start_key)
            node = node.children[idx]
        
        # Scan through leaf nodes using linked list
        while node is not None:
            self.stats["node_accesses"] += 1
            self.stats["range_nodes_visited"] += 1
            
            for i, key in enumerate(node.keys):
                if key > end_key:
                    return results
                if key >= start_key:
                    results.append((key, node.values[i]))
            
            node = node.next_leaf  # Follow linked list to next leaf
        
        return results
    
    # ---- INSERT: Add a key-value pair ----
    def insert(self, key, value):
        """Insert key-value pair into the B-Tree"""
        root = self.root
        
        # If root is full, split it first
        if len(root.keys) >= self.max_keys:
            new_root = BTreeNode(leaf=False)
            new_root.children.append(self.root)
            self._split_child(new_root, 0)
            self.root = new_root
            self.height += 1
        
        self._insert_non_full(self.root, key, value)
        self.total_keys += 1
    
    def _insert_non_full(self, node, key, value):
        idx = self._binary_search(node.keys, key)
        
        if node.leaf:
            # Insert in sorted position
            node.keys.insert(idx, key)
            node.values.insert(idx, value)
        else:
            # Check if child is full
            if len(node.children[idx].keys) >= self.max_keys:
                self._split_child(node, idx)
                if key > node.keys[idx]:
                    idx += 1
            self._insert_non_full(node.children[idx], key, value)
    
    def _split_child(self, parent, child_idx):
        """Split a full child node into two nodes"""
        self.stats["splits"] += 1
        
        full_child = parent.children[child_idx]
        mid = len(full_child.keys) // 2
        
        # Create new node with right half
        new_node = BTreeNode(leaf=full_child.leaf)
        
        # Move upper half of keys/values to new node
        mid_key = full_child.keys[mid]
        
        new_node.keys = full_child.keys[mid + 1:]
        full_child.keys = full_child.keys[:mid]
        
        if full_child.leaf:
            new_node.values = full_child.values[mid + 1:]
            full_child.values = full_child.values[:mid]
            # Maintain leaf linked list
            new_node.next_leaf = full_child.next_leaf
            if full_child.next_leaf:
                full_child.next_leaf.prev_leaf = new_node
            full_child.next_leaf = new_node
            new_node.prev_leaf = full_child
            # Keep mid key in left leaf for range scan correctness
            full_child.keys.append(mid_key)
            full_child.values.append(full_child.values[-1] if full_child.values else None)
        else:
            new_node.children = full_child.children[mid + 1:]
            full_child.children = full_child.children[:mid + 1]
        
        # Push mid_key up to parent
        parent.keys.insert(child_idx, mid_key)
        parent.children.insert(child_idx + 1, new_node)
    
    # ---- TREE INFO ----
    def get_info(self) -> dict:
        node_count = self._count_nodes(self.root)
        leaf_count = self._count_leaves(self.root)
        
        return {
            "total_keys": self.total_keys,
            "height": self.height,
            "order": self.order,
            "total_nodes": node_count,
            "leaf_nodes": leaf_count,
            "internal_nodes": node_count - leaf_count,
            "avg_keys_per_node": round(self.total_keys / max(1, node_count), 1),
            "max_keys_per_node": self.max_keys
        }
    
    def _count_nodes(self, node) -> int:
        if node is None:
            return 0
        count = 1
        if not node.leaf:
            for child in node.children:
                count += self._count_nodes(child)
        return count
    
    def _count_leaves(self, node) -> int:
        if node is None:
            return 0
        if node.leaf:
            return 1
        count = 0
        for child in node.children:
            count += self._count_leaves(child)
        return count

class CompositeIndex:
    """Index on multiple columns using tuple keys"""
    
    def __init__(self, column_names: list):
        self.column_names = column_names
        self.tree = SimpleBTree()
        self.name = f"idx_{'_'.join(column_names)}"
    
    def insert(self, row: dict, row_id: int):
        key = tuple(row.get(col) for col in self.column_names)
        self.tree.insert(key, row_id)
    
    def search_exact(self, values: dict) -> list:
        key = tuple(values.get(col) for col in self.column_names)
        return self.tree.search(key)
    
    def search_prefix(self, values: dict) -> list:
        prefix_cols = []
        for col in self.column_names:
            if col in values:
                prefix_cols.append(col)
            else:
                break
        
        if not prefix_cols:
            return []
        
        results = []
        prefix_key = tuple(values[col] for col in prefix_cols)
        prefix_len = len(prefix_key)
        
        for key, row_ids in self.tree.data.items():
            if key[:prefix_len] == prefix_key:
                results.extend(row_ids)
        
        return results
    
    def search_range(self, exact_values: dict, range_col: str,
                     range_start, range_end) -> list:
        """Search with exact match on leading cols + range on next col"""
        prefix_cols = []
        for col in self.column_names:
            if col in exact_values:
                prefix_cols.append(col)
            elif col == range_col:
                break
            else:
                break
        
        results = []
        prefix_key = tuple(exact_values[col] for col in prefix_cols)
        prefix_len = len(prefix_key)
        range_idx = len(prefix_cols)
        
        for key, row_ids in self.tree.data.items():
            if key[:prefix_len] == prefix_key:
                if range_idx < len(key):
                    if range_start <= key[range_idx] <= range_end:
                        results.extend(row_ids)
                else:
                    results.extend(row_ids)
        
        return results
    
    def delete(self, row: dict, row_id: int):
        key = tuple(row.get(col) for col in self.column_names)
        self.tree.delete(key, row_id)
    
    def can_cover(self, needed_columns: set) -> bool:
        return needed_columns.issubset(set(self.column_names))

    # ---- QUERY PLANNER (EXPLAIN) ----
    
    def explain(self, where: dict = None, select_cols: list = None,
                order_by: str = None, range_filter: dict = None) -> dict:
        """
        Generate query execution plan (like EXPLAIN in SQL)
        range_filter: {"column": col, "start": val, "end": val}
        """
        plan = {
            "table": self.table_name,
            "total_rows": len(self.rows),
            "where": where,
            "range_filter": range_filter,
            "select": select_cols or ["*"],
            "order_by": order_by,
            "access_method": None,
            "index_used": None,
            "estimated_rows_examined": 0,
            "covering_index": False,
            "notes": []
        }
        
        if not where and not range_filter:
            plan["access_method"] = "FULL_TABLE_SCAN"
            plan["estimated_rows_examined"] = len(self.rows)
            plan["notes"].append("No WHERE clause → must scan all rows")
            return plan
        
        # Find best matching index
        best_index = None
        best_match_score = 0
        
        filter_cols = set()
        if where:
            filter_cols.update(where.keys())
        if range_filter:
            filter_cols.add(range_filter["column"])
        
        needed_cols = set(select_cols) if select_cols and "*" not in select_cols else set(self.columns)
        
        for name, idx in self.indexes.items():
            match_count = 0
            for col in idx.column_names:
                if where and col in where:
                    match_count += 1
                elif range_filter and col == range_filter["column"]:
                    match_count += 0.5  # Range is less selective than exact
                    break
                else:
                    break
            
            if match_count > best_match_score:
                best_match_score = match_count
                best_index = (name, idx)
        
        if best_index and best_match_score > 0:
            name, idx = best_index
            is_covering = idx.can_cover(needed_cols | filter_cols)
            
            if is_covering:
                plan["access_method"] = "INDEX_ONLY_SCAN"
                plan["notes"].append(
                    "All needed columns found in index → no table row access"
                )
                self.query_stats["index_only_scans"] += 1
            else:
                plan["access_method"] = "INDEX_SCAN"
                plan["notes"].append(
                    "Index narrows search → then fetch full rows from table"
                )
                self.query_stats["index_scans"] += 1
            
            plan["index_used"] = name
            plan["covering_index"] = is_covering
            plan["estimated_rows_examined"] = "~subset (index-filtered)"
            plan["notes"].append(
                f"Matched {int(best_match_score)}/{len(idx.column_names)} "
                f"leading index columns"
            )
            
            # Check if ORDER BY can use this index
            if order_by and order_by in idx.column_names:
                plan["notes"].append(
                    f"ORDER BY '{order_by}' satisfied by index (no sort needed)"
                )
            elif order_by:
                plan["notes"].append(
                    f"ORDER BY '{order_by}' requires additional sorting step"
                )
        else:
            plan["access_method"] = "FULL_TABLE_SCAN"
            plan["estimated_rows_examined"] = len(self.rows)
            self.query_stats["full_scans"] += 1
            
            if self.indexes:
                plan["notes"].append(
                    f"No matching index for WHERE columns {filter_cols}"
                )
                plan["notes"].append(
                    "Available indexes don't match (leftmost prefix rule)"
                )
            else:
                plan["notes"].append("No indexes exist on this table")
        
        self.query_stats["total_queries"] += 1
        return plan
    
    # ---- QUERY EXECUTION ----
    
    def query(self, where: dict = None, select_cols: list = None,
              order_by: str = None, limit: int = None,
              range_filter: dict = None) -> dict:
        """
        Execute a query and return results with performance metrics
        """
        start_time = time.time()
        rows_examined = 0
        index_used = None
        access_method = "FULL_TABLE_SCAN"
        
        # ---- Step 1: Find matching row IDs ----
        candidate_row_ids = None
        
        if where:
            # Try to use an index
            best_index = None
            best_score = 0
            
            for name, idx in self.indexes.items():
                match_count = 0
                for col in idx.column_names:
                    if col in where:
                        match_count += 1
                    else:
                        break
                if match_count > best_score:
                    best_score = match_count
                    best_index = (name, idx)
            
            if best_index and best_score > 0:
                name, idx = best_index
                index_used = name
                
                # Check if all WHERE columns match all index columns
                if best_score == len(where):
                    candidate_row_ids = set(idx.search_exact(where))
                else:
                    candidate_row_ids = set(idx.search_prefix(where))
                
                access_method = "INDEX_SCAN"
        
        if range_filter and where:
            # Try composite index with exact + range
            for name, idx in self.indexes.items():
                leading_match = 0
                for col in idx.column_names:
                    if col in where:
                        leading_match += 1
                    elif col == range_filter["column"]:
                        # Found range column after exact matches
                        range_ids = set(idx.search_range(
                            where,
                            range_filter["column"],
                            range_filter["start"],
                            range_filter["end"]
                        ))
                        if candidate_row_ids is not None:
                            candidate_row_ids &= range_ids
                        else:
                            candidate_row_ids = range_ids
                        index_used = name
                        access_method = "INDEX_SCAN"
                        break
                    else:
                        break
        
        # ---- Step 2: Fetch and filter rows ----
        results = []
        
        if candidate_row_ids is not None:
            # Index-assisted: only examine matching rows
            for row_id in candidate_row_ids:
                if row_id not in self.rows:
                    continue
                row = self.rows[row_id]
                rows_examined += 1
                
                # Apply any remaining filters not covered by index
                match = True
                if range_filter:
                    col = range_filter["column"]
                    val = row.get(col)
                    if val is None or val < range_filter["start"] or val > range_filter["end"]:
                        match = False
                
                if match:
                    if select_cols and "*" not in select_cols:
                        results.append({col: row[col] for col in select_cols if col in row})
                    else:
                        results.append(row.copy())
        else:
            # Full table scan: examine every row
            access_method = "FULL_TABLE_SCAN"
            for row_id, row in self.rows.items():
                rows_examined += 1
                
                match = True
                if where:
                    for col, val in where.items():
                        if row.get(col) != val:
                            match = False
                            break
                
                if match and range_filter:
                    col = range_filter["column"]
                    val = row.get(col)
                    if val is None or val < range_filter["start"] or val > range_filter["end"]:
                        match = False
                
                if match:
                    if select_cols and "*" not in select_cols:
                        results.append({col: row[col] for col in select_cols if col in row})
                    else:
                        results.append(row.copy())
        
        # ---- Step 3: Sort if needed ----
        sort_time = 0
        if order_by and results:
            sort_start = time.time()
            reverse = False
            col = order_by
            if order_by.startswith("-"):
                reverse = True
                col = order_by[1:]
            results.sort(key=lambda r: r.get(col, ""), reverse=reverse)
            sort_time = time.time() - sort_start
        
        # ---- Step 4: Apply limit ----
        if limit and len(results) > limit:
            results = results[:limit]
        
        total_time = time.time() - start_time
        
        self.query_stats["rows_examined_total"] += rows_examined
        
        return {
            "results": results,
            "count": len(results),
            "performance": {
                "total_time_ms": round(total_time * 1000, 3),
                "sort_time_ms": round(sort_time * 1000, 3),
                "rows_examined": rows_examined,
                "rows_returned": len(results),
                "access_method": access_method,
                "index_used": index_used,
                "table_total_rows": len(self.rows)
            }
        }
    
    def get_query_stats(self) -> dict:
        return self.query_stats.copy()

# Analyzes query patterns and recommends optimal indexes
# This is what tools like pg_stat_statements + auto-explain do

import time
import random
from collections import defaultdict, Counter


class QueryAnalyzer:
    """
    Records queries and recommends indexes based on patterns
    Like a slow query log analyzer + index advisor
    """
    
    def __init__(self):
        self.query_log = []
        self.column_frequency = Counter()
        self.column_pair_frequency = Counter()
        self.slow_queries = []
        self.slow_threshold_ms = 50
    
    def record_query(self, where_cols: list, select_cols: list,
                     order_by: str, execution_time_ms: float,
                     rows_examined: int, access_method: str):
        """Record a query execution for analysis"""
        entry = {
            "where_cols": tuple(sorted(where_cols)) if where_cols else (),
            "select_cols": tuple(sorted(select_cols)) if select_cols else (),
            "order_by": order_by,
            "time_ms": execution_time_ms,
            "rows_examined": rows_examined,
            "access_method": access_method,
            "timestamp": time.time()
        }
        
        self.query_log.append(entry)
        
        # Track column usage frequency
        if where_cols:
            for col in where_cols:
                self.column_frequency[col] += 1
            
            # Track column pairs (for composite index recommendations)
            if len(where_cols) >= 2:
                for i in range(len(where_cols)):
                    for j in range(i + 1, len(where_cols)):
                        pair = tuple(sorted([where_cols[i], where_cols[j]]))
                        self.column_pair_frequency[pair] += 1
        
        # Track slow queries
        if execution_time_ms > self.slow_threshold_ms:
            self.slow_queries.append(entry)
    
    def recommend_indexes(self, existing_indexes: dict = None) -> list:
        """Generate index recommendations based on query patterns"""
        recommendations = []
        existing = existing_indexes or {}
        existing_col_sets = set()
        
        for idx in existing.values():
            existing_col_sets.add(tuple(idx.column_names))
        
        # ---- Recommendation 1: Single column indexes for frequent WHERE ----
        print(f"\n  📊 Query Analysis ({len(self.query_log)} queries recorded)")
        print(f"  {'─' * 55}")
        
        print(f"\n  Column frequency in WHERE clauses:")
        for col, count in self.column_frequency.most_common(10):
            pct = count / len(self.query_log) * 100
            bar = "█" * int(pct / 2)
            print(f"    {col:<20s} | {count:>5} queries ({pct:>5.1f}%) {bar}")
            
            if count >= 5 and (col,) not in existing_col_sets:
                # Check if it's already a prefix of an existing composite
                already_covered = False
                for cols in existing_col_sets:
                    if cols[0] == col:
                        already_covered = True
                        break
                
                if not already_covered:
                    recommendations.append({
                        "type": "SINGLE_COLUMN",
                        "columns": [col],
                        "reason": f"Used in {count} queries ({pct:.1f}%)",
                        "priority": "HIGH" if pct > 20 else "MEDIUM"
                    })
        
        # ---- Recommendation 2: Composite indexes for frequent pairs ----
        if self.column_pair_frequency:
            print(f"\n  Column pair frequency (composite index candidates):")
            for pair, count in self.column_pair_frequency.most_common(5):
                pct = count / len(self.query_log) * 100
                print(f"    {pair} | {count} queries ({pct:.1f}%)")
                
                if count >= 3 and pair not in existing_col_sets:
                    recommendations.append({
                        "type": "COMPOSITE",
                        "columns": list(pair),
                        "reason": f"Columns queried together {count} times",
                        "priority": "HIGH" if count > 10 else "MEDIUM"
                    })
        
        # ---- Recommendation 3: Covering indexes for slow SELECT queries ----
        slow_patterns = defaultdict(list)
        for sq in self.slow_queries:
            pattern = (sq["where_cols"], sq["select_cols"])
            slow_patterns[pattern].append(sq["time_ms"])
        
        if slow_patterns:
            print(f"\n  Slow query patterns (>{self.slow_threshold_ms}ms):")
            for (where, select), times in sorted(
                slow_patterns.items(),
                key=lambda x: -len(x[1])
            )[:5]:
                avg_time = sum(times) / len(times)
                print(f"    WHERE: {where} SELECT: {select}")
                print(f"    Occurrences: {len(times)}, Avg time: {avg_time:.1f}ms")
                
                if where and select:
                    covering_cols = list(where) + [
                        c for c in select if c not in where
                    ]
                    if len(covering_cols) <= 5:
                        recommendations.append({
                            "type": "COVERING",
                            "columns": covering_cols,
                            "reason": f"Would eliminate {len(times)} slow queries "
                                     f"(avg {avg_time:.1f}ms)",
                            "priority": "HIGH"
                        })
        
        # ---- Recommendation 4: Remove unused indexes ----
        if existing:
            for name, idx in existing.items():
                cols = tuple(idx.column_names)
                # Check if any query uses this index's leading column
                used = any(
                    cols[0] in q["where_cols"]
                    for q in self.query_log
                    if q["where_cols"]
                )
                if not used and len(self.query_log) >= 20:
                    recommendations.append({
                        "type": "DROP",
                        "index_name": name,
                        "columns": list(cols),
                        "reason": "No queries use this index's leading column",
                        "priority": "LOW"
                    })
        
        # ---- Print Recommendations ----

# ============================================================
# PROJECT 10D: PAGINATION — OFFSET vs CURSOR (KEYSET)
# Demonstrates why OFFSET pagination breaks at scale
# and how cursor-based pagination stays fast
# ============================================================

import time
import random


class PaginationDemo:
    """
    Demonstrates two pagination approaches:
    1. OFFSET: Skip N rows (slow at deep pages)
    2. CURSOR: WHERE id > last_seen (fast at any depth)
    """
    
    def __init__(self, num_rows: int = 500_000):
        print(f"  Generating {num_rows:,} rows with sequential IDs...")
        
        # Simulated table: sorted by created_at (indexed)
        self.rows = []
        for i in range(num_rows):
            self.rows.append({
                "id": i,
                "name": f"order_{i:06d}",
                "value": round(random.uniform(5, 50), 2),
                "created_at": 1700000000 + i * 60  # 1 minute apart
            })
        
        self.num_rows = num_rows
    
    def paginate_offset(self, page: int, page_size: int = 20) -> dict:
        """
        Traditional OFFSET pagination
        SQL: SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET (page*20)
        
        PROBLEM: Database must still SCAN AND SKIP all offset rows
        """
        start = time.time()
        
        offset = page * page_size
        
        # Simulate: database scans from beginning, skips offset rows
        rows_scanned = 0
        results = []
        
        for i, row in enumerate(self.rows):
            rows_scanned += 1
            if i >= offset:
                results.append(row)
                if len(results) >= page_size:
                    break
        
        elapsed = time.time() - start
        
        return {
            "page": page,
            "results": results,
            "count": len(results),
            "rows_scanned": rows_scanned,
            "time_ms": round(elapsed * 1000, 3)
        }
    
    def paginate_cursor(self, cursor_id: int, page_size: int = 20) -> dict:
        """
        Cursor-based (keyset) pagination
        SQL: SELECT * FROM orders WHERE id > cursor_id ORDER BY id LIMIT 20
        
        ADVANTAGE: With index on id, jumps directly to cursor position
        """
        start = time.time()
        
        # Simulate: binary search to find cursor position (index lookup)
        lo, hi = 0, len(self.rows)
        while lo < hi:
            mid = (lo + hi) // 2
            if self.rows[mid]["id"] <= cursor_id:
                lo = mid + 1
            else:
                hi = mid
        
        # Read next page_size rows from that position
        results = self.rows[lo:lo + page_size]
        rows_scanned = len(results) + 1  # +1 for the index lookup
        
        elapsed = time.time() - start
        
        next_cursor = results[-1]["id"] if results else None
        
        return {
            "cursor_id": cursor_id,
            "next_cursor": next_cursor,
            "results": results,
            "count": len(results),
            "rows_scanned": rows_scanned,
            "time_ms": round(elapsed * 1000, 3)
        }


def run_pagination_comparison():
    print("=" * 70)
    print("  📄 PAGINATION: OFFSET vs CURSOR — PERFORMANCE AT SCALE")
    print("=" * 70)
    
    demo = PaginationDemo(num_rows=500_000)
    
    # Test at different page depths
    pages_to_test = [0, 10, 100, 1000, 5000, 10000, 24999]
    page_size = 20
    
    print(f"\n  Table size: {demo.num_rows:,} rows")
    print(f"  Page size: {page_size}")
    print(f"\n  {'Page':>8s} | {'─── OFFSET ───':>30s} | {'─── CURSOR ───':>30s} | {'Speedup':>8s}")
    print(f"  {'':>8s} | {'Time':>10s} {'Scanned':>10s} | {'Time':>10s} {'Scanned':>10s} | {'':>8s}")
    print(f"  {'─' * 8} | {'─' * 10} {'─' * 10} | {'─' * 10} {'─' * 10} | {'─' * 8}")
    
    for page in pages_to_test:
        # OFFSET approach
        offset_result = demo.paginate_offset(page, page_size)
        
        # CURSOR approach (cursor_id = page * page_size - 1)
        cursor_id = page * page_size - 1
        cursor_result = demo.paginate_cursor(cursor_id, page_size)
        
        speedup = offset_result["time_ms"] / max(cursor_result["time_ms"], 0.001)
        
        print(f"  {page:>8,} | "
              f"{offset_result['time_ms']:>8.3f}ms {offset_result['rows_scanned']:>9,} | "
              f"{cursor_result['time_ms']:>8.3f}ms {cursor_result['rows_scanned']:>9,} | "
              f"{speedup:>6.1f}x")
    
    print(f"\n  💡 KEY INSIGHTS:")
    print(f"  {'─' * 55}")
    print(f"  → OFFSET: Time grows LINEARLY with page number")
    print(f"     Page 0:     scans 20 rows")
    print(f"     Page 10000: scans 200,000 rows (just to SKIP them)")
    print(f"     Page 24999: scans 500,000 rows (entire table!)")
    print(f"")
    print(f"  → CURSOR: Time is CONSTANT regardless of page depth")
    print(f"     Page 0:     index seek + 20 rows")
    print(f"     Page 10000: index seek + 20 rows (same!)")
    print(f"     Page 24999: index seek + 20 rows (still same!)")
    print(f"")
    print(f"  → OFFSET is fine for pages 1-10")
    print(f"  → CURSOR is mandatory for deep pagination (infinite scroll)")
    print(f"")
    print(f"  → TRADEOFF: Cursor can't jump to arbitrary page number")
    print(f"     → No 'Go to page 500' button")
    print(f"     → Only 'Next page' and 'Previous page'")
    print(f"     → Perfect for: feeds, timelines, search results")
    print(f"     → Not ideal for: numbered page navigation")


if __name__ == "__main__":
    run_pagination_comparison()

THE PIZZA COUNTER ANALOGY:
  → Your pizza shop has ONE counter
  → One person takes orders, makes pizza, handles payment
  → 5 customers in line → fine
  → 50 customers in line → slow
  → 500 customers in line → people leave
  → 5,000 customers in line → riot

TWO SOLUTIONS:
  → BIGGER COUNTER (Vertical Scaling)
    → Hire a faster person, bigger oven, wider counter
    → But the counter can only get SO big

  → MORE COUNTERS (Horizontal Scaling)
    → Open 10 counters, each with their own person
    → Need a TRAFFIC DIRECTOR to tell customers which counter
    → Theoretically unlimited — just open more counters

THIS is the most fundamental decision in system design
→ Every system eventually faces this wall
→ HOW you prepare determines survival or death

NOW WE ADD CAPACITY THINKING:

Step 1: MEASURE current performance
         → Requests per second (throughput)
         → Response time per request (latency)
         → CPU, Memory, Disk, Network usage
         → Error rate

Step 2: IDENTIFY the bottleneck
         → Is it CPU? (computation-heavy)
         → Is it Memory? (too much data in RAM)
         → Is it Disk I/O? (database reads/writes)
         → Is it Network? (large responses, bandwidth)
         → Is it Connections? (too many concurrent clients)
         → RULE: System is only as fast as its slowest part

Step 3: CALCULATE capacity needed
         → Expected users × requests per user = total requests
         → Total requests / time = requests per second needed
         → Compare with current capacity → gap = scaling needed

Step 4: CHOOSE scaling strategy
         → Vertical: upgrade hardware (quick, simple, limited)
         → Horizontal: add instances (complex, powerful, unlimited)
         → Usually: vertical FIRST, horizontal WHEN needed

Step 5: IMPLEMENT and verify
         → Make the change
         → Measure again
         → Did it solve the problem?
         → What's the NEW bottleneck?

    → 100 servers (same):         $2,000/month → 50,000 req/sec
    → LINEAR SCALING — cost grows proportionally to capacity

📌 WHAT COULD BE DONE

Option A: Just Vertical Scale
  → Buy the biggest machine money can buy
  → Works up to ~10,000-50,000 req/sec
  → Simple, no architecture changes
  → When: You're a small-medium business and always will be

Option B: Just Horizontal Scale
  → Many small identical servers behind a load balancer
  → Requires stateless application design
  → When: You expect massive unpredictable growth

Option C: Scale Different Layers Differently
  → Application servers: horizontal (easy to add)
  → Database: vertical first, then replicas, then sharding
  → Cache layer: horizontal (distributed cache)
  → When: Most real-world systems (THIS IS THE WAY)

Option D: Auto-Scaling
  → Cloud infrastructure automatically adds/removes servers
  → Based on CPU, memory, request count thresholds
  → When: Traffic is unpredictable (viral events, sales)
  → AWS Auto Scaling, Kubernetes HPA

Option E: Pre-scaling / Over-provisioning
  → Keep extra capacity always running
  → Handle sudden spikes without delay
  → Expensive but safe
  → When: Revenue loss from downtime > cost of extra servers

For THIS project → Option C (understanding all layers)
→ Because real systems scale different components differently

✅ WHAT SHOULD DO

1. ALWAYS measure BEFORE scaling
   → You cannot optimize what you cannot measure
   → Profile: WHERE is time spent?
   → 80% of time in database queries? Scale DB, not app server
   → 80% of time in computation? Scale CPU
   → Scaling the wrong thing = wasting money

2. ALWAYS know your numbers
   → How many requests/second can one server handle?
   → What's the average response time?
   → What's the p95 response time? (95th percentile)
   → What's the error rate?
   → How much headroom do you have?

3. ALWAYS design for horizontal scaling FROM THE START
   → Don't store state in the application server
   → Don't write to local filesystem
   → Don't use in-memory data that isn't shared
   → Even if you're running ONE server now
   → Stateless design costs nothing extra but enables everything later

4. ALWAYS have capacity planning
   → Current: 1,000 users, 100 req/sec
   → Growth: 20% month over month
   → In 6 months: 3,000 users, 300 req/sec
   → In 12 months: 9,000 users, 900 req/sec
   → When do you hit the wall? Plan BEFORE that day

5. ALWAYS scale incrementally
   → Don't jump from 1 server to 100
   → 1 → 2 → 4 → 8 → 16
   → At each step: measure, verify, identify new bottleneck

6. ALWAYS consider the COST of scaling
   → More servers = more money
   → But also: downtime = lost revenue
   → Calculate: cost of scaling vs cost of outage
   → Usually: preventing 1 hour downtime > months of extra servers

❌ WHAT SHOULD NOT DO

1. NEVER scale without measuring first
   ❌ "Let's add 10 more servers, that should fix it"
   → What if the bottleneck is one slow database query?
   → 10 more servers hitting the same slow query = still slow
   → Measure → identify → target → scale

2. NEVER assume vertical scaling will always work
   ❌ "We'll just get a bigger server when needed"
   → The biggest server in the world still has limits
   → At some point, you MUST go horizontal
   → Plan for it now, implement later

3. NEVER scale prematurely
   ❌ Building microservices for 100 users
   ❌ Sharding database for 10,000 rows
   ❌ 20 servers for a prototype
   → Premature optimization is the root of all evil
   → Build simple, measure, scale when proven necessary

4. NEVER ignore the database bottleneck
   → 20 app servers all hitting 1 database
   → App layer scales, but DB is still the single bottleneck
   → More app servers = MORE load on the SAME database
   → Can actually make things WORSE
   → Must scale DB layer too (Project 13, 18)

5. NEVER store application state on the server
   ❌ user_sessions = {} (in-memory dictionary)
   ❌ uploaded files stored on local disk
   ❌ computed results cached in server memory only
   → When you add Server 2, it doesn't have Server 1's state
   → User bounces between servers → inconsistent experience
   → Solution: External state stores (Project 5, 8)

6. NEVER forget about the network
   → More servers = more network traffic between them
   → App server → Database: network hop
   → App server → Cache: network hop
   → Each hop adds ~0.5-2ms latency
   → 10 hops in a request = 5-20ms just in network

BOTTLENECK IDENTIFICATION DECISION TREE:

  "System is slow. Where's the problem?"
       │
       ├── CPU > 80%? ──→ CPU-bound
       │     ├── Single slow algorithm? → Optimize code
       │     └── Too many requests? → Add servers / scale up CPU
       │
       ├── Memory > 85%? ──→ Memory-bound
       │     ├── Memory leak? → Fix code
       │     └── Legitimate data? → More RAM / external cache
       │
       ├── Disk I/O saturated? ──→ I/O-bound
       │     ├── Database queries? → Indexes / caching
       │     └── File operations? → SSD upgrade / object storage
       │
       ├── Network saturated? ──→ Network-bound
       │     ├── Large responses? → Compression / CDN
       │     └── Too many calls? → Batching / reduce chattiness
       │
       └── None of above? ──→ External dependency
             ├── Slow third-party API? → Async / timeout / cache
             └── Database lock contention? → Optimize queries / sharding

🚫 WHEN NOT TO USE

1. VERTICAL SCALING is NOT suitable when:
   → You need 99.99% uptime (single point of failure)
   → Traffic is 100x during peaks vs normal
   → You're already on the biggest available machine
   → Cost of next tier is prohibitive
   → You need geographic distribution

2. HORIZONTAL SCALING is NOT suitable when:
   → Application stores state in memory (must refactor first)
   → Database schema doesn't support concurrent access well
   → Strong consistency is required across all requests
      (horizontal = eventual consistency challenges)
   → Team is too small to manage distributed infrastructure
   → Traffic doesn't justify the complexity

3. AUTO-SCALING is NOT suitable when:
   → Startup time for new servers > acceptable delay
      (if server takes 5 min to start but spike lasts 2 min)
   → Application has expensive initialization (DB migrations, etc.)
   → Cost unpredictability is unacceptable
   → Workload is steady (just over-provision instead)

4. SCALING is NOT the answer when:
   → The problem is a single slow query → optimize the query
   → The problem is a memory leak → fix the code
   → The problem is unnecessary computation → cache it (Project 5)
   → The problem is network chattiness → batch requests
   → ALWAYS optimize before scaling

PART 1: HOW A SERVER HANDLES CONCURRENT REQUESTS
═══════════════════════════════════════════════════════

  Single-Threaded Server (Project 1 basic):
  
  Request 1 ─────[processing 50ms]─────→ Response 1
  Request 2       [waiting.............]─[processing 50ms]─→ Response 2
  Request 3       [waiting.............................]─[50ms]─→ Response 3
  
  Total: 150ms for 3 requests (sequential)

  Multi-Threaded Server:
  
  Request 1 ─────[processing 50ms]─────→ Response 1
  Request 2 ─────[processing 50ms]─────→ Response 2
  Request 3 ─────[processing 50ms]─────→ Response 3
  
  Total: 50ms for 3 requests (parallel)
  BUT: Limited by number of CPU cores
  4 cores → max 4 truly parallel requests
  Thread overhead: ~1MB stack per thread
  1000 threads = 1GB just for thread stacks

  Event-Driven Server (async):
  
  Request 1 ─[start]──[wait DB]──[resume]──→ Response 1
  Request 2 ────[start]──[wait DB]──[resume]──→ Response 2
  Request 3 ──────[start]──[wait DB]──[resume]──→ Response 3
  
  One thread handles ALL requests
  While waiting for I/O → handle other requests
  Very efficient for I/O-heavy workloads
  Node.js, Python asyncio, Go goroutines

PART 2: THE MATH OF CAPACITY PLANNING
═══════════════════════════════════════════════════════

  KEY FORMULAS:

  1. Throughput (requests per second):
     ┌─────────────────────────────────────────────────┐
     │ throughput = concurrent_workers / avg_latency    │
     │                                                  │
     │ Example:                                         │
     │   4 worker threads, 50ms avg response time       │
     │   throughput = 4 / 0.050 = 80 req/sec           │
     │                                                  │
     │   8 worker threads, 50ms avg response time       │
     │   throughput = 8 / 0.050 = 160 req/sec          │
     │                                                  │
     │   8 worker threads, 25ms avg (optimized!)        │
     │   throughput = 8 / 0.025 = 320 req/sec          │
     └─────────────────────────────────────────────────┘

  2. Required servers:
     ┌─────────────────────────────────────────────────┐
     │ servers_needed = total_req_per_sec /             │
     │                  req_per_sec_per_server          │
     │                                                  │
     │ Example:                                         │
     │   Need: 5,000 req/sec                           │
     │   Each server handles: 500 req/sec              │
     │   servers_needed = 5000 / 500 = 10 servers      │
     │                                                  │
     │   Add 50% headroom: 10 × 1.5 = 15 servers      │
     │   (Never run at 100% — leave room for spikes)   │
     └─────────────────────────────────────────────────┘

  3. Little's Law (queue theory):
     ┌─────────────────────────────────────────────────┐
     │ L = λ × W                                       │
     │                                                  │
     │ L = average number of requests in system        │
     │ λ = arrival rate (requests per second)          │
     │ W = average time each request spends in system  │
     │                                                  │
     │ Example:                                         │
     │   λ = 100 req/sec, W = 200ms = 0.2s            │
     │   L = 100 × 0.2 = 20 concurrent requests       │
     │   Need at least 20 worker threads               │
     │                                                  │
     │   If λ increases to 1000 req/sec:               │
     │   L = 1000 × 0.2 = 200 concurrent requests     │
     │   Need 200 workers → likely multiple servers    │
     └─────────────────────────────────────────────────┘

  4. Amdahl's Law (diminishing returns):
     ┌─────────────────────────────────────────────────┐
     │ speedup = 1 / ((1 - p) + p/n)                  │
     │                                                  │
     │ p = fraction that can be parallelized           │
     │ n = number of processors/servers                │
     │                                                  │
     │ If 90% of work is parallelizable:               │
     │   2 servers:  speedup = 1/(0.1 + 0.9/2) = 1.82×│
     │   4 servers:  speedup = 1/(0.1 + 0.9/4) = 3.08×│
     │   10 servers: speedup = 1/(0.1 + 0.9/10)= 5.26×│
     │   100 servers:speedup = 1/(0.1 + 0.9/100)=9.17×│
     │   ∞ servers:  speedup = 1/0.1 = 10× MAX        │
     │                                                  │
     │ The 10% non-parallelizable part LIMITS you      │
     │ Even infinite servers → only 10× improvement    │
     │ This is WHY optimization matters MORE than       │
     │ just throwing hardware at the problem            │
     └─────────────────────────────────────────────────┘

PART 4: HOW HORIZONTAL SCALING ACTUALLY WORKS
═══════════════════════════════════════════════════════

  Step 1: Application is STATELESS
    → No in-memory sessions
    → No local file storage
    → Any server can handle any request
    → Server is "disposable" — can be added/removed freely

  Step 2: Load Balancer distributes traffic
    → Client connects to load balancer IP
    → LB forwards to one of N servers
    → Algorithms: round-robin, least-connections, etc.
    → (Deep dive in Project 6)

  Step 3: Shared external state
    → Database: shared across all servers (Project 3)
    → Cache: shared Redis/Memcached (Project 5)
    → Sessions: stored in cache or database (Project 8)
    → Files: stored in object storage (Project 25)

  Step 4: Deployment consistency
    → All servers run IDENTICAL code
    → Deploy new version to all servers simultaneously
    → Rolling deployment: update one at a time

  The Request Flow:
  
  Client → DNS → Load Balancer → Server 3 → Cache → Database
                                    ↓
                                Response
                                    ↓
  Client ← Load Balancer ← Server 3

  Next request from SAME client might hit Server 7
  → Works because application is STATELESS
  → All state is in shared database/cache

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Scaling the Wrong Layer
  → App servers at 20% CPU, database at 95% CPU
  → Adding more app servers → MORE load on already-struggling DB
  → System gets WORSE, not better
  → FIX: Always identify the actual bottleneck first

PITFALL 2: Forgetting About State
  → Application stores user sessions in memory
  → Scale to 3 servers
  → User logs in on Server 1, next request goes to Server 2
  → Server 2 doesn't know the user → "Please log in again"
  → FIX: External session store (Project 8)

PITFALL 3: Thundering Herd
  → Cache expires → ALL servers simultaneously hit database
  → Database overwhelmed by sudden spike
  → FIX: Staggered TTLs, cache stampede protection (Project 5)

PITFALL 4: Noisy Neighbor
  → One heavy endpoint consumes all CPU
  → All other endpoints become slow
  → FIX: Resource isolation, rate limiting (Project 9),
    separate heavy work into background jobs (Project 11)

PITFALL 5: Scaling Without Monitoring
  → Add 5 servers → "problem solved"
  → Don't know if they're actually being used
  → Don't know if new bottleneck emerged
  → FIX: Always monitor after scaling (Project 15)

PITFALL 6: Connection Pool Exhaustion
  → Each server opens 20 database connections
  → 10 servers × 20 connections = 200 connections
  → Database max connections = 100
  → Half the servers can't connect
  → FIX: Connection pooling, pgBouncer (connection proxy)

PITFALL 7: Latency vs Throughput Confusion
  → "We handle 10,000 req/sec!" (throughput)
  → But each request takes 5 seconds (latency)
  → Users experience 5-second waits
  → FIX: Measure BOTH, optimize for user experience (latency)

class ResourceLimitedServer:
    """
    Simulates a server with configurable resource limits.
    Demonstrates how different hardware specs affect performance.
    """
    
    def __init__(self, cpu_cores=2, ram_mb=512, max_connections=100,
                 db_pool_size=10, name="Server-1"):
        self.cpu_cores = cpu_cores
        self.ram_mb = ram_mb
        self.max_connections = max_connections
        self.db_pool_size = db_pool_size
        self.name = name
        
        # Simulated resource tracking
        self.active_connections = 0
        self.active_db_connections = 0
        self.cpu_usage = 0.0
        self.memory_used_mb = 128  # Base OS overhead
        self.total_requests = 0
        self.total_errors = 0
        self.response_times = []
        self.lock = threading.Lock()
        
        # Resource usage per request (simulated)
        self.cpu_per_request = 100 / (cpu_cores * 50)  # % per concurrent req
        self.memory_per_connection = 1.5  # MB per connection
    
    def handle_request(self, request_type="read"):
        """Simulate handling a request with resource constraints"""
        start_time = time.time()
        
        with self.lock:
            # Check connection limit
            if self.active_connections >= self.max_connections:
                self.total_errors += 1
                return {
                    "success": False,
                    "error": "CONNECTION_REFUSED",
                    "message": f"Max connections ({self.max_connections}) reached",
                    "latency_ms": 0,
                    "server": self.name
                }
            
            # Check DB pool
            if self.active_db_connections >= self.db_pool_size:
                self.total_errors += 1
                return {
                    "success": False,
                    "error": "DB_POOL_EXHAUSTED",
                    "message": f"DB pool ({self.db_pool_size}) exhausted",
                    "latency_ms": 0,
                    "server": self.name
                }
            
            # Check memory
            projected_memory = (self.memory_used_mb +
                                self.memory_per_connection)
            if projected_memory > self.ram_mb:
                self.total_errors += 1
                return {
                    "success": False,
                    "error": "OUT_OF_MEMORY",
                    "message": f"Memory exhausted "
                               f"({self.memory_used_mb:.0f}/{self.ram_mb} MB)",
                    "latency_ms": 0,
                    "server": self.name
                }
            
            # Acquire resources
            self.active_connections += 1
            self.active_db_connections += 1
            self.memory_used_mb += self.memory_per_connection
            self.cpu_usage = min(100,
                                 self.active_connections * self.cpu_per_request)
        
        try:
            # Simulate processing time
            # Higher CPU usage = slower processing (contention)
            base_time_ms = 10 if request_type == "read" else 25
            
            # CPU contention multiplier
            cpu_multiplier = 1.0
            if self.cpu_usage > 70:
                cpu_multiplier = 1 + (self.cpu_usage - 70) / 10
            if self.cpu_usage > 90:
                cpu_multiplier = 3 + (self.cpu_usage - 90) / 5
            
            # DB contention multiplier
            db_utilization = self.active_db_connections / self.db_pool_size
            db_multiplier = 1 + max(0, (db_utilization - 0.7)) * 5
            
            actual_time_ms = base_time_ms * cpu_multiplier * db_multiplier
            # Add some randomness
            actual_time_ms *= random.uniform(0.8, 1.2)
            
            # Simulate the work
            time.sleep(actual_time_ms / 1000)
            
            elapsed_ms = (time.time() - start_time) * 1000
            
            with self.lock:
                self.total_requests += 1
                self.response_times.append(elapsed_ms)
            
            return {
                "success": True,
                "latency_ms": round(elapsed_ms, 2),
                "server": self.name,
                "cpu_usage": round(self.cpu_usage, 1),
                "memory_mb": round(self.memory_used_mb, 1),
                "active_connections": self.active_connections
            }
        
        finally:
            # Release resources
            with self.lock:
                self.active_connections -= 1
                self.active_db_connections -= 1
                self.memory_used_mb -= self.memory_per_connection
                self.cpu_usage = max(0,
                    self.active_connections * self.cpu_per_request)
    
    def get_stats(self):
        """Get current server statistics"""
        with self.lock:
            if not self.response_times:
                return {
                    "server": self.name,
                    "total_requests": 0,
                    "total_errors": 0
                }
            
            sorted_times = sorted(self.response_times)
            total = len(sorted_times)
            
            return {
                "server": self.name,
                "specs": {
                    "cpu_cores": self.cpu_cores,
                    "ram_mb": self.ram_mb,
                    "max_connections": self.max_connections,
                    "db_pool_size": self.db_pool_size
                },
                "total_requests": self.total_requests,
                "total_errors": self.total_errors,
                "error_rate_pct": round(
                    self.total_errors /
                    max(1, self.total_requests + self.total_errors) * 100, 2
                ),
                "latency": {
                    "avg_ms": round(statistics.mean(sorted_times), 2),
                    "p50_ms": round(sorted_times[int(total * 0.50)], 2),
                    "p95_ms": round(sorted_times[int(total * 0.95)], 2),
                    "p99_ms": round(sorted_times[min(
                        int(total * 0.99), total - 1
                    )], 2),
                    "min_ms": round(sorted_times[0], 2),
                    "max_ms": round(sorted_times[-1], 2)
                },
                "current_state": {
                    "cpu_usage_pct": round(self.cpu_usage, 1),
                    "memory_used_mb": round(self.memory_used_mb, 1),
                    "memory_total_mb": self.ram_mb,
                    "active_connections": self.active_connections
                }
            }
    
    def reset_stats(self):
        with self.lock:
            self.total_requests = 0
            self.total_errors = 0
            self.response_times = []

class SimpleLoadBalancer:
    """
    Distributes requests across multiple servers.
    Uses round-robin for simplicity (Project 6 covers advanced).
    """
    
    def __init__(self, servers):
        self.servers = servers
        self.current_index = 0
        self.lock = threading.Lock()
        self.total_distributed = 0
    
    def get_next_server(self):
        with self.lock:
            server = self.servers[self.current_index % len(self.servers)]
            self.current_index += 1
            self.total_distributed += 1
            return server
    
    def handle_request(self, request_type="read"):
        server = self.get_next_server()
        return server.handle_request(request_type)
    
    def get_all_stats(self):
        return {
            "load_balancer": {
                "total_distributed": self.total_distributed,
                "server_count": len(self.servers)
            },
            "servers": [s.get_stats() for s in self.servers]
        }

class LoadTester:
    """
    Sends concurrent requests to measure performance.
    This is how you find your system's breaking point.
    """
    
    def __init__(self):
        self.results = []
    
    def run_test(self, target, concurrent_users, total_requests,
                 request_type="read", label="Test"):
        """
        target: Server or LoadBalancer with handle_request method
        concurrent_users: simulated simultaneous users
        total_requests: total number of requests to send
        """
        print(f"\n  🔥 LOAD TEST: {label}")
        print(f"  {'─' * 50}")
        print(f"  Concurrent users: {concurrent_users}")
        print(f"  Total requests:   {total_requests}")
        print(f"  Request type:     {request_type}")
        print(f"  Running...", end="", flush=True)
        
        results = []
        errors = defaultdict(int)
        start_time = time.time()
        
        def send_request(_):
            result = target.handle_request(request_type)
            return result
        
        with ThreadPoolExecutor(max_workers=concurrent_users) as executor:
            futures = [
                executor.submit(send_request, i)
                for i in range(total_requests)
            ]
            
            for future in as_completed(futures):
                try:
                    result = future.result()
                    results.append(result)
                    if not result["success"]:
                        errors[result["error"]] += 1
                except Exception as e:
                    errors["EXCEPTION"] += 1
        
        elapsed = time.time() - start_time
        
        # Analyze results
        successful = [r for r in results if r["success"]]
        failed = [r for r in results if not r["success"]]
        
        latencies = [r["latency_ms"] for r in successful]
        
        if latencies:
            latencies.sort()
            total_latencies = len(latencies)
            
            analysis = {
                "label": label,
                "config": {
                    "concurrent_users": concurrent_users,
                    "total_requests": total_requests,
                    "request_type": request_type
                },
                "results": {
                    "total_time_sec": round(elapsed, 3),
                    "throughput_rps": round(len(successful) / elapsed, 1),
                    "successful": len(successful),
                    "failed": len(failed),
                    "error_rate_pct": round(
                        len(failed) / total_requests * 100, 2
                    ),
                    "errors_by_type": dict(errors)
                },
                "latency": {
                    "avg_ms": round(statistics.mean(latencies), 2),
                    "p50_ms": round(
                        latencies[int(total_latencies * 0.50)], 2
                    ),
                    "p95_ms": round(
                        latencies[int(total_latencies * 0.95)], 2
                    ),
                    "p99_ms": round(
                        latencies[min(
                            int(total_latencies * 0.99),
                            total_latencies - 1
                        )], 2
                    ),
                    "min_ms": round(latencies[0], 2),
                    "max_ms": round(latencies[-1], 2)
                }
            }
        else:
            analysis = {
                "label": label,
                "results": {
                    "total_time_sec": round(elapsed, 3),
                    "throughput_rps": 0,
                    "successful": 0,
                    "failed": len(failed),
                    "error_rate_pct": 100.0,
                    "errors_by_type": dict(errors)
                },
                "latency": {"note": "No successful requests"}
            }
        
        print(f" Done!")
        self._print_analysis(analysis)
        self.results.append(analysis)
        return analysis
    
    def _print_analysis(self, analysis):
        r = analysis["results"]
        print(f"\n  📊 Results:")
        print(f"  {'─' * 50}")
        print(f"  Total time:    {r['total_time_sec']}s")
        print(f"  Throughput:    {r['throughput_rps']} req/sec")
        print(f"  Successful:    {r['successful']}")
        print(f"  Failed:        {r['failed']}")
        print(f"  Error rate:    {r['error_rate_pct']}%")
        
        if r.get("errors_by_type"):
            print(f"  Error types:")
            for err_type, count in r["errors_by_type"].items():
                print(f"    → {err_type}: {count}")
        
        lat = analysis.get("latency", {})
        if "avg_ms" in lat:
            print(f"\n  ⏱️  Latency:")
            print(f"  {'─' * 50}")
            print(f"  Average:  {lat['avg_ms']}ms")
            print(f"  p50:      {lat['p50_ms']}ms")
            print(f"  p95:      {lat['p95_ms']}ms")
            print(f"  p99:      {lat['p99_ms']}ms")
            print(f"  Min:      {lat['min_ms']}ms")
            print(f"  Max:      {lat['max_ms']}ms")

class CapacityPlanner:
    """
    Uses mathematical models to predict scaling needs.
    Implements Little's Law, Amdahl's Law, and growth projections.
    """
    
    @staticmethod
    def littles_law(arrival_rate_rps, avg_latency_sec):
        """
        L = λ × W
        Calculates average number of concurrent requests in system.
        """
        concurrent = arrival_rate_rps * avg_latency_sec
        return {
            "formula": "L = λ × W (Little's Law)",
            "arrival_rate_rps": arrival_rate_rps,
            "avg_latency_sec": avg_latency_sec,
            "avg_concurrent_requests": round(concurrent, 2),
            "interpretation": (
                f"At {arrival_rate_rps} req/sec with {avg_latency_sec*1000}ms "
                f"avg latency, you'll have ~{concurrent:.0f} requests "
                f"being processed simultaneously"
            )
        }
    
    @staticmethod
    def required_throughput(daily_active_users, requests_per_user_per_day,
                            peak_multiplier=3):
        """Calculate required requests per second"""
        total_daily = daily_active_users * requests_per_user_per_day
        avg_rps = total_daily / 86400  # seconds in a day
        peak_rps = avg_rps * peak_multiplier
        
        return {
            "daily_active_users": daily_active_users,
            "requests_per_user_per_day": requests_per_user_per_day,
            "total_daily_requests": total_daily,
            "avg_requests_per_sec": round(avg_rps, 2),
            "peak_requests_per_sec": round(peak_rps, 2),
            "peak_multiplier": peak_multiplier,
            "interpretation": (
                f"Average: {avg_rps:.1f} req/sec, "
                f"Peak: {peak_rps:.1f} req/sec "
                f"(design for peak)"
            )
        }
    
    @staticmethod
    def servers_needed(required_rps, rps_per_server, redundancy_factor=1.5):
        """Calculate number of servers needed"""
        raw_servers = required_rps / rps_per_server
        with_redundancy = math.ceil(raw_servers * redundancy_factor)
        
        return {
            "required_rps": required_rps,
            "rps_per_server": rps_per_server,
            "raw_servers_needed": round(raw_servers, 2),
            "redundancy_factor": redundancy_factor,
            "recommended_servers": with_redundancy,
            "total_capacity_rps": with_redundancy * rps_per_server,
            "headroom_pct": round(
                (with_redundancy * rps_per_server - required_rps) /
                required_rps * 100, 1
            ),
            "interpretation": (
                f"Need {with_redundancy} servers "
                f"({with_redundancy * rps_per_server} req/sec capacity "
                f"for {required_rps} req/sec demand)"
            )
        }
    
    @staticmethod
    def amdahls_law(parallel_fraction, num_processors):
        """
        Calculate maximum speedup from parallelization.
        Shows diminishing returns of adding more servers.
        """
        results = []
        for n in num_processors:
            speedup = 1.0 / ((1 - parallel_fraction) +
                             parallel_fraction / n)
            efficiency = speedup / n * 100
            results.append({
                "processors": n,
                "speedup": round(speedup, 2),
                "efficiency_pct": round(efficiency, 1)
            })
        
        max_speedup = 1.0 / (1 - parallel_fraction)
        
        return {
            "formula": "S = 1 / ((1-p) + p/n) — Amdahl's Law",
            "parallel_fraction": parallel_fraction,
            "serial_fraction": round(1 - parallel_fraction, 4),
            "theoretical_max_speedup": round(max_speedup, 2),
            "results": results,
            "interpretation": (
                f"With {parallel_fraction*100}% parallelizable work, "
                f"max possible speedup = {max_speedup:.1f}× "
                f"regardless of how many servers you add"
            )
        }
    
    @staticmethod
    def growth_projection(current_users, monthly_growth_pct, months):
        """Project future capacity needs"""
        projections = []
        users = current_users
        
        for month in range(months + 1):
            projections.append({
                "month": month,
                "users": round(users),
                "estimated_rps": round(users * 10 / 86400, 2),
                "estimated_peak_rps": round(users * 10 / 86400 * 3, 2)
            })
            users *= (1 + monthly_growth_pct / 100)
        
        return {
            "starting_users": current_users,
            "monthly_growth_pct": monthly_growth_pct,
            "projections": projections,
            "users_at_end": round(
                current_users * (1 + monthly_growth_pct / 100) ** months
            )
        }
    
    @staticmethod
    def vertical_vs_horizontal_cost(target_rps):
        """Compare cost of vertical vs horizontal scaling"""
        
        # Vertical tiers (diminishing returns)
        vertical_tiers = [
            {"name": "Small",   "cores": 2,  "ram_gb": 4,
             "rps": 500,   "cost_month": 20},
            {"name": "Medium",  "cores": 4,  "ram_gb": 16,
             "rps": 1200,  "cost_month": 80},
            {"name": "Large",   "cores": 8,  "ram_gb": 32,
             "rps": 2500,  "cost_month": 200},
            {"name": "XLarge",  "cores": 16, "ram_gb": 64,
             "rps": 4000,  "cost_month": 500},
            {"name": "2XLarge", "cores": 32, "ram_gb": 128,
             "rps": 6000,  "cost_month": 1200},
            {"name": "4XLarge", "cores": 64, "ram_gb": 256,
             "rps": 8000,  "cost_month": 3000},
            {"name": "Metal",   "cores": 128,"ram_gb": 512,
             "rps": 10000, "cost_month": 7000},
        ]
        
        # Find cheapest vertical option
        vertical_solution = None
        for tier in vertical_tiers:
            if tier["rps"] >= target_rps:
                vertical_solution = {
                    "tier": tier["name"],
                    "specs": f"{tier['cores']} cores, {tier['ram_gb']}GB RAM",
                    "capacity_rps": tier["rps"],
                    "cost_per_month": tier["cost_month"],
                    "cost_per_rps": round(tier["cost_month"] / tier["rps"], 4)
                }
                break
        
        if not vertical_solution:
            vertical_solution = {
                "tier": "IMPOSSIBLE",
                "message": f"Cannot reach {target_rps} rps with single server",
                "max_available_rps": vertical_tiers[-1]["rps"],
                "cost_per_month": "N/A"
            }
        
        # Horizontal: use Small servers
        base = vertical_tiers[0]
        servers_needed = math.ceil(target_rps / base["rps"] * 1.5)
        lb_cost = 20  # Load balancer cost
        
        horizontal_solution = {
            "server_tier": base["name"],
            "servers_needed": servers_needed,
            "capacity_rps": servers_needed * base["rps"],
            "cost_per_server": base["cost_month"],
            "load_balancer_cost": lb_cost,
            "total_cost_per_month": (
                servers_needed * base["cost_month"] + lb_cost
            ),
            "cost_per_rps": round(
                (servers_needed * base["cost_month"] + lb_cost) /
                (servers_needed * base["rps"]), 4
            )
        }
        
        return {
            "target_rps": target_rps,
            "vertical": vertical_solution,
            "horizontal": horizontal_solution,
            "recommendation": (
                "Horizontal" if (
                    isinstance(vertical_solution.get("cost_per_month"), str) or
                    (isinstance(vertical_solution.get("cost_per_month"), (int, float)) and
                     vertical_solution["cost_per_month"] >
                     horizontal_solution["total_cost_per_month"])
                ) else "Vertical (simpler, sufficient)"
            )
        }

  → Optimization = doing less work (costs nothing)

THE BIGGEST OPTIMIZATION: DON'T DO WORK YOU'VE ALREADY DONE

REAL SCENARIO:
  → 10,000 users request the menu every minute
  → Menu changes once per day
  → Current: Each request → SQL query → disk read → format → respond
  → That's 10,000 identical database queries per minute
  → For data that hasn't changed

WHAT IF:
  → First request: query database, store result in MEMORY
  → Next 9,999 requests: serve from MEMORY (no DB hit)
  → Memory lookup: ~0.1ms
  → Database query: ~5-50ms
  → That's 50-500× faster
  → Database load drops by 99.99%

THIS IS CACHING:
  → Store frequently accessed data in fast storage (RAM)
  → Serve from cache instead of recalculating/querying
  → Cache hit = fast, cheap
  → Cache miss = normal path, then store for next time

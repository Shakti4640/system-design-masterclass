🎯 HYPOTHETICAL PROBLEM

You are the notification architect at a food delivery platform
serving 10 million active users across 3 countries

EVERY DAY, your platform generates these notification events:

  → 2 million order confirmations ("Your order is placed!")
  → 2 million delivery updates ("Driver is 5 min away!")
  → 2 million delivery completions ("Order delivered! Rate us")
  → 500,000 payment receipts ("$23.50 charged to Visa ****4821")
  → 200,000 promotional offers ("Free delivery this weekend!")
  → 50,000 operational alerts ("Payment service breaker OPEN")
  → 10,000 security alerts ("Login from new device detected")

  TOTAL: ~7 million notifications per day
  PEAK: 500 notifications per SECOND during dinner rush (6-9 PM)

EACH notification must potentially go through:
  → Push notification (mobile app)
  → SMS (critical: delivery, security)
  → Email (receipts, promotions)
  → In-app notification (notification bell)
  → Webhook (partner integrations)
  → Slack/PagerDuty (internal ops alerts)

THE DISASTERS THAT HAPPENED BEFORE YOU JOINED:

  DISASTER 1: The Newsletter Incident
    → Marketing sent 5 million promotional emails
    → These filled up the notification queue
    → Actual order confirmations waited 45 minutes
    → Customers thought orders weren't placed
    → 10,000 duplicate orders created
    → Revenue loss: $200,000

  DISASTER 2: The SMS Meltdown
    → SMS provider (Twilio) rate limit: 100 SMS/second
    → Flash sale notification → 500,000 SMS at once
    → First 100 went through, rest were rejected
    → No retry logic → 499,900 users never got SMS
    → Retry logic added → retries also hit rate limit
    → Exponential backoff not implemented → infinite loop

  DISASTER 3: The Notification Spam
    → Bug in delivery tracking service
    → Sent "Driver arriving" push every 2 seconds
    → User received 300 push notifications in 10 minutes
    → App store rating dropped from 4.5 to 3.2 overnight
    → 50,000 users uninstalled the app

  DISASTER 4: The Silent Failure
    → Push notification provider (FCM) went down for 2 hours
    → No fallback to SMS or email
    → No circuit breaker on the provider
    → System kept trying → all notification threads blocked
    → Even email notifications stopped (shared thread pool)
    → Complete notification blackout

  YOUR MISSION:
    → Design a notification service that handles ALL of this
    → Priority queues: urgent > normal > bulk
    → Multi-channel: push, SMS, email, webhook, in-app
    → Rate limiting per provider
    → Circuit breaker per channel
    → User preferences respected
    → Deduplication (no spam)
    → Reliable delivery with retry and fallback
    → Fan-out for mass notifications

🔍 HOW TO APPROACH IT

Think of it like a POSTAL SYSTEM for a country:

  SENDERS (services):
    → Different departments generate mail
    → Some mail is urgent (legal notices)
    → Some mail is bulk (advertisements)
    → Each needs different treatment

  POST OFFICE (notification service):
    → Receives ALL mail
    → Sorts by priority (express, standard, bulk)
    → Checks recipient preferences (no junk mail)
    → Routes to correct delivery channel

  DELIVERY CHANNELS:
    → Express courier (push notification — instant)
    → Registered mail (SMS — reliable, expensive)
    → Standard mail (email — cheap, slower)
    → Internal memo (in-app — free, requires app open)
    → Diplomatic pouch (webhook — partner systems)

  DELIVERY CONSTRAINTS:
    → Each courier has limited capacity (rate limits)
    → If courier is sick, use another (circuit breaker)
    → If recipient moved, try forwarding (channel fallback)
    → Keep receipt of delivery (audit trail)
    → Don't send same letter twice (deduplication)

COMPONENT 3: PRIORITY QUEUE SYSTEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Three priority levels:

  P0 — CRITICAL (process within 5 seconds)
    → Security alerts, OTP codes, payment failures
    → 1% of volume, 99% of urgency
    → Dedicated workers, no batching

  P1 — HIGH (process within 30 seconds)
    → Order confirmations, delivery updates, receipts
    → 70% of volume, normal operations
    → Standard workers, small batches

  P2 — LOW (process within 5 minutes)
    → Promotions, newsletters, weekly digests
    → 29% of volume, deferrable
    → Shared workers, large batches
    → Can be paused during peak hours

  QUEUE IMPLEMENTATION:
    → Separate queue per priority (Project 11)
    → Workers poll P0 first, then P1, then P2
    → Weighted: P0 gets 60% of worker time
    → P1 gets 30%, P2 gets 10%
    → Ensures critical never waits behind bulk

📌 WHAT COULD BE DONE

OPTION A: Simple Direct Send
  → Service directly calls push/SMS/email APIs
  → No queue, no priority, no retry
  → PROBLEM: Tight coupling, no resilience, no scaling
  → USE: Prototype only

OPTION B: Single Queue + Workers
  → All notifications in one queue
  → Workers process in FIFO order
  → PROBLEM: Bulk blocks critical (Newsletter Incident)
  → USE: Low-volume systems (<1000/day)

OPTION C: Priority Queues + Channel Workers
  → Separate queues by priority
  → Dedicated workers per channel
  → Retry with fallback
  → PROBLEM: Still single-region, moderate complexity
  → USE: Medium-scale (10K-100K/day) — SWEET SPOT

OPTION D: Full Event-Driven + Fan-Out
  → Event bus (Project 29 preview) triggers notifications
  → Priority queues per channel
  → Rate-limited workers with circuit breakers
  → User preference service
  → Template engine
  → Deduplication
  → Audit trail
  → USE: Large-scale (1M+/day) — Production grade

✅ WHAT SHOULD DO

1. ALWAYS use idempotency keys
   → Every notification request gets a unique key
   → Store key in KV store (Project 23) with TTL
   → Duplicate request → silently ignore
   → Prevents double-sends on retries or duplicate events

2. ALWAYS separate queues by priority
   → Critical notifications must NEVER wait behind bulk
   → P0 has dedicated workers
   → P2 can be paused during system stress

3. ALWAYS implement retry with exponential backoff
   → First retry: 1 second
   → Second retry: 5 seconds
   → Third retry: 30 seconds
   → Max retries: 3-5 depending on channel
   → After max retries → fallback to next channel

4. ALWAYS implement channel fallback
   → Push fails → try SMS
   → SMS fails → try email
   → Configurable fallback chain per notification type
   → At least ONE channel should be highly reliable

5. ALWAYS respect user preferences
   → User opted out → DO NOT SEND
   → Quiet hours → delay or skip
   → Channel preference → honor it
   → Legal requirement in many jurisdictions (GDPR, CAN-SPAM)

6. ALWAYS rate limit per user
   → Max N notifications per hour per user
   → Prevents spam from bugs
   → The "300 push notifications" disaster prevention

7. ALWAYS track delivery status
   → CREATED → QUEUED → SENT → DELIVERED → FAILED
   → Every state change recorded with timestamp
   → Audit trail for debugging and compliance

❌ WHAT SHOULD NOT DO

1. NEVER send notifications synchronously in request path
   → User places order → API response waits for push + SMS + email
   → 3 provider calls × 200ms each = 600ms added latency
   → If any provider slow → user request slow
   → ALWAYS queue and process async (Project 11 pattern)

2. NEVER use a single queue for all priorities
   → 500,000 promotional pushes queued
   → OTP code for login stuck behind them
   → User waiting 20 minutes for login OTP
   → The Newsletter Incident repeated

3. NEVER ignore provider rate limits
   → Twilio: 100 SMS/second → sending 1000/second = 90% rejected
   → FCM: 1000/second → sending 5000/second = rate limited
   → ALWAYS implement token bucket per provider (Project 9)
   → Queue excess, don't discard

4. NEVER share thread pool across channels
   → SMS provider hangs → all threads waiting
   → Push and email also blocked
   → Bulkhead pattern (Project 21): separate pool per channel

5. NEVER send without deduplication
   → Event processed twice → notification sent twice
   → User gets "Order confirmed" twice → confused, angry
   → ALWAYS deduplicate via idempotency key

6. NEVER hardcode notification content
   → "Your order ORD-123 is confirmed" → hardcoded
   → Use templates: "Your order {{order_id}} is confirmed"
   → Templates can be updated without code deployment
   → Different languages supported via template variants

7. NEVER skip the dead letter queue
   → Notification failed after all retries and fallbacks
   → If you drop it → notification permanently lost
   → Dead letter queue captures all failures
   → Ops team reviews and can manually retry

🚫 WHEN NOT TO USE

1. Real-time bidirectional communication
   → Notifications are ONE-WAY (server → user)
   → For two-way chat → WebSocket (Project 31)
   → Notification service for "you have a new message"
   → Chat system for the actual message exchange

2. Data synchronization
   → "Update user's local cache"
   → This is not a notification, it's data sync
   → Use event streaming (Project 29) instead

3. High-frequency updates (>1/second per user)
   → Live sports scores, stock prices
   → Push notification per second = spam
   → Use WebSocket or Server-Sent Events instead
   → Notification only for significant events

4. Large payload delivery
   → Sending a 10MB report as notification
   → Notifications carry SMALL payloads (<4KB)
   → Send notification with LINK to the report
   → "Your report is ready → download here"

5. Guaranteed exactly-once delivery
   → Push notifications are best-effort
   → Device might be offline, token expired
   → Design for at-least-once with deduplication
   → Not suitable where missing = catastrophic
   → For critical: use SMS as backup (higher reliability)

COMPONENT 1: NOTIFICATION LIFECYCLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Full journey of a single notification:

  CREATED (T=0ms)
    → API receives request
    → Assign notification_id (UUID)
    → Store in notifications table

  VALIDATED (T=2ms)
    → Check idempotency key → not duplicate ✓
    → Validate recipient exists ✓
    → Validate template exists ✓
    → Check user preferences ✓
    → Rate limit check → under limit ✓

  RENDERED (T=5ms)
    → Resolve template with data
    → Generate per-channel content:
      → Push: "Your order from Pizza Palace is confirmed! 🍕"
      → Email: HTML with order details, images, buttons
      → SMS: "Order ORD-9912 confirmed. ETA: 35 min"

  QUEUED (T=8ms)
    → Determine priority: P1 (order confirmation)
    → Create delivery tasks per channel:
      → Task 1: {channel: push, priority: P1, content: "..."}
      → Task 2: {channel: email, priority: P1, content: "..."}
    → Enqueue each task in priority queue
    → Return notification_id to caller (async)

  PROCESSING (T=50ms — picked up by worker)
    → Worker dequeues task from P1 queue
    → Check circuit breaker for channel (Project 21)
      → Breaker CLOSED → proceed
      → Breaker OPEN → fallback to next channel

  SENDING (T=60ms)
    → Call provider API (FCM, Twilio, SES)
    → Wait for response

  SENT (T=200ms)
    → Provider accepted the notification
    → Update status: SENT
    → Note: SENT ≠ DELIVERED (provider queues it)

  DELIVERED (T=500ms — async callback)
    → Provider confirms delivery to device
    → Update status: DELIVERED
    → (Not all providers support delivery confirmation)

  OR

  FAILED (T=200ms)
    → Provider returned error
    → Determine if retryable:
      → 429 Rate Limited → retryable (wait and retry)
      → 500 Server Error → retryable (provider issue)
      → 400 Bad Request → NOT retryable (our bug)
      → 410 Token Gone → NOT retryable (user uninstalled)
    → If retryable → schedule retry with backoff
    → If not retryable → try fallback channel
    → If all channels exhausted → DEAD LETTER QUEUE

COMPONENT 2: FAN-OUT ARCHITECTURE (Mass Notifications)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Sending to 500,000 users:

  APPROACH A: Individual Send (Naive)
    → Loop through 500,000 users
    → Create notification for each
    → 500,000 queue entries
    → PROBLEM: Takes forever to enqueue

  APPROACH B: Fan-Out with Segments (Production)

    Step 1: Create broadcast record
      broadcast_id: "BC-001"
      segment: "all_users_us"
      template: "flash_sale"
      total_recipients: 500,000

    Step 2: Partition into batches
      Batch 1: users 1 - 10,000
      Batch 2: users 10,001 - 20,000
      ...
      Batch 50: users 490,001 - 500,000

    Step 3: Enqueue batch jobs (50 jobs)
      Each batch job is ONE queue entry
      Workers pick up batch jobs in parallel

    Step 4: Worker processes batch
      → Fetch 10,000 user IDs from segment
      → For each user:
        → Check preferences (opted out? quiet hours?)
        → Check rate limit (already at max today?)
        → Render template with user data
        → Create individual delivery task
        → Enqueue to P2 (low priority)

    Step 5: Channel workers process P2
      → Respect provider rate limits

class PriorityQueueSystem:
    """
    Three-tier priority queue.
    Workers process P0 first, then P1, then P2.
    Uses Project 11 (Message Queue) concepts.
    """

    def __init__(self):
        self.queues = {
            Priority.CRITICAL: deque(),
            Priority.HIGH: deque(),
            Priority.LOW: deque()
        }
        self.dead_letter = deque()
        self.lock = threading.Lock()
        self.stats = {
            "enqueued": {p.name: 0 for p in Priority},
            "dequeued": {p.name: 0 for p in Priority},
            "dead_lettered": 0
        }

    def enqueue(self, task, priority=Priority.HIGH):
        with self.lock:
            self.queues[priority].append(task)
            self.stats["enqueued"][priority.name] += 1

    def dequeue(self):
        """
        Dequeue with priority: P0 first, then P1, then P2.
        Returns (task, priority) or (None, None).
        """
        with self.lock:
            for priority in [Priority.CRITICAL, Priority.HIGH,
                             Priority.LOW]:
                if self.queues[priority]:
                    task = self.queues[priority].popleft()
                    self.stats["dequeued"][priority.name] += 1
                    return task, priority
        return None, None

    def send_to_dead_letter(self, task):
        with self.lock:
            self.dead_letter.append({
                **task,
                "dead_lettered_at": datetime.now(
                    timezone.utc
                ).isoformat()
            })
            self.stats["dead_lettered"] += 1

    def get_depths(self):
        with self.lock:
            return {
                "P0_CRITICAL": len(self.queues[Priority.CRITICAL]),
                "P1_HIGH": len(self.queues[Priority.HIGH]),
                "P2_LOW": len(self.queues[Priority.LOW]),
                "DEAD_LETTER": len(self.dead_letter)
            }

class TokenBucketRateLimiter:
    """Token bucket rate limiter per provider."""

    def __init__(self, rate_per_second, burst_size=None):
        self.rate = rate_per_second
        self.burst = burst_size or rate_per_second
        self.tokens = float(self.burst)
        self.last_refill = time.time()
        self.lock = threading.Lock()

    def acquire(self, tokens=1, timeout=5.0):
        """Acquire tokens. Blocks until available or timeout."""
        deadline = time.time() + timeout
        while time.time() < deadline:
            with self.lock:
                self._refill()
                if self.tokens >= tokens:
                    self.tokens -= tokens
                    return True
            time.sleep(0.01)
        return False

    def _refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(
            self.burst,
            self.tokens + elapsed * self.rate
        )
        self.last_refill = now

class SimpleCircuitBreaker:
    """Simplified circuit breaker for channel providers."""

    def __init__(self, name, failure_threshold=5,
                 recovery_timeout=30):
        self.name = name
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failure_count = 0
        self.state = "CLOSED"  # CLOSED, OPEN, HALF_OPEN
        self.opened_at = None
        self.lock = threading.Lock()

    def allow_request(self):
        with self.lock:
            if self.state == "CLOSED":
                return True
            if self.state == "OPEN":
                if (time.time() - self.opened_at >
                        self.recovery_timeout):
                    self.state = "HALF_OPEN"
                    return True
                return False
            if self.state == "HALF_OPEN":
                return True
        return False

    def record_success(self):
        with self.lock:
            self.failure_count = 0
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"
                print(f"    ⚡ [{self.name}] HALF_OPEN → CLOSED "
                      f"(recovered)")

    def record_failure(self):
        with self.lock:
            self.failure_count += 1
            if self.failure_count >= self.failure_threshold:
                if self.state != "OPEN":
                    self.state = "OPEN"
                    self.opened_at = time.time()
                    print(f"    ⚡ [{self.name}] → OPEN "
                          f"(failures: {self.failure_count})")

class DeduplicationStore:
    """Tracks idempotency keys to prevent duplicate sends."""

    def __init__(self, ttl_seconds=86400):
        self.keys = {}
        self.ttl = ttl_seconds
        self.lock = threading.Lock()

    def is_duplicate(self, key):
        with self.lock:
            if key in self.keys:
                if time.time() < self.keys[key]:
                    return True
                else:
                    del self.keys[key]
            return False

    def mark_seen(self, key):
        with self.lock:
            self.keys[key] = time.time() + self.ttl

    def cleanup(self):
        with self.lock:
            now = time.time()
            expired = [k for k, v in self.keys.items() if now > v]
            for k in expired:
                del self.keys[k]

    # ============================================================
    # MAIN API: Send a notification
    # ============================================================

    def send(self, recipient_id, notification_type, priority_str,
             channels, template_name, data,
             idempotency_key=None, category=None):
        """
        Main entry point for sending a notification.
        Returns immediately after queuing (async processing).
        """
        self.stats["received"] += 1

        # Generate IDs
        notification_id = f"NOTIF-{uuid.uuid4().hex[:12]}"
        idempotency_key = idempotency_key or notification_id

        # Parse priority
        priority_map = {
            "critical": Priority.CRITICAL,
            "high": Priority.HIGH,
            "low": Priority.LOW
        }
        priority = priority_map.get(priority_str.lower(), Priority.HIGH)

        # Parse channels
        channel_list = []
        for ch in channels:
            if isinstance(ch, Channel):
                channel_list.append(ch)
            elif isinstance(ch, str):
                try:
                    channel_list.append(Channel(ch))
                except ValueError:
                    pass

        # ---- Step 1: Deduplication ----
        if self.dedup_store.is_duplicate(idempotency_key):
            self.stats["rejected_duplicate"] += 1
            return {
                "notification_id": notification_id,
                "status": "rejected",
                "reason": "duplicate"
            }
        self.dedup_store.mark_seen(idempotency_key)

        # ---- Step 2: User preference check ----
        category = category or notification_type
        if not self.pref_store.is_category_enabled(
                recipient_id, category):
            self.stats["rejected_preference"] += 1
            return {
                "notification_id": notification_id,
                "status": "rejected",
                "reason": f"user opted out of {category}"
            }

        # Filter channels by user preference
        enabled_channels = [
            ch for ch in channel_list
            if self.pref_store.is_channel_enabled(recipient_id, ch)
        ]

        if not enabled_channels:
            # Try fallback chain
            fallback = self.fallback_chains.get(category, [])
            enabled_channels = [
                ch for ch in fallback
                if self.pref_store.is_channel_enabled(recipient_id, ch)
            ]

        if not enabled_channels:
            self.stats["rejected_preference"] += 1
            return {
                "notification_id": notification_id,
                "status": "rejected",
                "reason": "no enabled channels"
            }

        # ---- Step 3: User frequency check ----
        prefs = self.pref_store.get_preferences(recipient_id)
        max_per_hour = prefs.get("frequency_cap", {}).get(
            "max_per_hour", 20
        )
        if not self.freq_limiter.check_and_increment(
                recipient_id, max_per_hour):
            self.stats["rejected_rate_limit"] += 1
            return {
                "notification_id": notification_id,
                "status": "rejected",
                "reason": "user frequency cap reached"
            }

        self.stats["validated"] += 1

        # ---- Step 4: Render templates ----
        rendered_content = {}
        for ch in enabled_channels:
            content = self.template_engine.render(
                template_name, ch, data
            )
            if content:
                rendered_content[ch] = content

        if not rendered_content:
            return {
                "notification_id": notification_id,
                "status": "rejected",
                "reason": "no template available for channels"
            }

        # ---- Step 5: Create notification record ----
        notification = {
            "notification_id": notification_id,
            "recipient_id": recipient_id,
            "type": notification_type,
            "priority": priority.name,
            "channels": [ch.value for ch in rendered_content.keys()],
            "template": template_name,
            "data": data,
            "status": DeliveryStatus.CREATED.value,
            "created_at": datetime.now(timezone.utc).isoformat(),
            "delivery_attempts": [],
            "category": category
        }

        with self.notification_lock:
            self.notifications[notification_id] = notification

        # ---- Step 6: Create delivery tasks and enqueue ----
        for channel, content in rendered_content.items():
            task = {
                "notification_id": notification_id,
                "recipient_id": recipient_id,
                "channel": channel,
                "content": content,
                "priority": priority,
                "category": category,
                "attempt": 0,
                "max_attempts": 3,
                "created_at": time.time(),
                "fallback_channels": [
                    ch for ch in self.fallback_chains.get(
                        category, []
                    )
                    if ch != channel and ch in rendered_content
                ]
            }
            self.queue_system.enqueue(task, priority)

        self.stats["queued"] += 1

        # Update notification status
        with self.notification_lock:
            notification["status"] = DeliveryStatus.QUEUED.value

        return {
            "notification_id": notification_id,
            "status": "queued",
            "channels": [ch.value for ch in rendered_content.keys()],
            "priority": priority.name
        }

    # ============================================================

    def broadcast(self, user_ids, notification_type, template_name,
                  data, channels, category=None, priority_str="low"):
        """
        Send notification to multiple users.
        Batches internally for efficiency.
        """
        broadcast_id = f"BC-{uuid.uuid4().hex[:8]}"
        results = {
            "broadcast_id": broadcast_id,
            "total_recipients": len(user_ids),
            "queued": 0,
            "rejected": 0,
            "reasons": defaultdict(int)
        }

        # Process in batches
        batch_size = 100
        for i in range(0, len(user_ids), batch_size):
            batch = user_ids[i:i + batch_size]
            for user_id in batch:
                result = self.send(
                    recipient_id=user_id,
                    notification_type=notification_type,
                    priority_str=priority_str,
                    channels=channels,
                    template_name=template_name,
                    data=data,
                    idempotency_key=f"{broadcast_id}:{user_id}",
                    category=category
                )
                if result["status"] == "queued":
                    results["queued"] += 1
                else:
                    results["rejected"] += 1
                    results["reasons"][result.get("reason", "unknown")] += 1

        results["reasons"] = dict(results["reasons"])
        return results

    # ============================================================
    # WORKER: Process delivery tasks from queues
    # ============================================================

    def start_workers(self, num_workers=3):
        """Start background workers to process delivery tasks."""
        self._workers_running = True
        self._worker_threads = []
        for i in range(num_workers):
            t = threading.Thread(
                target=self._worker_loop,
                args=(f"worker-{i+1}",),
                daemon=True
            )
            t.start()
            self._worker_threads.append(t)
        print(f"  [WORKERS] Started {num_workers} delivery workers")

    def stop_workers(self):
        """Stop all workers."""
        self._workers_running = False
        for t in self._worker_threads:
            t.join(timeout=2)

    def _worker_loop(self, worker_name):
        """Main worker loop — dequeue and deliver."""
        while self._workers_running:
            task, priority = self.queue_system.dequeue()

            if task is None:
                time.sleep(0.05)  # No tasks, wait briefly
                continue

            self._process_delivery_task(worker_name, task, priority)

    def _process_delivery_task(self, worker_name, task, priority):
        """Process a single delivery task through the full pipeline."""
        notification_id = task["notification_id"]
        recipient_id = task["recipient_id"]
        channel = task["channel"]
        content = task["content"]
        attempt = task["attempt"]
        max_attempts = task["max_attempts"]

        # ---- Step 1: Check circuit breaker ----
        breaker = self.breakers.get(channel)
        if breaker and not breaker.allow_request():
            # Circuit breaker open → try fallback
            self._try_fallback(worker_name, task, priority,
                               f"Circuit breaker OPEN for {channel.value}")
            return

        # ---- Step 2: Acquire rate limiter token ----
        rate_limiter = self.rate_limiters.get(channel)
        if rate_limiter and not rate_limiter.acquire(timeout=2.0):
            # Rate limited → re-queue with slight delay
            task["attempt"] = attempt  # Don't count as attempt
            time.sleep(0.1)
            self.queue_system.enqueue(task, priority)
            return

        # ---- Step 3: Send via provider ----
        provider = self.providers.get(channel)
        if not provider:
            self._try_fallback(worker_name, task, priority,
                               f"No provider for {channel.value}")
            return

        try:
            # Update status
            self._update_notification_status(
                notification_id, DeliveryStatus.PROCESSING,
                channel=channel.value, attempt=attempt + 1
            )

            # Actually send
            result = provider.send(recipient_id, content)

            # SUCCESS
            breaker.record_success()
            self.stats["sent"] += 1
            self.stats["delivered"] += 1
            self.stats["channel_stats"][channel.value]["sent"] += 1

            self._update_notification_status(
                notification_id, DeliveryStatus.DELIVERED,
                channel=channel.value,
                provider_message_id=result.get("message_id")
            )

        except ProviderError as e:
            # FAILURE
            breaker.record_failure()
            self.stats["channel_stats"][channel.value]["failed"] += 1

            # Record attempt
            self._record_delivery_attempt(
                notification_id, channel.value,
                attempt + 1, str(e), "failed"
            )

            if e.retryable and attempt + 1 < max_attempts:
                # Retry with exponential backoff
                backoff = min(30, (2 ** attempt) * 1)
                time.sleep(min(backoff, 0.5))  # Shortened for simulation
                task["attempt"] = attempt + 1
                self.queue_system.enqueue(task, priority)
            else:
                # Max retries exhausted → try fallback channel
                self._try_fallback(worker_name, task, priority,
                                   str(e))

        except Exception as e:
            # Unexpected error
            self.stats["failed"] += 1
            self._record_delivery_attempt(
                notification_id, channel.value,
                attempt + 1, str(e), "error"
            )
            self._try_fallback(worker_name, task, priority, str(e))

    def _try_fallback(self, worker_name, task, priority, reason):
        """Attempt delivery through fallback channels."""
        notification_id = task["notification_id"]
        fallback_channels = task.get("fallback_channels", [])

        if fallback_channels:
            # Try next fallback channel
            next_channel = fallback_channels[0]
            remaining_fallbacks = fallback_channels[1:]

            # Render content for fallback channel
            notif = self.notifications.get(notification_id, {})
            content = self.template_engine.render(
                notif.get("template", ""),
                next_channel,
                notif.get("data", {})
            )

            if content:
                fallback_task = {
                    **task,
                    "channel": next_channel,
                    "content": content,
                    "attempt": 0,
                    "fallback_channels": remaining_fallbacks,
                    "is_fallback": True,
                    "original_channel": task["channel"].value
                }
                self.queue_system.enqueue(fallback_task, priority)
                self.stats["fallback_used"] += 1

                self._record_delivery_attempt(
                    notification_id, task["channel"].value,
                    task["attempt"] + 1, reason,
                    f"fallback to {next_channel.value}"
                )
                return

        # All channels exhausted → dead letter queue
        self.queue_system.send_to_dead_letter(task)
        self.stats["dead_lettered"] += 1
        self.stats["failed"] += 1

        self._update_notification_status(
            notification_id, DeliveryStatus.DEAD_LETTER,
            reason=reason
        )

    # ============================================================
    # NOTIFICATION STATUS MANAGEMENT
    # ============================================================

    def _update_notification_status(self, notification_id, status,
                                    **extra):
        with self.notification_lock:
            if notification_id in self.notifications:
                notif = self.notifications[notification_id]
                notif["status"] = status.value
                notif["updated_at"] = datetime.now(
                    timezone.utc
                ).isoformat()
                for k, v in extra.items():
                    notif[k] = v

    def _record_delivery_attempt(self, notification_id, channel,
                                 attempt_num, message, result):
        with self.notification_lock:
            if notification_id in self.notifications:
                self.notifications[notification_id][
                    "delivery_attempts"
                ].append({
                    "channel": channel,
                    "attempt": attempt_num,
                    "message": message,
                    "result": result,
                    "timestamp": datetime.now(
                        timezone.utc
                    ).isoformat()
                })

    # ============================================================
    # QUERY: Get notification status
    # ============================================================

    def get_notification(self, notification_id):

    def get_user_notifications(self, user_id, limit=20):

THE FILE PROBLEM NOTIFICATIONS REVEAL:

  Your notification service works beautifully for TEXT:
    → "Your order is confirmed!"
    → "Payment of $23.50 received"

  But users want RICH notifications:
    → Payment receipt with PDF attachment
    → Order confirmation with restaurant photo
    → Profile picture in the notification
    → Delivery driver's photo
    → Invoice document

  AND your platform has OTHER file needs:
    → User uploads profile picture (2 MB)
    → Restaurant uploads menu images (50 images × 500 KB)
    → Driver uploads vehicle photo (3 MB)
    → System generates invoice PDFs (200 KB)
    → Chat messages with image attachments (5 MB)

  THE QUESTIONS:
    → Where do you STORE these files?
    → Not in PostgreSQL (BLOB storage is inefficient)
    → Not in Redis (too expensive for large binary data)
    → Not on the application server (stateful, doesn't scale)

    → How do users UPLOAD large files?
    → 100 MB video → single HTTP request → timeout
    → Need chunked upload, resume on failure

    → How do you DEDUPLICATE identical files?
    → 1000 users upload the same meme → store once

    → How do you SERVE files efficiently?
    → CDN (Project 12) for static content
    → Pre-signed URLs for private content
    → Thumbnails and resizing on the fly

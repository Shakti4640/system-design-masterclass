🎯 HYPOTHETICAL PROBLEM

Your restaurant chain API is now cached (Project 5) and you know
how to scale (Project 4). Time to actually DO horizontal scaling.

THE SITUATION:
  → 5 identical servers running your REST API
  → Each server: 2 cores, 4GB RAM, handles 500 req/sec
  → Total capacity: 2,500 req/sec
  → Database is shared (Project 3)
  → Cache is shared (Project 5 external cache concept)

THE PROBLEM:
  → Customer opens the app
  → App needs to call: GET /api/v1/menu-items
  → Which of the 5 servers should it call?

  ┌──────────┐
  │          │       Server 1 (192.168.1.10:8080)  ?
  │ Customer │       Server 2 (192.168.1.11:8080)  ?
  │   App    │ ───→  Server 3 (192.168.1.12:8080)  ?
  │          │       Server 4 (192.168.1.13:8080)  ?
  │          │       Server 5 (192.168.1.14:8080)  ?
  └──────────┘

NAIVE SOLUTIONS (all bad):

  Solution A: Hardcode one server
    → App always calls Server 1
    → Server 1 handles ALL traffic → overloaded
    → Servers 2-5 sit idle → wasted money
    → Server 1 crashes → entire system down

  Solution B: Client knows all servers
    → App has list of all 5 server IPs
    → App randomly picks one
    → Server 3 crashes → app sometimes picks dead server
    → Add Server 6 → must update EVERY app on EVERY phone
    → Scaling requires client changes → impossible

  Solution C: DNS round-robin
    → DNS returns different IP each time
    → Client caches DNS → keeps hitting same server
    → No health awareness → sends traffic to dead servers
    → No intelligence → can't avoid overloaded servers

THE REAL SOLUTION: A LOAD BALANCER
  → Single entry point (one IP address)
  → Knows about all backend servers
  → Distributes requests intelligently
  → Detects dead servers → stops sending traffic
  → Detects overloaded servers → sends less traffic
  → Add/remove servers → zero client changes

  ┌──────────┐     ┌────────────────┐     ┌──────────┐
  │          │     │                │     │ Server 1 │
  │ Customer │────→│ LOAD BALANCER  │────→│ Server 2 │
  │   App    │     │ (Traffic Cop)  │────→│ Server 3 │
  │          │←────│                │────→│ Server 4 │
  └──────────┘     └────────────────┘     │ Server 5 │
                    One IP address         └──────────┘
                    One entry point        All identical
                    All intelligence       All disposable

THE TRAFFIC COP ANALOGY:
  → Intersection with 5 lanes going to the same destination
  → Without traffic cop: everyone picks lane 1 → gridlock
  → With traffic cop: directs each car to the least busy lane
  → If lane 3 has accident → cop redirects to other lanes
  → New lane opens → cop starts directing traffic there
  → Nobody needs to know the lane layout — just follow the cop

         → Monitor backend server health

         → How does the LB decide which server gets the next request?
         → Round Robin, Weighted Round Robin, Least Connections, etc.

Step 3: IMPLEMENT health checking
         → How does the LB know if a server is alive?
         → Periodic health probes (HTTP GET /health)

         → What if LB itself crashes?

         → Layer 7: HTTP level (smart, can route by URL/header)

LOAD BALANCING ALGORITHMS:

═══════════════════════════════════════════════════════
ALGORITHM 1: ROUND ROBIN
═══════════════════════════════════════════════════════

  Simplest: each server takes turns

  Request 1 → Server 1
  Request 2 → Server 2
  Request 3 → Server 3
  Request 4 → Server 4
  Request 5 → Server 5
  Request 6 → Server 1  (back to start)
  Request 7 → Server 2
  ...

  ✅ Simple, predictable, fair distribution
  ❌ Ignores server load (busy server still gets requests)
  ❌ Ignores request weight (GET /health vs POST /complex-report)
  ❌ Assumes all servers are identical capacity

═══════════════════════════════════════════════════════
ALGORITHM 2: WEIGHTED ROUND ROBIN
═══════════════════════════════════════════════════════

  Servers have different capacities → give proportional traffic

  Server 1: weight=3 (powerful machine)
  Server 2: weight=1 (small machine)
  Server 3: weight=1 (small machine)

  Request 1 → Server 1
  Request 2 → Server 1
  Request 3 → Server 1
  Request 4 → Server 2
  Request 5 → Server 3
  Request 6 → Server 1  (back to start)
  ...

  ✅ Handles heterogeneous servers
  ❌ Still ignores current load
  ❌ Weight must be manually configured

═══════════════════════════════════════════════════════
ALGORITHM 3: LEAST CONNECTIONS
═══════════════════════════════════════════════════════

  Send to the server with fewest active connections

  State:
    Server 1: 12 active connections
    Server 2: 5 active connections   ← send here
    Server 3: 18 active connections
    Server 4: 8 active connections

  New request → Server 2 (only 5 connections)

  ✅ Naturally balances load based on actual server state
  ✅ Handles slow requests (busy server accumulates connections)
  ❌ Doesn't account for server capacity differences
  ❌ Requires tracking connection state

═══════════════════════════════════════════════════════
ALGORITHM 4: WEIGHTED LEAST CONNECTIONS
═══════════════════════════════════════════════════════

  Combines weights with connection count
  Score = active_connections / weight
  Send to lowest score

  Server 1: 12 connections, weight=4 → score = 3.0
  Server 2: 5 connections,  weight=2 → score = 2.5  ← send here
  Server 3: 8 connections,  weight=1 → score = 8.0

  ✅ Best of both worlds
  ✅ Handles heterogeneous servers AND current load

═══════════════════════════════════════════════════════
ALGORITHM 5: IP HASH
═══════════════════════════════════════════════════════

  Hash the client's IP address → always goes to same server

  hash("192.168.1.100") % 5 = 2 → always Server 2
  hash("10.0.0.50") % 5 = 4     → always Server 4

  ✅ Same client always reaches same server (sticky)
  ✅ Good for: in-memory session state (Project 8 warns against this)
  ❌ Uneven distribution if many clients behind same IP (NAT)
  ❌ Server removal → massive redistribution
  ❌ Encourages stateful servers (bad for scaling)

═══════════════════════════════════════════════════════
ALGORITHM 6: LEAST RESPONSE TIME
═══════════════════════════════════════════════════════

  Track each server's average response time
  Send to the fastest responder

  Server 1: avg response = 45ms
  Server 2: avg response = 23ms    ← send here
  Server 3: avg response = 120ms   ← something wrong?

  ✅ Automatically detects slow servers
  ✅ Routes around performance problems
  ❌ Complex to implement (needs response time tracking)
  ❌ Can oscillate (all traffic to fast server → it slows down)

═══════════════════════════════════════════════════════
ALGORITHM 7: RANDOM
═══════════════════════════════════════════════════════

  Pick a random server for each request

  ✅ Dead simple, no state to maintain
  ✅ Statistically even distribution over time
  ❌ Can be uneven for small request counts
  ❌ No intelligence about server state

  Surprisingly effective at scale!
  With 10,000 requests across 5 servers → nearly perfect distribution

📌 WHAT COULD BE DONE

Option A: Software Load Balancer (Nginx, HAProxy)
  → Runs on a regular server
  → Highly configurable
  → Very high performance (100,000+ req/sec)
  → When: Most web applications, microservices

Option B: Hardware Load Balancer (F5, Citrix)
  → Dedicated appliance
  → Extremely high performance
  → Very expensive ($10,000-$100,000+)
  → When: Enterprise, financial institutions

Option C: Cloud Load Balancer (AWS ALB/NLB, GCP LB)
  → Managed service, no maintenance
  → Auto-scales itself
  → Pay per usage
  → When: Cloud deployments (most modern apps)

Option D: DNS-Based Load Balancing
  → DNS returns different IPs for same domain
  → Simple but no health checking
  → Client caches DNS → sticky
  → When: Global distribution between regions (coarse-grained)

Option E: Client-Side Load Balancing
  → Client has list of servers, picks one
  → Used in microservice-to-microservice communication
  → When: Internal service mesh (Project 20)

For THIS project → Option A approach (build our own to understand)
→ Same principles apply to Nginx, HAProxy, AWS ALB

✅ WHAT SHOULD DO

1. ALWAYS implement health checks
   → LB must KNOW if a server is alive
   → Probe every server every 5-30 seconds
   → HTTP GET /health → 200 = alive, anything else = dead
   → Remove dead servers from rotation IMMEDIATELY
   → Re-add recovered servers after consecutive successes

2. ALWAYS have multiple LB instances
   → LB is the SINGLE entry point
   → If LB crashes → EVERYTHING is down
   → Run at least 2 load balancers
   → Use floating IP / VRRP for automatic failover
   → Or use cloud LB (inherently redundant)

3. ALWAYS use connection draining
   → When removing a server from rotation
   → Don't kill in-flight requests
   → Stop sending NEW requests to that server
   → Wait for existing requests to complete (timeout: 30s)
   → Then remove server

4. ALWAYS forward client information
   → Backend server needs to know the REAL client IP
   → Without forwarding: all requests appear from LB IP
   → Use headers: X-Forwarded-For, X-Real-IP
   → Essential for: logging, rate limiting (Project 9), analytics

5. ALWAYS set appropriate timeouts
   → Connection timeout: how long to wait for backend to accept
   → Response timeout: how long to wait for response
   → Idle timeout: how long to keep inactive connection open
   → Too short → legitimate slow requests fail
   → Too long → resources tied up by dead connections

6. ALWAYS log at the load balancer level
   → Every request: client IP, chosen backend, response time, status
   → This is your FIRST line of visibility
   → Can identify: slow backends, error patterns, traffic distribution

❌ WHAT SHOULD NOT DO

1. NEVER rely on a single load balancer
   ❌ One LB for entire production traffic
   → LB crash = total outage for ALL users
   → ALWAYS have redundancy (active-passive or active-active)

2. NEVER use sticky sessions without understanding the cost
   ❌ "Always send user X to Server 3"
   → Server 3 crashes → user X loses session
   → Server 3 overloaded → can't redistribute
   → Defeats the purpose of load balancing
   → Better: stateless servers + external session store (Project 8)

3. NEVER ignore health check configuration
   ❌ No health checks → LB sends traffic to dead servers
   ❌ Too aggressive checks → flapping (server bounces in/out)
   ❌ Too lenient checks → dead server receives traffic for minutes
   → Good defaults: check every 10s, 3 failures to remove,
     2 successes to re-add

4. NEVER let the LB become the bottleneck
   → LB must handle MORE traffic than all backends combined
   → If backends handle 5,000 req/sec total
   → LB must handle >> 5,000 req/sec
   → Monitor LB CPU/memory/connections too

5. NEVER expose backend server IPs to clients
   ❌ Response header: X-Backend-Server: 192.168.1.10
   → Security risk: attacker knows internal topology
   → If you must identify backends → use opaque IDs
   → X-Served-By: server-3 (no IP)

6. NEVER mix stateful and stateless servers behind same LB
   ❌ Server 1 stores sessions, Server 2 doesn't
   → Inconsistent behavior depending on which server handles request
   → ALL servers behind an LB must be IDENTICAL

🚫 WHEN NOT TO USE

1. Load Balancer is NOT needed when:
   → Single server handles all traffic comfortably
   → Internal tool with < 10 users
   → Development/staging environments (usually)
   → The complexity isn't justified by the traffic

2. Round Robin is NOT suitable when:
   → Servers have vastly different capacities
   → Some requests take 10ms, others take 10 seconds
   → You need session affinity

3. IP Hash is NOT suitable when:
   → Many clients behind same NAT (corporate networks)
     → All appear as same IP → all go to same server
   → You need even distribution
   → You plan to add/remove servers frequently
     → Hash redistribution disrupts existing sessions

4. Layer 4 LB is NOT suitable when:
   → You need URL-based routing (/api vs /static)
   → You need to inspect/modify HTTP headers
   → You need SSL termination at the LB
   → You need HTTP-level health checks

5. Sticky sessions are NOT suitable when:
   → You want true horizontal scalability
   → You can use external session storage instead
   → Server failures should be transparent to users

═══════════════════════════════════════════════════════
PART 1: THE COMPLETE REQUEST LIFECYCLE THROUGH LB
═══════════════════════════════════════════════════════

  Step 1: CLIENT INITIATES CONNECTION
    → Client resolves api.restaurant.com → LB IP (203.0.113.10)
    → Client TCP connects to 203.0.113.10:443
    → TCP three-way handshake with LB (see Project 1)

  Step 2: SSL/TLS TERMINATION (Layer 7)
    → LB decrypts HTTPS → reads HTTP request
    → Backend communication can be plain HTTP (faster)
    → Or re-encrypted (more secure)
    
    Client ──[HTTPS]──→ LB ──[HTTP]──→ Backend
                        ↑
                   SSL certificate
                   lives here

  Step 3: LB READS THE REQUEST (Layer 7)
    → Method: GET
    → Path: /api/v1/menu-items
    → Headers: Host, Accept, Authorization, etc.
    → Can make routing decisions based on ANY of these

  Step 4: LB SELECTS BACKEND
    → Consults algorithm (round robin, least connections, etc.)
    → Filters out unhealthy servers
    → Picks target: Server 3 (192.168.1.12:8080)

  Step 5: LB FORWARDS REQUEST
    → Opens (or reuses) TCP connection to Server 3
    → Adds/modifies headers:
      → X-Forwarded-For: client's real IP
      → X-Forwarded-Proto: https (original protocol)
      → X-Request-ID: unique tracking ID
    → Sends the HTTP request to Server 3

  Step 6: BACKEND PROCESSES
    → Server 3 receives request (looks like normal HTTP)
    → Processes it (cache check, DB query, etc.)
    → Sends HTTP response back to LB

  Step 7: LB RETURNS RESPONSE
    → LB receives response from Server 3
    → Can modify response headers
    → Can compress response (gzip)
    → Encrypts response (HTTPS)
    → Sends to client

  Step 8: LB LOGS THE REQUEST
    → Client IP, backend chosen, response time, status code
    → Aggregate for monitoring dashboards

PART 2: HEALTH CHECK MECHANICS
═══════════════════════════════════════════════════════

  LB maintains a health state for each backend:

  ┌──────────┬────────┬──────────┬──────────────────────┐
  │ Server   │ Status │ Failures │ Last Check           │
  ├──────────┼────────┼──────────┼──────────────────────┤
  │ Server 1 │ UP ✅  │ 0        │ 2025-01-01 12:00:05 │
  │ Server 2 │ UP ✅  │ 0        │ 2025-01-01 12:00:05 │
  │ Server 3 │ DOWN ❌│ 3        │ 2025-01-01 12:00:05 │
  │ Server 4 │ UP ✅  │ 1        │ 2025-01-01 12:00:05 │
  │ Server 5 │ UP ✅  │ 0        │ 2025-01-01 12:00:05 │
  └──────────┴────────┴──────────┴──────────────────────┘

  Health Check Process (every 10 seconds):
  
  1. LB sends: GET /health HTTP/1.1 to each server
  2. Server responds:
     → 200 OK → healthy
     → 503 Service Unavailable → unhealthy
     → Timeout (no response in 5s) → unhealthy
     → Connection refused → unhealthy
  
  3. State Transitions:
     → UP + health check passes → stays UP (reset failure count)
     → UP + health check fails → increment failures
     → UP + failures >= threshold (3) → transition to DOWN
     → DOWN + health check passes → increment successes
     → DOWN + successes >= threshold (2) → transition to UP
  
  This prevents flapping:
     → One failed check doesn't remove server (transient issue)
     → One passed check doesn't add server back (might be unstable)

═══════════════════════════════════════════════════════
PART 3: CONNECTION POOLING (LB to Backend)
═══════════════════════════════════════════════════════

  WITHOUT connection pooling:
    Every request: LB → TCP connect → send → receive → TCP close
    TCP handshake = 1.5 round trips = 1-3ms overhead PER REQUEST

  WITH connection pooling:
    LB maintains POOL of open connections to each backend
    Request arrives → grab connection from pool → send → return to pool
    No TCP handshake per request → much faster

  ┌──────────────────────────────────────────────────────┐
  │  Load Balancer Connection Pool                        │
  │                                                       │
  │  Server 1: [conn1] [conn2] [conn3] (3 idle, 2 active)│
  │  Server 2: [conn4] [conn5] (2 idle, 1 active)        │
  │  Server 3: [DOWN — no connections]                    │
  │  Server 4: [conn6] [conn7] [conn8] (1 idle, 4 active)│
  │  Server 5: [conn9] (1 idle, 0 active)                │
  └──────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════
PART 4: CONNECTION DRAINING
═══════════════════════════════════════════════════════

  When removing Server 2 from rotation (deployment, maintenance):
  
  Time 0:00 → Server 2 marked for draining
    → Stop sending NEW requests to Server 2
    → Server 2 has 15 active requests in-flight
  
  Time 0:05 → 10 requests completed naturally
    → 5 still in-flight
  
  Time 0:15 → 3 more completed
    → 2 still in-flight
  
  Time 0:25 → All completed
    → Server 2 safely removed
  
  Time 0:30 → Drain timeout (if any remain)
    → Force-close remaining connections
    → Accept minor disruption for those requests

  WITHOUT draining:
    → Remove Server 2 → 15 in-flight requests get TCP RST
    → 15 users see error → bad experience

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Load Balancer as Single Point of Failure
  → One LB → LB crash → everything down
  → Worse than having no LB at all (added dependency)
  → FIX: Always deploy LBs in pairs (active-passive or active-active)
  → Cloud LBs handle this automatically

PITFALL 2: Health Check Endpoint Does Too Much
  → /health checks database connection, cache, external APIs
  → External API is slow → health check times out
  → LB thinks server is dead → removes it
  → But server is actually fine for most requests
  → FIX: Two types of health checks:
    → Liveness: "Is the process running?" (simple, fast)
    → Readiness: "Can it handle requests?" (thorough)
    → LB uses liveness, deployment uses readiness (Project 15)

PITFALL 3: Thundering Herd on Server Recovery
  → Server 3 was down → comes back up
  → LB immediately sends FULL traffic share
  → Server 3 hasn't warmed up cache → all cache misses
  → Server 3 overwhelmed → goes down again
  → FIX: Slow start / warm-up period
    → Gradually increase traffic to recovered server
    → 10% → 25% → 50% → 100% over 60 seconds

PITFALL 4: Ignoring Backend Response Times
  → Server 4 responds in 5 seconds (disk issue)
  → LB keeps sending traffic (round robin doesn't care)
  → Users hitting Server 4 all have bad experience
  → FIX: Use Least Response Time algorithm
  → Or: set response timeout, mark slow servers as degraded

PITFALL 5: Wrong Timeout Configuration
  → LB timeout = 5 seconds
  → Report endpoint takes 30 seconds legitimately
  → LB kills the connection at 5 seconds → user gets error
  → FIX: Different timeouts for different endpoints
  → Or: separate LB for long-running requests

PITFALL 6: Losing Client IP
  → All requests appear from LB IP (192.168.1.1)
  → Backend logs show one "user" making all requests
  → Rate limiting doesn't work (can't identify clients)
  → FIX: Always set X-Forwarded-For header
  → Backend must read X-Forwarded-For instead of connection IP

PITFALL 7: SSL Certificate Issues
  → SSL termination at LB → backend sees HTTP
  → Backend generates URL: http://api.restaurant.com (not https)
  → Client gets mixed content or redirect loops
  → FIX: Backend reads X-Forwarded-Proto header
  → Or: configure backend to always generate HTTPS URLs

NEW FOR LOAD BALANCERS:

1. LB CAPACITY
   → Software LB (Nginx): ~100,000-500,000 req/sec per instance
   → Hardware LB (F5): millions of req/sec
   → Cloud LB (AWS ALB): auto-scales (virtually unlimited)
   → Usually LB is NOT the bottleneck
   → But: monitor it — it CAN become one

2. ADDING/REMOVING SERVERS
   → Zero-downtime scaling:
     → Add Server 6 → register with LB → starts receiving traffic
     → Remove Server 3 → drain → deregister → shut down
     → No client impact at any point

3. AUTO-SCALING INTEGRATION
   → Monitor: avg CPU across all servers
   → CPU > 70% → add server → register with LB
   → CPU < 30% → drain server → deregister → terminate
   → LB is the control point for fleet management

4. GEOGRAPHIC DISTRIBUTION
   → Global LB routes to nearest region
   → US users → US servers
   → EU users → EU servers
   → Asia users → Asia servers
   → DNS-based (Route 53) or Anycast
   → Deep dive in Project 12 (CDN)

5. LB CHAIN
   → Global LB → Regional LB → Application servers

class ServerStatus(Enum):
    HEALTHY = "healthy"
    DEGRADED = "degraded"
    UNHEALTHY = "unhealthy"
    DRAINING = "draining"


class BackendServer:
    """Simulates a backend application server."""
    
    def __init__(self, server_id, host, port, weight=1,
                 base_latency_ms=15, failure_rate=0.0):
        self.server_id = server_id
        self.host = host
        self.port = port
        self.weight = weight
        self.base_latency_ms = base_latency_ms
        self.failure_rate = failure_rate
        
        # State
        self.status = ServerStatus.HEALTHY
        self.active_connections = 0
        self.lock = threading.Lock()
        
        # Metrics
        self.total_requests = 0
        self.total_errors = 0
        self.response_times = deque(maxlen=1000)
        self.started_at = time.time()
    
    @property
    def address(self):
        return f"{self.host}:{self.port}"
    
    @property
    def is_available(self):
        return self.status in (ServerStatus.HEALTHY, ServerStatus.DEGRADED)
    
    def handle_request(self, request):
        """Simulate processing a request."""
        with self.lock:
            self.active_connections += 1
        
        try:
            # Simulate failures
            if random.random() < self.failure_rate:
                with self.lock:
                    self.total_errors += 1
                return {
                    "status": 500,
                    "error": "Internal Server Error",
                    "server": self.server_id
                }
            
            # Simulate processing time with load-based degradation
            load_factor = 1 + max(0, (self.active_connections - 10)) * 0.1
            latency = self.base_latency_ms * load_factor
            latency *= random.uniform(0.7, 1.3)
            
            if self.status == ServerStatus.DEGRADED:
                latency *= 3
            
            time.sleep(latency / 1000)
            
            with self.lock:
                self.total_requests += 1
                self.response_times.append(latency)
            
            return {
                "status": 200,
                "data": request.get("path", "/"),
                "server": self.server_id,
                "latency_ms": round(latency, 2)
            }
        finally:
            with self.lock:
                self.active_connections -= 1
    
    def health_check(self):
        """Respond to health probe."""
        if self.status == ServerStatus.UNHEALTHY:
            return False
        
        if self.status == ServerStatus.DRAINING:
            return False  # Don't accept new traffic
        
        # Simulate occasional health check failure
        if random.random() < self.failure_rate:
            return False
        
        return True
    
    def get_avg_response_time(self):
        with self.lock:
            if not self.response_times:
                return 0
            return statistics.mean(self.response_times)
    
    def get_stats(self):
        with self.lock:
            resp_times = list(self.response_times)
        
        total = self.total_requests + self.total_errors
        
        stats = {
            "server_id": self.server_id,
            "address": self.address,
            "status": self.status.value,
            "weight": self.weight,
            "active_connections": self.active_connections,
            "total_requests": self.total_requests,
            "total_errors": self.total_errors,
            "error_rate_pct": round(
                self.total_errors / max(1, total) * 100, 2
            )
        }
        
        if resp_times:
            resp_times.sort()
            n = len(resp_times)
            stats["latency"] = {
                "avg_ms": round(statistics.mean(resp_times), 2),
                "p50_ms": round(resp_times[int(n * 0.50)], 2),
                "p95_ms": round(resp_times[int(n * 0.95)], 2),
                "p99_ms": round(resp_times[min(int(n * 0.99), n-1)], 2)
            }
        
        return stats
    
    def reset_stats(self):
        with self.lock:
            self.total_requests = 0
            self.total_errors = 0
            self.response_times.clear()

class LoadBalancingAlgorithm:
    """Base class for all LB algorithms."""
    
    def __init__(self, name):
        self.name = name
    
    def select(self, servers, request=None):
        raise NotImplementedError


class RoundRobin(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Round Robin")
        self.index = 0
        self.lock = threading.Lock()
    
    def select(self, servers, request=None):
        if not servers:
            return None
        with self.lock:
            server = servers[self.index % len(servers)]
            self.index += 1
            return server


class WeightedRoundRobin(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Weighted Round Robin")
        self.index = 0
        self.lock = threading.Lock()
    
    def select(self, servers, request=None):
        if not servers:
            return None
        
        # Build weighted list
        weighted = []
        for server in servers:
            weighted.extend([server] * server.weight)
        
        if not weighted:
            return None
        
        with self.lock:
            server = weighted[self.index % len(weighted)]
            self.index += 1
            return server


class LeastConnections(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Least Connections")
    
    def select(self, servers, request=None):
        if not servers:
            return None
        return min(servers, key=lambda s: s.active_connections)


class WeightedLeastConnections(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Weighted Least Connections")
    
    def select(self, servers, request=None):
        if not servers:
            return None
        # Score = active_connections / weight (lower is better)
        return min(servers,
                   key=lambda s: s.active_connections / max(s.weight, 1))


class IPHash(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("IP Hash")
    
    def select(self, servers, request=None):
        if not servers:
            return None
        client_ip = request.get("client_ip", "0.0.0.0") if request else "0.0.0.0"
        hash_val = int(hashlib.md5(client_ip.encode()).hexdigest(), 16)
        return servers[hash_val % len(servers)]


class LeastResponseTime(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Least Response Time")
    
    def select(self, servers, request=None):
        if not servers:
            return None
        return min(servers, key=lambda s: s.get_avg_response_time())


class RandomSelection(LoadBalancingAlgorithm):
    def __init__(self):
        super().__init__("Random")
    
    def select(self, servers, request=None):
        if not servers:
            return None
        return random.choice(servers)

class HealthChecker:
    """
    Periodically checks backend server health.
    Implements state machine for server health transitions.
    """
    
    def __init__(self, check_interval=2.0, failure_threshold=3,
                 recovery_threshold=2):
        self.check_interval = check_interval
        self.failure_threshold = failure_threshold
        self.recovery_threshold = recovery_threshold
        
        # Per-server health tracking
        self.consecutive_failures = defaultdict(int)
        self.consecutive_successes = defaultdict(int)
        self.health_history = defaultdict(lambda: deque(maxlen=20))
        
        self.running = False
        self.thread = None
        self.lock = threading.Lock()
        self.servers = []
        self.on_server_down = None  # Callback
        self.on_server_up = None    # Callback
    
    def start(self, servers, on_down=None, on_up=None):
        self.servers = servers
        self.on_server_down = on_down
        self.on_server_up = on_up
        self.running = True
        self.thread = threading.Thread(target=self._check_loop, daemon=True)
        self.thread.start()
    
    def stop(self):
        self.running = False
        if self.thread:
            self.thread.join(timeout=5)
    
    def _check_loop(self):
        while self.running:
            for server in self.servers:
                if server.status == ServerStatus.DRAINING:
                    continue
                
                is_healthy = server.health_check()
                self._process_result(server, is_healthy)
            
            time.sleep(self.check_interval)
    
    def _process_result(self, server, is_healthy):
        sid = server.server_id
        timestamp = time.time()
        
        with self.lock:
            self.health_history[sid].append({
                "time": timestamp,
                "healthy": is_healthy
            })
            
            if is_healthy:
                self.consecutive_failures[sid] = 0
                self.consecutive_successes[sid] += 1
                
                # Recovery: DOWN → UP
                if (server.status == ServerStatus.UNHEALTHY and
                        self.consecutive_successes[sid] >= self.recovery_threshold):
                    server.status = ServerStatus.HEALTHY
                    if self.on_server_up:
                        self.on_server_up(server)
            else:
                self.consecutive_successes[sid] = 0
                self.consecutive_failures[sid] += 1
                
                # Failure: UP → DOWN
                if (server.status == ServerStatus.HEALTHY and
                        self.consecutive_failures[sid] >= self.failure_threshold):
                    server.status = ServerStatus.UNHEALTHY
                    if self.on_server_down:
                        self.on_server_down(server)
    
    def get_health_report(self):
        report = {}
        with self.lock:
            for server in self.servers:
                sid = server.server_id
                history = list(self.health_history[sid])
                recent_checks = len(history)
                recent_healthy = sum(1 for h in history if h["healthy"])
                
                report[sid] = {
                    "status": server.status.value,
                    "consecutive_failures": self.consecutive_failures[sid],
                    "consecutive_successes": self.consecutive_successes[sid],
                    "recent_checks": recent_checks,
                    "recent_healthy": recent_healthy,
                    "uptime_pct": round(
                        recent_healthy / max(1, recent_checks) * 100, 1
                    )
                }
        return report

class LoadBalancer:
    """
    Complete load balancer with:
    - Pluggable algorithms
    - Health checking
    - Access logging
    - Connection draining
    - Client IP forwarding
    - Per-server metrics
    """
    
    def __init__(self, algorithm=None, name="LoadBalancer"):
        self.algorithm = algorithm or RoundRobin()
        self.name = name
        self.servers = []
        self.health_checker = HealthChecker(
            check_interval=1.0,
            failure_threshold=3,
            recovery_threshold=2
        )
        
        # Metrics
        self.total_requests = 0
        self.total_errors = 0
        self.no_healthy_server_errors = 0
        self.access_log = deque(maxlen=5000)
        self.lock = threading.Lock()
        
        # Events
        self.events = deque(maxlen=100)
    
    def add_server(self, server):
        self.servers.append(server)
        self._log_event(f"Server {server.server_id} added ({server.address})")
    
    def remove_server(self, server_id, drain=True, drain_timeout=5):
        """Remove server with optional connection draining."""
        server = self._find_server(server_id)
        if not server:
            return False
        
        if drain:
            server.status = ServerStatus.DRAINING
            self._log_event(
                f"Server {server_id} draining "
                f"({server.active_connections} active connections)"
            )
            
            # Wait for connections to drain
            start = time.time()
            while (server.active_connections > 0 and
                   time.time() - start < drain_timeout):
                time.sleep(0.1)
            
            remaining = server.active_connections
            if remaining > 0:
                self._log_event(
                    f"Server {server_id} drain timeout "
                    f"({remaining} connections force-closed)"
                )
        
        self.servers.remove(server)
        self._log_event(f"Server {server_id} removed")
        return True
    
    def handle_request(self, request):
        """Process a single request through the load balancer."""
        start_time = time.time()
        
        with self.lock:
            self.total_requests += 1
            request_num = self.total_requests
        
        # Get healthy servers
        healthy_servers = [
            s for s in self.servers if s.is_available
        ]
        
        if not healthy_servers:
            with self.lock:
                self.total_errors += 1
                self.no_healthy_server_errors += 1
            
            log_entry = self._create_log_entry(
                request, None, 503, 0, request_num
            )
            self.access_log.append(log_entry)
            
            return {
                "status": 503,
                "error": "No healthy backend servers available",
                "lb": self.name
            }
        
        # Select backend server
        selected = self.algorithm.select(healthy_servers, request)
        
        if not selected:
            with self.lock:
                self.total_errors += 1
            return {
                "status": 503,
                "error": "Algorithm could not select a server",
                "lb": self.name
            }
        
        # Forward request with added headers
        enriched_request = dict(request)
        enriched_request["X-Forwarded-For"] = request.get("client_ip", "unknown")
        enriched_request["X-Forwarded-Proto"] = "https"
        enriched_request["X-Request-ID"] = f"req-{request_num}"
        enriched_request["X-LB-Name"] = self.name
        
        # Forward to backend
        response = selected.handle_request(enriched_request)
        
        elapsed_ms = (time.time() - start_time) * 1000
        
        # Add LB metadata to response
        response["lb_latency_ms"] = round(elapsed_ms, 2)
        response["backend"] = selected.server_id
        response["X-Request-ID"] = enriched_request["X-Request-ID"]
        
        if response.get("status", 200) >= 500:
            with self.lock:
                self.total_errors += 1
        
        # Log
        log_entry = self._create_log_entry(
            request, selected, response.get("status", 200),
            elapsed_ms, request_num
        )
        self.access_log.append(log_entry)
        
        return response
    
    def start_health_checks(self):
        self.health_checker.start(
            self.servers,
            on_down=self._on_server_down,
            on_up=self._on_server_up
        )
    
    def stop_health_checks(self):
        self.health_checker.stop()
    
    def _on_server_down(self, server):
        self._log_event(
            f"🔴 Server {server.server_id} marked UNHEALTHY "
            f"(removed from rotation)"
        )
    
    def _on_server_up(self, server):
        self._log_event(
            f"🟢 Server {server.server_id} recovered "
            f"(added back to rotation)"
        )
    
    def _find_server(self, server_id):
        for s in self.servers:
            if s.server_id == server_id:
                return s
        return None
    
    def _create_log_entry(self, request, server, status, latency_ms,
                          request_num):
        return {
            "request_id": f"req-{request_num}",
            "timestamp": time.time(),
            "client_ip": request.get("client_ip", "unknown"),
            "method": request.get("method", "GET"),
            "path": request.get("path", "/"),
            "backend": server.server_id if server else "NONE",
            "status": status,
            "latency_ms": round(latency_ms, 2)
        }
    
    def _log_event(self, message):
        self.events.append({
            "time": time.time(),
            "message": message
        })
        print(f"    [LB] {message}")
    
    def get_stats(self):
        healthy = sum(1 for s in self.servers if s.is_available)
        total_backend_requests = sum(s.total_requests for s in self.servers)
        
        stats = {
            "name": self.name,
            "algorithm": self.algorithm.name,
            "total_servers": len(self.servers),
            "healthy_servers": healthy,
            "unhealthy_servers": len(self.servers) - healthy,
            "total_requests": self.total_requests,
            "total_errors": self.total_errors,
            "no_healthy_server_errors": self.no_healthy_server_errors,
            "error_rate_pct": round(
                self.total_errors / max(1, self.total_requests) * 100, 2
            ),
            "servers": {}
        }
        
        for server in self.servers:
            s_stats = server.get_stats()
            pct = round(
                server.total_requests / max(1, total_backend_requests) * 100, 1
            )
            s_stats["traffic_share_pct"] = pct
            stats["servers"][server.server_id] = s_stats
        
        return stats
    
    def reset_all_stats(self):
        with self.lock:
            self.total_requests = 0
            self.total_errors = 0
            self.no_healthy_server_errors = 0
            self.access_log.clear()
        for server in self.servers:
            server.reset_stats()
    
    def get_recent_logs(self, count=10):
        return list(self.access_log)[-count:]

class LBLoadTester:
    """Sends concurrent requests through a load balancer."""
    
    def run_test(self, lb, num_requests, concurrent_users,
                 label="Test", client_ips=None):
        print(f"\n  🔥 {label}")
        print(f"     Requests: {num_requests} | Concurrent: {concurrent_users}")
        print(f"     Algorithm: {lb.algorithm.name}")
        print(f"     Running...", end="", flush=True)
        
        results = []
        start_time = time.time()
        
        def send_request(i):
            client_ip = (client_ips[i % len(client_ips)]
                         if client_ips
                         else f"10.0.{i % 256}.{(i * 7) % 256}")
            
            request = {
                "method": "GET",
                "path": "/api/v1/menu-items",
                "client_ip": client_ip
            }
            return lb.handle_request(request)
        
        with ThreadPoolExecutor(max_workers=concurrent_users) as executor:
            futures = [
                executor.submit(send_request, i)
                for i in range(num_requests)
            ]
            for future in as_completed(futures):
                results.append(future.result())
        
        elapsed = time.time() - start_time
        
        successful = [r for r in results if r.get("status", 500) < 500]
        failed = [r for r in results if r.get("status", 500) >= 500]
        
        latencies = [r["lb_latency_ms"] for r in successful
                     if "lb_latency_ms" in r]
        
        print(f" Done! ({elapsed:.2f}s)")
        
        analysis = {
            "label": label,
            "elapsed_sec": round(elapsed, 3),
            "throughput_rps": round(len(successful) / max(elapsed, 0.001), 1),
            "successful": len(successful),
            "failed": len(failed),
            "error_rate_pct": round(
                len(failed) / max(1, num_requests) * 100, 2
            )
        }
        
        if latencies:
            latencies.sort()
            n = len(latencies)
            analysis["latency"] = {
                "avg_ms": round(statistics.mean(latencies), 2),
                "p50_ms": round(latencies[int(n * 0.50)], 2),
                "p95_ms": round(latencies[int(n * 0.95)], 2),
                "p99_ms": round(latencies[min(int(n * 0.99), n-1)], 2)
            }
        
        # Traffic distribution
        server_dist = defaultdict(int)
        for r in successful:
            server_dist[r.get("backend", "unknown")] += 1
        analysis["distribution"] = dict(server_dist)
        
        self._print_analysis(analysis)
        return analysis
    
    def _print_analysis(self, a):
        print(f"     Throughput:  {a['throughput_rps']} req/sec")
        print(f"     Successful:  {a['successful']} | Failed: {a['failed']}"
              f" ({a['error_rate_pct']}% error)")
        
        lat = a.get("latency", {})
        if lat:
            print(f"     Latency avg: {lat['avg_ms']}ms | "
                  f"p95: {lat['p95_ms']}ms | p99: {lat['p99_ms']}ms")
        
        if a.get("distribution"):
            print(f"     Distribution:")
            total = sum(a["distribution"].values())
            for server_id, count in sorted(a["distribution"].items()):
                pct = round(count / max(1, total) * 100, 1)
                bar = "█" * int(pct / 2) + "░" * (50 - int(pct / 2))
                print(f"       {server_id:<12s} {count:>5d} "
                      f"({pct:>5.1f}%) {bar}")

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RIGHT NOW:
  → We have a load balancer at IP 203.0.113.10
  → Client app hardcodes this IP
  → What if we need to change the IP?
  → What if we have LBs in different regions?
  → What if we want users to type "restaurant.com" in a browser?

THE PROBLEM:
  → Humans remember NAMES: "restaurant.com"
  → Computers need NUMBERS: 203.0.113.10
  → Something must translate names to numbers

THE ANSWER: DNS (Domain Name System)
  → User types: api.restaurant.com
  → DNS resolves: api.restaurant.com → 203.0.113.10
  → Browser connects to 203.0.113.10 (our load balancer)
  → Everything else works as we've built it

BUT DNS IS MORE THAN JUST NAME→IP:
  → Hierarchical system (root → .com → restaurant.com)
  → Caching at every level (TTL-based, like our cache!)
  → Can return different IPs based on location (geo-routing)
  → Can be the FIRST layer of load balancing
  → Can point to CDN servers (Project 12)
  → Misconfiguration = entire system unreachable

WHAT PROJECT 7 TEACHES:
  → How DNS resolution works step by step
  → DNS record types (A, CNAME, MX, TXT, NS)
  → DNS TTL and caching
  → DNS as a load balancing mechanism
  → DNS failover for high availability
  → The foundation for CDN (Project 12) and service discovery (Project 20)

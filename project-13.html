PROJECT 13: THE BACKUP DIARY — DATABASE REPLICATION (READ REPLICAS)

📋 PHASE 2 (Easy) — Core Patterns

🎯 HYPOTHETICAL PROBLEM

Your pizza shop chain has grown massively:
  → 500,000 customers in the database
  → 10,000,000 orders (2 years of history)
  → 5,000 API requests/second hitting the database

Your SINGLE database server in New York handles EVERYTHING:

SCENARIO A: The Read/Write Bottleneck
  → 5,000 requests/second to ONE database
  → Breakdown:
    → 4,000 reads/sec (GET orders, GET menu, GET user profiles)
    → 1,000 writes/sec (POST orders, PUT user, UPDATE inventory)
  → Reads and writes COMPETE for:
    → CPU (query parsing, execution)
    → Memory (buffer pool, query cache)
    → Disk I/O (reading data pages, writing WAL)
    → Lock contention (writes lock rows that reads need)
  
  → Result:
    → Average query time: 5ms → climbed to 80ms
    → p99 query time: 500ms (unacceptable)
    → Database CPU: 92% utilized
    → Connection pool: 95% exhausted
    → One more traffic spike = database crash


SCENARIO B: The Single Point of Failure
  → Friday night, peak hours, 8,000 orders/hour
  → Database server's primary disk fails
  → ALL services go down:
    → Website: "Error loading menu"
    → App: "Cannot place order"
    → Kitchen displays: blank
    → Support: "Cannot look up orders"
  → Last backup: 6 hours ago
  → Restore from backup: 2 hours
  → 6 hours of orders LOST FOREVER
  → 2 hours of TOTAL downtime
  → Revenue lost: $80,000
  → Customer trust: destroyed


SCENARIO C: The Reporting Lock-Up
  → Finance team runs daily report at 9 AM:
    SELECT pizza_type, SUM(total_price), COUNT(*)
    FROM orders
    WHERE order_date >= '2025-01-01'
    GROUP BY pizza_type
    ORDER BY SUM(total_price) DESC
  
  → This query scans 500,000 rows (even with index — Project 10)
  → Takes 15 seconds to execute
  → During those 15 seconds:
    → Query holds read locks on thousands of rows
    → Write queries (new orders) must WAIT
    → Order placement latency spikes from 5ms to 2,000ms
    → Customers see "Order failed, try again"
  → Analytics queries KILL production performance


SCENARIO D: The Geographic Latency
  → Database in New York
  → New office opened in London
  → London staff query the database:
    → Network RTT: 80ms per round trip
    → Simple query: 5ms execution + 80ms network = 85ms
    → Complex page with 10 queries: 10 × 85ms = 850ms
  → Everything feels SLOW for London team
  → A database copy IN London would be 5ms per query


THE ANALOGY:
  → You keep ONE diary where you write EVERYTHING
  → Your entire business depends on this ONE diary
  → If the diary is destroyed → all records lost
  → If someone is reading the diary → nobody else can write in it
  → If 10 people need to read → they wait in line

  → SOLUTION: Make COPIES of the diary
  → One MASTER diary: the one you WRITE in (source of truth)
  → Multiple COPY diaries: people READ from these
  → Every time you write in the master → copies are updated
  → If master is destroyed → promote a copy to be the new master
  → 10 people reading → each reads their own copy → no waiting

  → The MASTER diary is the PRIMARY database
  → The COPY diaries are the REPLICA databases
  → "Updating copies" is REPLICATION
  → "Promoting a copy" is FAILOVER

🔍 HOW TO APPROACH IT

Step 1: Understand the READ/WRITE split
         → Reads: SELECT queries (read data, don't change it)
         → Writes: INSERT, UPDATE, DELETE (modify data)
         → Most applications are 80-95% reads
         → Replicas handle reads, primary handles writes

Step 2: Understand REPLICATION mechanics
         → Primary executes writes and records them in a log
         → Log is shipped to replicas
         → Replicas replay the log to apply changes
         → This is called Write-Ahead Log (WAL) replication

Step 3: Choose REPLICATION mode
         → Synchronous: Primary waits until replica confirms
         → Asynchronous: Primary doesn't wait (faster but lag possible)
         → Semi-synchronous: Primary waits for at least ONE replica

Step 4: Design READ routing
         → Application must know: reads → replica, writes → primary
         → Use a connection proxy or application-level routing
         → Handle "read-after-write" consistency

Step 5: Plan for FAILOVER
         → Primary dies → detect failure → promote replica → redirect traffic
         → Automatic vs manual failover
         → Data loss risk during async replication

Step 6: Monitor REPLICATION LAG
         → Time between write on primary and appearance on replica
         → Async: milliseconds to seconds of lag
         → Affects: "I just updated my profile but still see old data"

REPLICATION TYPES:

TYPE 1: ASYNCHRONOUS REPLICATION
════════════════════════════════
  
  Primary                          Replica
  ┌─────────┐                     ┌─────────┐
  │ WRITE   │                     │         │
  │ INSERT  │──→ WAL ──→ ship ──→│ REPLAY  │
  │ (done!) │     log     async   │ (later) │
  └─────────┘                     └─────────┘
       │
       ▼
  Response to client (immediately)
  
  → Primary does NOT wait for replica to confirm
  → Fastest for writes (no extra latency)
  → Risk: If primary crashes before log shipped → data loss
  → Replica may be BEHIND primary (replication lag)
  → Most common in production (MySQL default, PostgreSQL streaming)


TYPE 2: SYNCHRONOUS REPLICATION
═══════════════════════════════
  
  Primary                          Replica
  ┌─────────┐                     ┌─────────┐
  │ WRITE   │                     │         │
  │ INSERT  │──→ WAL ──→ ship ──→│ REPLAY  │
  │         │     log     sync    │         │
  │ (wait)  │←── ACK ────────────│ (done!) │
  │ (done!) │                     └─────────┘
  └─────────┘
       │
       ▼
  Response to client (after replica confirms)
  
  → Primary WAITS for replica to confirm write
  → Zero data loss (replica always has latest)
  → Slower writes (adds network RTT to every write)
  → If replica is slow/down → primary writes blocked
  → Used when data loss is unacceptable (financial systems)


TYPE 3: SEMI-SYNCHRONOUS REPLICATION
════════════════════════════════════
  
  Primary                    Replica 1          Replica 2
  ┌─────────┐               ┌─────────┐       ┌─────────┐
  │ WRITE   │──→ ship ──→   │ REPLAY  │       │         │
  │         │                │         │       │ (async) │
  │ (wait)  │←── ACK ───────│ (done!) │       │         │
  │ (done!) │               └─────────┘       └─────────┘
  └─────────┘
  
  → Wait for AT LEAST ONE replica to confirm
  → Balance: some durability + reasonable speed
  → If fastest replica is 2ms away → adds 2ms to writes
  → MySQL semi-sync replication works this way


WRITE-AHEAD LOG (WAL) MECHANICS:
════════════════════════════════

  Every write operation on primary:
  
  1. Transaction begins
  2. Write change to WAL (sequential append — very fast)
  3. Apply change to data files
  4. Transaction commits
  5. Ship WAL entries to replicas
  
  WAL entry format:
  ┌──────────────────────────────────────────────────┐
  │ LSN: [REDACTED:SSN]                                   │  ← Log Sequence Number
  │ Timestamp: 2025-01-15 10:30:45.123               │
  │ Transaction: TX-789                               │
  │ Operation: INSERT                                 │
  │ Table: orders                                     │
  │ Data: {order_id: 'ORD-5001', pizza: 'pepperoni', │
  │         user_id: 'user_042', total: 15.99}        │
  └──────────────────────────────────────────────────┘
  
  Replica receives WAL entries:
  1. Append to local WAL (for crash recovery)
  2. Replay: execute the same INSERT on local data
  3. Report position (LSN) back to primary
  4. Now replica has the same data as primary
  
  This is called LOG-BASED REPLICATION
  → Exact same changes applied in exact same order
  → Replica becomes an exact copy of primary


REPLICATION LAG:
═══════════════

  Timeline:
  
  t=0.000s: Primary receives INSERT (order ORD-5001)
  t=0.001s: Primary writes to WAL
  t=0.002s: Primary commits, responds to client "Order created!"
  t=0.003s: WAL entry shipped to network
  t=0.005s: Replica 1 receives WAL entry
  t=0.006s: Replica 1 replays INSERT
  
  Replication lag for Replica 1: 6ms - 2ms = 4ms
  
  During those 4ms:
  → Client that wrote sees: "Order ORD-5001 created!"
  → Client reading from Replica 1: "Order ORD-5001? Not found!"
  → This is the READ-AFTER-WRITE CONSISTENCY problem
  
  Under heavy load:
  → Replica falls behind → lag grows to seconds
  → User updates profile → refreshes page → sees OLD profile
  → User places order → checks order status → "Order not found"
  
  NORMAL lag: 1-50ms (barely noticeable)
  CONCERNING lag: 100ms-1s (user might notice)
  CRITICAL lag: >1s (application must handle this)

✅ WHAT SHOULD DO

1. ALWAYS route writes to primary, reads to replicas
   → Never write to a replica (it would diverge from primary)
   → Use a routing layer or proxy to enforce this
   → Application code should NOT know about specific servers

2. ALWAYS handle read-after-write consistency
   → User writes data → immediately reads it back
   → If read goes to replica → might not have the write yet
   → Solution 1: Route read-after-write to PRIMARY for N seconds
   → Solution 2: Include write timestamp, compare with replica lag
   → Solution 3: Read from primary for the USER who wrote
     (other users can read from replicas)

3. ALWAYS monitor replication lag
   → Lag > 1 second → alert
   → Lag > 10 seconds → critical alert
   → Lag growing steadily → replica can't keep up (scale issue)
   → Sudden lag spike → network issue or replica overloaded

4. ALWAYS have at least 2 replicas
   → 1 replica: if it fails during primary failover → no data
   → 2 replicas: if one fails, other is still available
   → More replicas = more read capacity + more redundancy

5. ALWAYS test failover BEFORE you need it
   → Schedule quarterly failover drills
   → Simulate primary crash → verify replica promotion works
   → Measure: How long does failover take? Any data loss?
   → Untested failover = no failover

6. ALWAYS place replicas in different failure zones
   → Primary and replica on same server → both die together
   → Primary in Zone A, Replica 1 in Zone B, Replica 2 in Zone C
   → Even better: replicas in different data centers

7. ALWAYS keep replicas at same schema version
   → Deploy schema changes carefully:
     → Step 1: Add new column (backward compatible)
     → Step 2: Deploy app code that uses new column
     → Step 3: Remove old column (if needed)
   → Never break replication with incompatible schema changes

❌ WHAT SHOULD NOT DO

1. NEVER write to replicas
   → Replicas are READ-ONLY
   → Writing to a replica creates data divergence
   → Primary doesn't know about the write → never replicated
   → Results in "split brain" data corruption

2. NEVER assume replicas are instantly up-to-date
   → Even with 1ms lag, reads might be stale
   → Design application to tolerate brief staleness
   → Critical reads (account balance, inventory) → read from primary
   → Non-critical reads (browsing, history) → read from replicas

3. NEVER use replicas for transactions that span read+write
   → Transaction: read balance → check sufficient → deduct
   → If read from replica (stale balance) → deduct on primary
   → Could overdraft because replica showed higher balance
   → Read AND write in same transaction → use PRIMARY

4. NEVER rely on replica count alone for performance
   → 10 replicas don't help if the QUERY is slow
   → Fix the query first (Project 10 — indexes)
   → Replicas multiply capacity, not speed per query

5. NEVER skip monitoring during "normal" operations
   → Replication can silently break
   → Replica falls behind → nobody notices → days of lag
   → Primary fails → promote replica → it's days behind → data loss
   → Monitor lag CONTINUOUSLY

6. NEVER promote a lagging replica without understanding the cost
   → Primary dies, Replica is 30 seconds behind
   → Promoting = losing 30 seconds of writes
   → Evaluate: is 30s of data loss acceptable?
   → Alternative: wait for replica to catch up (if possible)

7. NEVER use synchronous replication across distant data centers
   → Primary in NYC, sync replica in Tokyo
   → Every write: +200ms round trip to Tokyo
   → 1000 writes/sec → each takes 200ms → massive bottleneck
   → Use async for distant replicas, sync only for same-region

🚫 WHEN NOT TO USE

1. DO NOT use read replicas for write-heavy workloads
   → If 80%+ is writes → replicas barely help
   → All writes still go to ONE primary
   → Replicas just add replication overhead
   → For write scaling → use sharding (Project 18)

2. DO NOT use replication as a backup strategy alone
   → Replication copies everything INCLUDING mistakes
   → DELETE FROM orders (accidental) → replicated to all replicas
   → All replicas now have the deleted data too
   → You need POINT-IN-TIME backups for disaster recovery
   → Replication = high availability, NOT backup

3. DO NOT use synchronous replication when latency matters
   → Sync adds network round trip to every write
   → If replica is 100ms away → every write takes +100ms
   → Use async for performance, sync only for durability

4. DO NOT use replication when data is small and reads are cacheable
   → 1000-row table with infrequent updates
   → Cache layer (Project 5) handles this perfectly
   → Replication adds complexity with zero benefit
   → Only replicate when cache + single DB isn't enough

5. DO NOT replicate to solve a poorly designed schema
   → Slow queries from bad schema → still slow on replica
   → Fix schema and indexes first (Project 3, 10)
   → Then replicate for scale and availability

FAILOVER MECHANICS — WHAT HAPPENS WHEN PRIMARY DIES:
═══════════════════════════════════════════════════

  Normal state:
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ PRIMARY │────→│REPLICA 1│     │REPLICA 2│
  │ (alive) │────→│LSN: 0042│     │LSN: 0041│
  └─────────┘     └─────────┘     └─────────┘

  t=0: Primary CRASHES
  ┌─────────┐     ┌─────────┐     ┌─────────┐
  │ PRIMARY │     │REPLICA 1│     │REPLICA 2│
  │  (DEAD) │  ✗  │LSN: 0042│     │LSN: 0041│
  └─────────┘     └─────────┘     └─────────┘

  t=1-5s: Failure Detection
  → Health check (Project 15 concepts) detects primary is down
  → Heartbeat timeout: no response for 5 seconds

  t=5s: Replica Selection
  → Which replica is most up-to-date?
  → Replica 1: LSN 0042 (most recent)
  → Replica 2: LSN 0041 (1 transaction behind)
  → CHOOSE Replica 1 (highest LSN)

  t=5-10s: Promotion
  → Replica 1 promoted to new PRIMARY
  → Stops WAL receiver
  → Starts WAL sender
  → Begins accepting WRITES

  t=10s: Reconfiguration
  → Replica 2 now replicates from NEW primary (Replica 1)
  → Application connection strings updated
  → DNS updated to point to new primary (or proxy handles routing)

  After failover:
  ┌──────────────┐     ┌─────────┐
  │ NEW PRIMARY  │────→│REPLICA 2│
  │(was Replica 1)│    │LSN: [REDACTED:SSN]│ (catching up)
  │  LSN: [REDACTED:SSN]   │     └─────────┘
  └──────────────┘
  
  ┌─────────┐
  │OLD PRIMARY│ (dead, needs repair)
  │  (DEAD)  │
  └─────────┘

  DATA LOSS ANALYSIS:
  → Primary was at LSN 0042 when it died
  → If Replica 1 was also at LSN 0042 → ZERO data loss ✅
  → If Replica 1 was at LSN 0040 → 2 transactions lost ⚠️
  → This is why monitoring replication lag is CRITICAL


REPLICATION LAG — INTERNAL MECHANICS:
════════════════════════════════════

  What causes lag:
  
  1. NETWORK DELAY
     → WAL entries travel over network
     → Same datacenter: <1ms
     → Cross-region: 50-200ms
  
  2. REPLAY SPEED
     → Replica must execute each WAL entry
     → If writes are faster than replay → lag grows
     → Heavy writes (bulk import) → replicas fall behind
  
  3. RESOURCE CONTENTION
     → Replica also serves read queries
     → Heavy read queries steal CPU from replay process
     → Lag increases under read load
  
  4. LARGE TRANSACTIONS
     → One transaction inserts 1 million rows
     → Replica must replay all 1M inserts
     → During replay: replica appears to be "stuck"

  How lag is measured:
  
  Primary:  current WAL position = LSN 0042, time = 10:30:45.000
  Replica:  last replayed LSN = 0040, that LSN's time = 10:30:44.500
  
  Lag = 10:30:45.000 - 10:30:44.500 = 500ms
  
  Replica is 500ms behind primary
  → Any data written in last 500ms is NOT on this replica


READ-AFTER-WRITE CONSISTENCY — THE CORE CHALLENGE:
══════════════════════════════════════════════════

  Without handling:
  ┌────────┐   POST /orders   ┌─────────┐   WRITE    ┌─────────┐
  │ Client │─────────────────→│   App   │──────────→│ PRIMARY │
  └───┬────┘                  └─────────┘           └─────────┘
      │                                              LSN: 0042
      │ GET /orders/ORD-5001                    
      │                       ┌─────────┐   READ     ┌─────────┐
      └──────────────────────→│   App   │──────────→│ REPLICA │
                              └─────────┘           │LSN: 0040│
                                                    └─────────┘
                                                    "Not found!" ❌

  With read-after-write handling:
  ┌────────┐   POST /orders   ┌─────────┐   WRITE    ┌─────────┐
  │ Client │─────────────────→│   App   │──────────→│ PRIMARY │
  └───┬────┘                  │ Records │           └─────────┘
      │                       │write_ts │            LSN: 0042
      │ GET /orders/ORD-5001  │for user │
      │                       ┌─────────┐
      └──────────────────────→│   App   │
                              │ Checks: │
                              │ user did│
                              │ write   │   READ     ┌─────────┐
                              │ <5s ago │──────────→│ PRIMARY │
                              └─────────┘           └─────────┘
                                                    "Found!" ✅
  
  → For 5 seconds after a write, route that USER's reads to PRIMARY
  → After 5 seconds, replica has caught up → safe to read from replica

🚨 COMMON PITFALLS AND FAILURE MODES

PITFALL 1: Split Brain (Two Primaries)
  → Primary appears dead (network partition, not actual crash)
  → Replica promoted to new primary
  → Old primary comes back online → now TWO primaries
  → Both accept writes → data diverges → CORRUPTION
  → FIX: Fencing — old primary must be forcibly stopped
  → FIX: STONITH (Shoot The Other Node In The Head)
  → FIX: Quorum-based promotion (majority must agree)

PITFALL 2: Replication Lag Causes Application Bugs
  → User creates order → redirect to order details page
  → Order details reads from replica → "Order not found"
  → User panics, places order AGAIN → double order
  → FIX: Read-after-write routing to primary
  → FIX: Optimistic display (show from local state, not DB)

PITFALL 3: Replica Promotion Loses Data
  → Primary crashes at LSN 0042
  → Best replica is at LSN 0038
  → Promotion loses transactions 0039-0042
  → Those 4 transactions (orders, payments) are GONE
  → FIX: Semi-sync replication (at least 1 replica always current)
  → FIX: Accept the risk with monitoring and short lag targets

PITFALL 4: All Replicas Connected to Primary Directly
  → Primary ships WAL to 5 replicas
  → 5 concurrent WAL streams consume primary's network bandwidth
  → Primary slows down
  → FIX: Cascading replication
    → Primary → Replica 1 → Replica 2, 3
    → Primary only ships to 1-2, they forward to others

PITFALL 5: Schema Migration Breaks Replication
  → ALTER TABLE on primary adds column
  → WAL entry shipped to replica
  → Replica tries to replay but has incompatible schema version
  → Replication BREAKS
  → FIX: Use backward-compatible schema changes
  → FIX: Add columns with defaults, never remove columns in-flight

PITFALL 6: Long-Running Analytics Query on Replica
  → Finance runs 30-minute query on Replica 1
  → Replica needs to apply WAL entries during query
  → Some databases: WAL replay blocked by long query
  → Replica lag grows to 30 minutes
  → FIX: Use dedicated analytics replica
  → FIX: Set statement_timeout on replicas

PITFALL 7: Connection Pool Exhaustion on Failover
  → Primary dies → all connections to primary break
  → App retries → floods new primary with connection attempts
  → Connection storm overwhelms new primary
  → FIX: Connection pooling with retry backoff
  → FIX: Circuit breaker pattern (Project 21)

📈 SCALABILITY CONSIDERATIONS

READ SCALING:
  → 1 replica: 2x read capacity
  → 3 replicas: 4x read capacity
  → 5 replicas: 6x read capacity
  → Each replica handles full read workload independently
  → Load balancer (Project 6) distributes reads

WRITE SCALING LIMITATION:
  → ALL writes go to ONE primary
  → Replication does NOT scale writes
  → Primary write capacity = ceiling
  → To scale writes → sharding (Project 18)
  → This is the fundamental limit of replication

NUMBERS TO KNOW:
  → Replication lag (same datacenter): 1-10ms typical
  → Replication lag (cross-region): 50-500ms typical
  → Failover time (automatic): 10-30 seconds
  → Failover time (manual): 1-10 minutes

class WALEntry:
    """Single entry in the Write-Ahead Log"""
    
    def __init__(self, lsn: int, operation: str, table: str,
                 data: dict, timestamp: float = None):
        self.lsn = lsn  # Log Sequence Number (monotonically increasing)
        self.operation = operation  # INSERT, UPDATE, DELETE
        self.table = table
        self.data = data
        self.timestamp = timestamp or time.time()
    
    def to_dict(self):
        return {
            "lsn": self.lsn,
            "operation": self.operation,
            "table": self.table,
            "data": self.data,
            "timestamp": self.timestamp
        }


class WriteAheadLog:
    """
    Append-only log of all write operations
    The foundation of database replication
    """
    
    def __init__(self):
        self.entries = []
        self.current_lsn = 0
        self.lock = threading.Lock()
    
    def append(self, operation: str, table: str, data: dict) -> WALEntry:
        """Append a new entry to the WAL"""
        with self.lock:
            self.current_lsn += 1
            entry = WALEntry(self.current_lsn, operation, table, data)
            self.entries.append(entry)
            return entry
    
    def get_entries_after(self, lsn: int) -> list:
        """Get all entries with LSN > given LSN"""
        with self.lock:
            return [e for e in self.entries if e.lsn > lsn]
    
    def get_latest_lsn(self) -> int:
        with self.lock:
            return self.current_lsn

class NodeRole(Enum):
    PRIMARY = "PRIMARY"
    REPLICA = "REPLICA"
    CANDIDATE = "CANDIDATE"


class DatabaseNode:
    """
    A single database node that can serve as primary or replica
    Contains: data storage, WAL, replication logic
    """
    
    def __init__(self, node_id: str, role: NodeRole = NodeRole.REPLICA):
        self.node_id = node_id
        self.role = role
        self.alive = True
        
        # Data storage (simple in-memory tables)
        self.tables = defaultdict(dict)  # table_name → {row_id → row}
        
        # Write-Ahead Log
        self.wal = WriteAheadLog()
        
        # Replication state (for replicas)
        self.last_applied_lsn = 0
        self.replication_source = None  # Primary node reference
        self.replication_lag_ms = 0
        
        # Metrics
        self.metrics = {
            "reads": 0,
            "writes": 0,
            "wal_entries_received": 0,
            "wal_entries_applied": 0,
            "replication_errors": 0
        }
        
        self.lock = threading.Lock()
        
        # Simulated processing delay
        self.read_delay_ms = 2   # 2ms per read
        self.write_delay_ms = 5  # 5ms per write
    
    # ---- WRITE OPERATIONS (Primary only) ----
    
    def write(self, operation: str, table: str, row_id: str,
              data: dict) -> dict:
        """Execute a write operation (PRIMARY only)"""
        if self.role != NodeRole.PRIMARY:
            return {
                "success": False,
                "error": f"Cannot write to {self.role.value} node {self.node_id}"
            }
        
        if not self.alive:
            return {"success": False, "error": "Node is down"}
        
        time.sleep(self.write_delay_ms / 1000)
        
        with self.lock:
            # Step 1: Write to WAL
            wal_data = {"row_id": row_id, **data}
            entry = self.wal.append(operation, table, wal_data)
            
            # Step 2: Apply to data store
            if operation == "INSERT":
                self.tables[table][row_id] = data.copy()
            elif operation == "UPDATE":
                if row_id in self.tables[table]:
                    self.tables[table][row_id].update(data)
                else:
                    self.tables[table][row_id] = data.copy()
            elif operation == "DELETE":
                self.tables[table].pop(row_id, None)
            
            self.metrics["writes"] += 1
            
            return {
                "success": True,
                "lsn": entry.lsn,
                "node": self.node_id,
                "operation": operation,
                "timestamp": entry.timestamp
            }
    
    # ---- READ OPERATIONS (Any node) ----
    
    def read(self, table: str, row_id: str = None,
             filter_fn=None) -> dict:
        """Execute a read operation"""
        if not self.alive:
            return {"success": False, "error": "Node is down"}
        
        time.sleep(self.read_delay_ms / 1000)
        
        with self.lock:
            self.metrics["reads"] += 1
            
            if row_id:
                # Single row lookup
                row = self.tables.get(table, {}).get(row_id)
                if row:
                    return {
                        "success": True,
                        "data": row.copy(),
                        "node": self.node_id,
                        "role": self.role.value,
                        "data_lsn": self.last_applied_lsn 
                                    if self.role == NodeRole.REPLICA 
                                    else self.wal.get_latest_lsn()
                    }
                return {
                    "success": True,
                    "data": None,
                    "node": self.node_id,
                    "role": self.role.value
                }
            
            # Full table scan (with optional filter)
            rows = list(self.tables.get(table, {}).values())
            if filter_fn:
                rows = [r for r in rows if filter_fn(r)]
            
            return {
                "success": True,
                "data": rows,
                "count": len(rows),
                "node": self.node_id,
                "role": self.role.value
            }
    
    # ---- REPLICATION (Replica applies WAL from Primary) ----
    
    def apply_wal_entry(self, entry: WALEntry) -> bool:
        """Apply a single WAL entry to this replica's data"""
        if not self.alive:
            return False
        
        with self.lock:
            try:
                data = entry.data.copy()
                row_id = data.pop("row_id", None)
                
                if entry.operation == "INSERT":
                    self.tables[entry.table][row_id] = data
                elif entry.operation == "UPDATE":
                    if row_id in self.tables[entry.table]:
                        self.tables[entry.table][row_id].update(data)
                    else:
                        self.tables[entry.table][row_id] = data
                elif entry.operation == "DELETE":
                    self.tables[entry.table].pop(row_id, None)
                
                self.last_applied_lsn = entry.lsn
                self.metrics["wal_entries_applied"] += 1
                
                # Calculate replication lag
                self.replication_lag_ms = (time.time() - entry.timestamp) * 1000
                
                return True
                
            except Exception as e:
                self.metrics["replication_errors"] += 1
                return False
    
    # ---- FAILOVER (Promote replica to primary) ----
    
    def promote_to_primary(self):
        """Promote this replica to primary"""
        with self.lock:
            self.role = NodeRole.PRIMARY
            self.replication_source = None
            # WAL continues from last applied LSN
            self.wal.current_lsn = self.last_applied_lsn
    
    def demote_to_replica(self):
        """Demote this primary to replica"""
        with self.lock:
            self.role = NodeRole.REPLICA
    
    # ---- NODE LIFECYCLE ----
    
    def kill(self):
        """Simulate node failure"""
        self.alive = False
    
    def revive(self):
        """Simulate node recovery"""
        self.alive = True
    
    def get_status(self) -> dict:
        with self.lock:
            primary_lsn = None
            if self.replication_source and self.replication_source.alive:
                primary_lsn = self.replication_source.wal.get_latest_lsn()
            
            return {
                "node_id": self.node_id,
                "role": self.role.value,
                "alive": self.alive,
                "current_lsn": self.wal.get_latest_lsn() 
                              if self.role == NodeRole.PRIMARY 
                              else self.last_applied_lsn,
                "primary_lsn": primary_lsn,
                "replication_lag_ms": round(self.replication_lag_ms, 2),
                "tables": {t: len(rows) for t, rows in self.tables.items()},
                "metrics": self.metrics.copy()
            }

class ReplicationManager:
    """
    Manages WAL shipping from primary to replicas
    Supports: async, sync, semi-sync modes
    """
    
    def __init__(self, mode: str = "async"):
        self.mode = mode  # async, sync, semi-sync
        self.primary = None
        self.replicas = []
        self.running = False
        self.replication_threads = []
        
        # Simulated network delay between primary and each replica
        self.network_delays = {}  # replica_id → delay_ms
        
        self.metrics = {
            "wal_entries_shipped": 0,
            "replication_rounds": 0,
            "sync_waits": 0
        }
    
    def set_primary(self, node: DatabaseNode):
        self.primary = node
        node.role = NodeRole.PRIMARY
    
    def add_replica(self, node: DatabaseNode, network_delay_ms: float = 5):
        node.role = NodeRole.REPLICA
        node.replication_source = self.primary
        self.replicas.append(node)
        self.network_delays[node.node_id] = network_delay_ms
    
    def start(self):
        """Start replication threads"""
        self.running = True
        for replica in self.replicas:
            t = threading.Thread(
                target=self._replication_loop,
                args=(replica,),
                daemon=True
            )
            self.replication_threads.append(t)
            t.start()
    
    def stop(self):
        self.running = False
        for t in self.replication_threads:
            t.join(timeout=2)
    
    def _replication_loop(self, replica: DatabaseNode):
        """Continuously ship WAL entries from primary to replica"""
        while self.running:
            if not self.primary or not self.primary.alive:
                time.sleep(0.1)
                continue
            
            if not replica.alive:
                time.sleep(0.1)
                continue
            
            # Get new WAL entries since replica's last position
            new_entries = self.primary.wal.get_entries_after(
                replica.last_applied_lsn
            )
            
            if not new_entries:
                time.sleep(0.01)  # No new data, short sleep
                continue
            
            # Simulate network delay
            delay = self.network_delays.get(replica.node_id, 5)
            time.sleep(delay / 1000)
            
            # Apply entries to replica
            for entry in new_entries:
                if not self.running or not replica.alive:
                    break
                
                # Simulate replay processing time
                time.sleep(0.001)  # 1ms per entry
                
                success = replica.apply_wal_entry(entry)
                if success:
                    self.metrics["wal_entries_shipped"] += 1
                    replica.metrics["wal_entries_received"] += 1
            
            self.metrics["replication_rounds"] += 1
    
    def write_with_replication(self, operation: str, table: str,
                                row_id: str, data: dict) -> dict:
        """
        Write to primary with replication guarantees
        Behavior depends on mode: async, sync, semi-sync
        """
        if not self.primary or not self.primary.alive:
            return {"success": False, "error": "No primary available"}
        
        # Write to primary
        result = self.primary.write(operation, table, row_id, data)
        
        if not result["success"]:
            return result
        
        if self.mode == "async":
            # Don't wait for replicas — return immediately
            return result
        
        elif self.mode == "sync":
            # Wait for ALL replicas to confirm
            target_lsn = result["lsn"]
            self.metrics["sync_waits"] += 1
            
            timeout = 5.0  # 5 second timeout
            start = time.time()
            
            while time.time() - start < timeout:
                all_caught_up = True
                for replica in self.replicas:
                    if replica.alive and replica.last_applied_lsn < target_lsn:
                        all_caught_up = False
                        break
                
                if all_caught_up:
                    result["replication"] = "all_replicas_confirmed"
                    return result
                
                time.sleep(0.005)
            
            result["replication"] = "timeout_some_replicas_behind"
            return result
        
        elif self.mode == "semi-sync":
            # Wait for AT LEAST ONE replica to confirm
            target_lsn = result["lsn"]
            self.metrics["sync_waits"] += 1
            
            timeout = 5.0
            start = time.time()
            
            while time.time() - start < timeout:
                for replica in self.replicas:
                    if replica.alive and replica.last_applied_lsn >= target_lsn:
                        result["replication"] = f"confirmed_by_{replica.node_id}"
                        return result
                
                time.sleep(0.005)
            
            result["replication"] = "timeout_no_replica_confirmed"
            return result
        
        return result
    
    def get_replication_status(self) -> dict:
        primary_lsn = self.primary.wal.get_latest_lsn() if self.primary else 0
        
        replica_status = []
        for replica in self.replicas:
            lag_entries = primary_lsn - replica.last_applied_lsn
            replica_status.append({
                "node_id": replica.node_id,
                "alive": replica.alive,
                "applied_lsn": replica.last_applied_lsn,
                "lag_entries": lag_entries,
                "lag_ms": round(replica.replication_lag_ms, 2),
                "network_delay_ms": self.network_delays.get(replica.node_id, 0)
            })
        
        return {
            "mode": self.mode,
            "primary": self.primary.node_id if self.primary else None,
            "primary_lsn": primary_lsn,
            "replicas": replica_status,
            "metrics": self.metrics.copy()
        }

class ReadWriteRouter:
    """
    Routes queries to appropriate nodes:
    → Writes → Primary
    → Reads → Replicas (load balanced)
    → Read-after-write → Primary (for consistency)
    """
    
    def __init__(self, primary: DatabaseNode, replicas: list):
        self.primary = primary
        self.replicas = replicas
        self.replica_index = 0  # Round-robin counter
        
        # Track recent writes per user (for read-after-write)
        self.recent_writes = {}  # user_id → timestamp
        self.raw_consistency_window = 5.0  # seconds
        
        self.lock = threading.Lock()
        
        self.routing_stats = {
            "reads_to_primary": 0,
            "reads_to_replica": 0,
            "writes_to_primary": 0,
            "read_after_write_redirects": 0,
            "no_replica_available_fallbacks": 0
        }
    
    def write(self, operation: str, table: str, row_id: str,
              data: dict, user_id: str = None) -> dict:
        """Route write to primary, track for read-after-write"""
        result = self.primary.write(operation, table, row_id, data)
        
        if result["success"] and user_id:
            with self.lock:
                self.recent_writes[user_id] = time.time()
        
        with self.lock:
            self.routing_stats["writes_to_primary"] += 1
        
        return result
    
    def read(self, table: str, row_id: str = None,
             filter_fn=None, user_id: str = None,
             force_primary: bool = False) -> dict:
        """
        Route read to appropriate node:
        → force_primary=True: always primary
        → Recent write by same user: primary (read-after-write)
        → Otherwise: round-robin across healthy replicas
        """
        # Check if this user recently wrote
        if user_id and not force_primary:
            with self.lock:
                last_write = self.recent_writes.get(user_id, 0)
                if time.time() - last_write < self.raw_consistency_window:
                    force_primary = True
                    self.routing_stats["read_after_write_redirects"] += 1
        
        if force_primary:
            with self.lock:
                self.routing_stats["reads_to_primary"] += 1
            return self.primary.read(table, row_id, filter_fn)
        
        # Find healthy replica (round-robin)
        replica = self._get_healthy_replica()
        
        if replica is None:
            # No healthy replicas → fallback to primary
            with self.lock:
                self.routing_stats["no_replica_available_fallbacks"] += 1
                self.routing_stats["reads_to_primary"] += 1
            return self.primary.read(table, row_id, filter_fn)
        
        with self.lock:
            self.routing_stats["reads_to_replica"] += 1
        
        return replica.read(table, row_id, filter_fn)
    
    def _get_healthy_replica(self) -> Optional[DatabaseNode]:
        """Round-robin selection of healthy replicas"""
        with self.lock:
            if not self.replicas:
                return None
            
            # Try each replica starting from current index
            for _ in range(len(self.replicas)):
                replica = self.replicas[self.replica_index % len(self.replicas)]
                self.replica_index += 1
                
                if replica.alive:
                    return replica
            
            return None
    
    def update_primary(self, new_primary: DatabaseNode):
        """Update routing after failover"""
        self.primary = new_primary
    
    def get_routing_stats(self) -> dict:
        with self.lock:
            total_reads = (self.routing_stats["reads_to_primary"] + 
                          self.routing_stats["reads_to_replica"])
            replica_pct = 0
            if total_reads > 0:
                replica_pct = self.routing_stats["reads_to_replica"] / total_reads * 100
            
            return {
                **self.routing_stats,
                "total_reads": total_reads,
                "replica_read_pct": round(replica_pct, 1)
            }

class FailoverManager:
    """
    Monitors primary health and performs automatic failover
    when primary becomes unavailable
    """
    
    def __init__(self, repl_manager: ReplicationManager,
                 router: ReadWriteRouter,
                 check_interval: float = 1.0,
                 failure_threshold: int = 3):
        self.repl_manager = repl_manager
        self.router = router
        self.check_interval = check_interval
        self.failure_threshold = failure_threshold
        self.consecutive_failures = 0
        self.running = False
        self.thread = None
        
        self.failover_log = []
    
    def start(self):
        self.running = True
        self.thread = threading.Thread(target=self._monitor_loop, daemon=True)
        self.thread.start()
    
    def stop(self):
        self.running = False
        if self.thread:
            self.thread.join(timeout=2)
    
    def _monitor_loop(self):
        while self.running:
            primary = self.repl_manager.primary
            
            if primary and not primary.alive:
                self.consecutive_failures += 1
                
                if self.consecutive_failures >= self.failure_threshold:
                    self._perform_failover()
                    self.consecutive_failures = 0
            else:
                self.consecutive_failures = 0
            
            time.sleep(self.check_interval)
    
    def _perform_failover(self):
        """Promote the most up-to-date replica to primary"""
        start_time = time.time()
        old_primary = self.repl_manager.primary
        
        # Find best replica (highest LSN, alive)
        best_replica = None
        best_lsn = -1
        
        for replica in self.repl_manager.replicas:
            if replica.alive and replica.last_applied_lsn > best_lsn:
                best_lsn = replica.last_applied_lsn
                best_replica = replica
        
        if best_replica is None:
            self.failover_log.append({
                "timestamp": time.time(),
                "status": "FAILED",
                "reason": "No healthy replica available"
            })
            return
        
        # Calculate data loss
        old_primary_lsn = old_primary.wal.get_latest_lsn()
        data_loss_entries = old_primary_lsn - best_lsn
        
        # Promote
        best_replica.promote_to_primary()
        
        # Update replication manager
        self.repl_manager.primary = best_replica
        self.repl_manager.replicas.remove(best_replica)
        
        # Reconfigure remaining replicas to follow new primary
        for replica in self.repl_manager.replicas:
            replica.replication_source = best_replica
        
        # Update router
        self.router.update_primary(best_replica)
        
        failover_time = (time.time() - start_time) * 1000
        
        event = {
            "timestamp": time.time(),

import time
import threading
import random
from collections import defaultdict


# Reusing classes from Part A (DatabaseNode, ReplicationManager, etc.)
# For standalone execution, minimal versions below:

class SimpleNode:
    """Minimal database node for comparison testing"""
    
    def __init__(self, node_id, role="REPLICA"):
        self.node_id = node_id
        self.role = role
        self.alive = True
        self.data = {}
        self.lsn = 0
        self.last_applied_lsn = 0
        self.lock = threading.Lock()
    
    def write(self, key, value, delay_ms=2):
        time.sleep(delay_ms / 1000)
        with self.lock:
            self.lsn += 1
            self.data[key] = value
            return self.lsn
    
    def apply_wal(self, lsn, key, value, network_delay_ms=5):
        time.sleep(network_delay_ms / 1000)
        time.sleep(0.001)  # Replay time
        with self.lock:
            self.data[key] = value
            self.last_applied_lsn = lsn
            return True


def benchmark_replication_mode(mode, num_writes=50, network_delay_ms=5):
    """
    Benchmark a replication mode
    Returns: write latencies, total time, data safety level
    """
    primary = SimpleNode("primary", "PRIMARY")
    replicas = [
        SimpleNode(f"replica-{i}") for i in range(3)
    ]
    
    latencies = []
    data_confirmed_replicas = []
    
    for i in range(num_writes):
        write_start = time.time()
        
        # Write to primary
        key = f"order-{i:04d}"
        value = {"item": f"pizza_{i}", "price": 10 + i % 20}
        lsn = primary.write(key, value, delay_ms=2)
        
        if mode == "async":
            # Don't wait — fire and forget to replicas
            for replica in replicas:
                t = threading.Thread(
                    target=replica.apply_wal,
                    args=(lsn, key, value, network_delay_ms),
                    daemon=True
                )
                t.start()
            
            confirmed = 0  # Don't know yet
        
        elif mode == "sync":
            # Wait for ALL replicas
            threads = []
            results = [False] * len(replicas)
            
            def apply_and_track(idx, replica):
                results[idx] = replica.apply_wal(
                    lsn, key, value, network_delay_ms)
            
            for idx, replica in enumerate(replicas):
                t = threading.Thread(
                    target=apply_and_track,
                    args=(idx, replica)
                )
                threads.append(t)
                t.start()
            
            for t in threads:
                t.join(timeout=2)
            
            confirmed = sum(results)
        
        elif mode == "semi-sync":
            # Wait for AT LEAST ONE replica
            event = threading.Event()
            first_confirmed = [False]
            
            def apply_and_signal(replica):
                replica.apply_wal(lsn, key, value, network_delay_ms)
                first_confirmed[0] = True
                event.set()
            
            for replica in replicas:
                t = threading.Thread(
                    target=apply_and_signal,
                    args=(replica,),
                    daemon=True
                )
                t.start()
            
            event.wait(timeout=2)
            confirmed = 1 if first_confirmed[0] else 0
        
        write_latency = (time.time() - write_start) * 1000
        latencies.append(write_latency)
        data_confirmed_replicas.append(confirmed)
    
    # Wait for async to finish
    time.sleep(0.5)
    
    return {
        "mode": mode,
        "num_writes": num_writes,
        "latencies": latencies,
        "avg_latency_ms": sum(latencies) / len(latencies),
        "min_latency_ms": min(latencies),
        "max_latency_ms": max(latencies),
        "p50_ms": sorted(latencies)[len(latencies) // 2],
        "p99_ms": sorted(latencies)[int(len(latencies) * 0.99)],
        "total_time_ms": sum(latencies),
        "data_safety": "NONE" if mode == "async" else 
                       "ONE_REPLICA" if mode == "semi-sync" else 
                       "ALL_REPLICAS"
    }

# ============================================================
# PROJECT 13C: REPLICATION LAG MONITOR
# Real-time monitoring of replication lag across replicas
# Shows how lag behaves under different write loads
# Foundation for Project 15 (Health Checks & Monitoring)
# ============================================================

import time
import threading
import random
from collections import deque


class ReplicationLagMonitor:
    """
    Monitors replication lag and provides alerting
    Tracks: current lag, lag history, lag trends
    """
    
    def __init__(self, check_interval: float = 0.5):
        self.check_interval = check_interval
        self.nodes = {}  # node_id → {"primary_lsn_fn", "replica_lsn_fn"}
        self.lag_history = {}  # node_id → deque of (timestamp, lag_entries, lag_ms)
        self.alerts = []
        self.running = False
        self.thread = None
        
        # Alert thresholds
        self.warning_threshold_ms = 1000    # 1 second
        self.critical_threshold_ms = 10000  # 10 seconds
        self.history_size = 100
    
    def register_node(self, node_id, get_primary_lsn, get_replica_lsn,
                      get_lag_ms):
        self.nodes[node_id] = {
            "get_primary_lsn": get_primary_lsn,
            "get_replica_lsn": get_replica_lsn,
            "get_lag_ms": get_lag_ms
        }
        self.lag_history[node_id] = deque(maxlen=self.history_size)
    
    def start(self):
        self.running = True
        self.thread = threading.Thread(target=self._monitor_loop, daemon=True)
        self.thread.start()
    
    def stop(self):
        self.running = False
        if self.thread:
            self.thread.join(timeout=2)
    
    def _monitor_loop(self):
        while self.running:
            for node_id, fns in self.nodes.items():
                try:
                    primary_lsn = fns["get_primary_lsn"]()
                    replica_lsn = fns["get_replica_lsn"]()
                    lag_ms = fns["get_lag_ms"]()
                    lag_entries = primary_lsn - replica_lsn
                    
                    self.lag_history[node_id].append({
                        "timestamp": time.time(),
                        "lag_entries": lag_entries,
                        "lag_ms": lag_ms,
                        "primary_lsn": primary_lsn,
                        "replica_lsn": replica_lsn
                    })
                    
                    # Check alerts
                    if lag_ms > self.critical_threshold_ms:
                        self.alerts.append({
                            "level": "CRITICAL",
                            "node": node_id,
                            "lag_ms": lag_ms,
                            "lag_entries": lag_entries,
                            "timestamp": time.time()
                        })
                    elif lag_ms > self.warning_threshold_ms:
                        self.alerts.append({
                            "level": "WARNING",
                            "node": node_id,
                            "lag_ms": lag_ms,
                            "lag_entries": lag_entries,
                            "timestamp": time.time()
                        })
                except Exception:
                    pass
            
            time.sleep(self.check_interval)
    
    def get_current_status(self) -> dict:
        status = {}
        for node_id, history in self.lag_history.items():
            if history:
                latest = history[-1]
                
                # Calculate trend (increasing, stable, decreasing)
                if len(history) >= 5:
                    recent = list(history)[-5:]
                    lags = [h["lag_entries"] for h in recent]
                    if lags[-1] > lags[0] + 5:
                        trend = "INCREASING ↑"
                    elif lags[-1] < lags[0] - 5:
                        trend = "DECREASING ↓"
                    else:
                        trend = "STABLE ─"
                else:
                    trend = "MEASURING..."
                
                # Health status
                lag_ms = latest["lag_ms"]
                if lag_ms < self.warning_threshold_ms:
                    health = "HEALTHY 🟢"
                elif lag_ms < self.critical_threshold_ms:
                    health = "WARNING 🟡"
                else:
                    health = "CRITICAL 🔴"
                
                status[node_id] = {
                    "lag_entries": latest["lag_entries"],
                    "lag_ms": round(latest["lag_ms"], 1),
                    "primary_lsn": latest["primary_lsn"],
                    "replica_lsn": latest["replica_lsn"],
                    "health": health,
                    "trend": trend
                }
        
        return status
    
    def print_dashboard(self):
        """Print a text-based monitoring dashboard"""
        status = self.get_current_status()
        
        print(f"\n  ┌{'─' * 62}┐")
        print(f"  │{'REPLICATION LAG MONITOR':^62s}│")
        print(f"  ├{'─' * 62}┤")
        
        for node_id, info in status.items():
            print(f"  │ {node_id:<20s} │ {info['health']:<15s} │ "
                  f"Lag: {info['lag_entries']:>4d} entries │ "
                  f"{info['lag_ms']:>7.1f}ms │")
            
            # Visual lag bar
            bar_len = min(int(info['lag_ms'] / 100), 40)
            bar = "█" * bar_len
            print(f"  │ {'':20s} │ {info['trend']:<15s} │ "
                  f"{bar:<22s}│")
        
        print(f"  └{'─' * 62}┘")
        
        # Recent alerts
        recent_alerts = [a for a in self.alerts[-5:]]
        if recent_alerts:
            print(f"  Recent Alerts:")
            for alert in recent_alerts:
                icon = "🔴" if alert["level"] == "CRITICAL" else "🟡"
                print(f"    {icon} [{alert['level']}] {alert['node']}: "
                      f"lag={alert['lag_ms']:.0f}ms "
                      f"({alert['lag_entries']} entries)")

  → 1 primary + 3 replicas = 20TB total storage
  → But the REAL problem: even ONE server can't hold 5TB efficiently

  → Solution: SPLIT the data across multiple servers

  → If you add a new server, how much data needs to move?
  → If a server dies, how do you redistribute its data?
  → If you use simple hash(key) % N → adding a server reshuffles EVERYTHING

  → Solved elegantly with a hash ring

  → Virtual nodes: ensure even distribution
